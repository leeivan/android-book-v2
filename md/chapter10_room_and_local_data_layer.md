# 第10章 Room、Repository 与本地数据层

只会把数据写进文件并不等于具备了本地数据层能力。当数据需要查询、筛选、关系建模和离线读取时，就需要更清晰的结构化存储方案。本章以 Room 为主线，同时解释 Repository 在代码组织中的位置。

## 学习目标

- 理解 Room 为什么是现代 Android 的本地结构化存储主线。
- 理解实体、DAO、数据库类和事务之间的关系。
- 理解 Repository 如何隔离界面层与数据来源。
- 理解 ContentProvider 在现代项目中的真实定位。

## 10.1 Room 的角色

### 10.1.1 为什么不把 SQLite API 作为主线

- SQLite 仍然重要，但直接使用原生 API 学习成本高、错误暴露晚。
- Room 更适合作为结构化数据的默认教学入口。

### 10.1.2 Room 的基本组成

- Entity。
- Dao。
- Database。
- Migration。

## 10.2 Repository 的角色

### 10.2.1 为什么需要 Repository

- 界面层不应直接拼装所有数据来源。
- Repository 帮助协调本地与远程数据，保持依赖方向清晰。

### 10.2.2 从数据源到页面状态

- Room 提供本地能力。
- Repository 提供业务读取入口。
- ViewModel 负责将数据转化为界面状态。

## 10.3 Room 进阶主题

### 10.3.1 关系与事务

- 一对多、多对多。
- 为什么事务边界会影响数据一致性。

### 10.3.2 分页与性能

- Paging 的适用场景。
- 查询设计与列表加载的基本关系。

## 10.4 ContentProvider 的定位

### 10.4.1 什么时候需要 ContentProvider

- 当数据需要对系统或其他应用暴露统一访问入口时。
- 它不是所有应用都必须实现的组件。

### 10.4.2 与 Room 的关系

- Room 面向应用内部数据组织。
- ContentProvider 面向跨进程或跨应用数据暴露。

## 10.5 本章实践任务

- 建立一个最小 Room 数据库。
- 通过 Repository 向界面提供数据。
- 实现一个列表页的新增、查询和刷新。

## 10.6 小结

结构化数据的重点不在于“把数据存进去”，而在于能否用一致、可维护的方式组织读取、修改和展示流程。本章为最终综合案例提供了本地数据层基础。
