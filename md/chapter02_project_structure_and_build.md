# 第2章 项目结构、Gradle 与构建发布基础

很多初学者能运行项目，却不知道项目为什么能运行；能修改界面，却不知道依赖、源集、签名和构建变体分别由谁负责。本章的任务就是把“Android 工程”从黑盒拆开，让读者知道一个项目究竟由哪些部分组成。

## 学习目标

- 理解 Android 项目的目录结构与文件职责。
- 理解 Gradle、Android Gradle Plugin 和依赖管理的基本分工。
- 知道调试版、发布版、构建类型和产品变体的关系。
- 理解 APK、AAB、签名与发布的基础流程。

## 2.1 Android 工程由什么组成

### 2.1.1 模块与源集

- `app` 模块是最常见的应用模块。
- `main`、`androidTest`、`test` 对应不同代码用途。
- 资源、源码、清单和测试代码并不是一堆杂乱文件，而是按源集组织的。

### 2.1.2 资源与清单

- `res/` 负责静态资源组织。
- `AndroidManifest.xml` 负责声明组件、权限、特性和入口。
- Manifest 不是“必须存在的配置文件”这么简单，它定义了应用与系统的契约边界。

## 2.2 Gradle 与构建系统

### 2.2.1 Gradle 与 AGP 的角色

- Gradle 负责通用构建任务调度。
- AGP 负责 Android 相关规则，例如资源处理、Manifest 合并和打包流程。
- IDE 只是调用构建系统的界面，构建本身可以脱离 IDE 执行。

### 2.2.2 依赖管理

- 学会识别 `implementation`、`testImplementation` 等常见配置。
- 知道为什么要区分应用依赖、测试依赖和插件依赖。
- 理解版本升级会同时影响构建、运行和兼容性。

### 2.2.3 构建变体

- `debug` 与 `release` 是最基础的构建类型。
- 产品变体用于表达“同一个项目的不同业务版本”。
- 构建变体帮助开发、测试和发布形成明确边界。

## 2.3 打包、签名与发布

### 2.3.1 APK 与 AAB

- APK 是安装产物。
- AAB 更适合作为商店分发产物。
- 二者面向的场景不同，不应混为一谈。

### 2.3.2 调试签名与发布签名

- 调试包可以自动签名。
- 发布包必须使用受控的正式签名。
- 签名既是技术流程，也是发布资产管理问题。

### 2.3.3 基础发布链路

- 构建。
- 签名。
- 测试。
- 分发。

## 2.4 本章实践任务

- 阅读一个空白项目的 Gradle 配置。
- 识别 `main`、`test`、`androidTest` 的职责差异。
- 构建调试包并查看生成产物位置。

## 2.5 小结

学完本章后，读者应能把 Android 项目看作一个有明确结构、明确构建规则和明确发布边界的工程，而不是“Android Studio 里的一堆文件”。
