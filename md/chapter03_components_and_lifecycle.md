# 第3章 Activity、Fragment、ViewModel 与生命周期

Android 开发的难点之一，不在于组件名字多，而在于组件的状态变化和系统回调会不断打断应用执行流程。本章以 Activity、Fragment 和 ViewModel 为主线，帮助读者建立“界面不是一直存在、状态必须被正确托管”的基本认识。

## 学习目标

- 理解 Activity 和 Fragment 的职责边界。
- 理解生命周期回调为何直接影响界面状态与资源管理。
- 理解 ViewModel 在状态保存和配置变更中的价值。
- 能够设计一个基本稳定的页面状态结构。

## 3.1 Activity 的角色

### 3.1.1 Activity 不是完整应用

- Activity 更接近“一个可见界面容器”。
- 一个应用可以只有少量 Activity，但包含很多页面状态和交互场景。

### 3.1.2 生命周期

- `onCreate`、`onStart`、`onResume`、`onPause`、`onStop`、`onDestroy`。
- 生命周期不是为了背诵，而是为了决定初始化、监听、释放和保存状态的时机。

## 3.2 Fragment 的角色

### 3.2.1 为什么需要 Fragment

- 复用界面结构。
- 适应多窗格和不同屏幕尺寸。
- 配合导航形成更清晰的页面组织方式。

### 3.2.2 Fragment 生命周期与宿主关系

- Fragment 依赖宿主存在。
- Fragment 自身也有视图生命周期。
- 理解宿主生命周期和视图生命周期的区别，是避免空引用和泄漏的关键。

## 3.3 ViewModel 与状态管理

### 3.3.1 为什么不能把一切状态都塞进 Activity

- Activity 和 Fragment 会被系统销毁重建。
- 状态和界面容器耦合过深，会导致重复请求、状态丢失和逻辑混乱。

### 3.3.2 ViewModel 的作用

- 承担界面相关状态。
- 配置变更时保留状态。
- 让界面层更专注于渲染与事件分发。

### 3.3.3 状态保存的基本策略

- 临时界面状态与长期业务状态要区分。
- ViewModel、`savedInstanceState` 和持久化存储各自有不同职责。

## 3.4 本章实践任务

- 创建一个包含 Activity 和 Fragment 的简单页面。
- 在配置变更后保持页面关键状态不丢失。
- 用 ViewModel 托管界面展示数据。

## 3.5 小结

本章的核心不是多记几个回调方法，而是理解 Android 界面开发首先是“生命周期管理问题”。只有把组件、状态和回调关系理顺，后续的布局、导航和交互代码才会稳定。
