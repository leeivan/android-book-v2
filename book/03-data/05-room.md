# Room 数据库

Room 是现代 Android 中最重要的本地结构化存储能力之一。它的价值不在于“让数据库看起来更高级”，而在于把大量重复、易错且分散的数据库代码组织成更清晰的工程结构。本章把 Room 作为本地数据库主线来讲。

## 学习目标

- 理解 Room 的核心组成。
- 理解 Room 与原生 SQLite API 的关系。
- 为数据层设计、离线读取和状态流更新提供基础。

## 前置知识

- 已理解 SQLite 的基本角色。
- 已具备列表和本地数据展示的基本认知。

## 正文

### 1. Room 解决了哪些实际问题

如果直接使用 SQLite API，开发者很快会遇到几个典型痛点：

- SQL 与代码分散，维护成本高。
- 数据实体和查询结构容易失配。
- 错误发现往往偏晚。

Room 的价值就在于把这些问题收束成更清晰的结构：实体、DAO 和数据库类分别承担不同职责，让结构化数据访问更接近可维护工程，而不是零散脚本。

### 2. Room 的三个核心组成

理解 Room 时，最重要的是先分清三个角色：

- Entity：描述数据长什么样。
- DAO：描述如何访问数据。
- Database：描述这些实体和访问入口如何汇总成数据库实例。

只要这三个角色清楚，后续不管是关系建模、事务还是分页，都能比较自然地挂接进去。

### 3. Room 为什么适合和现代 Android 架构配合

Room 不只是数据库工具，它还天然适合和以下能力结合：

- ViewModel。
- Flow 或 LiveData。
- Repository。
- Paging。

这意味着 Room 往往不会单独出现，而是成为本地数据层的核心组成部分。

### 4. 数据建模时最该警惕什么

使用 Room 后，很多人会误以为“只要能写出注解就行”。真正更重要的是数据建模本身：

- 实体是否表达了真实业务对象。
- 查询是否围绕实际读取场景设计。
- 关系是否有必要，不要为了“形式完整”过度建模。

Room 让数据库更容易写，但不会代替你做建模决策。

### 5. 最小实践任务

1. 为一个待办列表设计最小 Room 实体。
2. 为它设计一个插入方法和一个查询全部方法。
3. 思考这个实体以后如果加上筛选、排序或完成状态，需要怎样调整 DAO。

### 6. 常见误区

- 只关注注解写法，不关注实体和查询设计。
- 把 Room 当成“会自动生成所有正确数据库结构”的黑盒。
- 在还没明确场景前就过度设计复杂关系。

## 小结

Room 把 Android 本地数据库开发从“低层 API 使用”提升成了“结构清晰的数据层设计”。后续进入数据层和架构章节时，Room 会成为重要支点。
