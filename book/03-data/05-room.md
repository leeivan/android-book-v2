# Room 数据库

如果说 SQLite 提供了 Android 本地结构化存储的底层能力，那么 Room 提供的就是更适合现代工程使用的数据库表达方式。它的价值不在于“让数据库看起来更高级”，而在于把那些重复、分散且容易出错的底层数据库代码收束成更清晰的结构：实体负责描述数据，DAO 负责表达访问，数据库类负责统一组织入口。本章要建立的，就是这种结构化数据库思维。

## 学习目标

- 理解 Room 的核心组成和职责边界。
- 理解 Room 与 SQLite 的关系。
- 理解 Room 为什么适合与 ViewModel、Flow 和 Repository 结合。
- 建立对实体建模、查询设计和迁移问题的初步认识。

## 前置知识

- 已理解 SQLite 的基本角色和核心概念。
- 已具备列表和本地结构化数据展示的基本认知。

## 正文

### 1. Room 解决了哪些实际问题

直接使用 SQLite API 时，开发者很快会遇到几个典型痛点：

- SQL 和业务代码分散。
- 对象映射容易出错。
- 表结构变化后迁移成本高。
- 数据读取和更新路径不够清晰。

Room 的价值就在于把这些问题组织成更稳定的结构，而不是让你在各个角落零散地写数据库逻辑。

### 2. Room 的三个核心组成

理解 Room 时，最重要的是先分清三个角色。

#### 2.1 Entity：数据长什么样

Entity 描述的是表结构对应的数据模型。它的重点不是“注解怎么写”，而是这个对象是否真正表达了业务里的稳定记录。

#### 2.2 DAO：如何读写这些数据

DAO 用来表达查询、插入、更新和删除等数据访问行为。它解决的是“我怎样和这类数据交互”，而不是“页面要显示什么”。

#### 2.3 Database：如何把实体和访问入口汇总起来

Database 类把整个本地数据库组织起来，形成统一入口。它的职责是“总装配”，而不是承载业务逻辑。

只要这三者职责清楚，Room 的整体结构就不会显得抽象。

### 3. 为什么 Room 特别适合现代 Android 架构

Room 不只是数据库工具，它还天然适合和这些能力组合：

- ViewModel。
- Flow 或 LiveData。
- Repository。
- Paging。

这意味着 Room 很少孤立存在。它通常是本地数据层的中心节点，负责把“数据库中的变化”转化为界面可观察的状态来源。

### 4. 数据建模时最需要警惕什么

很多人学 Room 时容易把注意力全部放在注解和 API 上，反而忽略真正更重要的问题：数据建模。

例如：

- 一个 Entity 是否表达了真实业务对象。
- 查询是否围绕真实读取场景设计。
- 是否为了“看起来完整”而过度引入复杂关系。

Room 让数据库代码更好写，但它不会替你做建模决策。建模一旦混乱，再优雅的注解也救不了后续维护。

### 5. 查询设计为什么比“把数据存进去”更重要

很多本地数据库设计失败，不是因为存不进去，而是因为：

- 最常见的读取场景没有被提前考虑。
- 查询频繁却没有围绕真实用例设计。
- 页面需要的数据不得不做多次低效拼装。

因此设计 Room 时，要优先问：这个页面最常见会怎么读，而不是先问：我能不能把所有字段都塞进一个实体。

### 6. Room 与迁移：数据库不是永远不变的

只要应用持续迭代，数据库结构迟早会变化。你可能会新增字段、调整默认值、增加索引，甚至重新组织表结构。Room 的一个重要价值，就是让数据库变化能进入更明确的工程管理，而不是依赖临时脚本和手工处理。

入门阶段你不必掌握所有迁移细节，但必须建立一个意识：数据库结构不是一次性设计完就永远不变的。

### 7. 最小示例：一个待办实体如何映射到页面

假设一个待办条目至少包含：

- 唯一标识。
- 标题。
- 完成状态。
- 创建时间。

当它进入 Room 后，你会立刻看到一条更完整的链路：

- Entity 描述条目结构。
- DAO 提供列表读取和状态更新方法。
- ViewModel 订阅数据变化。
- RecyclerView 或 Compose 列表根据最新数据刷新界面。

这就是 Room 在现代 Android 项目里的真实价值：它不是“把数据存一下”，而是成为稳定数据流的一部分。

### 8. 最小实践任务

1. 为待办列表设计一个最小 Room 实体。
2. 为它设计一个插入方法、一个按时间排序的查询方法和一个更新完成状态的方法。
3. 用自己的话说明 Entity、DAO 和 Database 各自解决什么问题。
4. 思考如果后续要增加分类字段，这次结构变更会影响哪些层。

### 9. 常见误区

- 只关注注解写法，不关注实体和查询设计。
- 把 Room 当成会自动替你解决所有数据库问题的黑盒。
- 在还没明确实际读取场景时就过度设计复杂关系。
- 认为数据库结构一旦定下就永远不会变化。

## 小结

Room 把 Android 本地数据库开发从“低层 API 使用”提升成了“结构清晰的数据层设计”。真正重要的不是注解本身，而是你是否能围绕真实业务场景，把实体、查询、状态流和页面展示组织成一条稳定的数据链路。
