# 数据层设计

当本地存储、JSON、列表界面和页面状态都开始出现时，一个更高层的问题就自然出现了：这些数据到底该怎样组织？本章不是讲单一 API，而是讲数据层思维。它会把前面几章中看似分散的存储问题，整理成一条更接近真实工程的数据流。

## 学习目标

- 理解数据层在应用架构中的位置。
- 理解 Repository、缓存和状态同步的基本问题。
- 为网络开发、架构和综合项目建立桥梁。

## 前置知识

- 已理解 DataStore、文件、SQLite、Room 和 JSON 的基本作用。
- 已具备列表展示和界面事件处理基础。

## 正文

### 1. 为什么单有存储和网络还不够

即使你已经会做本地存储、会调接口、会解析 JSON，也不代表数据层已经成立。因为真正的工程问题是：

- 数据从哪里来。
- 数据先读本地还是先读远程。
- 失败后怎么回退。
- 界面拿到的是原始数据还是整理后的状态。

如果这些问题没有被统一组织，页面一多，应用就会很快进入“每个页面都各自处理数据”的混乱状态。

### 2. 数据层为什么是界面层和数据源之间的缓冲带

数据层的核心价值是隔离变化。它让界面层不必直接关心：

- 接口细节。
- 数据库存取细节。
- 缓存策略。
- 错误恢复逻辑。

这样一来，界面层更接近“我需要什么数据”，而不是“我应该按什么顺序调几个接口、查几张表、合并几种状态”。

### 3. Repository 的位置

Repository 是数据层里最常见的组织方式之一。它的意义不在于名字，而在于职责：

- 对外提供稳定的数据读取入口。
- 对内协调本地与远程来源。
- 统一处理缓存、错误和回退策略。

在小项目里，Repository 也许看起来像“多绕了一层”；但只要页面和数据来源稍微增多，这一层就会迅速体现价值。

### 4. 缓存与同步问题为什么必须提前思考

很多项目早期只关心“先把接口调通”，等真正需要离线显示、失败兜底和重复进入页面时，才发现数据流已经难以整理。更稳妥的做法是尽早回答：

- 哪些数据可以缓存。
- 缓存多久有效。
- 远程失败时是否使用本地兜底。
- 页面刷新时是全量覆盖还是增量更新。

这些设计不是优化题，而是数据层架构题。

### 5. 最小实践任务

1. 以“新闻列表”或“待办事项”举例，画出一条简单数据流：界面 -> ViewModel -> Repository -> 本地/远程数据源。
2. 标出其中哪些地方负责读取、缓存、转换和回传状态。
3. 说明如果远程失败，界面最理想的兜底行为是什么。

### 6. 常见误区

- 每个页面都直接调自己的数据源。
- 把缓存逻辑散落在多个界面里。
- 只考虑成功路径，不设计失败和回退策略。

## 小结

数据层设计让前面学过的存储与网络能力真正开始协同工作。后续进入网络开发和应用架构章节后，这套思路会成为组织复杂项目的主线之一。
