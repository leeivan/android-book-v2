# 数据层设计

前面几章分别讲了配置、文件、SQLite、Room 和 JSON，但真正的 Android 项目不会把这些能力孤立使用。现实里的问题通常是：页面需要数据时，应该向谁要；本地和远程数据如何协作；缓存和真实来源如何划分；模型转换应该放在哪一层。所有这些问题，最终都会汇聚到“数据层设计”上。

这一章很容易被写成几句抽象架构口号，例如“加一个 Repository”“保持单一可信来源”“不要让页面直接访问数据库”。这些结论当然没有错，但如果不把它们放回真实数据流里，读者很难真正理解为什么数据层会成为 Android 工程里的关键稳定器。本章要做的，就是把这些原则还原为真实项目中能落地的边界设计。

## 学习目标

- 理解数据层为什么是 Android 工程的关键稳定器。
- 理解 Repository 在现代项目中的基本作用。
- 理解本地数据、远程数据和页面状态之间应如何分工。
- 建立“单一可信来源”和“边界清晰”的数据设计意识。

## 前置知识

- 已理解配置、文件、数据库和 JSON 的基本角色。
- 已具备 ViewModel 和页面状态的初步认知。

## 正文

### 1. 为什么页面不应该直接面对所有数据来源

如果页面既直接访问网络，又直接写数据库，还顺手读取配置和文件，那么短期看似方便，长期几乎一定会出问题：页面职责会迅速膨胀，同一数据逻辑会在多个页面重复出现，本地与远程优先级难以统一，测试和替换实现都会变得困难。UI 层本来应该专注在显示数据和接收用户事件，一旦它开始承担多数据源协调，复杂度就会很快失控。

Android 官方的数据层架构文档对此说得很清楚：UI 层包含 UI 状态和 UI 逻辑，而数据层包含应用数据和大部分业务逻辑。这个分层的目的不是让结构“看起来更专业”，而是让不同层可以围绕各自职责独立演进。

### 2. Repository 为什么会成为现代主线

官方数据层指南明确建议：应用应为不同类型的数据创建 repository，其他层不应直接访问数据源，数据层的入口应始终是 repository。Repository 的价值，不是“多加一层显得高级”，而是统一数据入口、隐藏本地与远程实现细节、把数据获取和更新逻辑从页面层抽离出来。

只要页面只依赖 Repository 提供的数据能力，整个工程就会更容易维护，也更容易替换数据来源。页面不需要知道数据来自 Room、DataStore、文件还是网络，只需要消费“当前这份可用数据”以及相关状态。这正是 Repository 的现实价值。

### 3. 单一可信来源为什么这么重要

Android 官方架构总览把“单一可信来源（single source of truth）”描述得非常明确：每种数据类型都应有一个单一可信来源，只有它能修改该数据，其他部分通过不可变形式读取。这个原则在数据层里尤其重要。因为只要同一份数据同时由页面缓存一份、数据库保存一份、网络返回又单独保留一份，而且三者更新时机还不一致，你很快就会遇到“到底哪份才是最新”的问题。

数据层文档还进一步指出：Repository 暴露给上层的数据，应直接来自可信来源；在离线优先场景中，本地数据库通常是推荐的可信来源。也就是说，Repository 的任务之一就是组合多个数据源，并解决它们之间的冲突，让上层不必自己判断“这一次该信哪份数据”。

### 4. 数据源和 Repository 的边界应该怎么分

官方数据层指南建议，每个数据源只负责一种来源，例如文件、网络或本地数据库；而 Repository 负责对外暴露数据、集中变更、抽象来源，并在多数据源之间做冲突解决。这个边界很关键。数据源更像“会和某一类外部系统打交道的桥”，Repository 则像“对上层提供统一数据能力的门面”。

只要这条边界清楚，很多架构决策都会自然得多。比如 Room DAO 属于本地数据源的一部分，Retrofit service 属于远程数据源的一部分，DataStore repository 或 preferences data source 属于配置来源的一部分；而把它们统一组合成“任务数据由谁来提供”的，应该是更上层的 Repository，而不是页面自己去到处拼。

### 5. 本地和远程数据如何协作，才不会让页面失控

很多应用既依赖远程接口，又需要本地缓存。这时最重要的问题不是“先调哪个接口”，而是：首次打开时是否应先展示本地已有内容，刷新时本地和远程如何同步，网络失败时页面是否还能显示旧数据。这类问题没有唯一模板，但官方关于 offline-first 的文档给出了一条非常稳定的原则：有网络访问的 Repository 在离线优先应用里通常都应拥有本地数据源，并且本地数据源应作为规范可信来源，也就是这里反复强调的单一可信来源。

这意味着页面通常不应该直接等网络返回后再决定显示什么，而应优先订阅可信来源，由 Repository 决定何时刷新本地、何时回退旧数据、何时上报错误状态。只要这个协调工作留在页面层，复杂度会迅速膨胀，而且不同页面很容易各写一套不一致策略。

### 6. 模型转换为什么通常属于数据层责任

网络接口模型、本地实体模型和页面展示模型往往不会完全一样。如果把这些转换逻辑散落到页面里，最终会导致页面充满数据清洗代码，同一转换被多处重复实现，结构调整时影响面过大。因此，模型转换通常更适合由数据层承担。页面层最好只看到更接近自身需求的结果，而不是同时理解网络 JSON、数据库 Entity 和 UI 文案规则。

这也是为什么数据层不仅仅是“取数据”，还承担了数据整理和抽象职责。数据源面对的是原始来源，Repository 面对的是上层使用场景，中间往往就需要模型转换来完成这层语义收束。

### 7. 最小示例：一个 Repository 如何组织本地与远程

下面这个例子用极简结构展示 Repository 怎样把本地 Room 数据源和远程数据源组织成统一入口：

```kotlin
class TaskRepository(
    private val localDataSource: TaskDao,
    private val remoteDataSource: TaskRemoteDataSource
) {
    fun observeTasks(): Flow<List<TaskEntity>> {
        return localDataSource.observeAll()
    }

    suspend fun refreshTasks() {
        val remoteTasks = remoteDataSource.fetchTasks()
        remoteTasks.forEach { task ->
            localDataSource.upsert(task.toEntity())
        }
    }
}
```

这个例子故意保持简单，但已经体现了几个关键点。上层观察的是本地数据流，而不是直接盯着网络结果；刷新动作由 Repository 协调，而不是由页面自己同时调网络和写数据库；本地数据库在这里更接近单一可信来源。你未来当然会加入错误处理、冲突策略和模型转换细节，但这条结构主线不应改变。

### 8. 数据层为什么直接影响测试和迭代效率

只要数据层边界清晰，页面可以更容易做状态测试，Repository 可以更容易替换模拟数据源，本地和远程策略也可以独立演进。Android 官方的数据层文档同样强调：把数据层从 UI 分离出来，有助于多屏复用数据、在 UI 之外复现业务逻辑，并提升可测试性。

也就是说，数据层不是“写给架构图看的”，而是直接影响开发效率和长期维护成本。一个边界清晰的数据层，会让新需求主要在同一条数据流里演进；一个边界混乱的数据层，则会迫使你在每个页面里都重新协调一遍数据来源。

### 9. 实践任务

起点条件：

- 已有一个列表页面，或者至少已经设计过需要本地与远程协作的数据场景。

步骤：

1. 选一个列表页面，画出它当前的数据流：页面、ViewModel、本地数据源、远程数据源分别在哪里。
2. 标出页面是否直接依赖了多个数据来源。
3. 重新设计一个更清晰的 Repository 边界，明确它应对外暴露什么。
4. 判断这条链路中的单一可信来源是什么，以及为什么是它。
5. 补一条模型转换规则：网络模型如何变成本地实体，或者本地实体如何变成页面模型。

预期结果：

- 你能把“页面直接调一切”的数据流改造成更稳定的分层结构。
- 你会更清楚数据源、Repository 和页面状态之间的边界。
- 你能说出某条数据链路里到底哪一层应被信任为最终结果。

自检方式：

- 你能解释：为什么页面不应直接面对多个数据来源。
- 你能判断：某段逻辑应属于数据源、Repository，还是页面层。
- 你能说出：单一可信来源在当前应用里是数据库、内存缓存，还是其他更合适的层。

调试提示：

- 如果某个页面同时直接拿网络、数据库和配置，通常说明数据层入口还没收束。
- 如果你总在页面里写“如果网络失败就回退本地，否则刷新数据库”这类逻辑，通常说明 Repository 还没承担起协调职责。
- 如果同一份数据在多个地方各存一份却没人能说清谁最新，先回到 SSOT 原则。

### 10. 常见误区

- 页面同时直接读网络、数据库和配置。
- 觉得 Repository 只是“为了分层而分层”。
- 同一数据有多个互相竞争的状态来源。
- 模型转换逻辑到处分散，页面层充满数据清洗代码。

## 小结

数据层设计的核心，不是层数有多少，而是数据边界是否清晰、来源是否统一、页面是否只依赖自己真正需要的结果。只要 Repository、数据源和单一可信来源这几条主线站稳，本地与远程协作、离线能力、页面状态管理和后续架构演进都会更容易组织。

到这里，数据篇的主线已经基本连起来了：配置、文件、SQLite、Room 和数据层不再是孤立工具，而是一套围绕边界和来源组织起来的完整数据体系。后续进入网络、架构和并发章节时，这条线会继续向外延伸。
