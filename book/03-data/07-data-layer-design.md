# 数据层设计

前面几章分别讲了配置、文件、SQLite、Room 和 JSON，但真正的 Android 项目不会把这些能力孤立使用。现实里的问题通常是：页面需要数据时，应该向谁要；本地和远程数据如何协作；缓存和真实来源如何划分；模型转换应该放在哪一层。所有这些问题，最终都会汇聚到“数据层设计”上。

## 学习目标

- 理解数据层为什么是 Android 工程的关键稳定器。
- 理解 Repository 在现代项目中的基本作用。
- 理解本地数据、远程数据和页面状态之间应如何分工。
- 建立“单一可信来源”和“边界清晰”的数据设计意识。

## 前置知识

- 已理解配置、文件、数据库和 JSON 的基本角色。
- 已具备 ViewModel 和页面状态的初步认知。

## 正文

### 1. 为什么页面不应该直接面对所有数据来源

如果页面既直接访问网络，又直接写数据库，还顺手读取配置和文件，那么短期看似方便，长期几乎一定会出问题：

- 页面职责会迅速膨胀。
- 同一数据逻辑会在多个页面重复出现。
- 本地与远程优先级难以统一。
- 测试和替换实现都会变得困难。

这就是为什么现代 Android 项目会强调数据层边界。页面需要的是“可消费的数据结果”，而不是自己去管理所有来源。

### 2. Repository 为什么是常见主线

Repository 的价值，不是“多加一层显得高级”，而是：

- 统一数据入口。
- 隐藏本地与远程实现细节。
- 把数据获取和更新逻辑从页面层抽离出来。

当页面只依赖 Repository 提供的数据能力时，整个工程会更容易维护，也更容易替换数据来源。

### 3. 单一可信来源为什么重要

如果同一份数据同时由页面缓存一份、数据库保存一份、网络返回又单独保留一份，而且三者更新时机还不一致，你很快就会遇到“到底哪份才是最新”的问题。

这就是“单一可信来源”概念的重要性。它要求你尽量明确：

- 这类数据最终以哪一层为准。
- 页面展示应优先读哪里。
- 更新时应由哪一层负责写回。

一旦这个判断不清，数据层会很快变成状态来源大战。

### 4. 本地与远程数据如何协作

很多应用既依赖远程接口，又需要本地缓存。这时最重要的问题不是“先调哪个接口”，而是：

- 首次打开时，是否应先展示本地已有内容。
- 刷新时，本地和远程如何同步。
- 网络失败时，页面是否还能显示旧数据。

这类问题没有唯一模板，但有一个稳定原则：页面不应直接承担本地与远程协调职责，这应由数据层统一决定。

### 5. 模型转换为什么应属于数据层责任

网络接口模型、本地实体模型和页面展示模型往往不会完全一样。如果把这些转换逻辑散落到页面里，最终会导致：

- 页面充满数据清洗代码。
- 同一转换被多处重复实现。
- 结构调整时影响面过大。

因此，模型转换通常更适合由数据层承担。页面层最好只看到更接近自己需求的结果。

### 6. 数据层设计为什么会影响测试和迭代效率

只要数据层边界清晰：

- 页面可以更容易做状态测试。
- Repository 可以更容易替换模拟数据源。
- 本地和远程策略可以独立演进。

也就是说，数据层不是“写给架构图看的”，而是直接影响开发效率和长期维护成本。

### 7. 最小示例：新闻列表的数据流应该如何组织

一个新闻列表页面通常会经历这样一条链路：

- Repository 决定先读本地还是先发起远程刷新。
- 本地数据库或缓存提供当前可用内容。
- 网络返回新结果后更新本地。
- 页面订阅统一状态来源并自动刷新。

这条链路的重点不是 API，而是你能否看出：页面根本不应该知道这些协调细节。

### 8. 最小实践任务

1. 选一个列表页面，画出它当前的数据流。
2. 标出页面是否直接依赖了多个数据来源。
3. 重新设计一个更清晰的 Repository 边界。
4. 明确这条链路中的单一可信来源是什么。

### 9. 常见误区

- 页面同时直接读网络、数据库和配置。
- 觉得 Repository 只是“为了分层而分层”。
- 同一数据有多个互相竞争的状态来源。
- 模型转换逻辑到处分散，页面层充满数据清洗代码。

## 小结

数据层设计的核心，不是层数有多少，而是数据边界是否清晰、来源是否统一、页面是否只依赖自己真正需要的结果。只要这一层站稳，后面的架构、并发和离线能力都会更容易组织。
