# 文件存储

一提到本地存储，很多人第一反应就是数据库，但现实里的 Android 应用经常还会处理另一类完全不同的数据：图片缓存、下载文件、导出文档、音视频、临时附件。这些内容不适合用键值或表结构表达，而更接近“文件”本身。本章要解决的，是如何用文件视角看待数据，以及为什么现代 Android 对文件访问有越来越清晰的边界要求。

文件存储这一章最容易学偏的地方，是还停留在“有路径就能读写”的旧思路里。现代 Android 并不是完全不让你处理文件，而是要求你更清楚地回答几个问题：这份文件只是当前应用自己使用，还是需要共享给其他应用；它是缓存，还是用户明确希望长期保留的资产；它是媒体，还是普通文档；它是否应该由用户亲自通过系统入口授予访问权。只要这些问题没理清，文件设计很快就会出问题。

## 学习目标

- 理解什么类型的数据更适合文件存储。
- 区分应用私有文件、缓存文件和共享文件的边界。
- 理解现代 Android 文件访问为什么更强调受控入口。
- 为媒体访问、导出分享和缓存设计打基础。

## 前置知识

- 已建立基础的数据分类框架。
- 已知道“文件”和“结构化记录”是两类不同问题。

## 正文

### 1. 什么数据适合文件存储

文件存储最适合那些更接近“独立内容单元”的数据，例如图片、视频、音频、导出文档、离线下载资源和临时缓存文件。它们通常不需要按字段查询，也不需要复杂关系建模，而更关注内容本身、大小、生命周期和共享方式。只要你的问题更接近“这份内容应该保存在哪里、何时清理、是否允许共享”，它就更像文件问题，而不是数据库问题。

反过来，如果你真正关心的是“如何按状态筛选、如何按时间排序、如何做条件查询”，那通常已经是结构化数据问题。文件和数据库可以一起出现在同一业务里，但它们解决的不是同一层问题。

### 2. 应用私有文件和共享文件，必须先分开理解

Android 官方的存储总览把 app-specific storage 和 shared storage 清楚分开，这是现代文件设计最重要的边界之一。应用私有文件只服务于当前应用使用，例如缓存图片、中间导出结果、本地临时附件。它们最适合放在应用私有目录中，因为不需要暴露给其他应用，生命周期更容易控制，也更符合权限模型。官方文档也明确指出：应用私有目录中的文件会在应用卸载时被移除，因此不应把用户预期独立保留的资产放在那里。

共享文件则不同。如果文件需要被其他应用访问，或者本身就是系统媒体库的一部分，就不能再简单按“私有内部文件”来处理。你需要进一步考虑：用户是否明确授权，应通过哪种系统入口选择或暴露文件，分享和导出应如何保持边界清晰。这时候问题已经不只是“把 bytes 写到哪”，而是“谁有权访问它”。

### 3. 缓存文件和长期文件为什么不能混用

这是文件设计中非常常见的错误。缓存文件通常具备几个特点：可以被清理，可以重新下载或重新生成，不应被当成长期业务资产。而长期文件通常意味着用户有预期它会保留，丢失后有实际成本，可能需要更明确的目录和命名策略。如果把两者混在一起，最终通常会出现两个结果：要么你因为怕误删而不敢清缓存，要么你真的把用户关心的内容当缓存删掉了。

文件系统的问题很多时候不是“读写失败”，而是生命周期管理失控。你必须从一开始就知道：哪些目录可以大胆清理，哪些目录必须谨慎处理；哪些文件只是性能辅助，哪些文件属于业务结果。只有这条边界清楚，后面缓存策略和导出策略才不会互相冲突。

### 4. 为什么现代 Android 不鼓励随意拿路径到处传

文件访问在 Android 上越来越强调受控边界。原因很简单：文件不只是数据问题，还涉及隐私、共享和访问权限。官方数据存储文档明确建议：如果文件只给当前应用使用，就优先放在 app-specific storage；如果需要访问用户的媒体，优先考虑 Photo Picker；如果需要处理文档和其他共享文件，则使用 Storage Access Framework。

这意味着今天学习文件存储时，必须从一开始就把“边界”纳入考虑。不是说路径完全没用了，而是很多场景里，文件访问的主入口已经变成 URI、系统选择器和用户授权，而不是开发者自己假定某个路径可长期直接访问。

### 5. Photo Picker、MediaStore 和 SAF 分别在解决什么问题

这三个名字很容易被一起提到，但它们解决的问题并不完全相同。Photo Picker 是现代 Android 访问用户照片和视频时最推荐的入口之一。官方 Photo Picker 文档强调，它提供了一种更安全、内建的方式，让用户只把选中的图片或视频授予给应用，而不是把整个媒体库都暴露出来。它特别适合“让用户选一张或几张图片/视频”这种场景。

MediaStore 更适合围绕系统共享媒体库来读取或写入媒体内容。它面对的是“媒体”这类受系统集中管理的共享内容。Storage Access Framework 则更适合普通文档和文件，包括创建文档、打开文档、选择目录等。官方 SAF 文档也明确指出，这种方式由用户亲自参与选择文件或目录，因此通常不需要额外系统权限。简单说：选照片视频优先想 Photo Picker，处理一般文档优先想 SAF，围绕系统媒体库组织共享媒体时再更多考虑 MediaStore。

### 6. 应用私有文件的最小保存路径

对于只给当前应用自己使用的文件，最稳妥的默认思路是先落到应用私有目录。Android 官方存储总览给出的路径非常明确：内部私有文件可以通过 `getFilesDir()` 或 `getCacheDir()` 访问，外部 app-specific 目录可以通过 `getExternalFilesDir()` 或 `getExternalCacheDir()` 获取。在 Android 4.4 及以上系统上，应用访问自己的 app-specific external storage 一般也不需要额外存储权限。

下面这个示例演示如何在缓存目录里写一个临时导出文件：

```kotlin
fun Context.createTempExportFile(content: String): File {
    val exportFile = File(cacheDir, "tasks-export.txt")
    exportFile.writeText(content)
    return exportFile
}
```

这个例子的重点不是 API 难度，而是边界选择：这是一个当前应用生成、可重建、可清理的临时导出结果，因此放在 `cacheDir` 是合理的。如果这份文件变成“用户明确保存并希望长期保留的导出文档”，那它就不该继续停留在缓存目录里，而应通过更合适的共享或创建文档入口处理。

### 7. 最小示例：用 Photo Picker 让用户选图

对于“让用户从相册中选一张图或一段视频”的场景，官方当前更推荐通过 Activity Result API 使用 Photo Picker。它的优势在于访问边界清晰，而且在支持的设备上会优先走系统内建选择器；当 Photo Picker 不可用时，库还会自动回退到 `ACTION_OPEN_DOCUMENT`。

```kotlin
private val pickMedia =
    registerForActivityResult(PickVisualMedia()) { uri: Uri? ->
        if (uri != null) {
            // 使用用户选中的媒体 URI
        }
    }

fun pickPhoto() {
    pickMedia.launch(
        PickVisualMediaRequest(PickVisualMedia.ImageOnly)
    )
}
```

这个例子的教学点是：文件访问主入口已经不再总是“我自己拿路径去扫目录”，而是“由系统提供受控入口，用户明确选择，应用拿到一个可用的 URI”。这也是现代 Android 文件与媒体访问最需要建立的直觉。

还要注意，应用拿到一个 `Uri` 并不等于拿到了一条可以永久依赖的真实文件路径。对很多现代入口来说，`Uri` 本身就是访问边界的一部分。若业务只是临时读取、显示或上传，直接围绕 `Uri` 处理通常更合适；若业务要求长期离线持有这份内容，则应在合适授权前提下把内容复制进应用私有目录，或在文档场景中明确处理持续访问策略，而不是假设原始位置会一直可读。

### 8. 文件命名和目录策略，也是设计问题

很多文件系统问题，不是读写失败，而是随着项目增长逐渐失控。例如临时文件和正式导出文件放在一起，文件名无法表达来源和用途，清理策略没有分层。文件存储同样需要设计：哪些目录是缓存，哪些目录是长期保存，哪些内容可以重建，哪些内容必须谨慎删除。

只要目录和命名策略从一开始就乱，后面即使 API 都写对了，系统也还是会越来越难维护。文件设计不只是 I/O 操作，更是生命周期和边界管理。

### 9. 实践任务

起点条件：

- 已有一个包含导出、附件或图片选择需求的练习应用，或者使用“待办/笔记”作为示例。

步骤：

1. 列出应用中所有文件型数据，不要只写“文件”，要写到具体用途，例如缓存缩略图、导出文档、用户附件。
2. 区分哪些是缓存、哪些是长期文件、哪些可能需要共享给其他应用。
3. 为它们分别设计目录和清理策略。
4. 找出至少一个应优先通过 Photo Picker 或 SAF 解决的场景，而不是直接处理路径。
5. 如果有导出需求，明确它是临时导出结果还是用户希望长期保留的文件。

预期结果：

- 你能把应用私有文件、缓存文件和共享文件分开理解。
- 你能说清哪些场景应优先用 Photo Picker、SAF 或 MediaStore。
- 你会开始用“生命周期和授权边界”来设计文件系统，而不是只看读写是否成功。

自检方式：

- 你能解释：为什么现代 Android 不鼓励随意拿文件路径到处传。
- 你能判断：一个文件应该放 app-specific 目录，还是应该走共享访问入口。
- 你能说出：某份文件丢失后，对用户来说是可重建缓存，还是不可随意清理的资产。

调试提示：

- 如果你一开始就想请求大范围存储权限，先停下来检查是否其实可以用 app-specific storage、Photo Picker 或 SAF。
- 如果文件一多就不敢清理，通常说明缓存和长期资产没有分层。
- 如果业务需要让用户“明确保存到某处”，优先考虑创建文档类系统入口，而不是内部私有路径。

### 10. 常见误区

- 把所有文件都放在同一目录下。
- 缓存和长期文件混用。
- 只关心能不能读写，不关心共享边界。
- 仍然把“到处传文件路径”当成默认思路。

## 小结

文件存储解决的是“独立内容单元如何保存和管理”的问题。现代 Android 中，文件设计必须同时考虑生命周期、共享边界、用户授权和清理策略，而不能再停留在“有路径就能读写”的早期思路上。

下一章我们会把视角切到结构化存储，也就是 SQLite。到那时，问题会从“文件放哪里”转向“记录如何稳定增删改查、排序和查询”。
