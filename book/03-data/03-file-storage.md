# 文件存储

一提到本地存储，很多人第一反应就是数据库，但现实里的 Android 应用经常还会处理另一类完全不同的数据：图片缓存、下载文件、导出文档、音视频、临时附件。这些内容不适合用键值或表结构表达，而更接近“文件”本身。本章要解决的，是如何用文件视角看待数据，以及为什么现代 Android 对文件访问有越来越清晰的边界要求。

## 学习目标

- 理解什么类型的数据更适合文件存储。
- 区分应用私有文件、缓存文件和共享文件的边界。
- 理解现代 Android 文件访问为什么更强调受控入口。
- 为媒体访问、导出分享和缓存设计打基础。

## 前置知识

- 已建立基础的数据分类框架。
- 已知道“文件”和“结构化记录”是两类不同问题。

## 正文

### 1. 什么数据适合文件存储

文件存储最适合那些更接近“独立内容单元”的数据，例如：

- 图片、视频、音频。
- 导出文档。
- 离线下载资源。
- 临时缓存文件。

它们往往不需要按字段查询，不需要复杂关系建模，而更关注路径、大小、生命周期和共享方式。

### 2. 应用私有文件和共享文件为什么必须分开理解

这是现代 Android 文件设计最重要的边界之一。

#### 2.1 应用私有文件

这类文件只服务于当前应用自身使用，例如缓存图片、内部导出中间结果、本地临时附件。它们最适合放在应用私有目录中，因为：

- 不需要额外暴露给其他应用。
- 生命周期更容易控制。
- 与权限模型更一致。

#### 2.2 共享文件

如果文件需要被其他应用访问，或者本身就是系统媒体库的一部分，那么就不能再简单按“私有内部文件”来处理。你需要进一步考虑：

- 用户是否明确授权。
- 应通过哪种系统入口选择或暴露文件。
- 分享和导出应如何保持边界清晰。

### 3. 缓存文件和长期文件为什么不能混用

这是文件设计中非常常见的错误。缓存文件通常具备下面几个特点：

- 可以被清理。
- 可以重新下载或重新生成。
- 不应被当成长期业务资产。

而长期文件通常意味着：

- 用户有预期它会保留。
- 丢失后有实际成本。
- 可能需要更明确的目录和命名策略。

如果把两者混在一起，最终的结果通常是：不敢清缓存，或者误删长期文件。

### 4. 现代 Android 为什么不鼓励随意拿文件路径到处传

文件访问在 Android 上越来越强调受控边界。原因很简单：文件不只是数据问题，还涉及隐私、共享和访问权限。现代 Android 更鼓励开发者：

- 优先使用应用私有目录。
- 当需要用户选择外部文件时，走系统文件选择器或照片选择器。
- 当需要共享内容时，使用受控的共享机制，而不是到处裸传路径。

学习文件存储时，必须从一开始就把“边界”纳入考虑。

### 5. 文件命名和目录策略为什么也是设计问题

很多文件系统问题，不是读写失败，而是随着项目增长逐渐失控。例如：

- 临时文件和正式导出文件放在一起。
- 文件名无法表达来源和用途。
- 清理策略没有分层。

因此，文件存储同样需要设计：

- 哪些目录是缓存。
- 哪些目录是长期保存。
- 哪些内容可以重建。
- 哪些内容必须谨慎删除。

### 6. 最小示例：待办应用的导出文件和图片附件

假设一个待办应用支持：

- 导出清单为文档。
- 为待办项添加图片附件。

这两个功能都涉及文件，但它们的边界并不相同。导出文档更接近“用户明确要保存和分享的内容”，而图片附件则可能既涉及应用内部缓存，也涉及用户从系统中选择媒体。这个例子说明，文件问题不能只用一个“保存到本地”来笼统概括。

### 7. 最小实践任务

1. 列出你的应用里所有文件型数据。
2. 区分哪些是缓存、哪些是长期文件、哪些可能需要共享。
3. 为它们分别设计目录和清理策略。
4. 思考哪些场景应该走系统文件选择或照片选择入口，而不是直接操作路径。

### 8. 常见误区

- 把所有文件都放在同一目录下。
- 缓存和长期文件混用。
- 只关心能不能读写，不关心共享边界。
- 仍然把“到处传文件路径”当成默认思路。

## 小结

文件存储解决的是“独立内容单元如何保存和管理”的问题。现代 Android 中，文件设计必须同时考虑生命周期、共享边界和清理策略，而不能再停留在“有路径就能读写”的早期思路上。
