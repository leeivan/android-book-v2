# SharedPreferences 与 DataStore

设置项和轻量配置数据，是 Android 应用中最常见的一类持久化需求。它们看起来简单，却几乎存在于所有应用里：主题模式、登录引导状态、上次筛选条件、用户偏好、开关项等。本章的重点，不是把两种 API 并列介绍，而是说明：为什么今天更推荐把 DataStore 作为新项目主线，以及它和早期 SharedPreferences 的边界差异在哪里。

## 学习目标

- 理解键值型配置数据适合解决什么问题。
- 理解 SharedPreferences 的历史价值和现实边界。
- 理解 DataStore 为什么更适合作为现代主线。
- 理解“简单配置”为什么不该滥用数据库或文件存储。

## 前置知识

- 已建立基本的数据分类认知。
- 知道配置型数据和业务型数据不是一回事。

## 正文

### 1. 什么数据适合键值型存储

键值型存储最适合那些：

- 数据量小。
- 结构简单。
- 不需要复杂查询。
- 读取频繁。

例如：

- 是否开启通知。
- 当前语言偏好。
- 用户是否完成新手引导。

如果一份数据一开始就需要排序、筛选、分页或多表关系，它通常就已经不适合键值型存储了。

### 2. SharedPreferences 为什么长期流行

SharedPreferences 之所以长期流行，是因为它非常直观：读写简单、适合轻量配置、几乎所有 Android 开发者都见过它。对早期项目而言，它确实解决了大量基础设置项保存问题。

但“简单易上手”并不等于“今天仍然是最佳主线”。随着应用对一致性、异步读取和错误处理的要求变高，SharedPreferences 的局限会越来越明显。

### 3. DataStore 为什么成为现代推荐做法

DataStore 的意义，不只是“新的 Preferences API”，更重要的是它在设计上更适合现代 Android：

- 它更契合异步和响应式数据读取方式。
- 更适合和协程、Flow 一起使用。
- 在数据更新组织上更容易纳入现代状态流体系。

也就是说，DataStore 不只是替代写法，而是更适合现代项目状态管理和数据流组织的方案。

### 4. Preferences DataStore 与传统偏好存储的关系

学习 DataStore 时要特别注意，不要把“Jetpack DataStore”与某个名字相似的扩展机制混为一谈。我们这里讲的主线，是 Jetpack 提供的现代配置存储能力，它的目标是替代 SharedPreferences 作为新项目的首选方案之一。

入门阶段最需要抓住的，不是所有实现细节，而是两个核心问题：

- 这是轻量配置数据，不是结构化业务数据库。
- 读取和更新应该服务于状态流，而不是零散同步调用。

### 5. 为什么“简单配置”也值得认真建模

很多项目里，设置项之所以后来难维护，不是因为数据多，而是因为一开始就没有把它们当正式状态来设计。例如：

- 相同配置在多个地方重复保存。
- 布尔值命名含义不清。
- 页面状态和持久配置混杂。

键值型存储虽然简单，但并不意味着可以随意写。只要命名、边界和来源不清，后面问题一样会堆起来。

### 6. 一个实际判断：什么时候不该再用键值存储

如果你发现某类数据开始具备下面这些特征，就应该重新考虑是否还适合键值存储：

- 有明确列表结构。
- 需要按字段搜索。
- 需要历史记录或分页。
- 需要多个字段组合表达一个复杂实体。

这通常意味着数据已经进入数据库问题，而不是配置问题。

### 7. 最小示例：主题模式和引导状态为什么适合 DataStore

例如：

- 是否使用暗色模式。
- 是否展示过首次引导。

这类数据共同特点是：小、简单、读取频繁，而且需要随着设置变化及时反馈到界面。它们非常适合作为 DataStore 的练习入口。

### 8. 最小实践任务

1. 列出你的应用中所有“设置项类”数据。
2. 判断它们是否都真的适合键值型存储。
3. 选两个最典型的配置项，重新命名并明确其含义和默认值。
4. 思考这些配置是否需要作为状态流暴露给界面层。

### 9. 常见误区

- 把 SharedPreferences 当成所有小数据的永久默认答案。
- 用键值存储承载已经明显结构化的业务数据。
- 配置项命名含糊，导致后续维护困难。
- 把配置读取写成零散同步调用，而不是纳入状态流组织。

## 小结

键值型存储解决的是“轻量配置如何持久化”的问题。今天的新项目更适合以 DataStore 作为主线，而不是继续默认从 SharedPreferences 起步。真正重要的不是 API 名字，而是你能否清楚区分配置数据和业务数据，并让配置读取自然融入现代状态管理模型。
