# 数据存储概述

Android 应用里的数据并不是只有一种形式。设置项、登录状态、缓存文件、图片、离线列表、数据库记录、用户手动选择的文档，这些表面上都叫“数据”，但它们的生命周期、读取方式、权限边界和维护成本完全不同。本章先不急着讲某个具体 API，而是先建立一套判断框架：什么数据应该放哪里，为什么不能用“一种存储打天下”。

## 学习目标

- 建立“什么数据该放哪里”的判断框架。
- 理解 Android 中几类最常见的数据存储形态。
- 理解现代 Android 为什么更强调存储边界和访问控制。
- 为 DataStore、文件、SQLite 和 Room 打基础。

## 前置知识

- 已完成基础篇和 UI 部分的学习。
- 理解页面状态、列表数据和用户输入的基本概念。

## 正文

### 1. 存储问题首先不是 API 问题，而是分类问题

很多初学者第一次接触存储时，只问一个问题：数据怎么存进去？但真正有价值的存储设计，至少还要同时回答：

- 这份数据是否需要长期保存。
- 这份数据是简单键值，还是结构化记录。
- 它是否需要离线查询。
- 它是否可能被其他应用或系统共享。
- 它是否涉及隐私、权限或大体量文件。

只有这些问题先说清楚，后面选择 DataStore、文件、SQLite、Room、MediaStore 还是系统文档接口，才不会混乱。

### 2. Android 中最常见的四类数据

从学习和工程决策角度看，Android 数据大致可以先分成四类：

#### 2.1 键值型配置

例如：

- 是否开启深色模式。
- 是否看过新手引导。
- 当前筛选条件。

这类数据通常量小、结构简单，读取频繁，但不适合复杂查询。

#### 2.2 文件型数据

例如：

- 图片缓存。
- 导出文档。
- 应用下载的离线资源。

它们更接近二进制内容或独立文件，而不是需要按字段查询的数据表。

#### 2.3 结构化业务数据

例如：

- 待办事项列表。
- 聊天消息记录。
- 文章收藏和历史阅读记录。

这类数据通常需要增删改查、排序、筛选和离线读取，因此更适合数据库模型。

#### 2.4 共享媒体与文档访问

例如：

- 访问系统媒体库里的图片和视频。
- 让用户从文件选择器里选一个文档。
- 从系统提供的照片选择器中选图。

这类数据不仅是“存在哪里”的问题，还涉及平台权限和访问边界。

### 3. 为什么现代 Android 的存储更强调边界

早期 Android 对文件访问相对宽松，很多教程里会出现“直接拿路径到处读写”的写法。现代 Android 越来越强调下面这些方向：

- 应用私有目录优先。
- 共享存储要走受控入口。
- 用户选择的文档要尊重用户授权边界。
- 媒体访问应优先使用平台提供的受控接口。

这意味着今天学习存储时，不能只看“能不能读写成功”，还必须考虑这种做法是否符合平台长期方向。

### 4. 做存储决策时最值得先问的四个问题

当你面对一份新数据时，先不要着急选 API。可以先按下面顺序判断：

1. 它只是简单配置，还是需要结构化查询。
2. 它更像记录，还是更像文件。
3. 它是否需要与其他应用或系统共享。
4. 它是短期缓存，还是长期业务资产。

只要这四个问题回答清楚，后面的技术选择通常就会自然收敛。

### 5. “缓存”和“持久化业务数据”为什么不能混为一谈

这是数据设计里非常常见的混乱源。缓存的重点通常是：

- 可以重建。
- 允许丢失。
- 更关注速度和重复利用。

而持久化业务数据的重点通常是：

- 不能随意丢失。
- 需要明确数据结构和一致性。
- 需要支持长期读取和修改。

如果把两者混在一起，你很快就会发现数据清理、恢复和同步策略都变得模糊。

### 6. 最小示例：待办应用的数据应该怎么分层

以一个待办应用为例，至少会出现三类不同数据：

- 用户是否开启提醒：适合键值型配置。
- 待办条目本身：适合结构化数据库。
- 用户导出的分享文件：适合文件存储。

这个例子说明，哪怕是一个看起来很小的应用，也很少只需要一种存储方案。

### 7. 最小实践任务

1. 列出你正在设计的应用中的至少三类不同数据。
2. 判断哪些适合键值存储，哪些适合数据库，哪些属于文件。
3. 用一张简单表格写出：数据类型、是否长期保存、是否共享、推荐存储方式。

### 8. 常见误区

- 所有数据都想塞进同一种存储方案。
- 只关心能不能存，不关心未来怎么读、怎么维护。
- 不区分缓存和长期业务数据。
- 忽视权限与共享边界。

## 小结

本章的目标不是让你立刻掌握某个 API，而是先建立存储决策框架。后面几章会在这个框架上展开不同数据类型各自更合适的实现方式。
