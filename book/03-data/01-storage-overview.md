# 数据存储概述

Android 应用里的数据并不是只有一种形式。设置项、登录状态、缓存文件、图片、离线列表、数据库记录、用户手动选择的文档，这些表面上都叫“数据”，但它们的生命周期、读取方式、权限边界和维护成本完全不同。本章先不急着讲某个具体 API，而是先建立一套判断框架：什么数据应该放哪里，为什么不能用“一种存储打天下”。

数据存储这一部分最容易学偏的地方，是把它学成几个 API 的串讲：今天讲 SharedPreferences，明天讲文件，后天讲 SQLite。这样当然能记住一些工具名字，但很难回答真正重要的问题：为什么这份数据应该进 DataStore 而不是数据库，为什么这类文件应该放应用私有目录而不是共享存储，为什么有些内容应该通过 Photo Picker 或 SAF 让用户选择，而不是直接处理路径。本章的任务，就是先把这些判断框架立起来。

## 学习目标

- 建立“什么数据该放哪里”的判断框架。
- 理解 Android 中几类最常见的数据存储形态。
- 理解现代 Android 为什么更强调存储边界和访问控制。
- 为 DataStore、文件、SQLite 和 Room 打基础。

## 前置知识

- 已完成基础篇和 UI 部分的学习。
- 理解页面状态、列表数据和用户输入的基本概念。

## 正文

### 1. 存储问题首先不是 API 问题，而是分类问题

很多初学者第一次接触存储时，只问一个问题：数据怎么存进去？但真正有价值的存储设计，至少还要同时回答几件事：这份数据是否需要长期保存，它是简单键值还是结构化记录，它是否需要离线查询，它是否可能被其他应用或系统共享，它是否涉及隐私、权限或大体量文件。只有这些问题先说清楚，后面选择 DataStore、文件、SQLite、Room、MediaStore 还是系统文档接口，才不会混乱。

Android 官方的数据与文件存储总览，也正是按“应用私有文件、共享存储、偏好配置、本地数据库”这几类来划分，而不是先从工具名字出发。对读者来说，这种划分方式很重要，因为它提醒你：存储方案的第一原则是“边界正确”，而不是“哪种 API 用起来最顺手”。

### 2. Android 中最常见的四类数据

从学习和工程决策角度看，Android 数据可以先分成四大类。第一类是键值型配置，例如是否开启深色模式、是否看过新手引导、当前筛选条件。这类数据通常量小、结构简单、读取频繁，但不适合复杂查询。第二类是文件型数据，例如图片缓存、导出文档、应用下载的离线资源。它们更接近独立内容单元，而不是需要按字段查询的数据表。

第三类是结构化业务数据，例如待办事项列表、聊天消息记录、文章收藏和历史阅读记录。这类数据通常需要增删改查、排序、筛选和离线读取，因此更适合数据库模型。第四类是共享媒体与文档访问，例如访问系统媒体库里的图片和视频、让用户从文件选择器里选一个文档、从系统提供的照片选择器中选图。这类数据不仅是“存在哪里”的问题，还涉及平台权限、授权边界和用户控制。

### 3. 现代 Android 为什么更强调存储边界

早期 Android 对文件访问相对宽松，很多教程里会出现“直接拿路径到处读写”的写法。现代 Android 则越来越强调几个方向：应用私有目录优先，共享存储要走受控入口，用户选择的文档要尊重用户授权边界，媒体访问应优先使用平台提供的受控接口。Android 官方存储用例文档也明确建议：如果文件只给当前应用使用，就优先放在 app-specific storage；如果需要访问媒体或文档，则应使用 Photo Picker、MediaStore 或 Storage Access Framework。

这意味着今天学习存储时，不能只看“能不能读写成功”，还必须考虑这种做法是否符合平台长期方向。一个能运行的方案，如果需要多余权限、绕过用户授权边界，或者把本该私有的数据直接暴露到共享空间，那么它就不是一个好的现代 Android 方案。

### 4. 做存储决策时，先问四个判断问题

当你面对一份新数据时，先不要着急选 API。更稳妥的顺序通常是：它只是简单配置，还是需要结构化查询；它更像记录，还是更像文件；它是否需要与其他应用或系统共享；它是短期缓存，还是长期业务资产。只要这四个问题回答清楚，后面的技术选择通常就会自然收敛。

例如，“用户是否完成新手引导”显然更接近轻量配置；“待办列表”显然更接近结构化业务数据；“用户从相册中选的一张照片”则天然牵涉共享媒体与授权边界；“图片缓存缩略图”往往只是可重建缓存，而不应被当成长期业务资产。对真实项目来说，很多存储问题其实在这一步就已经决定了大方向。

### 5. “缓存”和“持久化业务数据”为什么不能混为一谈

这是数据设计里非常常见的混乱源。缓存的重点通常是可以重建、允许丢失、更关注速度和重复利用；而持久化业务数据的重点通常是不能随意丢失、需要明确数据结构和一致性、需要支持长期读取和修改。如果把两者混在一起，你很快就会发现数据清理、恢复和同步策略都变得模糊。

举个简单例子，同样是“待办应用里的图片”。如果它只是某个列表缩略图缓存，就更接近缓存数据；如果它是用户手动为待办项添加的证据图片，那它就更接近长期业务资产。两者看起来都像文件，但生命周期和清理策略完全不同。这也是为什么存储分类不能只靠文件后缀或类型名，而要看业务语义。

### 6. 最小示例：同一个待办应用，也会同时用到多种存储

以一个待办应用为例，至少会出现三到四类不同数据：用户是否开启提醒，适合键值型配置；待办条目本身，适合结构化数据库；用户导出的分享文件，适合文件存储；用户手动挑选的封面图片，则可能通过 Photo Picker 或 SAF 获取，再根据业务边界决定是否复制到应用私有目录中。

这个例子说明，哪怕是一个看起来很小的应用，也很少只需要一种存储方案。真正成熟的工程不是“选一个存储全都塞进去”，而是为不同数据选择最合适的边界和生命周期。

### 7. 实践任务

起点条件：

- 你已经有一个正在设计中的应用，或者愿意用“待办/笔记/清单”作为练习对象。

步骤：

1. 列出应用中的至少五类不同数据，不要只写“数据”两个字，要写到具体业务层级。
2. 为每类数据回答四个问题：是否长期保存、是否结构化、是否需要共享、是否可重建。
3. 初步判断它更适合键值存储、文件存储、数据库还是共享存储入口。
4. 特别找出一类“看起来像文件、但其实是业务资产”的数据，再写下它和缓存的区别。

预期结果：

- 你会发现同一个应用往往天然需要多种存储方案。
- 你会开始用“生命周期和边界”来分类数据，而不是只看 API 熟不熟。
- 你能在后续章节里更快判断某份数据应该落到哪一类方案。

自检方式：

- 你能解释：为什么“所有数据都塞进同一种存储”通常是坏主意。
- 你能区分：一份数据是配置、文件、结构化记录，还是共享媒体/文档访问问题。
- 你能判断：某份数据丢失后是“可以重建”，还是会直接破坏用户预期。

调试提示：

- 如果你发现某类数据“既像文件又像数据库”，先回到业务语义，问它主要解决的是内容保存还是结构化查询。
- 如果你总想直接处理路径，先检查这份数据是否其实属于共享媒体或用户授权文档访问场景。
- 如果某类数据未来需要筛选、排序和历史记录，通常就不该再停留在简单键值或单文件整体保存层。

### 8. 常见误区

- 所有数据都想塞进同一种存储方案。
- 只关心能不能存，不关心未来怎么读、怎么维护。
- 不区分缓存和长期业务数据。
- 忽视权限与共享边界。

## 小结

本章真正建立的是存储判断框架，而不是某个具体 API 的用法。只要你能先分清配置、文件、结构化记录和共享访问这几类问题，后面学习 DataStore、文件系统、SQLite 和 Room 时，就不会把所有工具都混成“都是本地存储”。

下一章我们会先处理最轻量的一类数据，也就是设置项与偏好配置，并把 DataStore 作为现代 Android 的默认主线来展开。

### 教材化延伸：为什么存储总览不能只记“有哪些方案”

官方文档会清楚告诉你 SharedPreferences、DataStore、文件、Room、MediaStore、SAF 各自是什么，但教材更需要回答“什么数据该放哪一层、为什么不能只按技术名选方案”。如果没有先建立数据边界视角，读者很容易把所有问题都变成“哪个 API 更方便”，而不是“这个数据的生命周期、共享边界和一致性要求是什么”。

### 资料路线

- 先用本章的选择表从数据边界出发分类，而不是从 API 名字出发分类。
- 再对照官方 data storage use cases 文档，确认不同存储方案的正式适用范围。
- 最后阅读样例项目中配置数据、本地实体和媒体文件分别落在哪一层，理解真实工程里的拆分方式。
