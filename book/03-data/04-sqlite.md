# SQLite

虽然现代 Android 项目更常把 Room 作为本地数据库主线，但 SQLite 仍然是理解 Android 结构化存储的底层基础。本章不把 SQLite 当作主教学入口来讲复杂实现，而是帮助读者理解数据库思维和 Room 背后的平台基础。

## 学习目标

- 理解 SQLite 在 Android 平台中的位置。
- 理解表、查询、事务等基础数据库概念。
- 为 Room 章节提供底层背景。

## 前置知识

- 已理解文件、配置型存储与结构化数据的区别。
- 对列表和数据展示已有基础认知。

## 正文

### 1. 为什么仍然要理解 SQLite

即使你在新项目中很少直接手写 SQLite API，也仍然有必要理解它，原因有三点：

- Room 最终仍建立在 SQLite 之上。
- 旧项目中直接操作 SQLite 的代码仍然大量存在。
- 许多数据库设计问题并不会因为用了 Room 就自动消失。

所以，本章的重点不是“你以后一定要手写 SQLite”，而是“你应该知道结构化数据底层是怎样工作的”。

### 2. SQLite 适合处理什么数据

当数据具有以下特征时，SQLite 或基于 SQLite 的上层方案就更合适：

- 记录之间存在明确字段结构。
- 需要按条件查询、排序和过滤。
- 需要事务一致性。
- 数据需要长期离线保存。

这类问题如果用键值存储或文件硬拼，短期可能能跑，但复杂度会上升得很快。

### 3. 数据库思维比 API 细节更重要

学习 SQLite 时，最值得先建立的是数据库思维：

- 表代表什么实体。
- 主键为什么重要。
- 查询为什么应围绕读取场景设计。
- 事务为什么关系到一致性。

如果只盯着执行语句的 API，而不理解这些结构问题，就很容易把数据库写成“能跑但难维护的临时仓库”。

### 4. 为什么新项目通常不以原生 SQLite API 为主线

直接使用 SQLite API 的问题不在于它不能用，而在于：

- 样板代码多。
- 错误暴露晚。
- 与现代状态流和架构组织衔接成本较高。

因此，本书下一章会把 Room 作为真正的主线入口，而本章主要承担“打底层认知”的作用。

### 5. 最小实践任务

1. 以“待办事项”或“用户收藏”为例，设计一张最小数据表。
2. 确定这张表的主键和几个核心字段。
3. 写出两个最常见的读取场景，例如“按时间排序列出全部数据”“按关键字搜索”。

### 6. 常见误区

- 只会建表，不考虑查询场景。
- 把数据库当成万能存储箱。
- 以为换成 Room 后数据库设计问题就自动消失。

## 小结

理解 SQLite 的意义，在于理解结构化数据底层如何工作。掌握了这层基础，后面学习 Room 时你看到的就不只是注解和工具，而是一套对数据库问题的更高层封装。
