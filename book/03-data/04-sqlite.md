# SQLite

在 Android 本地数据体系里，SQLite 仍然是一个绕不开的名字。即使今天新项目通常更推荐以 Room 作为数据库主线，理解 SQLite 依然非常重要，因为它决定了 Android 本地结构化存储的底层能力模型。本章的重点不是把 SQLite 学成纯数据库课程，而是帮助你理解：它在 Android 里解决什么问题、为什么原生直接使用门槛较高，以及为什么 Room 能建立在它之上。

## 学习目标

- 理解 SQLite 在 Android 本地结构化存储中的位置。
- 理解表、行、查询、索引和事务这些最基本概念。
- 理解直接使用原生 SQLite API 的主要痛点。
- 为 Room 和数据层设计打下数据库基础认知。

## 前置知识

- 已理解什么数据适合结构化存储。
- 已知道键值型和文件型存储不能替代数据库。

## 正文

### 1. 为什么结构化数据需要数据库

一旦你的数据开始具备下面这些特征，数据库问题就出现了：

- 需要按字段查询。
- 需要排序和筛选。
- 需要更新局部字段。
- 需要在多条记录之间保持一致性。

例如待办列表、聊天记录、订单草稿、离线文章，这些都不适合用一个简单文件整体保存，因为你不是只需要“把它存下来”，而是需要“稳定地增删改查”。

### 2. SQLite 在 Android 中扮演什么角色

SQLite 是 Android 本地结构化存储最基础的底层引擎之一。它的价值在于：

- 轻量。
- 嵌入式。
- 不依赖独立数据库服务。
- 足以支撑大量移动端本地数据场景。

也就是说，在 Android 设备上做本地数据库，并不是去连接一个远端数据库进程，而是在应用环境里直接使用嵌入式数据库能力。

### 3. 学 SQLite 时最少需要掌握哪些概念

对 Android 开发者来说，最基础、最必要的概念包括：

- 表：数据按结构组织的基本容器。
- 行：一条记录。
- 列：记录中的字段。
- 查询：如何按条件取出数据。
- 索引：如何提升某些读取效率。
- 事务：如何保证多步写入的一致性。

你不需要在入门阶段就把数据库理论学得极深，但如果连这些概念都没有，后面学 Room 也只会停留在注解记忆层面。

### 4. 为什么直接使用原生 SQLite API 门槛较高

SQLite 本身很强，但直接在 Android 中使用原生 SQLite API，会很快遇到几个问题：

- SQL 与代码分散，维护成本高。
- 实体对象和查询结果映射容易出错。
- 数据库结构变化后的迁移管理更困难。
- 代码中容易充斥重复模板。

这并不是 SQLite 不好，而是说明：真实项目需要更高一层的抽象来降低这些样板和易错点。

### 5. SQL 仍然值得理解

即使你后面会主要使用 Room，也不意味着可以跳过 SQL 认知。原因很简单：结构化数据设计的核心依然是：

- 数据如何建模。
- 查询如何组织。
- 哪些读取路径是高频路径。

如果只会写注解、不会理解查询和结构设计，你在数据量稍大时很快就会陷入性能和结构问题。

### 6. 最小示例：待办表为什么天然适合 SQLite 思维

一个待办表至少会包含：

- `id`
- `title`
- `is_done`
- `created_at`

即使只是这样一个简单表，你也会立刻遇到数据库视角下的问题：

- 如何按创建时间排序。
- 如何筛选未完成项。
- 如何更新某一条的完成状态。

这说明数据库的核心不是“能存多行”，而是“能按结构稳定地处理记录”。

### 7. 最小实践任务

1. 为一个待办列表写出最小数据表结构。
2. 思考你最常见的三个查询场景，例如全部读取、按状态筛选、按时间排序。
3. 判断哪些字段未来可能需要索引或迁移。
4. 记录为什么这类数据不适合直接用文件或键值型存储。

### 8. 常见误区

- 把 SQLite 学成“数据库命令大全”而忽略实际场景。
- 认为有了 Room 就完全不需要理解 SQL。
- 一开始就设计过度复杂的表关系。
- 只考虑存储，不考虑未来如何查询和更新。

## 小结

SQLite 是 Android 本地结构化存储的重要底层基础。理解它的价值，不是为了让你长期手写底层数据库代码，而是为了让你后面学习 Room 时真正明白：Room 简化了什么，又没有替你解决什么。
