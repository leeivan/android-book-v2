# SQLite

在 Android 本地数据体系里，SQLite 仍然是一个绕不开的名字。即使今天新项目通常更推荐以 Room 作为数据库主线，理解 SQLite 依然非常重要，因为它决定了 Android 本地结构化存储的底层能力模型。本章的重点不是把 SQLite 学成纯数据库课程，而是帮助你理解：它在 Android 里解决什么问题、为什么原生直接使用门槛较高，以及为什么 Room 能建立在它之上。

这一章最容易学偏的地方，是把 SQLite 误解成“已经过时，所以可以跳过”，或者反过来把它学成一整套独立数据库命令大全。前一种会让你后面学 Room 时只会背注解，后一种则容易脱离 Android 真实工程场景。更合理的路径，是先抓住 SQLite 在 Android 中的角色，再理解 Room 到底为你抽象掉了哪些痛点。

## 学习目标

- 理解 SQLite 在 Android 本地结构化存储中的位置。
- 理解表、行、查询、索引和事务这些最基本概念。
- 理解直接使用原生 SQLite API 的主要痛点。
- 为 Room 和数据层设计打下数据库基础认知。

## 前置知识

- 已理解什么数据适合结构化存储。
- 已知道键值型和文件型存储不能替代数据库。

## 正文

### 1. 为什么结构化数据需要数据库

一旦你的数据开始具备这些特征，数据库问题就出现了：需要按字段查询，需要排序和筛选，需要更新局部字段，需要在多条记录之间保持一致性。例如待办列表、聊天记录、订单草稿、离线文章，这些都不适合用一个简单文件整体保存，因为你不是只需要“把它存下来”，而是需要“稳定地增删改查”。

也就是说，数据库真正解决的不是“能存很多条”，而是“能按结构处理记录”。只要你的业务开始要求查询和更新，问题就已经不再是存储格式问题，而是数据结构问题。

### 2. SQLite 在 Android 中扮演什么角色

Android 官方的 SQLite 文档把它定位为适合“重复或结构化数据”的本地数据库能力。它的价值在于轻量、嵌入式、不依赖独立数据库服务，并且足以支撑大量移动端本地数据场景。换句话说，在 Android 设备上做本地数据库，并不是去连接一个远端数据库进程，而是在应用环境里直接使用嵌入式数据库能力。

这也是为什么 SQLite 长期仍然是 Android 本地结构化存储的底层基础。即使你以后主要通过 Room 间接使用它，SQLite 的模型依然决定了表如何组织、查询如何表达、索引如何影响读取路径，以及事务如何保证一致性。

### 3. 学 SQLite 时，最少需要掌握哪些概念

对 Android 开发者来说，最基础、最必要的概念包括：表、行、列、查询、索引和事务。表是按结构组织数据的基本容器；行是一条记录；列是记录里的字段；查询负责按条件取出数据；索引帮助某些读取路径更快；事务用来保证多步写入的一致性。

你不需要在入门阶段就把数据库理论学得极深，但如果连这些概念都没有，后面学 Room 也只会停留在注解记忆层面。尤其要注意的是，数据库设计从来不是“先想怎么存”，而是“先想会怎么查、怎么改、哪些字段会成为高频路径”。

### 4. 结构设计和查询设计，要一起考虑

很多数据库设计失败，不是因为字段不够多，而是因为最常见的读取场景从一开始就没有被考虑进去。一个待办表看起来很简单，但只要你问几个真实问题，结构设计就开始显现：要不要按创建时间排序，要不要筛选未完成项，要不要按分类分组，要不要按更新时间同步。这些问题最终都会反馈到表结构、索引和查询写法上。

因此，学习 SQLite 时要尽早养成一个习惯：建模和查询不是先后完全分离的两步。一个结构如果无法支撑最常见的读取场景，它就不是一个好结构。你后面学 Room 时，这个判断依然成立，只是表达方式从手写 SQL 变成了注解和 DAO 方法。

### 5. 为什么直接使用原生 SQLite API 门槛较高

Android 官方的 SQLite 页面对这一点讲得非常直接：这些 API 虽然强大，但属于低层接口，需要大量时间和精力使用。官方特别指出两个痛点：原始 SQL 查询缺少编译期验证，随着数据图变化，受影响的查询要手工更新；另外，SQL 结果与对象之间的转换需要大量样板代码。

除此之外，真实项目中还会再叠加几个工程痛点：数据库结构变化后的迁移管理更困难，线程处理需要自己保证，数据库创建和升级逻辑容易散落在不同角落。官方文档也明确提醒，`getWritableDatabase()` 和 `getReadableDatabase()` 这类调用可能是耗时操作，应放在后台线程。这些问题并不代表 SQLite 不好，而是说明真实项目需要更高一层的抽象来降低样板和易错点。

### 6. `SQLiteOpenHelper` 在原生 SQLite 里解决什么问题

如果直接使用原生 SQLite，Android 官方给出的标准入口通常是 `SQLiteOpenHelper`。它帮助你把数据库创建、升级等生命周期管理集中起来。官方文档也明确说明：通过 `SQLiteOpenHelper` 获取数据库引用时，系统会在需要时执行创建和更新逻辑，而不是在应用启动时无条件完成全部工作。

下面这个例子展示的是最小化的数据库帮助类骨架：

```kotlin
private const val DATABASE_NAME = "tasks.db"
private const val DATABASE_VERSION = 1

class TaskDbHelper(context: Context) : SQLiteOpenHelper(
    context,
    DATABASE_NAME,
    null,
    DATABASE_VERSION
) {
    override fun onCreate(db: SQLiteDatabase) {
        db.execSQL(
            """
            CREATE TABLE task (
                id INTEGER PRIMARY KEY,
                title TEXT NOT NULL,
                is_done INTEGER NOT NULL DEFAULT 0,
                created_at INTEGER NOT NULL
            )
            """.trimIndent()
        )
    }

    override fun onUpgrade(db: SQLiteDatabase, oldVersion: Int, newVersion: Int) {
        // 根据版本做迁移
    }
}
```

这个例子的重点不是让你以后长期手写这些代码，而是让你看见原生 SQLite 的真实形态：表结构由 SQL 定义，数据库生命周期由 helper 管理，迁移要自己负责，后续查询、映射和线程控制也大多要自己处理。这正是 Room 后面要帮你简化的部分。

这里尤其要注意版本演进。数据库表一旦进入正式发布，表结构就不再只是当前版本的草稿，而是需要随应用升级一起维护的公共契约。新增列、拆分表、修改默认值时，都要明确旧数据如何迁移；如果一开始没把“将来怎么升级”纳入设计，数据库往往不是坏在第一次建表，而是坏在第三次迭代以后。

### 7. SQL 仍然值得理解，因为 Room 并不会替你做建模

即使你后面会主要使用 Room，也不意味着可以跳过 SQL 认知。原因很简单：结构化数据设计的核心依然是数据如何建模、查询如何组织、哪些读取路径是高频路径。Room 会帮你做编译期校验、对象映射和样板收束，但它不会替你决定业务表结构是否合理，也不会自动知道你的查询设计是否高效。

如果只会写注解，不会理解查询和结构设计，你在数据量稍大时很快就会陷入性能和结构问题。这也是为什么这一章虽然不要求你长期手写原生 SQLite，却仍然必须建立数据库底层认知。

### 8. 最小示例：待办表为什么天然适合 SQLite 思维

一个待办表至少会包含 `id`、`title`、`is_done` 和 `created_at`。即使只是这样一个简单表，你也会立刻遇到数据库视角下的问题：如何按创建时间排序，如何筛选未完成项，如何更新某一条的完成状态，哪些字段需要索引支持。这说明数据库的核心不是“能存多行”，而是“能按结构稳定地处理记录”。

正因为这些问题天然存在，待办表非常适合做 SQLite 入门练习。它足够简单，不会让业务复杂度淹没数据库问题；又足够真实，能让你看到表结构、查询和更新是如何联动的。

### 9. 实践任务

起点条件：

- 已有一个待办、笔记或清单类的小应用设想。
- 已理解配置型和文件型数据不适合解决结构化查询。

步骤：

1. 为一个待办列表写出最小数据表结构，至少包含唯一标识、标题、完成状态和创建时间。
2. 写下最常见的三个查询场景，例如全部读取、按状态筛选、按时间排序。
3. 思考这些查询在数据量变大时，哪些字段可能需要索引。
4. 额外考虑：如果未来新增分类字段或更新时间，这个表结构会怎么演进。

预期结果：

- 你会开始把数据库设计理解为“结构和查询一起设计”。
- 你能说清为什么这类数据不适合继续停留在文件或键值存储层。
- 你会对 Room 后续解决的问题有更具体的感受。

自检方式：

- 你能解释：为什么有了 Room 也仍然值得理解 SQLite 和 SQL。
- 你能区分：SQLite 解决的是底层结构化存储问题，Room 解决的是工程抽象问题。
- 你能判断：某个字段是“顺手加上去”，还是与高频查询直接相关。

调试提示：

- 如果你在画表时只想着“能存哪些字段”，先停下来补上“最常见怎么查”。
- 如果你想一开始就设计很多复杂关系，先回到当前真实读取场景，不要为了完整而过度设计。
- 如果你觉得 SQLite 很繁琐，那正是 Room 后面值得出现的原因，但不要因此跳过底层概念。

### 10. 常见误区

- 把 SQLite 学成“数据库命令大全”而忽略实际场景。
- 认为有了 Room 就完全不需要理解 SQL。
- 一开始就设计过度复杂的表关系。
- 只考虑存储，不考虑未来如何查询和更新。

## 小结

SQLite 是 Android 本地结构化存储的重要底层基础。理解它的价值，不是为了让你长期手写底层数据库代码，而是为了让你后面学习 Room 时真正明白：Room 简化了什么，又没有替你解决什么。只要这一层站稳，后面的实体建模、查询设计和迁移策略就不会变成只会套注解的表面工作。

下一章我们会在 SQLite 这层基础上进入 Room，看看现代 Android 项目是如何把实体、DAO、数据库入口和数据流组织成更稳定的结构。
