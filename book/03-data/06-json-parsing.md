# JSON 解析

移动应用很少只处理本地生成的数据。只要开始请求网络接口、读取配置文件、处理导入导出，JSON 就几乎不可避免地进入你的工程。很多人会把 JSON 解析理解成“把字符串转成对象”，这当然是它的直接作用，但真正重要的是：你如何让外部数据安全、清晰地进入自己的应用模型，而不是把接口字段原样扩散到整个代码库中。

这一章最容易学偏的地方，是把 JSON 解析当成某个库的语法演示。这样能写出例子，却很难回答真实项目中的问题：接口字段缺失了怎么办，字段名和页面语义不一致怎么办，接口模型是否应该直接进数据库或页面，错误和默认值由谁兜底。本章会把 JSON 解析放回数据层边界里理解，而不是只看“能不能反序列化成功”。

## 学习目标

- 理解 JSON 在 Android 数据流中的常见位置。
- 理解“接口数据模型”和“业务展示模型”为什么不应完全混在一起。
- 理解 JSON 解析不仅是技术转换，也是边界控制问题。
- 为 Retrofit、Repository 和数据层设计打基础。

## 前置知识

- 已具备基础的数据分类认知。
- 已知道应用常常需要和网络或外部数据源交互。

## 正文

### 1. JSON 为什么几乎无处不在

在 Android 开发里，JSON 常见于网络接口返回、本地导入导出文件、配置数据，以及调试和日志中的结构化内容。它之所以常见，不是因为它完美，而是因为它足够通用、可读且跨平台友好。Kotlin 官方的序列化文档也把“通过网络传输或存入数据库/文件的数据转换”视为序列化的典型场景，而 JSON 正是其中最常见的一种格式。

但正因为 JSON 常见，很多项目会不自觉地让它侵入整个代码库：接口怎么返回，应用里就怎么传；网络字段叫什么，页面里就跟着叫什么。短期看这样很省事，长期却会让外部接口结构和内部业务结构牢牢耦合。

### 2. 解析 JSON 的真正难点，不在语法

入门时，很多人把注意力都放在“怎么把字段映射到类”，但真实难点往往在更后面：接口字段可能缺失，类型可能变化，某些字段名字并不适合直接进入 UI 层，同一个接口模型在不同页面里也可能需要不同展示方式。这说明 JSON 解析的关键不是“对象能不能生成”，而是“外部数据怎样进入你的内部模型”。

Kotlin 序列化和其他 JSON 库都能帮你减少样板代码，但它们不会替你决定边界。如果解析成功就直接把 DTO 传到页面，工具用得再现代，结构依然可能很脆弱。

### 3. DTO、Entity、UI Model 为什么要分层理解

现代数据层设计里，一个很有价值的意识是：外部数据模型不一定等于内部业务模型。你可以把它简单理解为：DTO 更接近接口原始结构，Entity 更接近本地存储结构，UI Model 更接近页面展示需要。并不是每个小项目都必须把模型切得很细，但至少要知道：直接把接口模型一路传到 UI，是一种短期方便、长期脆弱的做法。

这条边界之所以重要，是因为三个模型面对的问题不同。DTO 关心的是“接口长什么样”，Entity 关心的是“本地怎么稳定存”，UI Model 关心的是“页面实际需要什么”。只要你把它们混成一个对象，任何一层变化都可能波及另外两层。

### 4. 容错、默认值和空安全，决定了边界是否稳

外部数据永远比本地内存对象更不可靠。接口可能返回空字段、旧字段、异常值，甚至整体结构变化。因此解析层需要承担一个重要角色：把外部不稳定性尽量限制在边界处，不要让错误一路扩散到页面层。这也是为什么 JSON 解析通常要和空安全、默认值策略、错误包装、模型转换一起考虑。

如果这一步没有做好，页面层就会不断出现“为什么这里突然是 null”“为什么一个字段换名就全项目报错”“为什么空列表和解析失败完全分不清”。解析层的真正价值，就是把这些不确定性拦在入口附近。

### 5. 解析工具只是手段，边界设计才是核心

无论你使用 `kotlinx.serialization`、Moshi、Gson，甚至 Android 自带的 `org.json`，真正的重点始终是：模型是否表达清楚，解析后的数据是否进入正确层级，错误和缺失字段是否被妥善处理。Kotlin 官方序列化文档强调的是“把外部格式转换为运行时对象”，而 Android 自带的 `JSONTokener` 文档甚至特别提醒其解析器出于兼容性是宽松的，成功解析并不等于输入完全符合严格 JSON 规范。

也就是说，工具能减少样板代码，但不能替你完成边界设计。解析库越强，你越容易在早期忽略这个问题；但项目一复杂，边界仍然得靠你自己把住。

### 6. 最小示例：用 DTO 到 UI Model 的转换保护页面

下面这个例子用 `kotlinx.serialization` 展示一个最小 DTO 到 UI Model 的转换。重点不是库本身，而是边界位置：

```kotlin
@Serializable
data class ArticleDto(
    val id: Long,
    val title: String,
    @SerialName("published_at") val publishedAt: String? = null,
    @SerialName("image_url") val imageUrl: String? = null,
    @SerialName("source_name") val sourceName: String? = null
)

data class ArticleUiModel(
    val id: Long,
    val title: String,
    val subtitle: String,
    val coverUrl: String?
)

fun ArticleDto.toUiModel(): ArticleUiModel {
    return ArticleUiModel(
        id = id,
        title = title,
        subtitle = sourceName ?: "未知来源",
        coverUrl = imageUrl
    )
}
```

这个例子里，接口字段保持在 DTO 层，页面拿到的是更接近展示语义的 `ArticleUiModel`。这样一来，当接口字段变化时，不会直接波及整个 UI 层。你仍然需要补更完整的日期转换、错误处理和格式校验，但结构主线已经建立起来了。

### 7. 最小实践任务

起点条件：

- 已有一个简单接口返回样例，或者愿意使用文章列表/新闻列表作为练习对象。

步骤：

1. 找一个简单 JSON 结构，用自己的话描述每个字段的实际业务含义，而不是只抄字段名。
2. 把其中一部分字段定义成 DTO，再决定哪些字段应进入 UI Model，哪些应在中间被转换或过滤。
3. 为可能缺失的字段设计默认值策略，例如空标题、空来源、空图片链接。
4. 思考如果接口新增、删除或重命名字段，你的页面会受到什么影响，哪些层需要修改。

预期结果：

- 你会开始把 JSON 解析理解成“外部数据进入内部模型”的过程。
- 你能区分接口模型和页面模型的不同职责。
- 你会更早意识到默认值和容错策略的重要性。

自检方式：

- 你能解释：为什么 JSON 解析不只是“字符串转对象”。
- 你能判断：某个字段是否适合直接暴露给 UI。
- 你能说出：默认值和转换逻辑应尽量挡住哪些外部不稳定性。

调试提示：

- 如果接口字段一改页面就跟着大量报错，通常说明 DTO 和 UI 边界没有立住。
- 如果你开始在页面里处理大量空值和字段拼装，说明解析层和转换层职责可能过弱。
- 如果你只关注“库能不能解析”，而没问“解析后数据该去哪一层”，后面问题往往会堆到 Repository 和 UI。

### 8. 常见误区

- 把 JSON 解析理解成单纯字符串转对象。
- 接口模型直接在整个项目里到处传递。
- 不处理空值、缺失字段和异常结构。
- 把解析工具当成设计本身。

## 小结

JSON 解析的重点，不是你会不会用某个库，而是你是否能把外部数据安全、清晰地吸收到自己的应用结构中。只要这条边界处理得好，网络层、数据层和页面层之间就不容易互相污染；如果边界没立住，再多现代工具也只是让问题被包装得更漂亮。

到这里，第 3 部分的数据主线就基本补齐了。后续进入网络篇时，你会不断看到这里建立的 DTO、模型转换和边界意识继续发挥作用。
