# JSON 解析

移动应用很少只处理本地生成的数据。只要开始请求网络接口、读取配置文件、处理导入导出，JSON 就几乎不可避免地进入你的工程。很多人会把 JSON 解析理解成“把字符串转成对象”，这当然是它的直接作用，但真正重要的是：你如何让外部数据安全、清晰地进入自己的应用模型，而不是把接口字段原样扩散到整个代码库中。

## 学习目标

- 理解 JSON 在 Android 数据流中的常见位置。
- 理解“接口数据模型”和“业务展示模型”为什么不应完全混在一起。
- 理解 JSON 解析不仅是技术转换，也是边界控制问题。
- 为 Retrofit、Repository 和数据层设计打基础。

## 前置知识

- 已具备基础的数据分类认知。
- 已知道应用常常需要和网络或外部数据源交互。

## 正文

### 1. JSON 为什么几乎无处不在

在 Android 开发里，JSON 常见于：

- 网络接口返回。
- 本地导入导出文件。
- 配置数据。
- 调试和日志中的结构化内容。

它之所以常见，不是因为它完美，而是因为它足够通用、可读且跨平台友好。

### 2. 解析 JSON 的真正难点不在语法

入门时，很多人把注意力都放在“怎么把字段映射到类”，但真实难点往往在更后面：

- 接口字段可能缺失。
- 类型可能变化。
- 某些字段名字并不适合直接进入 UI 层。
- 同一个接口模型在不同页面里需要不同展示方式。

这说明 JSON 解析的关键不是“对象能不能生成”，而是“外部数据怎样进入你的内部模型”。

### 3. DTO、Entity、UI Model 为什么要分层理解

现代数据层设计里，一个很有价值的意识是：外部数据模型不一定等于内部业务模型。

你可以把它简单理解为：

- DTO 更接近接口原始结构。
- Entity 更接近本地存储结构。
- UI Model 更接近页面展示需要。

并不是每个小项目都必须把模型切得很细，但至少要知道：直接把接口模型一路传到 UI，是一种短期方便、长期脆弱的做法。

### 4. 容错和默认值为什么很重要

外部数据永远比本地内存对象更不可靠。接口可能返回空字段、旧字段、异常值，甚至整体结构变化。因此解析层需要承担一个重要角色：把外部不稳定性尽量限制在边界处，不要让错误一路扩散到页面层。

这也是为什么 JSON 解析通常要和：

- 空安全。
- 默认值策略。
- 错误包装。
- 模型转换。

一起考虑。

### 5. 解析工具只是手段，不是设计本身

无论你使用哪种 JSON 解析工具，真正的重点始终是：

- 模型是否表达清楚。
- 解析后的数据是否进入正确层级。
- 错误和缺失字段是否被妥善处理。

工具能减少样板代码，但不能替你完成边界设计。

### 6. 最小示例：文章列表接口为什么不该直接绑定到页面

假设新闻接口返回的字段包括：

- `id`
- `title`
- `published_at`
- `image_url`
- `source_name`

你当然可以把这些字段直接绑定到页面，但更稳妥的做法通常是：

- 先在网络层保持接口模型。
- 在数据层转换成更符合业务含义的内部模型。
- 再根据页面需求组织展示数据。

这样一来，当接口字段变化时，不会直接波及整个 UI 层。

### 7. 最小实践任务

1. 找一个简单 JSON 结构，用自己的话描述每个字段的实际业务含义。
2. 判断哪些字段适合直接进入页面，哪些更适合先做转换。
3. 为可能缺失的字段设计默认值策略。
4. 思考如果接口新增或删除字段，你的页面会受到什么影响。

### 8. 常见误区

- 把 JSON 解析理解成单纯字符串转对象。
- 接口模型直接在整个项目里到处传递。
- 不处理空值、缺失字段和异常结构。
- 把解析工具当成设计本身。

## 小结

JSON 解析的重点，不是你会不会用某个库，而是你是否能把外部数据安全、清晰地吸收到自己的应用结构中。只有边界处理得好，网络层和页面层才不会彼此污染。
