# 事件处理

一个页面一旦从“能显示”进入“能交互”，事件处理就成了 UI 设计的中心问题。按钮点击、输入变化、列表项操作、弹窗确认、手势触发，这些都属于事件。很多初学者学事件处理时，容易把它学成“给控件绑定回调”的机械动作。但从工程角度看，事件处理真正要解决的是：用户动作如何进入系统，页面如何响应，状态如何更新，以及这些逻辑应该放在哪一层才不会很快失控。

## 学习目标

- 理解 UI 事件从用户动作到页面反馈的基本链路。
- 理解监听器、状态更新和业务逻辑为什么不应混成一团。
- 建立“页面负责接收事件，状态层负责组织结果”的基本意识。
- 为 Fragment、Navigation、RecyclerView 和 Compose 状态驱动打基础。

## 前置知识

- 已理解布局基础和常用控件的职责。
- 已具备 Activity 和页面基本结构认知。

## 正文

### 1. 事件处理的本质是什么

事件处理表面上是“某个控件被点击了”，本质上却是一次用户意图进入系统的过程。一个完整的事件链通常包括：

1. 用户做出动作。
2. UI 层接收到动作。
3. 页面根据当前状态决定如何响应。
4. 状态被更新。
5. 界面依据新状态重新展示。

如果把这条链路看清楚，很多“回调地狱”问题都会更容易避免。

### 2. 为什么事件逻辑不能全写在点击回调里

很多入门示例都会在 `setOnClickListener` 里直接写业务逻辑，这在最小例子里可以接受，但一旦页面复杂起来，就会立刻出现问题：

- 业务逻辑散落在多个监听器中。
- 同一状态被多个地方分别修改。
- 页面一重建，逻辑和状态边界难以维护。

更合理的做法通常是：UI 负责捕获事件和触发动作，真正的状态变化与业务判断放到更稳定的状态层或 ViewModel 中。

### 3. 页面层到底该负责什么

页面层最适合承担这些职责：

- 绑定控件和监听。
- 把用户动作转换成明确的页面事件。
- 根据状态渲染界面。
- 处理轻量的 UI 反馈，例如显示 Snackbar、弹出输入法或导航跳转。

页面层不适合承担的是长期业务状态和复杂分支逻辑。只要这一点守不住，页面代码会非常快地膨胀。

### 4. 输入事件为什么比点击更容易失控

点击事件通常是离散触发，而输入事件往往是连续变化。文本输入、搜索建议、表单校验、密码强度提示，背后都可能带来频繁状态更新。

因此处理输入事件时，尤其要注意：

- 不要每次输入变化都立即触发昂贵操作。
- 区分“当前文本变化”和“用户提交意图”。
- 把输入值和校验结果作为状态管理，而不是零散判断。

这类问题后面在 Compose 和状态流章节里会再次出现。

### 5. 页面反馈为什么也属于事件设计的一部分

事件处理不只是“接到点击之后执行方法”，还包括用户在动作之后能否得到清晰反馈。例如：

- 按钮点击后是否立即禁用，避免重复提交。
- 表单错误是否在正确位置显示。
- 提交成功后是否导航，还是仅刷新界面。

如果反馈设计缺失，页面就会给用户一种“没反应”或“反应很奇怪”的感觉。

### 6. 最小示例：按钮点击不应直接包揽所有逻辑

一个常见的反例是：点击保存按钮后，在监听器里直接做输入校验、数据库写入、网络请求、页面跳转和提示弹窗。这样的代码短期能跑，长期几乎注定难维护。

更合理的链路通常是：

- 监听器只负责收集当前输入。
- 页面把这次动作交给状态层处理。
- 状态层决定校验结果和下一步行为。
- 页面根据返回状态显示错误、进度或成功结果。

重点不是某个具体 API，而是责任边界。

### 7. 事件处理与单向数据流

当页面复杂起来后，一个很重要的稳定器就是单向数据流：

- 事件从 UI 发出。
- 状态层处理事件。
- 新状态返回给 UI。
- UI 依据状态重绘。

这个模式的价值在于让“谁触发了什么、为什么界面会变”变得可追踪。它会在 Compose、ViewModel 和 Flow 章节里不断被强化。

### 8. 最小实践任务

1. 做一个包含输入框和提交按钮的简单页面。
2. 把点击事件分成三步：收集输入、校验、更新状态，而不是全写在一个监听器里。
3. 为输入错误、加载中和提交成功分别设计不同页面反馈。
4. 记录页面里有哪些事件属于“离散动作”，哪些属于“连续输入变化”。

### 9. 常见误区

- 认为事件处理只是给控件挂回调。
- 把所有业务逻辑都塞进点击监听器。
- 输入事件和提交事件不区分，导致频繁无意义处理。
- 只处理动作，不处理用户反馈。

## 小结

事件处理的核心不是“监听器怎么写”，而是用户动作如何进入系统、状态如何更新以及反馈如何返回界面。只要这条链路组织清楚，页面复杂度上升时也不容易失控。
