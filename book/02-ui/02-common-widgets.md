# 常用控件

学习控件时，最容易走偏的方向是把它学成一份“控件大全”。这种方式短期看似覆盖很多，实际上帮助不大，因为真实页面并不是靠一堆孤立控件堆出来的，而是靠语义清晰的输入区、信息区和操作区组合出来的。本章不会把重点放在列举所有控件，而是聚焦那些最常见、最值得优先理解的基础控件，并说明它们在真实页面里如何分工。

## 学习目标

- 理解常见基础控件在页面中的典型职责。
- 知道文本展示、用户输入、开关选择和图像承载的常见做法。
- 理解控件选择和页面语义、可访问性之间的关系。
- 为事件处理、表单设计和列表项设计打基础。

## 前置知识

- 已理解布局基础和资源组织方式。
- 已能运行并修改一个简单页面。

## 正文

### 1. 为什么控件学习不能停留在“会拖控件”

控件的价值，从来不是“把一个东西摆到屏幕上”，而是让用户更自然地理解信息和完成任务。也就是说，控件选择本质上是交互设计问题，而不仅是 API 问题。

例如：

- 一段说明文字应该用文本展示，而不是按钮样式伪装。
- 一个二元状态更适合开关或勾选控件，而不是手写点击文本切换。
- 一个图片封面更适合图像容器，而不是背景图硬贴。

看起来只是控件不同，背后反映的其实是页面语义是否清晰。

### 2. 文本类控件：页面的信息骨架

`TextView` 是最基础也最重要的 UI 控件之一。很多页面的结构感，本质上是靠文本层级建立起来的，例如标题、副标题、说明、标签、错误提示、按钮文案。

学习文本控件时，重点应放在：

- 文本层级是否清楚。
- 字号和留白是否表达了主次。
- 文案是否来自资源而不是硬编码。

在正式项目里，一个页面好不好读，往往首先取决于文本结构是否清晰，而不是颜色有多花。

### 3. 输入类控件：用户任务的入口

输入控件通常包括 `EditText` 及其基于 Material 组件的封装。它们的关键不是“能不能输入文字”，而是：

- 是否提供明确标签。
- 是否有合适的输入类型。
- 是否处理校验与错误提示。
- 是否与页面状态保持一致。

一个好的输入区不仅能收集内容，还应降低用户犯错的概率。

### 4. 操作类控件：把用户动作表达清楚

按钮类控件的重点不是“点了能触发事件”，而是要表达动作的优先级和风险等级。常见问题包括：

- 所有按钮样式都一样，看不出主操作和次操作。
- 删除、提交、取消没有清晰区分。
- 按钮文案只写“确定”，却不说明具体动作。

这说明按钮控件的选择和样式，本质上服务于任务表达，而不是单纯点击能力。

### 5. 选择与状态类控件：适合表达有限分支

勾选框、单选项、Switch 等控件特别适合表达有限且明确的选择。关键不是“能切换状态”，而是：

- 这个状态是否足够简单，值得直接露出。
- 用户是否能在当前上下文里理解开和关的含义。
- 状态变更后页面是否立即反馈。

如果一个状态切换的后果非常复杂，却仍然用一个简单开关承载，就很容易让用户困惑。

### 6. 图像类控件：不仅是“显示图片”

`ImageView` 或其他图像承载控件的意义，也不只是显示一张图。它经常参与：

- 列表封面展示。
- 用户头像或商品图。
- 状态图标表达。
- 空状态和占位反馈。

因此使用图像类控件时，要同时考虑尺寸、裁切方式、占位图、加载失败和内容描述，而不仅是图片路径本身。

### 7. Material 组件为什么值得优先考虑

现代 Android 项目里，很多基础控件的现实用法其实已经不再是最原始的系统控件，而是 Material 组件体系。原因不只是“看起来更现代”，更重要的是它提供了更稳定的视觉规范、状态反馈和主题集成能力。

这也是为什么本书后续示例会尽量优先使用 Material 风格组件，而不是只展示最早期的原始控件。

### 8. 最小示例：一个简单表单由哪些控件组成

一个最小登录或资料表单，通常会包含：

- 标题文本。
- 输入框。
- 错误提示文本。
- 主操作按钮。
- 可选的开关或勾选项。

这个组合本身就说明，真实页面里的控件几乎总是成组出现，而不是孤立存在。因此学习控件时，应始终带着“它在页面任务里扮演什么角色”的问题。

### 9. 最小实践任务

1. 做一个简单资料填写页面，至少包含标题、两个输入框、一个开关和一个提交按钮。
2. 为每个控件写一句话，说明它在这个页面里承担什么任务。
3. 给输入控件补上标签和错误提示，而不是只留一个空白输入框。
4. 检查页面中的控件文案是否都来自资源文件。

### 10. 常见误区

- 把控件学习理解成控件名字背诵。
- 所有操作都用按钮，所有状态都用文本伪装。
- 输入框没有标签、没有校验、没有错误反馈。
- 只关心能显示，不关心可访问性和语义。

## 小结

控件不是页面的零件仓库，而是页面语义和任务流程的具体承载。学会根据任务选择控件，比记住更多控件名字更重要。下一章我们会继续讨论，当这些控件被用户真正操作时，事件应该如何组织。
