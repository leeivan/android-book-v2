# 常用 UI 组件

在布局解决“界面结构”之后，下一步就是理解构成界面的具体元素：文本如何展示、输入如何发生、用户如何选择、状态如何反馈。本章不追求把所有控件逐个罗列，而是用更贴近真实页面的方式来组织常用 UI 组件。

## 学习目标

- 认识常见 UI 组件的典型使用场景。
- 理解输入、展示与反馈之间的关系。
- 为事件处理、列表和 Material 设计章节提供控件基础。

## 前置知识

- 已理解布局与界面结构的基本关系。
- 已能修改布局文件并重新运行应用。

## 正文

### 1. 不要把控件学习理解成“记控件名字”

Android 控件很多，如果只是按名称逐个死记，学习成本会很高，而且很难真正形成设计能力。更高效的方式是按功能分组理解：

- 展示类控件：文本、图片、提示。
- 输入类控件：文本输入、开关、勾选、选择。
- 操作类控件：按钮、菜单入口、轻量操作触发器。
- 反馈类控件：进度、提示、错误、确认。

这样做的好处是，一旦进入真实页面设计，你想的会是“这里需要用户输入什么”“这里应该怎样反馈”，而不是“我会不会某个控件名字”。

### 2. 文本与输入类控件的核心问题

TextView 和 EditText 看似基础，但它们代表了两种完全不同的职责：

- TextView 负责展示。
- EditText 负责收集输入。

初学者最容易忽略的，是输入控件从来不是“摆上去就行”。它还涉及：

- 输入类型。
- 校验与错误提示。
- 焦点管理。
- 键盘弹出和收起时的体验。

也就是说，输入控件本身就已经把界面、交互和用户体验三件事绑在了一起。

### 3. 按钮与选择类控件该怎么理解

按钮类控件解决的是“触发动作”，选择类控件解决的是“表达状态或偏好”。两者经常出现在同一页面里，但职责不同。

例如：

- 按钮通常意味着用户希望立即做某个动作。
- Switch、CheckBox、RadioButton 更接近状态选择。

如果把这两类控件混用，页面就会变得含义模糊。好的界面往往不是控件多，而是控件角色清晰。

### 4. 反馈类控件为什么重要

很多初学者会花大量时间研究输入和点击，却忽略一个更常见的问题：用户做完操作后，系统如何回应？

最基本的反馈包括：

- 加载中。
- 成功。
- 失败。
- 需要确认。

如果没有反馈，即使功能逻辑没问题，用户也会觉得界面“没反应”。这也是为什么后续事件处理和 Material Design 章节，会反复强调反馈设计。

### 5. 新项目里应优先形成什么控件习惯

- 控件名称和用途要清晰。
- 文本、颜色和尺寸尽量资源化。
- 避免用一个控件承担多个模糊职责。
- 优先考虑可访问性和输入体验，而不是只关心视觉。

### 6. 最小实践任务

1. 设计一个包含标题、说明、输入框、选择项和提交按钮的简单表单。
2. 为输入框设置合适的输入类型和提示文本。
3. 模拟用户提交成功、失败和等待中三种状态，分别设计界面反馈。

### 7. 常见误区

- 学控件时只记属性，不看使用场景。
- 把“能点击”误当成“交互设计已经完成”。
- 只设计输入，不设计反馈。

## 小结

控件学习真正要建立的是“界面角色意识”。不同控件不是随便摆上去凑页面，而是分别承担展示、输入、动作和反馈的职责。把这层职责分清，后续的事件处理和界面设计才会稳定。
