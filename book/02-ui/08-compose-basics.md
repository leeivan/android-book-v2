# Jetpack Compose 基础

即使本书的基础 UI 主线仍以 View/XML 为主，现代 Android 开发已经无法绕开 Compose。本章的目标不是立刻把整本书都切到 Compose，而是帮助读者建立声明式 UI 的第一层认知，使其能够读懂、理解并逐步迁移到 Compose 项目。

## 学习目标

- 理解 Compose 与 View/XML 的核心差异。
- 建立声明式 UI 的基本认知。
- 为后续阅读现代 Android 项目和官方示例提供基础。

## 前置知识

- 已理解 View/XML 布局与控件的基本组织方式。
- 已具备事件处理和状态变化的基础概念。

## 正文

### 1. 为什么 Compose 值得单独介绍

Compose 的意义不在于“又多了一套 UI 框架”，而在于它改变了界面组织的思路。View/XML 更偏向先定义结构，再通过代码操作视图对象；Compose 更强调根据当前状态直接声明界面应该长成什么样。

这意味着，学习 Compose 时，重点不只是 API，而是思维转变：

- 界面更直接地由状态驱动。
- 组件组合更自然。
- 界面代码和状态组织的联系更紧。

### 2. View/XML 与 Compose 的核心差异

最重要的差异可以概括为一句话：

- View/XML 更强调“操作现有视图”。
- Compose 更强调“根据状态重新声明界面”。

这并不意味着旧方式无效，而是意味着新项目在组织复杂界面时，有了另一种更统一的表达路径。

### 3. 为什么本书不直接全量切到 Compose

因为现实世界里二者长期共存：

- 很多现有项目仍以 View/XML 为主。
- 很多企业代码库短期不会全部迁移。
- Android 工程师往往需要同时读懂两类项目。

因此，本书选择先用更适合建立平台认知的 View/XML 打基础，再在这里引入 Compose，帮助读者获得现代项目阅读能力和迁移能力。

### 4. 学 Compose 时最该带着什么问题

- 界面状态放在哪里。
- 组件如何组合，而不是如何嵌套 XML。
- 交互如何通过状态更新反映回界面。

这三个问题，决定你是在“背 Compose 组件名”，还是在真正学习声明式界面。

### 5. 最小实践任务

1. 阅读一个最简单的 Compose 页面示例，找出“状态”和“界面声明”的对应关系。
2. 用自己的话解释：为什么 Compose 更强调状态驱动。
3. 对比一个简单 View/XML 页面和一个简单 Compose 页面在结构组织上的差异。

### 6. 常见误区

- 把 Compose 当成一套“新的 XML 控件表”。
- 只看语法糖，不看状态驱动思想。
- 以为学了 Compose 就不需要理解 Android 平台基础。

## 小结

Compose 是现代 Android UI 的重要方向，但它不是对平台基础的替代，而是另一种更现代的界面组织方式。理解这一点，才能在后续阅读现代项目时不被表面语法差异带偏。
