# Jetpack Compose 基础

即使本书前面的 UI 主线仍以 View/XML 为主，现代 Android 开发已经无法绕开 Compose。本章的目标不是立刻把整本书都切到 Compose，而是帮助你建立声明式 UI 的第一层认知：界面不再主要通过“拿到视图对象然后修改它”来组织，而是直接根据当前状态声明“界面此刻应该长成什么样”。理解这一点，比记住任何单个 Compose 组件都更重要。

## 学习目标

- 理解 Compose 与 View/XML 在界面组织思路上的核心差异。
- 建立声明式 UI、状态驱动和重组的基本认知。
- 理解为什么 Compose 适合作为现代 Android UI 入口之一。
- 为后续阅读现代项目和理解 Material3 in Compose 打基础。

## 前置知识

- 已理解 View/XML 的布局、控件和事件处理方式。
- 已具备页面状态变化的基本概念。

## 正文

### 1. 为什么 Compose 值得单独成为一章

Compose 的意义不在于“Android 又多了一套 UI API”，而在于它改变了界面组织思路。View/XML 更强调：

- 先定义结构。
- 拿到视图对象。
- 在事件发生后不断修改对象状态。

Compose 更强调：

- 先定义状态。
- 根据状态直接声明界面。
- 状态变化后重新计算界面应该呈现的结果。

这不是语法差异，而是建模方式差异。

### 2. 什么叫声明式 UI

所谓声明式 UI，并不是“代码更短”这么简单。它的核心意思是：你描述的是结果，而不是一连串命令式操作步骤。

也就是说，你更关注：

- 当前状态是什么。
- 在这个状态下界面应该显示什么。

而不是先拿到一个视图，再分别调用很多方法把它改成想要的样子。

### 3. 状态为什么成为 Compose 的中心

一旦 UI 由状态驱动，状态就成了 Compose 世界里最重要的基础设施。你在 Compose 中最需要持续问自己的问题是：

- 这段界面依赖什么状态。
- 状态归谁管理。
- 状态变化后，哪些界面会重新计算。

这也是为什么 Compose 学习天然会和 ViewModel、单向数据流和状态提升这些概念绑在一起。

### 4. 重组不是“重画一切”，而是状态驱动下的重新计算

很多初学者一听到 Compose 会“重组”，就误以为每次状态变化都会把整个页面全盘重建。更合理的理解是：状态变化后，Compose 会重新执行相关的界面描述逻辑，并决定哪些部分需要更新。

你不需要在入门阶段就精通所有底层机制，但必须建立一个基本认识：Compose 不是靠你手工管理每个控件状态，而是靠状态和组合关系来决定更新。

### 5. 为什么本书没有一开始就全量切到 Compose

原因不是 Compose 不重要，而是现实世界里 View/XML 和 Compose 会长期共存：

- 大量现有项目仍以 View/XML 为主。
- 很多企业代码库短期不会整体迁移。
- Android 工程师往往需要同时读懂两类项目。

因此，本书选择先用 View/XML 建立平台和界面结构认知，再在这里单独引入 Compose，帮助读者获得现代项目阅读能力和迁移能力。

### 6. 最小示例：一个状态驱动的计数器

下面这个极简例子可以帮助你观察 Compose 的思路：

```kotlin
@Composable
fun CounterScreen() {
    var count by remember { mutableStateOf(0) }

    Column {
        Text(text = "当前计数：$count")
        Button(onClick = { count++ }) {
            Text("增加")
        }
    }
}
```

这个示例的重点不是 `Button` 或 `Text`，而是你能否看出：

- `count` 是状态。
- 界面显示取决于 `count`。
- 点击按钮后，状态变化，界面自然更新。

这正是 Compose 最核心的学习入口。

### 7. 状态提升为什么很关键

随着页面变复杂，你很快会发现：并不是所有状态都应该放在同一个 Composable 内部。Compose 的一个重要实践，是把状态放在更合适的位置，再把当前值和事件回调传给下层组件。

这就是状态提升。它的价值在于：

- 让组件更可复用。
- 让状态来源更清晰。
- 更容易与 ViewModel 结合。

### 8. Compose 与 Material3 为什么天然适合一起学习

在现代 Android 项目中，Compose 往往会和 Material3 一起出现。原因很简单：

- Compose 提供声明式 UI 组织方式。
- Material3 提供现代设计语言和组件体系。

两者结合后，界面结构、状态驱动和设计系统会更容易统一。

### 9. 最小实践任务

1. 先实现一个最小计数器页面，观察状态变化如何驱动界面更新。
2. 把页面拆成显示区和操作区两个 Composable，体验组合式结构。
3. 尝试把状态从子组件提升到父组件，观察数据流是否更清晰。
4. 用自己的话解释 Compose 与 View/XML 最大的思维差异是什么。

### 10. 常见误区

- 把 Compose 当成一套“新的控件表”。
- 只看语法糖，不理解状态驱动思想。
- 状态随意分散在多个组件里，导致数据流不清晰。
- 以为学了 Compose 就不需要理解 Android 平台基础。

## 小结

Compose 是现代 Android UI 的重要方向，但它不是平台基础的替代，而是一种更现代的界面组织方式。只要你先抓住状态驱动和组合式思维，后面阅读 Compose 项目就不会只看到一堆陌生 API，而能看清界面的组织逻辑。
