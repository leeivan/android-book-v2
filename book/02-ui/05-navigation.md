# Navigation

页面一旦变多，导航就不再只是“从 A 跳到 B”这么简单。返回栈如何组织、参数如何传递、页面层级如何表达、返回行为是否符合用户预期，这些问题都会集中暴露出来。Navigation 章节的重点，不是教你背若干跳转 API，而是帮助你理解 Android 页面流转的结构化组织方式。

## 学习目标

- 理解导航为什么是页面结构问题，而不仅是跳转问题。
- 理解返回栈和页面层级对体验的影响。
- 认识 Jetpack Navigation 作为现代页面流转主线的价值。
- 为结果回调、深链接和多页面项目结构打基础。

## 前置知识

- 已理解 Activity、Fragment 和生命周期基本概念。
- 已具备简单页面交互经验。

## 正文

### 1. 为什么导航不是“点了就跳”

如果应用只有两个页面，直接写一个跳转当然很简单。但只要页面开始变多，下面这些问题就会迅速出现：

- 返回键应该回到哪里。
- 同一个目标页是否要复用还是重复创建。
- 页面参数应该怎么组织。
- 页面层级如何映射到应用结构。

这说明导航不是单个动作，而是整个页面流转结构的表达方式。

### 2. 返回栈为什么是导航的核心概念

用户之所以会对某些应用产生“返回很奇怪”的感觉，常常不是因为某个页面写错了，而是因为返回栈设计混乱。返回栈本质上反映的是用户刚刚经历的页面路径，而不是开发者想当然的页面集合。

一个良好的导航设计，应该让用户在前进和返回时都能保持方向感。

### 3. 为什么现代项目更适合用结构化导航

过去很多示例会把跳转散落在各个 Activity 或 Fragment 中，这种方式在小项目里还能应付，但项目一大，很容易出现：

- 页面路径不清楚。
- 参数传递分散。
- 返回行为难以统一。

Jetpack Navigation 的价值就在于把页面目的地、参数和流转关系放到更统一的结构中理解，让导航不再只是零散调用。

### 4. Navigation 组件真正带来的是什么

它带来的不只是“少写一些跳转代码”，更重要的是：

- 页面目的地有了更清晰的组织方式。
- 导航宿主和内容目的地之间的职责更稳定。
- 返回行为和参数管理更容易统一。

这对于中大型项目尤为重要，因为页面越多，导航越需要结构而不是技巧。

### 5. 导航设计为什么要和页面职责一起考虑

很多导航问题，其实本质上是页面职责不清导致的。如果一个页面既像首页、又像弹窗、又像流程中间页，那么导航关系一定会混乱。

因此设计导航时，必须同时思考：

- 这个页面是独立目的地还是局部内容。
- 它在流程里是入口、分支还是终点。
- 用户从哪里来，完成后要回哪里去。

只要这三点明确，导航设计通常会更稳定。

### 6. 最小示例：列表页到详情页为什么不是简单跳转

列表页进入详情页看似只是一次点击跳转，但它背后至少涉及：

- 当前列表项标识如何传递。
- 返回时是否回到原列表状态。
- 详情页是单独目的地还是同一容器中的内容替换。

这说明哪怕是最常见的“列表到详情”，也不是单纯写一句跳转方法就结束了。

### 7. 最小实践任务

1. 画出一个三页面小应用的导航图，例如首页、详情页、编辑页。
2. 为每个页面标明它在流程中的角色和返回预期。
3. 思考哪些页面应该是独立目的地，哪些只适合做局部内容。
4. 检查你的设计是否能让系统返回键行为保持一致。

### 8. 常见误区

- 把导航理解成零散跳转代码。
- 不先画流程和层级，直接在页面里到处写跳转。
- 忽略返回栈，导致返回行为与用户预期不一致。
- 页面职责不清，强行套导航组件也救不了结构问题。

## 小结

导航的重点不是会不会跳转，而是页面流转是否有结构。只要你开始从“目的地、返回栈和流程角色”的角度理解导航，后面的复杂页面组织才不会混乱。
