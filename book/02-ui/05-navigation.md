# Navigation

页面一旦变多，导航就不再只是“从 A 跳到 B”这么简单。返回栈如何组织、参数如何传递、页面层级如何表达、返回行为是否符合用户预期，这些问题都会集中暴露出来。Navigation 章节的重点，不是教你背若干跳转 API，而是帮助你理解 Android 页面流转的结构化组织方式。

很多导航问题表面上看像“代码没跳对”，实际上是页面职责、回退语义和流程设计没有理清。如果不先建立导航结构，再多的跳转技巧都只能在小项目里勉强维持。本章会把导航从“写一行跳转代码”推进到“如何组织目的地、返回栈和参数”的层面来理解。

## 学习目标

- 理解导航为什么是页面结构问题，而不仅是跳转问题。
- 理解返回栈和页面层级对体验的影响。
- 认识 Jetpack Navigation 作为现代页面流转主线的价值。
- 为结果回调、深链接和多页面项目结构打基础。

## 前置知识

- 已理解 Activity、Fragment 和生命周期基本概念。
- 已具备简单页面交互经验。

## 正文

### 1. 为什么导航不是“点了就跳”

如果应用只有两个页面，直接写一个跳转当然很简单。但只要页面开始变多，下面这些问题就会迅速出现：返回键应该回到哪里，同一个目标页是否要复用还是重复创建，页面参数应该怎么组织，页面层级如何映射到应用结构。这说明导航不是单个动作，而是整个页面流转结构的表达方式。

用户之所以会觉得某些应用“返回很奇怪”，往往不是因为某个 API 用错了，而是因为整个流程没有被清楚设计。用户关注的不是你的代码路径，而是“我从哪里来、现在在流程中的什么位置、返回之后应该回到哪里”。这正是导航设计必须优先回答的问题。

### 2. 返回栈为什么是导航的核心概念

Android 官方关于 Navigation back stack 的文档明确指出，`NavController` 维护着一条 back stack，用户访问过的目的地都会被压入或弹出这条栈。这个定义看似简单，却决定了导航体验的核心：返回行为不应该围绕“开发者脑中有哪些页面”，而应围绕“用户刚刚走过怎样的路径”。

一个良好的导航设计，应该让用户在前进和返回时都能保持方向感。比如，列表进入详情，再进入编辑，返回时通常应回到详情，再回到列表；而登录流程完成后，Back 不应再把用户带回登录页。这些都是 back stack 设计问题，而不是单个按钮点击问题。

### 3. 为什么现代项目更适合用结构化导航

过去很多示例会把跳转散落在各个 Activity 或 Fragment 中，这种方式在小项目里还能应付，但项目一大，很容易出现页面路径不清楚、参数传递分散、返回行为难以统一的问题。Jetpack Navigation 的价值就在于把页面目的地、参数和流转关系放到更统一的结构中理解，让导航不再只是零散调用。

这也是为什么现代 Android 项目在基于 Fragment 的页面结构中，通常更推荐使用 Navigation 组件来承载主要流程。它不是“帮你少写几行代码”的小工具，而是把导航提升为一套结构化能力：目的地有图，起点明确，参数可声明，back stack 可推理，宿主和内容目的地的边界也更稳定。

### 4. Navigation 组件真正带来的是什么

Navigation 组件真正带来的，不只是一个 `navigate()` 方法，而是三个更有价值的东西。第一，页面目的地有了统一组织方式，流程关系不再散落在各处。第二，导航宿主和内容目的地之间的职责更稳定，常见的宿主是 `NavHostFragment`。第三，参数管理和返回行为更容易统一，这对中大型项目尤其重要。

对于基于 Fragment 的页面来说，Navigation 还承担了一层隐性价值：它能替你更规范地管理很多原本需要自己直接操作 `FragmentManager` 和 back stack 的细节。Android 官方在 FragmentManager 文档里也明确建议，应用导航应优先使用 Navigation library，因为它遵循了 working with fragments、back stack 和 fragment manager 的最佳实践。

### 5. 页面职责不清，导航一定会跟着混乱

很多导航问题，本质上不是导航组件本身的问题，而是页面职责不清。如果一个页面既像首页、又像弹窗、又像流程中间页，那么导航关系一定会混乱。设计导航时，必须同时思考：这个页面是独立目的地还是局部内容，它在流程里是入口、分支还是终点，用户从哪里来，完成后要回哪里去。

只要这三点明确，导航设计通常就会稳定得多。换句话说，导航不是后期补上的“页面之间连线”，而是页面结构本身的一部分。目的地没有定义清楚，任何导航库都救不了混乱的流程。

### 6. 参数传递为什么应该尽量轻

Android 官方关于 pass data between destinations 的文档明确建议，目的地之间应只传递最少必要的数据，通常优先传递 ID 或 key，而不是整个复杂对象。这条建议非常重要。因为导航参数属于页面流转边界的一部分，它不适合承担大对象同步、复杂业务状态搬运或跨页面长期共享。

更稳妥的思路通常是：导航只负责传“足够定位内容”的最小信息，例如任务 ID、用户 ID、筛选条件 key；真正的详细数据由目标页在自己的状态层中再去加载。这样做不仅更符合 Navigation 的设计，也更有利于配置变化和进程重建后的恢复。

### 7. 最小示例：列表页进入详情页，不只是一次点击跳转

下面这个例子演示最常见的“列表进入详情”导航。重点不是语法，而是目的地关系和最小参数传递。

先在导航图里声明从列表页到详情页的 action，并给详情页定义一个 `taskId` 参数。然后在列表页点击事件中只传当前项的 ID：

```kotlin
val action = TaskListFragmentDirections
    .actionTaskListFragmentToTaskDetailFragment(taskId = task.id)

findNavController().navigate(action)
```

这个例子之所以重要，是因为它体现了两个现代导航原则。第一，列表页并不需要把整个任务对象打包传过去，只传一个足够定位内容的 ID。第二，导航动作不再是随手拼一个 `Bundle`，而是作为页面结构的一部分被明确声明。如果你后来想调整返回行为、参数默认值或流程结构，这种组织方式会比零散跳转稳定得多。

### 8. 返回行为和导航行为，不要混成一回事

在 Android 里，Back 和 Up 看起来都像“返回”，但它们的语义并不完全相同。官方 Navigation back stack 文档明确指出，Back 更接近“回到用户刚刚走过的历史路径”，而 Up 更接近“回到应用层级中的上一层”。这两者在简单应用里可能看起来一致，但在复杂流程里会分化。

对当前阶段的读者来说，最重要的不是一次掌握所有差异，而是先建立一种警觉：导航设计不能只考虑“怎么前进”，还必须考虑“用户怎么退回来”。只要返回语义没有设计，流程越多，体验越容易崩。

### 9. 实践任务

起点条件：

- 已有一个基于 Fragment 的简单页面结构。
- 工程中可以继续增加一个或两个 Fragment 作为目的地。

步骤：

1. 设计一个三页面小流程，例如列表页、详情页、编辑页。
2. 先画出导航图，明确每个页面是入口、分支还是终点。
3. 为详情页只设计最小必要参数，例如 item ID，而不是整个对象。
4. 思考从编辑完成返回时，应该回到详情页还是直接回列表页，并写下理由。
5. 检查你的设计是否能让系统返回键行为保持一致。

预期结果：

- 你对“页面目的地”和“页面内容区域”有更清晰区分。
- 你能明确说出一个简单流程的 back stack 预期。
- 你开始把导航理解成结构，而不是散落在页面里的跳转代码。

自检方式：

- 你能解释：为什么导航不是“点了就跳”。
- 你能判断：某个页面到底应该是独立目的地，还是更适合做局部内容。
- 你能说出：为什么参数传递应尽量轻，只传最小必要数据。

调试提示：

- 如果返回行为总让你意外，先回头画流程和 back stack，而不是先改 API。
- 如果你想传越来越大的对象，通常说明目标页的数据加载边界没有理清。
- 如果页面之间跳转越来越散，通常说明导航结构没有被集中组织。

### 10. 常见误区

- 把导航理解成零散跳转代码。
- 不先画流程和层级，直接在页面里到处写跳转。
- 忽略返回栈，导致返回行为与用户预期不一致。
- 页面职责不清，强行套导航组件也救不了结构问题。

## 小结

导航的重点不是会不会跳转，而是页面流转是否有结构。只要你开始从“目的地、back stack、参数边界和流程角色”的角度理解导航，页面之间的关系就会比单纯调用跳转 API 稳定得多。

下一章及后续相关章节里，我们会继续在这个导航基础上讨论列表、主从结构和更复杂的页面组织。导航一旦建立起结构感，后面的多页面工程才不会越来越乱。
