# 常见错误排查

几乎每个 Android 学习者都会在相似的地方反复踩坑：构建失败、资源找不到、权限弹窗不出现、页面状态重复、导航参数错误、通知不生效、WorkManager 不执行、混淆后崩溃。真正重要的不是把所有错误背下来，而是建立一套足够稳定的排查顺序，让你在面对陌生问题时也能快速缩小范围。

这份附录的目标，就是帮你把零散报错经验收束成一条更系统的排查思路：先看哪一层，再看哪一类边界，再决定用什么工具和日志去验证。

## 学习目标

- 建立一套常见 Android 问题的排查顺序。
- 理解构建、资源、权限、状态、后台任务等错误的高频根因。
- 理解为什么很多“奇怪错误”其实来自边界设计不清。
- 提高面对陌生问题时的定位速度。

## 前置知识

- 已读完本书前面主要章节，至少接触过构建、UI、数据、网络和系统组件。

## 正文

### 1. 先区分错误属于哪一层

很多问题之所以越查越乱，是因为一开始就没有先分类。更有效的顺序通常是先问：

- 这是构建期错误还是运行期错误。
- 是 UI 表现异常，还是数据链路异常。
- 是系统权限和组件边界问题，还是业务逻辑问题。

只要先分层，排查范围通常会立刻缩小一大半。

### 2. 构建错误优先看“配置、依赖、变体”

常见构建问题往往来自：

- 依赖版本冲突。
- 插件版本不匹配。
- 变体配置错误。
- 资源合并或 Manifest 合并异常。

如果一开始就盯着具体报错行而不回到构建边界，往往会陷入反复试错。构建错误通常先看最近改了什么构建配置，再看依赖和版本链路。

### 3. 运行期 UI 问题优先看“状态来源”

很多看似是“界面没显示对”的问题，真实原因并不在布局，而在状态来源混乱。例如：

- 页面重复收到旧状态。
- Loading、Empty、Error 混在一起。
- 导航返回后状态没有恢复。

这类问题最值得先查的是：状态到底从哪来，谁在更新它，旧任务是否还在覆盖新结果。

### 4. 系统能力问题优先看“权限与入口”

像通知不显示、相册选图失败、定位结果为空、文件访问失败，这类问题首先应检查：

- 是否真的有权限。
- 是否选择了正确系统入口。
- 当前 Android 版本有没有新的平台限制。

很多错误并不是代码逻辑错，而是系统边界根本没满足。

### 5. 后台任务问题优先看“任务分类是否正确”

WorkManager 不执行、前台服务被限制、广播收不到，很多时候不是 API 不稳定，而是任务本身分类就错了：

- 本该是页面内异步工作，却被硬做成后台任务。
- 本该是前台服务，却没有用户可见通知。
- 本该是 WorkManager，却要求立刻执行。

因此后台问题排查时，第一步通常不是改代码，而是重新审视任务性质。

### 6. release 才出现的问题优先看“debug 与 release 差异”

有些问题只在正式构建出现，常见原因包括：

- 混淆规则遗漏。
- 环境配置切换错误。
- 日志或调试开关遮蔽了真实行为。
- 签名、资源或版本信息异常。

这类问题最忌讳的，就是继续只在 debug 环境里猜。它要求你把 release 视为独立验证对象。

### 7. 一个实用的排查顺序

当你遇到陌生问题时，可以先按这条顺序来：

1. 先确认问题属于哪一层。
2. 再确认最近改动了什么。
3. 查看是否有明确日志和堆栈。
4. 检查边界条件：权限、生命周期、变体、系统版本、状态来源。
5. 用最小复现路径重建问题。

这个顺序不会保证你立刻解决所有错误，但能让你避免在多个方向同时乱试。

### 8. 实践任务

起点条件：

- 已遇到过或正在面对某个 Android 开发问题。

步骤：

1. 把问题先分类到构建、运行期 UI、数据链路、系统能力、后台任务或发布差异之一。
2. 写下最近一次与该问题最相关的改动。
3. 列出该问题最可能关联的边界条件。
4. 用最小复现路径再现问题，而不是继续在完整复杂页面中猜。
5. 记录最终根因，整理成自己的错误库。

预期结果：

- 你会从“按层分类”而不是“凭直觉乱试”开始排查。
- 你能更快定位高频错误的真正边界。

自检方式：

- 你能解释：为什么很多 Android 问题首先要先分类而不是先修。
- 你能判断：某个问题更像状态来源问题、系统边界问题还是构建问题。

调试提示：

- 如果报错看起来很怪，先回到最近改动和层级分类，而不是马上怀疑系统。
- 如果问题只能在复杂路径出现，优先尝试做最小复现。


## 小结

常见错误排查真正要建立的，不是报错记忆库，而是一套面对陌生问题时也能保持稳定的定位方法。只要“先分类、再缩范围、再验证”的习惯建立起来，你的排查速度会明显提升。
