# 常见错误排查

很多学习者在 Android 入门阶段最痛苦的体验，不是知识点太多，而是错误一出现就完全不知道该从哪里下手。构建失败、依赖冲突、页面白屏、权限被拒、崩溃堆栈看不懂、列表不更新、通知不弹、WorkManager 不执行，这些问题单独看都能解决，真正让人崩溃的是它们总像随机事件一样出现。这个附录的价值，不是给你列一张“常见报错大全”，而是帮你建立排查顺序。

参考资料里讲调试和故障处理时，都在强调“先缩小范围，再验证假设”。这个顺序比死记报错文本重要得多，因为框架版本会变，报错文案会变，但排查逻辑不会变。本章就围绕这条逻辑来组织。

## 学习目标

- 理解 Android 常见错误更适合按层定位，而不是按文案死记。
- 学会区分构建错误、运行时错误、状态错误和系统能力错误。
- 学会建立“先缩小范围，再验证假设”的排查顺序。
- 把常见错误处理转化为可重复的方法，而不是经验运气。

## 前置知识

- 已接触构建、页面、数据层、权限、通知和后台任务。
- 已会使用基础日志、断点和 Android Studio 调试工具。

## 正文

### 1. 排查错误时，最先做的不是改代码，而是先分层

很多人一遇到错误，就马上开始改配置、删缓存、重启 IDE。更成熟的第一步通常是先判断问题属于哪一层：

- 构建层：项目都编不过。
- 运行层：能编过，但一运行就崩。
- 状态层：不崩，但表现不对。
- 系统交互层：权限、通知、后台、组件入口没有按预期工作。

只要层级判断对了，后面的排查路径通常会快很多。

### 2. 构建错误通常先查依赖、版本和环境

当问题发生在构建期，最常见的原因往往集中在：

- 依赖版本不兼容。
- Gradle / AGP / Kotlin 版本组合不合适。
- 本地环境或缓存状态异常。
- 模块引用、变体或签名配置有误。

这类问题最怕“看到报错就随便搜一条配置改上去”。更稳妥的做法是先确认：这是版本组合问题，还是环境状态问题，还是脚本本身写错了。只要这三类没分清，后面越修越乱是常态。

### 3. 运行时崩溃先看“在哪一层崩了”

崩溃堆栈最容易让初学者害怕，但你不需要一开始就看懂每一行。更有效的做法通常是先问：

- 崩溃发生在页面层、数据层还是系统组件层？
- 是空指针、状态不合法、权限问题，还是第三方库错误？
- 是第一次进入就崩，还是某个特定操作后才崩？

只要先从“上下文 + 层级”切入，堆栈就会更容易读。你不是在理解整个系统，而是在理解某条路径为什么在这个点失败了。

### 4. 不崩但不对，通常是状态和数据流问题

很多 Android bug 其实不崩，例如：

- 列表刷新了但 UI 没更新。
- 搜索切词后旧结果把新结果覆盖了。
- 页面旋转后 Snackbar 重复弹。
- 明明点了完成，列表却没有变化。

这类问题往往不是语法错误，而是状态设计错误。排查时更该先问：

- 这份状态到底由谁持有？
- 这次更新是从哪条链路来的？
- 有没有多个地方同时在改同一份状态？

只要你开始从“状态归谁管”这个角度去看，不崩类 bug 会比想象中更容易定位。

### 5. 系统能力问题更要先看前提条件

权限不生效、通知不弹、后台任务不执行、外部 Intent 匹配不到，这类问题表面上像“系统 API 有坑”，实际上很多都来自前提条件没有成立。例如：

- 权限请求时机不对。
- 通知渠道没有建立。
- 后台任务本来就不该用当前机制。
- Intent 的 action / data / category 不匹配。

系统能力类问题，先检查“前提条件是否成立”，通常比先改实现细节更有效。

### 6. 一套更稳妥的通用排查顺序

面对大多数问题时，你都可以先按这个顺序走：

1. 用一句话描述现象。
2. 判断它发生在哪一层。
3. 提出两个最可能的原因。
4. 为每个原因选择一条最省力的验证方式。
5. 只改最小一处验证，不要同时乱动很多地方。

这套顺序最大的价值，是防止你在没缩小范围之前就不断制造新变量。很多错误不是因为本身难，而是因为排查过程把现场破坏掉了。

### 7. “最近改了什么”为什么总是值得先回看

很多 bug 并不是“突然出现”，而是最近某次改动带来的副作用。尤其在这些时刻更值得先回看提交历史：

- 升级依赖以后。
- 调整构建配置以后。
- 重构页面状态以后。

先回看最近改动，常常能比盲目全局搜更快缩小范围。排查不是侦探小说，不需要假装一切信息都未知。

### 8. 什么时候该怀疑工具和环境，而不是业务代码

并不是所有问题都来自业务逻辑。有些问题更可能是：

- Gradle 缓存异常。
- 模拟器或真机环境状态异常。
- IDE 索引与构建状态不同步。

成熟的排查并不是默认“代码一定错了”，而是能区分“环境异常”和“代码异常”。只要这一点不分清，很多时间都会浪费在错误方向上。

### 9. 实践任务

起点条件：

- 已有一个真实 bug，或你最近遇到过的常见错误。

步骤：

1. 用一句话准确描述问题现象。
2. 把它归类到构建、运行、状态或系统交互层。
3. 写下两个最可能原因。
4. 为每个原因选一条最省力的验证方式，例如日志、断点、Inspector 或配置核对。
5. 记录最后真正原因属于哪一层，以及下次更快定位时该先看哪里。

预期结果：

- 你会更少陷入“哪里都改一点”的混乱排查。
- 你能逐渐把错误处理变成可重复方法。
- 面对新问题时也更容易知道从哪下手。

自检方式：

- 你能解释为什么常见错误应先分层再排查。
- 你能判断某个问题更像状态错误还是系统边界错误。
- 你能说明为什么一次只验证一个假设更有效。

调试提示：

- 一次改很多地方，最后很难知道到底哪一步真的解决了问题。
- 报错文案看不懂时，先别慌，先从层级和上下文缩小范围。
- 系统能力问题先查前提条件，往往比先改实现更有效。

### 10. 常见误区

- 一遇到错误就立刻清缓存、重装、乱改配置。
- 不分层，所有问题都按同一种方式排查。
- 一次改很多地方，导致结果不可解释。
- 只记错误文案，不建立排查逻辑。

## 小结

常见错误排查真正有价值的，不是记住多少报错文本，而是建立一套稳定的缩小范围方法。只要你先分层、再提假设、再验证最可能原因，很多原本看起来随机的错误，都会变成可解释、可处理的工程问题。

## 练习题

1. 概念理解题：为什么 Android 常见错误更适合按构建、运行、状态和系统交互四层来定位，而不是只按报错文案去搜索？
2. 编码实现题：回顾你最近遇到的一个真实问题，按“现象 -> 分层 -> 两个假设 -> 验证方式”写出一份最小排查记录。
3. 扩展思考题：如果要为团队整理一份常见错误排查清单，你会优先沉淀哪些高频问题和验证顺序，才能真正提高协作效率？
