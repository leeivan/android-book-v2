# 常见错误排查

Android 学习过程中最容易打断节奏的，往往不是概念难，而是环境、构建、权限、网络和生命周期问题层层叠加，导致你一时分不清问题到底出在哪。这个附录的目标，不是给出零散口诀，而是帮你建立一套常见故障的初步排查框架，让你在遇到问题时先有方向，再谈具体解法。

## 学习目标

- 对常见 Android 开发错误形成初步分类。
- 学会先判断问题层级，再决定排查方向。
- 建立最小排查闭环，而不是盲目重试。
- 避免把所有问题都归因于“IDE 抽风”。

## 前置知识

- 已阅读前文主要章节。
- 已接触过基本的构建、运行和调试流程。

## 正文

### 1. 先问：这是哪一层的问题

排查时，第一步永远是分类。常见层级包括：

- 环境与依赖问题。
- 构建配置问题。
- 运行时崩溃问题。
- UI 与生命周期问题。
- 网络与权限问题。
- 发布与设备差异问题。

只要先把层级分出来，后面的检索和验证都会轻很多。

### 2. 常见环境与构建问题有哪些特征

这类问题通常表现为：

- Gradle 同步失败。
- 找不到 SDK 或依赖。
- 构建脚本报错。
- 模拟器或真机安装失败。

这时应优先检查：

- 工具版本是否匹配。
- 依赖源和网络是否可用。
- 本地 SDK 组件是否完整。
- 当前构建变体和签名是否正确。

而不是先去改业务代码。

### 3. 常见运行时问题如何判断

运行时常见问题包括：

- 页面启动崩溃。
- 参数为空。
- 主线程做耗时任务。
- 数据库或网络调用时序错误。
- 页面重建后状态异常。

这类问题更适合结合堆栈、日志和断点一起处理，而不是凭感觉猜原因。

### 4. 常见系统交互问题为什么特别容易误判

权限、通知、文件访问、后台任务这类问题常常不是“代码没写”，而是平台约束没满足。例如：

- 没有在正确时机申请权限。
- 忽略通知渠道或用户授权状态。
- 用了不合适的后台执行模型。
- 文件访问方式不符合当前平台边界。

排查这些问题时，要优先回到系统规则，而不是只盯着函数返回值。

### 5. 设备差异和系统版本差异应如何看

很多问题并不是“偶发”，而是：

- 只在某些 API 级别出现。
- 只在某些厂商设备上暴露。
- 只在特定权限状态或后台状态下出现。

因此排查时应同步记录：

- 系统版本。
- 设备型号。
- 当前权限状态。
- 应用前后台状态。

没有上下文的问题描述，通常都不够可复现。

### 6. 建立自己的错误笔记库为什么有效

真正高效的开发者，不是从不出错，而是能快速沉淀经验。你可以把常见错误按类别记录下来，包括：

- 现象。
- 根因。
- 解决方式。
- 如何避免再次发生。

长期来看，这会显著提升你的问题定位速度，也能减少同类错误反复出现。

### 7. 一个最小排查顺序

当你遇到问题时，至少可以先按下面顺序检查：

1. 这属于哪一层。
2. 是否能稳定复现。
3. 日志和堆栈提供了什么关键线索。
4. 最近改动最可能影响了哪条链路。
5. 是代码问题、配置问题还是平台约束问题。

只要这条顺序建立起来，排查就不会一开始就陷入盲改。

### 8. 最小实践任务

1. 回忆最近三个 Android 开发错误。
2. 把它们分别归类到环境、构建、运行时、系统交互或设备差异问题。
3. 为每个问题写下“下次先检查哪三件事”。
4. 开始建立一个自己的错误记录模板。

### 9. 常见误区

- 一遇到问题就认为是 IDE 故障。
- 不看堆栈和日志，直接盲改代码。
- 不记录复现条件，导致问题难以稳定追踪。
- 没有沉淀自己的排查经验。

## 小结

常见错误排查的核心，不是记住所有问题答案，而是先建立分类和验证方法。只要方法对了，大多数错误都能比想象中更快定位。
