# 参考资料与后续学习路径

学完一本书，最容易出现两种情况。第一种是“知道很多名词，但不知道下一步怎么继续深入”；第二种是“到处找资料，结果资料太多反而不知道先学什么”。这个附录的价值，不是再给你堆一长串链接，而是帮你建立一种更有顺序的学习方式: 不同阶段该优先看什么类型的资料，不同问题该去什么地方找一手信息，不同能力该如何从教材过渡到项目实践。

换句话说，这一章不是资源仓库，而是学习路线图。真正有效的后续学习，不在“收集更多”，而在“知道下一阶段最值得看什么”。

## 学习目标

- 理解不同类型资料分别适合解决什么学习问题。
- 理解从教材到官方文档、样例项目、库文档和真实项目的过渡顺序。
- 学会围绕当前问题选择资料，而不是盲目堆链接。
- 为后续长期自学建立更稳定的资料使用方法。

## 前置知识

- 已完成本书主线内容学习，或至少完成其中大部分。
- 愿意继续通过官方资料和真实项目深化理解。

## 正文

### 1. 资料太多时，真正缺的往往不是链接，而是顺序

很多学习者在一个主题卡住时，会同时打开:

- 官方文档
- Codelab
- Blog
- YouTube 视频
- GitHub 示例

结果信息并没有变少，反而变得更乱。真正更有效的方法通常是先问: 我现在缺的是哪一种帮助？

- 缺概念框架？
- 缺可跟练步骤？
- 缺真实项目结构参考？
- 缺权威 API 细节？

只要问题类型确定，资料选择就会容易很多。

### 2. 教材、官方文档、样例项目分别适合什么

这三类资料最容易被混在一起，但它们承担的角色并不相同。

教材适合帮你建立连续主线，解决“为什么学这个、它和其他知识怎么连起来”。

官方文档适合解决准确性和最新边界问题，尤其是在 API、平台行为和限制细节上。

样例项目适合帮助你看“真实结构到底长什么样”，尤其适合把抽象架构落地成目录、模块和代码组织。

理解这三类资料的角色，能避免你把文档当教材、把样例项目当教程。

### 3. Codelab 和课程资料最适合做什么

如果你已经知道一个主题的重要性，但还不会亲手落地，那么 Codelab 和课程型资料通常最有帮助。它们的价值在于:

- 给你一条较短的实践路径。
- 帮你快速建立“能跑起来”的第一版经验。

但它们也有边界: 它们通常更像练习场，而不是完整工程范式。所以做完以后，仍然需要再回头看官方文档和真实项目。

### 4. 真实项目和官方样例为什么特别值得反复读

很多知识点在单章教程里都能看懂，一进真实项目就又迷路了。原因不是你没学会，而是你还没见过这些知识如何被组织进长期维护的工程里。官方样例和高质量真实项目最大的价值，就在于展示:

- 模块怎么分。
- 状态怎么流。
- 依赖怎么注入。
- 数据层怎么连接 UI 层。

这类资料非常适合在你完成教材主线后作为“结构参照物”。

### 5. 库文档什么时候该看

当你已经知道某个工具为什么存在，并准备在项目里真正使用它时，库文档就变得非常重要。例如:

- OkHttp / Retrofit 文档
- Kotlin Flow / coroutines 文档
- Hilt 官方文档

库文档最擅长回答的是“这个库现在真实支持什么、约束是什么、推荐写法是什么”，而不是帮你从零建立完整概念主线。所以阅读顺序上，通常更适合在你已经有主线概念之后使用。

### 6. 一个更成熟的后续学习顺序

如果你已经学完本书，下一步可以优先按这个顺序继续:

1. 用 Codelab 跟练一个你还不够熟的核心主题。
2. 用官方文档把细节边界补准。
3. 用官方样例或高质量项目看真实结构。
4. 再用自己的小项目把它重新写一遍。

这比“随机多看十篇文章”更容易形成真正可迁移的能力。

### 7. 不同主题优先看什么资料

如果你要补的是:

- 页面与状态管理: 优先官方架构文档和官方样例。
- Compose: 优先 Compose Codelab + 官方文档。
- 网络和数据层: 优先库文档 + 官方架构文档。
- 工程实践: 优先官方构建、测试、发布文档和真实项目结构。

资料选择应该围绕主题性质，而不是围绕“哪里看起来更热闹”。

### 8. 一个最小资源路线图

下面这组资源非常适合作为本书之后的继续学习主线:

- Android Developers: <https://developer.android.com/>
- Android Codelabs: <https://developer.android.com/get-started/codelabs>
- Android Basics with Compose: <https://developer.android.com/courses/android-basics-compose/course>
- Kotlin Docs: <https://kotlinlang.org/docs/home.html>
- Now in Android: <https://github.com/android/nowinandroid>
- Architecture Samples: <https://github.com/android/architecture-samples>
- OkHttp Docs: <https://square.github.io/okhttp/>
- Retrofit Docs: <https://square.github.io/retrofit/>
- Material 3: <https://m3.material.io/>

真正要记住的不是这些链接本身，而是它们分别解决什么问题。

### 9. 实践任务

起点条件:

- 已完成本书主体内容，准备继续深入一个或两个方向。

步骤:

1. 先明确自己下一阶段最想补的主题。
2. 判断你当前缺的是概念、跟练、真实结构，还是 API 细节。
3. 从教材、文档、Codelab、样例项目中选最匹配的一类资料。
4. 给自己定一个最小学习闭环: 看资料 -> 跟练 -> 回到自己的项目。
5. 删掉那些暂时看起来有趣、但和当前主题无关的资料。

预期结果:

- 你会从“收藏资料”转向“使用资料”。
- 你会更清楚不同资料的角色和优先级。
- 后续学习会更像有路线的深化，而不是漫无目的扩散。

自检方式:

- 你能解释教材、官方文档和样例项目分别适合什么。
- 你能判断当前自己最该补的是哪种资料。
- 你能为下一阶段学习制定一个最小闭环。

调试提示:

- 如果总在收集链接却没有实际跟练，说明路线还没收紧。
- 如果一上来就只看 API 文档，容易缺少概念主线。
- 如果只看教程不看真实项目，结构理解会长期停在浅层。

### 10. 常见误区

- 以为资源越多越好。
- 不区分教材、文档、样例和库文档的角色。
- 只看资料，不回到自己的项目练。
- 每个方向都想同时补，结果哪个都没走深。

## 小结

后续学习最重要的，不是再拥有多少链接，而是知道在当前阶段最该看哪种资料、为什么看、看完以后去哪里实践。只要把教材、官方文档、Codelab、样例项目和库文档放回各自角色里，学习路径就会明显清晰很多。
