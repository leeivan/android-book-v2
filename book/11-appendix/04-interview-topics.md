# Android 面试主题

很多读者学到后期会开始关心面试，但一旦把“面试准备”理解成背题，就很容易走偏。因为 Android 面试真正考的，通常不是你能不能背出某个 API 名字，而是你能不能把概念、场景、权衡和工程判断串起来讲清楚。也就是说，面试题表面上是问答，实质上在看你是否真正理解这门技术在真实项目里的结构。

参考目录里的书虽然不是面试书，但它们提供了一个很重要的提醒：真正成熟的技术表达，几乎总是围绕“问题是什么、为什么重要、现代做法是什么、代价和边界在哪里”来展开。本附录的目标也不是给你一份死记答案，而是帮你把本书主线重新整理成一套可输出的面试表达框架。

## 学习目标

- 理解 Android 面试更像是结构化表达，而不是题库背诵。
- 理解高频主题为什么总围绕生命周期、状态、并发、数据层、系统组件和工程实践。
- 学会把“概念 + 场景 + 取舍”组合成更成熟的回答方式。
- 把本书内容转化成可输出的知识框架。

## 前置知识

- 已完成本书大部分主线内容学习。
- 已理解页面、架构、并发、系统组件和工程实践。

## 正文

### 1. 面试官通常在问三种东西

很多题目表面不同，背后其实都在考三类能力：

- 你是否理解概念。
- 你是否知道它在真实场景里怎么用。
- 你是否能解释为什么这样设计，而不是那样设计。

只要抓住这三类能力，准备方式就会从“背答案”变成“搭框架”。这也是为什么纯粹背 API 列表，往往在追问阶段很快失效。

### 2. 生命周期与状态管理为什么永远是高频主题

Android 最鲜明的特点之一就是组件生命周期复杂，而页面状态又持续变化。因此面试中高频出现这些问题并不奇怪：

- Activity / Fragment 生命周期
- 状态保存
- 配置变化
- ViewModel 和页面状态

真正有说服力的回答，不是把回调顺序背出来，而是能解释“这些变化为什么会导致状态管理问题，以及现代推荐做法是什么”。只要你能把问题落到真实页面，比如搜索页、列表页、编辑页，答案就会明显更稳。

### 3. 架构题真正考的是边界感

无论是问 MVVM、Repository、UseCase、Clean Architecture 还是模块化，核心都不是“会不会背层名”，而是看你是否知道：

- 哪些逻辑应该留在页面层。
- 哪些逻辑该进数据层。
- 哪些动作值得抽成 UseCase。
- 为什么某些边界能降低复杂度。

回答这类题时，最有力的方式通常不是抽象定义，而是拿一个真实页面或真实功能流来解释。

### 4. 并发与协程题为什么会追问取消和作用域

现在问协程，几乎不会只停在 `launch` 和 `async` 区别。更常见的追问是：

- 任务应该归谁管。
- 页面销毁后怎么办。
- 为什么要用 `viewModelScope`。
- Flow 为什么和状态管理绑在一起。

这说明并发题真正考的，已经不是语法，而是“异步任务如何进入架构”。如果你只能说出协程启动方式，却说不清作用域和取消语义，回答通常会显得很浅。

### 5. 数据层问题通常在考“谁是可信来源”

Room、DataStore、Repository、网络缓存、offline-first 这些题背后，常常都在追问同一件事：页面到底应该从哪一层拿数据，多个来源冲突时谁说了算。如果你能把“单一可信来源”和“数据策略”讲清楚，很多数据层题都会更容易展开。

这也是为什么面试里很喜欢追问“为什么不能直接把 Retrofit 结果给页面”“为什么要有 Repository”。这些问题在本质上都和数据边界相关。

### 6. 系统组件题最常见的坑是只背 API，不讲边界

例如：

- Intent
- Service
- BroadcastReceiver
- Notification
- 权限

这类题如果只答“怎么用”，很容易显得浅。更成熟的回答通常会补上：

- 它解决什么边界问题。
- 为什么不能滥用。
- 现代 Android 下有哪些限制或替代方案。

系统组件题很少只是在考你记不记得方法名，它更像在看你有没有真正踩过平台边界。

### 7. 工程实践题最能拉开层次

中高级面试中，真正拉开差距的往往不是基础 API，而是：

- 测试怎么做。
- CI/CD 怎么接。
- 性能问题如何定位。
- 安全边界怎么管。
- 发布和维护怎么组织。

这类题目特别能体现你是否真正做过完整项目，而不只是写过页面和接口。因此如果你只复习 UI 和架构，而忽略工程化主题，准备会明显偏科。

### 8. 一个更成熟的回答结构

面对大多数 Android 面试题，一个稳定的回答顺序通常是：

1. 先给出这个主题要解决的核心问题。
2. 再说明它在 Android 场景里为什么重要。
3. 然后说现代主流做法。
4. 最后补一两个常见误区或 trade-off。

例如被问“为什么要用 ViewModel”时，比起直接说“它可以在配置变化后保留数据”，更成熟的表达通常是先说“页面实例不稳定，但页面状态需要稳定承载”，然后再引出 ViewModel 作为状态持有者的价值。

### 9. 一个更实际的准备方法

如果你准备面试，不要只刷题。更有效的方式通常是：

- 从生命周期、架构、并发、数据层、系统组件、工程实践中各选一个主题。
- 各自写一段 2 到 3 分钟的口头回答。
- 每段都带一个真实项目场景。
- 再补一个常见误区或权衡点。

这样做会比单纯收集题目更接近真实面试，因为面试官常常会顺着你的回答继续追问场景和取舍。

### 10. 实践任务

起点条件：

- 已准备开始系统复盘 Android 主线知识。

步骤：

1. 从生命周期、架构、并发、数据层、系统组件、工程实践中各选一个主题。
2. 按“问题 -> 场景 -> 推荐做法 -> 权衡”的结构各写一段回答。
3. 检查自己是否只会背定义，还是能落到真实页面和项目。
4. 选一个最不熟的主题，补上一个真实例子。
5. 口头讲 3 分钟，看自己在哪一段最容易卡住。

预期结果：

- 你会从背题转向搭建知识表达框架。
- 你会更容易把本书内容串成可输出的回答。
- 你也更容易发现自己真正薄弱的主题。

自检方式：

- 你能解释为什么面试题真正考的是结构化理解。
- 你能用真实场景回答，而不只是背定义。
- 你能说明某项现代推荐做法背后的 trade-off。

调试提示：

- 如果回答一开口就是 API 名字，先退回去说“它解决什么问题”。
- 如果只会给标准答案，不会给真实场景，说明理解还没落地。
- 如果每道题都答得像百科词条，面试表现通常会比较弱。

### 11. 常见误区

- 把面试准备理解成背题库。
- 只记定义，不讲场景和权衡。
- 忽视工程实践和发布维护类主题。
- 不会把本书主线知识串起来输出。

## 小结

Android 面试准备真正要做的，不是收集更多题，而是把核心主题整理成结构化理解。只要你能围绕“它解决什么问题、为什么重要、现代推荐做法是什么、常见误区和 trade-off 在哪里”去表达，本书前面学到的内容就会逐渐从阅读知识变成可输出的能力。

## 练习题

1. 概念理解题：为什么 Android 面试高频主题看似分散，背后却总是在考概念理解、真实场景和设计取舍三种能力？
2. 表达练习题：从生命周期、架构、并发、数据层、系统组件或工程实践中任选一个主题，按“问题 -> 场景 -> 推荐做法 -> trade-off”组织一段 2 到 3 分钟回答。
3. 扩展思考题：如果让你把自己做过的一个项目经验转化成面试回答，你会怎样把业务背景、技术选择和踩坑复盘串成一段更有说服力的叙述？
