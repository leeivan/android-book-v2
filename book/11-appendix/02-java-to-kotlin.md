# Java 到 Kotlin 迁移

很多 Android 开发者并不是从零开始学 Kotlin，而是在已经写过 Java Android 项目的前提下迁移。真正困难的地方往往也不在语法，而在思维方式：如果你只是把 Kotlin 当成“语法更短的 Java”，那么代码虽然能写出来，但很难真正发挥现代 Android 技术栈的优势。

这份附录想解决的，就是迁移阶段最常见的思维断层：哪些 Java 写法在 Kotlin 中不再自然，哪些 Kotlin 特性值得优先接受，怎样让迁移尽量稳而不是一下子重写整个项目。

## 学习目标

- 理解 Java 到 Kotlin 迁移最常见的思维变化。
- 理解哪些 Kotlin 特性最值得优先接受。
- 理解如何渐进迁移，而不是一次性推倒重写。
- 减少“语法迁过去了，风格还停留在 Java”的问题。

## 前置知识

- 已有 Java 或 Java-based Android 项目经验。
- 已愿意接受 Kotlin 作为现代 Android 主线语言。

## 正文

### 1. 迁移最大的问题不是语法，而是默认值

Java 的很多写法建立在这些默认值之上：

- 可空性主要靠运行时约定。
- 大量样板代码默认存在。
- 可变对象与 setter 使用很常见。

Kotlin 则把很多默认值反过来了：

- 默认不可空。
- 数据模型鼓励更轻量。
- 更鼓励不可变和表达式式写法。

这意味着迁移最重要的，不是“找到语法对应项”，而是先接受这些默认值变化。

### 2. 不要一开始就追求“全量重写”

对已有项目来说，更稳妥的迁移方式通常是渐进式：

- 新代码优先使用 Kotlin。
- 边改边迁移离当前需求最近的类。
- 保持 Java 与 Kotlin 互操作可读可维护。

一次性大规模翻译不仅风险高，也很容易把 Java 风格原封不动搬进 Kotlin，最后得到的是“不太像 Kotlin 的 Kotlin 项目”。

### 3. 最先值得接受的 Kotlin 能力

对 Java 开发者来说，最值得优先建立的习惯通常包括：

- 空安全。
- `data class`。
- `when` 和表达式风格。
- 扩展函数。
- 更明确的不可变状态表达。

这些能力会直接影响你写 ViewModel、页面状态对象、DTO 和 Flow 代码的方式，也是现代 Android 项目里最常见的 Kotlin 风格。

### 4. 互操作能力让渐进迁移成为可能

Kotlin 和 Java 的互操作能力非常强，这也是 Android 团队能渐进迁移的重要原因。真正重要的是利用这点降低迁移风险，而不是借口无限拖延。换句话说：

- 互操作能力让你不必一次全改。
- 但它不意味着永远停留在双风格混杂状态。

一个更成熟的目标，是让项目逐步把“新的主线”收束到 Kotlin，而不是长期维持两套思维并行。

### 5. 迁移时最常见的风格陷阱

常见问题包括：

- 把所有可空性都粗暴改成可空类型。
- 继续大量写工具类和静态风格代码。
- 把数据类仍按可变 bean 模式使用。
- 只做语法翻译，不重构状态和边界设计。

这些问题说明，迁移不是格式转换，而是一次逐步调整代码表达方式的机会。

### 6. 实践任务

起点条件：

- 已有一段 Java Android 代码或旧项目模块。

步骤：

1. 选一个相对独立的类，先迁移而不是大面积改动。
2. 迁移时重点观察可空性、数据类和不可变字段的机会。
3. 检查迁移后是否只是“Java 写法翻译成 Kotlin”。
4. 再选一个状态类或 ViewModel 相关类，体验 Kotlin 对状态建模的帮助。

预期结果：

- 你会用更低风险方式推进迁移。
- 你能逐渐建立 Kotlin 风格，而不是只完成语法替换。

自检方式：

- 你能解释：为什么 Kotlin 迁移最难的是默认值和风格变化。
- 你能判断：某段代码迁过去后是否仍然是 Java 思维。

调试提示：

- 如果迁移后可空类型泛滥，说明空安全边界还没真正建立。
- 如果 Kotlin 文件里依然大量是 Java Bean 风格，说明风格转换还没开始。

## 小结

Java 到 Kotlin 迁移的核心，不是一次性完成翻译，而是逐步接受 Kotlin 作为现代 Android 主线语言的表达方式。只要迁移节奏和风格判断都稳住，项目会比“一把重写”更安全，也比“永远不动”更有未来。
