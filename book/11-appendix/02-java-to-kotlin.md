# Java 到 Kotlin

很多 Android 开发者并不是从零开始接触 Kotlin，而是从 Java 项目迁移过来。这个迁移过程最大的挑战，不是语法替换，而是思维方式变化。你不只是把 `public class` 改成 `class`，而是在重新适应一套更强调空安全、表达式风格和简洁建模的语言。本附录的重点，是帮助 Java 背景读者更快跨过“能看懂 Kotlin”这道坎。

## 学习目标

- 帮助有 Java 基础的读者快速建立 Kotlin 对照理解。
- 知道迁移时哪些习惯需要主动调整。
- 理解 Kotlin 带来的不只是简写，更是更明确的类型表达。
- 建立更稳妥的增量迁移顺序。

## 前置知识

- 具备 Java 基础。
- 已开始阅读本书中的 Kotlin 示例。

## 正文

### 1. 迁移时不要只盯语法对照

很多人在迁移时喜欢做一件事：逐句找 Java 对应写法。这在最开始有帮助，但如果长期停留在这一步，你会一直把 Kotlin 当成“写法不一样的 Java”，很难真正用好它。

更好的做法是优先理解 Kotlin 想强化什么：

- 可空性明确。
- 数据模型更轻量。
- 函数表达更自然。
- 不可变性更受鼓励。

### 2. 最常见的思维差异

从 Java 迁移到 Kotlin，最先需要调整的通常是：

- 从默认可变转向优先不可变。
- 从“所有对象都可能为 null”转向显式可空声明。
- 从样板化 getter/setter 转向更直接的数据建模。
- 从层层回调转向协程与状态流。

这些变化会显著影响你写 ViewModel、数据模型和 UI 状态的方式。

### 3. 为什么数据类和扩展函数特别常见

Kotlin 让很多过去需要大量样板代码的模式变得更轻。数据类适合表达纯数据模型，扩展函数适合给现有类型补充实用能力。

它们的价值不是“语法优雅”，而是让代码更贴近真实意图，尤其适合 Android 中大量数据与状态对象的场景。

### 4. 协程为什么是迁移中的关键断层

过去很多 Java 风格 Android 项目会依赖回调、线程池和 Handler 组织异步逻辑。迁移到 Kotlin 后，协程往往会成为最明显的思维升级点。

这不仅影响代码风格，也会影响你如何设计：

- Repository。
- UseCase。
- ViewModel。
- 页面状态流。

所以 Java 到 Kotlin 的迁移，本质上往往也是异步模型的迁移。

### 5. 迁移顺序为什么要渐进

一个更稳妥的迁移顺序通常是：

1. 先能读懂 Kotlin。
2. 再把新增代码写成 Kotlin。
3. 然后从边界清晰的小模块开始替换旧 Java。
4. 最后再处理深层公共模块和更复杂逻辑。

如果一开始就要求全量重写，风险和成本通常都过高。

### 6. 哪些 Java 习惯最值得主动戒掉

迁移过程中，最常见的问题不是不会写 Kotlin，而是继续沿用 Java 旧习惯，例如：

- 一切都用可变变量。
- 可空性不明确。
- 到处写工具类静态方法。
- 过度依赖继承而不是组合。

只要这些习惯不调整，Kotlin 最终只会退化成“语法更短的 Java”。

### 7. 最小实践任务

1. 把一个简单 Java 数据模型改写成 Kotlin 数据类。
2. 把一个带空判断的 Java 方法改写成 Kotlin 安全调用风格。
3. 找出一个回调式异步场景，思考它迁移到协程后会如何简化。
4. 检查你当前 Kotlin 代码里是否仍大量保留 Java 风格写法。

### 8. 常见误区

- 只做语法层翻译，不理解 Kotlin 的设计意图。
- 继续大量沿用 Java 的可变和空值习惯。
- 一开始就想全量迁移整个项目。
- 迁移了语言，却不迁移异步和状态设计方式。

## 小结

Java 到 Kotlin 的迁移，本质上是开发习惯和建模方式的迁移。只要你先抓住空安全、不可变性、更清晰的函数表达和现代异步主线，后续过渡会顺畅很多。
