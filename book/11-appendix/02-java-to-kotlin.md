# Java 到 Kotlin

很多 Android 开发者并不是“从零开始学 Kotlin”，而是带着多年 Java 习惯转过来的。真正困难的地方，通常也不是语法本身，而是思维方式的迁移：过去习惯可变对象、setter、空值默认存在、匿名内部类和样板代码撑住结构；到了 Kotlin 以后，代码突然更强调空安全、不可变值、表达式风格和更轻量的状态对象。如果仍然用完全相同的 Java 思维去写 Kotlin，代码表面换了语言，结构和可读性却不会真正变好。

参考目录里的 Kotlin 书籍都反复强调一个事实：Java 到 Kotlin 的迁移，真正困难的不是“新关键字太多”，而是默认写法和默认判断都需要调整。本章不做逐行语法对照，而是帮助你完成“写法迁移”到“思维迁移”。只有这一步完成，Kotlin 在 Android 中的真正优势才会显现出来。

## 学习目标

- 理解 Java 开发者转向 Kotlin 时最常见的思维阻力。
- 理解哪些 Java 习惯在 Kotlin 中应保留，哪些应主动放下。
- 理解 Kotlin 的空安全、不可变风格和状态表达为什么更适合现代 Android。
- 学会以“渐进迁移”而不是“一次重写”的方式完成语言转型。

## 前置知识

- 已有 Java 或类 Java 语言基础。
- 已接触 Kotlin 的基本语法。

## 正文

### 1. 迁移真正困难的不是语法，而是默认思维

Java 开发者在 Kotlin 中最容易出现的问题，不是不会写，而是会下意识写成：

- 到处 `var`
- 到处可空
- 大量 setter
- 工具类和静态风格堆叠

这些写法在 Java 时代可能很自然，但在 Kotlin 里往往会削弱语言本来能带来的结构收益。迁移的真正关键，是意识到默认习惯本身需要被调整。

### 2. Kotlin 并不是要求你否定 Java

迁移语言最常见的误区之一，就是把 Kotlin 理解成“Java 的否定版”。其实 Java 训练出来的很多习惯依然很有价值，例如：

- 清晰的类职责
- 稳定的接口边界
- 对命名和可读性的重视

Kotlin 并不是让你把这些都丢掉，而是要求你把一些过去模糊的边界表达得更明确，例如空值边界、可变与不可变边界、状态对象和行为对象边界。真正成熟的迁移，不是推翻过去，而是保留结构经验、更新表达方式。

### 3. 空安全是 Java 到 Kotlin 最大的思维变化

Java 里很多对象默认都可能是 `null`，开发者更多靠经验和判空来兜底。Kotlin 要求你正面回答这个问题：这个值到底允不允许为空？

这会迫使你在设计阶段就明确：

- 哪些字段必须存在。
- 哪些字段是可选值。
- 某个 API 的调用方该承担什么空值责任。

对 Android 来说，这种约束尤其有价值，因为页面状态和网络结果里空值非常常见。很多以前运行时才爆炸的空指针，在 Kotlin 里会提前变成编译期提醒。

### 4. `data class` 和不可变状态会改变你组织代码的方式

Java 开发者很容易习惯写很多 Bean、getter、setter。Kotlin 的 `data class` 和 `copy()` 会鼓励另一种状态组织方式：

- 把状态整体表达出来。
- 通过复制产生新状态，而不是四处局部修改。

这对现代 Android 的 ViewModel、Flow、Compose 都特别重要，因为它更符合状态驱动 UI 的方式。也就是说，Kotlin 在 Android 中的优势，并不只是少写样板代码，而是更适合状态建模。

### 5. 不要把 Kotlin 写成“没有分号的 Java”

这是迁移阶段最值得警惕的现象。典型表现包括：

- 明明可以用 `val` 却习惯性写成 `var`
- 明明可以用表达式，却仍然保持过度命令式结构
- 一看到扩展函数、高阶函数就下意识绕开
- 继续依赖大量可变字段和 setter 组织页面状态

这样做不是“更稳”，而是让你失去了 Kotlin 在状态表达、异步代码和模型建模上的优势。

### 6. 渐进迁移比一次性重写更现实

真实项目里，最稳妥的做法通常不是一口气把整个 Java 项目重写成 Kotlin，而是：

- 新代码优先使用 Kotlin。
- 边界清晰的新模块优先迁移。
- 与状态建模强相关的层优先迁移，例如 ViewModel、UI state、UseCase。

这种渐进式路径比一次性重构更安全，也更容易让团队逐步建立 Kotlin 习惯。参考资料里的项目实践大多也沿着这个顺序，而不是鼓励“一次性翻盘”。

### 7. Java 背景反而会成为迁移优势

如果你已经有多年 Java 经验，其实并不吃亏。因为你通常已经具备：

- 对长期维护成本的敏感度
- 对职责边界的判断能力
- 对命名和结构清晰度的要求

真正要更新的，不是这些工程感，而是具体表达方式。成熟的迁移姿态，往往不是“完全忘掉 Java”，而是把 Java 的结构经验和 Kotlin 的表达优势结合起来。

### 8. 一个更现实的迁移练习

如果你手头有一段明显“写得像 Java”的 Kotlin 代码，不必一上来全面重写。更有效的改法通常是：

1. 先把能改成 `val` 的变量改掉。
2. 重新明确哪些字段真的需要可空。
3. 把只承载数据的类改成 `data class`。
4. 把一串 setter 风格状态修改收束为一次状态替换。

这类小步重构非常适合帮助团队建立 Kotlin 写法直觉，也更容易在真实项目里推进。

### 9. 实践任务

起点条件：

- 已有一段 Java 风格较重的 Kotlin 代码，或正在从 Java 向 Kotlin 迁移。

步骤：

1. 找一段明显“写得像 Java”的 Kotlin 代码。
2. 检查其中哪些变量其实可以改为 `val`。
3. 检查哪些对象更适合改成 `data class`。
4. 找出一段空值边界模糊的代码，重新明确可空 / 不可空设计。
5. 只重写一小段，观察可读性和状态表达是否改善。

预期结果：

- 你会更清楚自己是在迁移语言，还是在迁移思维。
- 你能逐步把 Kotlin 的优势真正带进 Android 项目。
- 你会更克制地做渐进迁移，而不是一次性重写。

自检方式：

- 你能解释 Java 到 Kotlin 最大的变化为什么是空安全和不可变思维。
- 你能判断一段 Kotlin 为什么“虽然能跑，但还是像 Java”。
- 你能说明为什么渐进迁移通常更现实。

调试提示：

- Kotlin 里如果仍处处是可变对象和 setter，优先回看状态建模方式。
- 一看到高阶函数就拒绝使用，通常说明思维迁移还没真正开始。
- 如果迁移计划一开始就是“全部重写”，风险往往会被低估。

### 10. 常见误区

- 以为 Java 到 Kotlin 只是语法映射。
- 把 Kotlin 写成“没有分号的 Java”。
- 一次性重写全部模块。
- 只学语言表面，不调整状态和边界设计方式。

## 小结

Java 到 Kotlin 的真正迁移，核心不在语法，而在默认思维。只要空安全、不可变状态和更轻量的模型表达真正进入你的 Android 代码，Kotlin 才会发挥出现代主线语言的优势。渐进迁移、边改边学，通常比一次性重写更现实，也更容易成功。

## 练习题

1. 概念理解题：为什么说 Java 到 Kotlin 的最大变化不是关键字，而是空安全、不可变状态和表达式风格带来的思维迁移？
2. 编码实现题：找一段明显“写得像 Java”的 Kotlin 代码，至少完成一次 `var` 到 `val`、普通类到 `data class` 或多次 setter 到一次状态替换的重构。
3. 扩展思考题：如果你的团队准备从 Java 逐步迁移到 Kotlin，你会如何安排模块优先级和迁移节奏，才能降低一次性重写的风险？
