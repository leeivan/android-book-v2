# Flow

Flow 是现代 Kotlin 和 Android 中非常重要的异步数据流工具。很多学习者第一次接触它时，会把它理解成“比 LiveData 更现代的可观察对象”。这种说法不算完全错，但远远不够。Flow 的真正价值，在于它提供了一种表达连续异步数据、转换数据流、组合多个来源、并与协程体系自然协同的方式。本章的任务，就是把 Flow 从“API 列表”还原成“数据流建模工具”。

在 Android 官方架构建议里，Flow 被放在层与层之间通信的重要位置。原因很直接：现代 Android 项目大量状态都不是“一次结果然后结束”，而是会随着时间持续变化。只要你开始从“状态如何持续流动”来理解 Flow，它就会和 ViewModel、Room、Repository、Compose 自然接起来。

## 学习目标

- 理解 Flow 适合解决什么类型的问题。
- 理解冷流、热流、StateFlow 和 SharedFlow 的基本意义。
- 理解 Flow 为什么特别适合与 ViewModel、Room 和 Repository 协同。
- 理解状态流与事件流为什么必须区分。

## 前置知识

- 已理解协程和异步任务的基本组织方式。
- 已理解 ViewModel 是页面状态容器。

## 正文

### 1. Flow 更适合表达“持续变化的结果”

如果一个能力只会返回一次结果，那么普通 `suspend` 函数往往就足够了。Flow 更擅长的场景是：结果并不是一次性的，而是一串随着时间变化的值。例如：

- 数据库内容变化。
- 搜索关键字驱动的实时查询。
- 页面状态随着请求过程持续更新。
- 多个数据源合并后不断产生新结果。

换句话说，Flow 不是“另一个拿结果的办法”，而是一种把结果变化过程正式建模出来的方式。

### 2. 为什么 Flow 比回调链更适合复杂异步场景

旧式回调代码的问题不在于“拿不到数据”，而在于：

- 多个回调很难组合。
- 错误处理分散在不同层级。
- 取消和生命周期边界不清楚。
- 一条数据变化后怎样影响另一条数据难以表达。

Flow 的优势在于，它把这些变化统一放进一个可以组合、变换和取消的流模型里。也正因此，它特别适合复杂的 Android 页面和数据层场景。

### 3. 冷流和热流为什么要分开理解

这是 Flow 学习里最容易被忽略、却又极其关键的一层。

冷流更强调“被收集时才开始工作”。典型例子是某些按需执行的计算或请求过程。

热流更强调“数据源本身就持续存在并不断更新”。例如页面状态、共享事件流、某些全局状态。

如果这一点没弄清，后面就很容易把状态、事件和一次性操作混在一起，最终出现重复消费、状态不一致和难以调试的问题。

### 4. StateFlow 为什么特别适合作为页面状态主线

Android 官方的 StateFlow 指南明确指出，StateFlow 是一个 state-holder observable flow。它非常适合承接“总有当前值”的场景，而页面状态恰好就是这类场景：

- 当前是否加载中。
- 当前内容列表是什么。
- 当前是空态还是错误态。

这些都是“当前值明确”的状态，因此 StateFlow 和 ViewModel 的组合特别自然。它让 UI 随时都能获取当前屏幕应该呈现的样子，而不是只知道某个历史事件曾发生过。

### 5. SharedFlow 更适合什么

并不是所有页面变化都属于“当前状态”。例如：

- 导航到详情页。
- 弹出 Snackbar。
- 触发文件选择器。
- 通知页面“某个一次性动作已经发生”。

这些更像一次性事件，而不是一个应长期持有的当前值。SharedFlow 常被用来承载这类事件流，但前提是你先分清“稳定状态”和“一次性事件”不是同一类东西。

### 6. Flow 为什么特别适合数据层

Flow 不只是 UI 工具，它在数据层的价值同样很大。Android 官方架构建议推荐 Repository 暴露数据流，一个重要原因就是很多数据来源本身就是可持续变化的：

- Room 数据变化会自动触发新结果。
- 搜索关键词变化会影响查询结果。
- 远程刷新写回本地后，会引发新的本地流结果。

这意味着 Flow 不只是“页面层观察工具”，它也是数据层、状态层和 UI 层之间的一种统一语言。

### 7. 组合与转换是 Flow 的真正强项

Flow 真正强大的地方，不只是能发值，而是能自然地做：

- `map` 这类数据转换。
- `filter`、`distinctUntilChanged` 这类过滤。
- `combine` 这类多来源合并。
- `debounce` 这类节流与去抖。
- `flatMapLatest` 这类随输入变化切换最新结果。

这些操作看似进阶，实际上在搜索、筛选、动态列表、分页和多条件页面里都非常常见。只要页面开始有持续输入和多数据来源，Flow 的组合能力就会迅速体现价值。

### 8. 搜索页为什么天然适合 Flow

搜索页通常具备这些特征：

- 输入会持续变化。
- 每次变化都可能触发新查询。
- 旧查询结果可能应被取消或忽略。
- 结果状态需要不断更新到 UI。

这种场景如果用裸回调，很容易写乱；而用 Flow，可以把“输入变化”“取消旧任务”“输出新结果”建模成清晰的数据流。这也是理解 Flow 最好的练习之一。

### 9. 实践任务

起点条件：

- 已有一个存在连续变化结果的页面，例如搜索页、筛选页或列表页。

步骤：

1. 选一个页面，列出其中哪些数据属于持续变化的结果。
2. 判断哪些适合建模为 StateFlow，哪些更像一次性事件。
3. 检查项目里是否存在大量回调链，其实更适合用数据流表达。
4. 尝试用一个简单的 Flow 操作链把输入变化和结果变化连接起来。
5. 检查当前页面是否把状态和事件错误地混在同一个模型里。

预期结果：

- 你会开始从“数据随时间如何变化”而不是“怎么收一个值”来理解 Flow。
- 你能更清楚地分辨状态流和事件流。
- 你会为后续 WorkManager、Repository 和系统组件章节建立更自然的状态组织方式。

自检方式：

- 你能解释：为什么 Flow 不是“另一个 LiveData”这么简单。
- 你能说出：冷流和热流的关键差异。
- 你能判断：某个页面需求到底更适合 StateFlow 还是 SharedFlow。

调试提示：

- 如果状态和事件混用，旋转或重建后通常会出现重复消费问题。
- 如果某个异步结果明明会持续变化，却还用单次回调拼接，后续复杂度会迅速上升。
- 如果你说不清一个 Flow 是谁拥有、谁收集、为什么会变化，说明数据流建模还不够清晰。

### 10. 常见误区

- 把 Flow 仅仅理解成更新版可观察对象。
- 不区分冷流、热流、状态流和事件流。
- 只会收集 Flow，不理解为什么要用它来建模数据变化。
- 状态和事件混用，导致页面语义混乱。

## 小结

Flow 的价值，不只是让异步代码看起来更现代，而是提供了一种更适合表达连续变化数据的建模方式。只要你开始从“数据如何流动”和“状态如何持续演化”的角度理解它，它就会成为现代 Android 架构中非常稳定的一条主线。

### 教材化延伸：为什么 Flow 不能只看操作符

很多 Flow 教程会迅速进入 `map`、`filter`、`combine`、`collectLatest` 等操作符，但教材更需要先让读者知道“为什么页面状态适合流式描述”“为什么冷流和热流会影响订阅者行为”“为什么取消和背压会影响 UI 更新体验”。没有这层直觉，操作符越多越容易混乱。
