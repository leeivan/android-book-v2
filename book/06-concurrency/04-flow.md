# Flow

Flow 是现代 Kotlin 和 Android 中非常重要的异步数据流工具。很多学习者第一次接触它时，会把它理解成“比 LiveData 更现代的可观察对象”。这种说法不算完全错，但远远不够。Flow 的真正价值，在于它提供了一种表达连续异步数据、转换数据流、组合多个来源、并与协程体系自然协同的方式。本章的任务，就是把 Flow 从“API 列表”还原成“数据流建模工具”。

## 学习目标

- 理解 Flow 用来解决什么问题。
- 理解冷流、热流、StateFlow 和 SharedFlow 的基本意义。
- 理解 Flow 为什么适合与 ViewModel、Room 和 Repository 协同。
- 为页面状态流和事件流建模打基础。

## 前置知识

- 已理解协程和异步任务的基本组织方式。
- 已理解 ViewModel 是页面状态容器。

## 正文

### 1. 什么问题更适合用 Flow 来表达

Flow 最适合表达那些“不是一次结果，而是一串随时间变化的结果”的问题。例如：

- 数据库内容变化。
- 搜索关键字驱动的实时查询。
- 页面状态随着网络请求而连续更新。
- 多个数据来源的组合结果。

如果一个能力只会返回一次结果，它当然也可以在协程里处理；但一旦结果具有连续变化特征，Flow 的价值就会迅速凸显。

### 2. 为什么 Flow 比“回调列表”更适合复杂异步场景

很多旧式异步代码的问题，不在于拿不到数据，而在于：

- 多个回调链很难组合。
- 错误处理分散。
- 取消和生命周期边界不清楚。
- 一个数据来源变化后，其他部分如何联动很难表达。

Flow 提供的不是单一 API，而是一种把这些变化组织成流的方式，因此它特别适合复杂异步场景。

### 3. 冷流和热流为什么要区分

这是 Flow 学习里最容易被忽略、却又非常关键的一层。

- 冷流更强调“被收集时才开始产生数据”。
- 热流更强调“数据源本身持续存在和更新”。

为什么要区分？因为页面状态、共享事件和数据库监听，并不是同一种流。只要这一点没分清，后面很容易把状态、事件和一次性操作混在一起。

### 4. StateFlow 为什么特别适合作为页面状态主线

页面通常总会有一个“当前状态”：加载中、显示内容、错误、空状态等。这种“始终存在一个当前值”的特点，非常适合用 StateFlow 来表达。

它的现实价值在于：

- 页面随时都能拿到当前状态。
- 状态更新更符合单向数据流。
- 与 ViewModel 和 Compose 都很容易对齐。

因此，在现代 Android 项目中，StateFlow 经常成为页面状态暴露的主线工具。

### 5. SharedFlow 和一次性事件如何理解

并不是所有页面变化都属于“当前状态”。例如：

- 导航到详情页。
- 弹出 Snackbar。
- 触发文件选择器。

这类更像“一次性事件”，而不是一个持续存在的稳定状态。SharedFlow 常常更适合承载这类需求，但前提是你先清楚地区分状态和事件。

### 6. Flow 为什么非常适合与数据层结合

Flow 在数据层的价值尤其明显，因为很多数据来源本身就是变化流：

- Room 数据随数据库更新而变化。
- 搜索关键词变化会影响查询结果。
- 本地与远程协同可能会产生多次状态更新。

这意味着 Flow 不是仅服务于 UI，它也非常适合成为 Repository 和数据层之间的连接语言。

### 7. 组合与转换是 Flow 的真正强项

Flow 的重要价值之一，是它允许你更自然地做：

- 数据转换。
- 条件过滤。
- 多流组合。
- 节流与去抖。

这类能力看似“高级”，其实在真实项目里非常常见。只要页面一复杂、搜索一实时、状态一增多，Flow 的组合能力就会变得非常有价值。

### 8. 最小示例：搜索页为什么天然适合 Flow

搜索页通常具备这样的特点：

- 输入会持续变化。
- 请求结果需要随着关键字不断更新。
- 旧结果可能应被取消或忽略。

这类场景用单次回调很容易写乱，而用 Flow 来表达输入变化和结果流动，会更自然。这个例子非常适合帮助你理解 Flow 的真实价值。

### 9. 最小实践任务

1. 选一个存在连续变化结果的页面，例如搜索页或列表页。
2. 判断哪些数据适合建模为 StateFlow，哪些更像一次性事件。
3. 检查你的项目里是否存在大量回调链，其实更适合用数据流表达。
4. 用自己的话解释：为什么 Flow 不是“另一个 LiveData”这么简单。

### 10. 常见误区

- 把 Flow 仅仅理解成更新版可观察对象。
- 不区分冷流、热流、状态流和事件流。
- 只会收集 Flow，不理解为什么要用它来建模数据变化。
- 状态和事件混用，导致页面语义混乱。

## 小结

Flow 的价值，不只是让异步代码看起来更现代，而是提供了一种更适合表达连续变化数据的建模方式。只要你开始从“数据流动”和“状态组织”的角度理解它，它就会成为现代 Android 架构中非常稳定的一条主线。
