# Kotlin Coroutines

协程已经成为现代 Kotlin Android 开发中最核心的异步工具之一。它的价值不在于“把线程换个写法”，而在于让异步代码更接近顺序思考方式，同时又能与生命周期、取消、状态流和数据层更自然地协同。本章的重点，是把协程放回它真正的定位：结构化并发工具，而不是“更时髦的后台线程”。

## 学习目标

- 理解协程解决的核心问题。
- 理解 `launch`、`async`、`withContext` 和取消机制的基本意义。
- 理解结构化并发为什么特别适合 Android。
- 为 Flow、ViewModel 和后台任务章节提供主线工具。

## 前置知识

- 已理解主线程约束和异步任务边界。
- 已知道现代 Android 更偏向 Kotlin-first。

## 正文

### 1. 协程解决的不是“少写几行代码”

协程最重要的价值，是让异步任务的组织方式更清晰。过去开发者经常要围绕：

- 回调嵌套。
- 手工切线程。
- 临时状态同步。
- 任务取消难题。

来写大量胶水代码。协程让这些问题至少可以在更统一的模型下表达。

### 2. 协程为什么特别适合 Android

Android 中大量核心问题都具有异步特征：

- 网络请求。
- 数据库读写。
- 文件访问。
- 页面初始化后的数据加载。
- 多个来源结果合并。

而这些问题又几乎都与生命周期和页面状态有关。协程正适合在这种“需要异步，又需要结构清晰”的环境中发挥作用。

### 3. 协程和线程的关系不应被混淆

一个非常重要的认识是：协程不是线程，也不是“开了协程就一定在后台线程”。协程更像是一种组织异步任务和挂起恢复的编程模型，它可以在合适的调度环境下运行，但不等于自动拥有正确线程边界。

这就是为什么理解协程时，必须把“线程在哪”“生命周期归谁”“如何取消”一起看。

### 4. 结构化并发为什么是协程的核心价值

结构化并发的价值，在于让异步任务不再像到处乱飞的临时工作，而是有明确父子关系和生命周期归属。对 Android 而言，这一点尤为关键，因为：

- 页面会重建和销毁。
- 页面对应的异步任务不应无限继续。
- 用户动作触发的多个任务需要有边界。

这让“协程属于谁、何时取消”成为设计的一部分，而不是事后补救。

### 5. `launch`、`async` 和 `withContext` 应如何理解

入门阶段不必沉迷 API 细节，但要建立最基本的理解：

- `launch` 更适合发起不直接返回值的协程任务。
- `async` 更适合表达需要结果的并发计算。
- `withContext` 更适合在协程中切换执行上下文并返回结果。

真正重要的不是背名字，而是知道你现在需要的是“启动一个任务”“并发等待结果”还是“在现有协程里切换工作环境”。

### 6. 取消为什么在 Android 中尤其重要

用户离开页面、旋转屏幕、切换搜索关键词、重复点击刷新，这些都是 Android 中非常常见的取消场景。如果异步任务没有明确取消边界，后果通常包括：

- 旧请求覆盖新结果。
- 页面销毁后仍尝试更新状态。
- 无意义任务继续消耗资源。

协程之所以比裸线程更适合 Android，很大一部分原因就在于它对取消的支持更自然。

### 7. 协程最适合放在哪一层启动

这个问题没有唯一答案，但有一个稳定判断标准：谁拥有任务的生命周期，谁就更适合启动和管理它。

常见实践通常是：

- 页面层不直接承担大部分异步任务。
- 页面级异步流程多由 ViewModel 启动和管理。
- 数据层内部的具体 I/O 操作由 Repository 或更下层负责。

只要把这条线守住，协程就更容易服务架构，而不是制造新的混乱。

### 8. 最小示例：页面加载文章列表的协程思路

一个典型列表页的协程链路通常会是：

- 页面发出加载动作。
- ViewModel 启动协程。
- Repository 在合适上下文中执行网络或数据库读取。
- 结果返回后更新页面状态流。

这个例子的重点不是协程语法，而是“异步任务如何进入状态链路”。

### 9. 最小实践任务

1. 找一个原本需要异步执行的操作，说明为什么协程比手工线程更适合表达它。
2. 判断这个协程应放在页面层、ViewModel 还是数据层启动。
3. 写出这个任务在页面关闭、重复触发和失败时分别应如何处理。
4. 检查你的项目里是否存在“到处直接启动协程”的情况。

### 10. 常见误区

- 把协程当成后台线程的别名。
- 到处直接启动协程，不考虑生命周期和归属。
- 只关注语法，不关注取消和结构化并发。
- 以为用了协程就自动拥有合理架构。

## 小结

协程是现代 Android 异步主线之一。理解它的关键，不是会不会写 `launch`，而是能否让异步任务真正融入生命周期、页面状态和数据层边界之中。
