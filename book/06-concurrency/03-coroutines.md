# Kotlin Coroutines

协程已经成为现代 Kotlin Android 开发中最核心的异步工具之一。它的价值不在于“把线程换个写法”，而在于让异步代码更接近顺序思考方式，同时又能与生命周期、取消、状态流和数据层更自然地协同。本章的重点，是把协程放回它真正的定位：结构化并发工具，而不是“更时髦的后台线程”。

Android 官方的 Kotlin 协程文档反复强调两件事：一是协程非常适合主线程安全地组织异步工作，二是它们必须和生命周期、取消、调度一起看。只要把这两点抓住，协程就不会再只是语法技巧，而会变成你组织页面状态和数据流的主线工具。

## 学习目标

- 理解协程真正解决的是什么问题。
- 理解 `launch`、`async`、`withContext` 和取消机制的基本意义。
- 理解结构化并发为什么特别适合 Android。
- 理解协程在 ViewModel、Repository 和页面层中的合理位置。

## 前置知识

- 已理解主线程约束和异步任务边界。
- 已知道现代 Android 更偏向 Kotlin-first。

## 正文

### 1. 协程解决的不是“少写几行代码”

很多初学者第一次接触协程，会把它理解成“写异步代码更像同步代码”。这确实是一个直观好处，但并不是核心价值。真正更重要的是，协程提供了一种统一组织异步任务的方式，让这些问题可以在同一模型下处理：

- 任务启动和归属。
- 线程切换。
- 取消。
- 异常传播。
- 多任务之间的结构关系。

相比回调嵌套和手工线程管理，这种统一模型才是协程真正让 Android 开发受益的地方。

### 2. 协程为什么特别适合 Android

Android 的大多数核心工作都具有异步特征：

- 网络请求。
- 数据库读写。
- 文件访问。
- 页面初始化后的数据加载。
- 多来源数据的组合与取消。

这些工作又几乎总与生命周期和页面状态相关。协程的优势，正是既能表达异步，又能较自然地和生命周期感知组件结合。这也是为什么官方架构建议里总会把协程、Flow、ViewModel 放在同一条推荐路径上。

### 3. 协程不是线程，调度也不是自动正确的

这是必须尽早建立的概念。协程不是线程，也不等于“开了协程就自动跑在后台”。协程是一种在挂起点之间调度执行的并发模型，而真正在哪个线程执行，仍取决于调度器和上下文。

这意味着：

- 在主线程启动协程不等于一直阻塞主线程。
- 在协程里做重工作，也不等于自动安全。
- `withContext(Dispatchers.IO)` 之类的切换仍然是必要的边界表达。

如果把协程误当成线程本身，就很容易写出“用了协程却依然卡主线程”的代码。

### 4. 结构化并发才是协程最适合 Android 的原因

结构化并发的核心思想是：协程不是到处乱飞的孤立任务，而应有明确的父子关系和生命周期归属。对 Android 来说，这一点尤其关键，因为：

- 页面会销毁和重建。
- 用户会重复触发同一动作。
- 旧任务不应无限覆盖新状态。
- 屏幕离开后很多任务就不再有意义。

官方推荐在 ViewModel 中使用 `viewModelScope`，本质上就是在把异步任务绑定到明确的状态拥有者。这样一来，页面重建和取消行为都更自然。

### 5. `launch`、`async` 和 `withContext` 应如何理解

入门阶段不需要死背所有 API 细节，但必须建立三个稳定判断：

- `launch` 更适合启动一个不直接返回结果的协程任务。
- `async` 更适合并发获取结果，并通过 `await()` 合并。
- `withContext` 更适合在现有协程内部切换执行上下文并返回结果。

真正重要的不是名字，而是你此刻需要的任务形态：是发起动作、并发等待结果，还是在现有流程中切换工作环境。

### 6. 取消在 Android 中尤其重要

用户离开页面、旋转设备、切换搜索关键词、重复点击刷新，这些都是 Android 中非常常见的取消场景。如果任务没有明确取消边界，后果通常会是：

- 旧请求覆盖新结果。
- 页面销毁后仍尝试更新状态。
- 无意义任务继续浪费网络、电量和 CPU。

协程比裸线程更适合 Android，其中一个核心原因就在于取消机制更自然。官方文档也明确强调，取消是协程协作式的一部分，而不是临时补丁。

### 7. 协程最适合在哪一层启动

一个稳定的判断标准是：谁拥有任务的生命周期，谁就更适合启动和管理它。

因此现代 Android 中更常见的实践是：

- 页面层不直接承担大多数异步任务。
- 页面级异步流程由 ViewModel 使用 `viewModelScope` 发起。
- 数据层内部的具体 I/O 切换由 Repository 或更下层负责。

这样做的价值是，异步任务会和状态归属保持一致，而不是散落在 UI 回调和工具类之间。

### 8. 一个更接近真实项目的最小示例

下面这个例子展示协程为什么特别适合页面状态链路：

```kotlin
class ArticleListViewModel(
    private val repository: ArticleRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow(ArticleListUiState())
    val uiState: StateFlow<ArticleListUiState> = _uiState

    fun refresh() {
        viewModelScope.launch {
            _uiState.update { it.copy(isLoading = true, errorMessage = null) }
            val result = withContext(Dispatchers.IO) {
                repository.fetchLatestArticles()
            }
            _uiState.update {
                it.copy(isLoading = false, articles = result)
            }
        }
    }
}
```

这个例子里最重要的，不是协程语法本身，而是：

- 任务归属于 ViewModel。
- I/O 工作与页面状态更新被清晰分开。
- 页面只消费状态，不直接持有异步流程。

### 9. 实践任务

起点条件：

- 已有一个需要异步执行的页面操作，例如列表加载、搜索、刷新或表单提交。

步骤：

1. 找一个原本需要异步执行的操作，说明为什么协程比手工线程更适合表达它。
2. 判断这个协程应由页面、ViewModel 还是数据层启动。
3. 明确它在页面关闭、重复触发和失败时分别应如何处理。
4. 把其中一个 I/O 操作放进合适的调度上下文中。
5. 检查项目里是否存在到处直接启动协程、却没有明确归属的情况。

预期结果：

- 你会更清楚协程和线程、状态、生命周期之间的关系。
- 异步任务的归属会更明确，取消和重试也更容易设计。
- 你能为 Flow 和 WorkManager 建立稳定基础。

自检方式：

- 你能解释：协程为什么不是线程本身。
- 你能判断：某个任务更适合用 `launch`、`async` 还是 `withContext` 表达。
- 你能说出：为什么结构化并发特别适合 Android 页面场景。

调试提示：

- 如果用了协程页面还是卡，优先检查重工作是否仍跑在主线程上下文。
- 如果旧请求频繁覆盖新结果，优先检查取消边界和作用域归属。
- 如果项目里每层都随手 `launch`，说明协程还没真正融入架构。

### 10. 常见误区

- 把协程当成后台线程的别名。
- 到处直接启动协程，不考虑生命周期和归属。
- 只关注语法，不关注取消和结构化并发。
- 以为用了协程就自动拥有合理架构。

## 小结

协程是现代 Android 异步主线之一。理解它的关键，不是会不会写 `launch`，而是能否让异步任务真正融入生命周期、页面状态和数据层边界之中。只要这层结构立起来，协程就会成为你组织复杂异步流程时最稳定的工具。
