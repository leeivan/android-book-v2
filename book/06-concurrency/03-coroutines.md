# Kotlin Coroutines

很多读者第一次认真使用协程，并不是因为想学一门新的并发理论，而是因为旧方式已经开始撑不住了。页面里嵌套回调越来越长，线程切换越来越乱，取消越来越难处理，网络请求和数据库读写一多，状态更新也开始互相覆盖。协程之所以成为现代 Android 异步主线，不是因为它“写起来更像同步代码”这么简单，而是因为它提供了一套更统一的方式来组织任务、取消、线程切换和结果归属。

这一章要解决的，不只是几个 API 的使用方式，而是让你真正理解: 协程在 Android 中到底在替代什么，为什么结构化并发比“到处启动异步任务”更重要，为什么 ViewModel、Repository、Flow 和协程会被放在同一条架构主线上。

## 学习目标

- 理解协程真正解决的是任务组织问题，而不是语法简化问题。
- 理解 `launch`、`async`、`withContext`、取消和作用域在 Android 中的意义。
- 理解结构化并发为什么特别适合页面和状态驱动架构。
- 学会判断协程更适合在哪一层启动和管理。

## 前置知识

- 已理解主线程边界和耗时任务分类。
- 已接触 ViewModel、Repository 和页面状态概念。

## 正文

### 1. 协程为什么不是“线程换个写法”

很多人第一次看到协程，会被它的顺序写法吸引，误以为它最大的价值是“少写回调”。这当然是优点，但真正更重要的是，协程把异步任务的几个关键问题统一到了同一个模型中:

- 任务是谁启动的。
- 它属于哪个作用域。
- 它在哪个上下文执行。
- 什么时候应该取消。
- 失败后错误如何传播。

相比手工线程或回调链，这种统一性才是协程在 Android 中真正的杀手级优势。

### 2. Android 为什么特别适合采用协程

Android 上的大量工作都具备两个共同特点:

- 它们是异步的。
- 它们和生命周期、页面状态强相关。

例如页面首次加载、搜索输入、数据库刷新、提交表单、并发合并多个来源，这些任务都不只是“后台跑一下”，还要考虑页面是否还活着、旧任务要不要取消、结果该回到哪个状态上。协程和作用域结合后，正好特别适合描述这类问题。

### 3. 协程不是线程，本身也不会自动替你切线程

这条边界必须尽早建立。协程是一种并发模型，不是线程本身。它之所以灵活，是因为它能在不同调度器和上下文中以更低成本组织任务，但真正在哪个线程执行，仍然取决于上下文。

这意味着:

- 在主线程启动协程，不等于全程阻塞主线程。
- 在协程里做重工作，也不等于自动安全。
- `withContext(Dispatchers.IO)` 之类的上下文切换仍然是必要的边界表达。

如果把协程误当成“自动后台线程”，你很容易写出表面用了协程、实际仍然卡 UI 的代码。

### 4. 结构化并发为什么是协程最适合 Android 的原因

协程真正改变 Android 开发的，不是 `suspend` 关键字，而是结构化并发。它要求任务有明确的父子关系和归属，而不是到处散着启动。

对 Android 来说，这一点太重要了，因为:

- 页面会销毁和重建。
- 用户会重复触发动作。
- 旧任务不该随意覆盖新状态。
- 页面离开后很多结果已经没有意义。

这也是为什么现代实践里更强调 `viewModelScope`、`lifecycleScope` 和明确的作用域，而不是随手 new 线程或全局起任务。

### 5. `launch`、`async` 和 `withContext` 该怎样理解

入门阶段没必要死背所有签名，但必须建立三个稳定判断。

`launch` 更适合“发起一个动作”。它常用于页面刷新、提交、保存这类不需要直接返回值给调用方的场景。

`async` 更适合“并发拿结果后再汇合”。如果你只是为了普通异步而滥用 `async`，代码很容易变复杂。

`withContext` 更适合“在已有协程里切换到某个上下文完成一段工作再回来”。它在 Android 中非常适合表达“这段 I/O 不该跑在当前上下文里”。

真正重要的不是 API 名字，而是你此刻面对的任务关系。

### 6. 取消在 Android 中不是附属功能，而是核心能力

页面切换、搜索关键词变化、重复点击刷新、界面销毁，这些都是非常典型的取消场景。只要取消边界没有设计好，协程再优雅也会出问题:

- 旧请求覆盖新结果。
- 页面销毁后仍试图更新状态。
- 无意义的任务继续消耗资源。

协程之所以比传统回调和裸线程更适合 Android，其中一个根本原因就是取消变得更自然、更可表达。

### 7. 协程最适合在哪一层启动

一个很实用的判断标准是: 谁拥有这段任务的生命周期，谁就更适合启动它。

因此现代 Android 中常见的做法是:

- 页面内短期、纯 UI 生命周期相关的任务，可以放在 `lifecycleScope`。
- 页面级业务动作更常由 ViewModel 在 `viewModelScope` 中启动。
- 数据层负责在内部切换合适上下文完成 I/O。

这样一来，任务归属、状态归属和取消边界会比较一致，不容易出现“谁都能发起异步，但谁都不真正负责它”的情况。

### 8. 一个更接近真实项目的最小结构

下面这个例子展示协程如何自然地进入页面状态链路:

```kotlin
class ArticleListViewModel(
    private val repository: ArticleRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow(ArticleListUiState())
    val uiState: StateFlow<ArticleListUiState> = _uiState

    fun refresh() {
        viewModelScope.launch {
            _uiState.update { it.copy(isLoading = true, errorMessage = null) }

            val result = withContext(Dispatchers.IO) {
                repository.refreshArticles()
            }

            when (result) {
                is ApiResult.Success -> _uiState.update {
                    it.copy(isLoading = false, articles = result.data)
                }
                is ApiResult.Failure -> _uiState.update {
                    it.copy(isLoading = false, errorMessage = result.message)
                }
            }
        }
    }
}
```

这段代码最重要的教学点不是语法，而是:

- 任务由 ViewModel 发起。
- I/O 与状态更新边界清楚。
- 页面只消费状态，不直接参与异步过程。

### 9. 协程不会自动给你合理架构

这一点特别需要强调。很多项目迁移到协程后，以为异步问题已经解决，结果只是把原来的回调混乱换成了新的 `launch` 混乱。只要:

- 谁都能随手启动协程。
- 状态没有统一出口。
- 取消边界不清楚。
- 数据层和页面层职责仍旧混乱。

那协程只是让异步写起来更顺手了一点，并没有真正带来稳定结构。

### 10. 实践任务

起点条件:

- 已有一个需要异步执行的页面操作，例如加载、搜索、刷新或提交。

步骤:

1. 选一个异步操作，写下它应该归属哪一层。
2. 判断它是否需要返回值、是否需要并发等待多个结果、是否需要上下文切换。
3. 用 `launch`、`async` 或 `withContext` 中更合适的一种表达它。
4. 为这个任务写出至少一个取消场景。
5. 检查它最终更新的是哪份页面状态。

预期结果:

- 你会把协程看成任务组织工具，而不是线程替代名词。
- 你能更自然地把异步任务和状态边界放在一起思考。
- 后面学习 Flow 和 WorkManager 时，你会更容易看出它们与协程的关系。

自检方式:

- 你能解释协程为什么不是线程本身。
- 你能判断某个任务更适合用 `launch`、`async` 还是 `withContext`。
- 你能说出结构化并发为什么比“随手启动任务”更适合 Android。

调试提示:

- 如果页面仍然卡顿，优先检查重工作是否真的离开了主线程上下文。
- 如果旧结果覆盖新结果，优先检查取消和任务归属。
- 如果每一层都随手 `launch`，说明协程还没有真正进入架构主线。

### 11. 常见误区

- 把协程当成后台线程别名。
- 只学 `launch` 语法，不思考作用域和取消。
- 到处直接启动协程，不考虑任务归属。
- 以为用了协程就自动拥有良好异步结构。

## 小结

协程在现代 Android 中真正提供的，不只是更顺手的异步写法，而是一套更统一的任务组织方式。它把作用域、取消、上下文和结果归属放进了同一个模型里，因此特别适合和页面状态、ViewModel、Repository 一起工作。只要这条结构主线建立起来，协程就会成为 Android 异步开发里最稳的一块基础设施。
