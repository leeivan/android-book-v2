# 后台任务

很多团队第一次遇到后台任务问题，不是因为“想学一个新 API”，而是因为项目开始出现一些难回答的问题：用户把应用切到后台后，上传要不要继续；草稿同步失败了要不要重试；定位记录为什么一锁屏就断；明明只是一次页面请求，为什么却被做成了常驻服务。后台任务设计真正难的地方，从来不是类名，而是判断一件事到底属于哪一类工作。

如果这个判断做错，后面的所有实现都会跟着出错。该跟页面一起结束的任务，被硬做成后台长跑；应该让用户明确感知的任务，被偷偷藏在后台；本来可以择机重试的任务，却被设计成立刻占用资源的持续执行。到了现代 Android，这些错误已经不能简单靠“先跑起来再说”掩盖，因为平台本身对后台执行、电量消耗、用户可感知性都有更严格的约束。

这一章不急着进入 `Service`、前台服务或 `WorkManager` 的细节，而是先把后台工作本身分清楚。只要分类清楚，后面章节里的机制选择就会自然很多。

## 学习目标

- 理解“异步工作”和“后台任务”不是一回事。
- 能根据任务价值、时效性和用户感知需求给任务分类。
- 理解为什么现代 Android 必须对后台工作保持克制。
- 为后续 `Service`、前台服务和 `WorkManager` 打下判断基础。

## 前置知识

- 已理解线程、协程和页面生命周期。
- 已知道页面销毁后，很多任务本来就应该终止。
- 已接触网络请求、上传、同步、播放或定位这类实际需求。

## 正文

### 1. 先从一个真实决策场景开始

假设你正在做一个支持附件发送的即时消息页面。用户点击“发送”后，可能会立刻离开当前聊天页，甚至直接切到别的应用。这时你要面对三个完全不同的任务：

第一，发送按钮触发后的输入校验和界面 loading，这是当前页面内的异步工作。只要页面离开，这部分 UI 反馈通常就没有继续存在的必要。

第二，附件上传和消息投递，页面离开后依然可能有价值。用户并不一定要求它在 1 秒内完成，但通常希望它最终能完成，或者至少知道失败了。

第三，如果这是语音通话、导航或持续定位，而不是消息发送，那么任务就不只是“后台继续做点事”这么简单，而是用户明确依赖它持续运行，系统也必须让用户知道应用仍在做什么。

同样都发生在“页面之外”，它们却属于三类完全不同的工作。后台设计的第一课，就是不要把这三类东西统称成“后台任务”。

### 2. 异步工作不等于后台任务

很多初学者一看到协程、线程池、网络请求，就下意识把它们归到后台任务里。这个分类太粗了。异步只是执行方式，后台任务则是任务生命周期和系统关系的问题。

如果一个任务只服务于当前页面，而且页面结束后结果就没有意义，那么它只是页面内异步工作。比如打开列表页时的首次加载、表单提交前的校验、点击按钮后的短暂操作。它们的重点是“不阻塞主线程”，而不是“在后台长期存在”。

一旦把“异步”误当成“后台”，开发者就很容易为并不需要持续执行的任务引入过度复杂的机制。页面已经关闭，任务结果也没人再需要，却还在系统里占资源，这种设计本质上就是分类错误。

### 3. Android 为什么越来越严格限制后台执行

过去的移动应用常把后台当成一块比较宽松的执行空间，只要不在主线程做事，很多逻辑都想办法往后台塞。现代 Android 逐渐收紧后台执行，原因并不难理解：电量有限、内存有限、用户对隐私和通知打扰更敏感，设备也不可能允许每个应用都在后台长期自由活动。

所以今天设计后台任务时，要先接受一个现实前提：后台执行不是默认权利，而是需要被证明“确实有价值、符合用户预期、符合平台规则”的能力。教材如果只讲“怎么调 API”，不先讲这个前提，读者在真实项目里很快就会踩平台限制。

### 4. 用三个问题给任务分类

判断一项工作是否属于真正的后台任务，可以先问三个问题。

第一个问题是：页面关闭后，这件事还有没有价值。如果没有价值，它大概率只是页面内异步工作；如果仍然有价值，例如草稿同步、离线上传、定期拉取远程配置，它才值得进入后台设计视角。

第二个问题是：它是否必须立刻完成。如果用户正在等待结果，任务往往属于当前交互链路，不应该被一股脑地推给系统“以后再说”。如果它允许延迟，并且稍晚完成并不会破坏体验，那么它更接近系统调度型任务。

第三个问题是：用户是否应该明确知道它正在持续运行。如果答案是“应该”，那就不能把它偷偷藏在后台。音乐播放、导航、运动记录、通话类任务之所以要特别对待，就是因为它们天然带有强烈的用户感知属性。

这三个问题比直接问“该用哪个 API”更重要，因为它们回答的是任务本质。

### 5. 页面内异步工作：离开页面后通常就应结束

先看最常见、也最容易被误判的一类任务。比如新闻列表首次加载、登录按钮点击后的请求、提交表单后的短暂等待，这些任务往往和当前页面直接绑定。页面销毁后，结果已经没有消费方，继续保留通常没有意义。

对这类任务，更重要的是：

- 不阻塞主线程。
- 能在页面关闭或取消时停止。
- 结果明确回到当前页面状态。

它们通常不需要上升到后台任务设计，更不需要为了“保险”引入服务或持久调度机制。成熟的工程判断，不是让所有任务都更“强大”，而是知道什么时候该及时结束。

### 6. 可延迟执行的工作：重点是最终完成，而不是立即完成

第二类任务是真正会通向 `WorkManager` 等机制的那类任务。典型例子包括草稿同步、日志或崩溃信息上传、离线收藏同步、非实时数据刷新。这些任务的特点是：页面结束后仍然有价值，但用户并不会一直盯着它等结果。

这类任务更适合系统择机执行，因为它们通常能够接受一些现实条件：

- 等网络恢复再执行。
- 等设备空闲或电量条件更好时再执行。
- 失败后允许重试。

在教学里必须强调这一点：所谓“后台任务”，很多时候真正需要的不是“现在立刻后台长跑”，而是“最终可靠地完成”。这和前台持续运行完全不是一回事。

这类任务还有一个很重要的工程要求：尽量做到幂等。也就是说，同一项同步或上传即使因为重试、重排队或进程恢复被执行多次，结果也不应该变得更糟。后台工作一旦离开页面、交给系统调度，重复执行就不再是例外情况，而是必须提前接受的现实。

### 7. 用户可感知的持续工作：重点是透明和可控

第三类任务是最不能偷懒分类的。导航、音乐播放、运动记录、录音、长时间位置跟踪，这些任务一旦运行，用户通常会明显依赖它们持续存在。它们不是“有空再跑”，而是“用户正在指望它别停”。

这类工作有两个鲜明特征：

- 一旦中断，用户马上能感知到损失。
- 用户理应知道应用正在持续工作。

这也是为什么现代 Android 要求这类任务更明确地进入前台服务或其他受约束机制，而不是悄悄在后台维持。用户感知不是附属问题，而是任务分类本身的一部分。

### 8. 后台任务设计本质上也是产品设计

很多开发者一谈后台任务，马上只想到线程、调度和生命周期。但后台任务设计其实直接关系到用户是否理解你的应用行为。应用为什么还在运行，为什么会出现通知，为什么会继续耗电、占网、记录位置，这些都不是技术细节，而是用户体验和产品信任的一部分。

所以教材里要把后台设计讲得更像“系统与用户的契约”，而不是一份类库列表。你做的不只是让任务跑起来，而是在回答：这件事值得继续吗，值得继续多久，失败时谁来承担后果，用户是否被充分告知。

### 9. 一个对比：草稿同步和导航服务为什么不是一回事

把草稿同步和导航放在一起看，区别会非常明显。

草稿同步通常在页面退出后依然有价值，但它并不要求每一秒都持续运行。网络差一点、晚几分钟、失败后稍后重试，这些通常都还在可接受范围内。对用户来说，重点是“最终别丢内容”。

导航服务则完全不同。用户此刻正在依赖实时路线、语音播报和位置信息，一旦中断，体验立刻崩溃。它不仅需要持续执行，还需要让用户明确知道应用正在进行持续服务。

同样都发生在页面之外，一个更接近持久调度，一个更接近前台持续服务。这就是为什么后台任务不能按“是不是在页面外”来简单判断。

### 10. 一个实用的决策顺序

当你面对一个新需求，不确定它该落在哪类机制上时，可以按下面这个顺序想。

先问：页面关闭后，这件事还值不值得继续。如果不值得，就不要把它送进后台体系。

再问：结果是不是必须立刻交付给正在等待的用户。如果必须，那它更像当前交互流程的一部分。

再问：用户是否应该持续知道它在运行。如果应该，这通常意味着任务不能偷偷藏在后台。

最后再问：它能不能接受系统择机执行，失败后能不能安全重试。如果可以，它通常更适合交给持久后台调度。

这套顺序的好处是，你先确定任务性质，再选工具，而不是先记一堆 API 名字再硬套场景。

### 11. 从错误设计里反推分类原则

很多后台 bug 本质上都能追溯到错误分类。

如果一个本应跟页面一起结束的任务被做成长期后台执行，你会看到资源浪费、状态回写对象已不存在、错误难以追踪。

如果一个用户明确依赖的持续任务被误做成普通后台工作，你会看到任务在锁屏、切后台或系统回收后中断，用户会直接感觉应用“不可靠”。

如果一个本来可以择机重试的工作被强行做成立即执行的持续服务，你会看到通知打扰过重、电量消耗不合理、实现复杂度居高不下。

所以“后台机制怎么选”从来不是孤立 API 问题，而是任务分类问题的后果。

### 12. 实践任务

起点条件：

- 已有一个包含网络请求、上传、同步、播放或定位需求的项目，或者至少有这类需求草图。

步骤：

1. 列出项目中的三个异步任务。
2. 分别判断它们属于页面内异步工作、可延迟后台工作，还是用户可感知的持续工作。
3. 为每个任务写一句理由，说明页面关闭后它是否仍有价值。
4. 再写一句理由，说明用户是否应该知道它仍在运行。
5. 选出一个当前分类可能有问题的任务，重新设计它的实现方向。

预期结果：

- 你会先描述任务性质，再讨论 API。
- 你能更自然地区分页面协程、持久后台调度和前台持续服务。
- 你会发现不少所谓“后台需求”其实只是页面内异步工作。

自检方式：

- 你能解释为什么异步工作不等于后台任务。
- 你能判断某个任务是否允许系统择机执行。
- 你能说明用户为什么应该或不应该感知某项任务仍在运行。

调试提示：

- 页面已经退出，任务结果也没人再需要，却还在跑，优先检查是否被错误设计成后台任务。
- 用户明明依赖任务持续存在，却没有任何通知或可见提示，优先检查分类是否应该转向前台服务。
- 如果你总是因为“怕任务丢”而让所有东西后台常驻，通常说明任务分类还没有想清楚。

### 13. 常见误区

- 把所有异步工作都统称为后台任务。
- 页面退出后仍让原本无意义的任务继续运行。
- 不区分可延迟工作和用户可感知的持续工作。
- 只从实现难度看后台任务，不从用户价值和平台约束判断。

## 小结

后台任务设计的起点不是 API，而是分类。先判断页面关闭后任务是否仍有价值，再判断它是否必须立刻完成，最后判断用户是否应该感知它在持续运行。只要这三步做对，后面的 `Service`、前台服务和 `WorkManager` 选择就会顺很多。反过来，如果分类一开始就错，越往后实现只会越重、越脆弱、越难维护。
