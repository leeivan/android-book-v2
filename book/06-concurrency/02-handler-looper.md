# Handler 与 Looper

在现代 Android 项目里，很多新代码已经不再以 Handler 作为异步主线，但它仍然是理解 Android 消息机制和阅读历史代码的重要基础。本章的重点不是把它当成新项目默认方案，而是帮助读者理解它在平台中的位置。

## 学习目标

- 理解 Handler、Looper 和 MessageQueue 的关系。
- 理解旧式线程间通信在 Android 中如何实现。
- 为历史代码迁移和现代并发对照提供基础。

## 前置知识

- 已理解主线程与工作线程的职责差异。
- 已知道后台结果需要回到界面层。

## 正文

### 1. Handler 机制解决什么问题

在协程和现代状态流广泛使用之前，Android 中常见的线程间通信方式之一，就是通过消息循环把后台结果调度回目标线程。Handler、Looper 和消息队列共同承担了这件事。

### 2. 为什么它依然值得理解

原因主要有三点：

- Android 平台本身很多行为仍建立在消息循环机制之上。
- 旧项目里大量代码仍然使用 Handler。
- 理解它有助于理解“线程切换”并不是魔法，而是有明确调度模型的。

### 3. 为什么它不再是新项目默认异步主线

因为现代 Kotlin 项目有更自然的异步表达方式，例如协程和 Flow，它们在可读性、状态组织和与现代架构协同上通常更合适。

因此，本章的正确定位是：

- 用来理解 Android 消息机制。
- 用来阅读和迁移历史代码。
- 不作为新项目默认并发写法主线。

### 4. 最小实践任务

1. 用自己的话解释 Handler、Looper 和消息队列的关系。
2. 找出一个历史项目中可能出现 Handler 的位置。
3. 思考：如果换到现代 Kotlin 项目，这段逻辑是否更适合由协程处理。

### 5. 常见误区

- 把 Handler 当成所有异步问题的默认答案。
- 只会照抄用法，不理解消息循环模型。
- 看到历史代码中有 Handler 就想立刻无脑删除，而不先理解其职责。

## 小结

Handler 与 Looper 是 Android 并发机制的重要历史与底层基础。理解它，不是为了回到旧写法，而是为了更清晰地看懂平台和迁移代码。
