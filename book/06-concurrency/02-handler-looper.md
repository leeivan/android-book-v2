# Handler 与 Looper

在现代 Android 开发中，协程和 Flow 已经成为异步主线，但这并不意味着 Handler 和 Looper 可以完全跳过。它们仍然是理解 Android 线程消息模型的关键基础，也是很多系统组件和旧代码背后的运行逻辑。本章的重点，不是鼓励你在新项目里大量回到 Handler 写法，而是让你明白 Android 的消息循环机制到底是什么，以及今天应该如何看待它的边界。

## 学习目标

- 理解 Looper、MessageQueue 和 Handler 的基本关系。
- 理解 Android 主线程为什么是消息循环驱动的。
- 知道 Handler 在现代项目中的保留价值和使用边界。
- 为理解协程调度、主线程切换和旧代码迁移打基础。

## 前置知识

- 已理解主线程和耗时工作边界。
- 已知道 Android UI 相关工作天然依赖主线程。

## 正文

### 1. 为什么还需要理解 Handler 和 Looper

很多现代 Android 代码表面上已经不直接写 Handler，但 Android 自己的线程模型并没有消失。主线程之所以能持续处理点击、绘制、回调和消息，本质上依赖的仍然是一套消息循环机制。

如果不理解这层机制，很多问题都会显得神秘，例如：

- 为什么某些任务必须切回主线程。
- 为什么延迟执行会和消息队列相关。
- 为什么旧代码里会到处出现 Handler。

### 2. Looper、MessageQueue 和 Handler 的关系

可以用一个非常朴素的方式理解三者：

- Looper 负责循环取消息。
- MessageQueue 负责存放待处理消息。
- Handler 负责发送消息或把任务投递进队列。

这个模型之所以重要，是因为它解释了 Android 主线程并不是“自动处理一切”，而是在一个持续消息循环中处理工作。

### 3. 主线程为什么本质上是消息驱动的

用户点击、绘制刷新、生命周期回调、某些异步结果返回，本质上都需要被安排到合适时机处理。消息循环机制的意义，就是让这些工作按顺序进入线程执行，而不是无组织地互相打断。

因此，理解主线程时，不应只把它看成“UI 线程”，还应把它看成“消息循环线程”。

### 4. Handler 的经典用途是什么

在早期和底层场景中，Handler 常见用途包括：

- 切回主线程执行 UI 相关工作。
- 延迟执行某个任务。
- 在线程之间传递简单消息。

这些用途今天仍然存在，但在很多应用层代码中，已经有更现代、更易维护的替代方案，例如协程和生命周期感知状态流。

### 5. 为什么新项目不应默认回到 Handler 思路

Handler 的问题不在于它失效了，而在于它本身并不解决这些更高层问题：

- 生命周期绑定。
- 结构化取消。
- 异步结果如何进入页面状态。
- 多异步任务如何更清晰地组合。

因此，在现代项目里直接用 Handler 组织大部分异步逻辑，往往会重新回到手工管理时序和状态的老路。

### 6. 但 Handler 仍然值得理解和识别

你仍然需要理解 Handler，至少因为：

- 旧项目里很常见。
- 某些底层或系统交互场景仍会出现。
- 理解它有助于你更清楚协程切线程和主线程调度背后的本质。

换句话说，本章的重点是“知道它是什么、为什么存在、今天何时仍合理”，而不是“回到它做主线”。

### 7. 最小示例：为什么主线程延迟任务能工作

如果你把一个任务投递到主线程稍后执行，本质上就是把它放进了主线程的消息队列中，等待 Looper 在合适时机取出并执行。这个例子说明，Android UI 线程的行为不是凭空发生的，而是有明确消息模型支撑。

### 8. 最小实践任务

1. 用自己的话画出 Looper、MessageQueue 和 Handler 的关系。
2. 找一段旧代码，识别其中 Handler 到底承担了什么职责。
3. 判断这段逻辑在现代项目里是否更适合用协程或其他更高层方案表达。
4. 观察你的项目中是否还把 Handler 作为大多数异步逻辑的默认入口。

### 9. 常见误区

- 以为协程出现后，Handler 和 Looper 就完全不重要了。
- 在新项目里仍默认用 Handler 管理复杂异步流程。
- 只记 API，不理解消息循环模型。
- 把 Handler 当成“异步万能工具”。

## 小结

Handler 与 Looper 的核心价值，是帮助你理解 Android 的消息循环模型。今天的新项目不应再把它作为主要异步组织方式，但理解它，能让你对主线程、调度和旧代码迁移有更扎实的认识。
