# 线程基础

并发问题是 Android 开发里最常见、也最容易被“会用 API”掩盖的问题之一。页面卡顿、主线程阻塞、重复请求、状态竞争、加载时机混乱，很多表面上看像 UI 或网络问题的故障，底层都和线程与执行模型有关。本章的目标，是先把最基础的一条线拉直：主线程为什么重要，什么工作不该阻塞它，以及为什么“会开线程”远远不等于“并发设计是对的”。

## 学习目标

- 理解主线程在 Android 中的特殊地位。
- 理解 CPU 密集和 I/O 密集任务的基本区别。
- 理解为什么阻塞主线程会直接破坏 UI 体验。
- 为 Handler、协程、Flow 和后台任务章节打基础。

## 前置知识

- 已理解页面、生命周期和网络数据的基本特征。
- 已知道 Android 页面会频繁响应用户输入和状态变化。

## 正文

### 1. 为什么 Android 特别强调主线程

Android 的界面渲染、事件分发和大部分 UI 相关回调，都围绕主线程组织。对开发者来说，这意味着一个非常现实的结论：主线程不是普通线程，它承担着维持页面可交互和可响应的核心责任。

只要主线程被长时间占用，用户最先感受到的就是：

- 点击没反应。
- 滚动卡顿。
- 页面切换迟缓。
- 动画不流畅。

因此，并发章节首先要解决的不是“怎样开更多线程”，而是“怎样保护主线程”。

### 2. 什么工作不该放在主线程

一个最基本的判断标准是：凡是明显可能耗时、等待或大量计算的工作，都不应直接阻塞主线程。常见包括：

- 网络请求。
- 数据库读取或写入。
- 大文件读写。
- 图片解码和大规模计算。

这条规则看似简单，却是 Android 并发设计里最核心的底线之一。

### 3. CPU 密集和 I/O 密集为什么值得区分

所有异步工作都可以丢到后台线程，这种说法不够精确。更有价值的理解是：

- CPU 密集型任务主要消耗计算资源。
- I/O 密集型任务主要花时间等待网络、磁盘或其他外部资源。

为什么要区分？因为不同类型任务的调度策略和性能瓶颈并不一样。即使你在入门阶段不做复杂调优，也应先建立这层意识。

### 4. “开个线程”为什么不是完整答案

早期教程里经常会把异步问题简化成“新开一个线程就好”。这种理解很快就会暴露问题：

- 线程何时结束。
- 页面关闭后任务是否还在运行。
- 异步结果如何回到正确页面。
- 多个任务同时运行时如何协调。

也就是说，真正的并发问题不仅是“在哪个线程执行”，还包括“谁管理任务、谁取消任务、谁消费结果”。

### 5. UI 卡顿很多时候不是因为线程少，而是因为工作放错地方

这点非常值得强调。很多项目卡顿并不是因为“线程不够多”，而是因为：

- 本应后台做的工作跑在主线程。
- 状态变化太频繁，导致主线程反复重绘。
- 数据处理和 UI 更新交织在一起。

因此，优化并发时，最先要做的不是加更多线程，而是重新审视工作边界。

### 6. 并发设计为什么会影响架构

一旦你开始认真处理线程和异步问题，就会发现它和架构几乎不可分：

- ViewModel 适合管理哪些异步任务。
- Repository 应如何组织远程与本地读取。
- 页面关闭时哪些任务应取消。
- 后台长期工作又该交给谁。

这也是为什么现代 Android 的并发主线越来越围绕协程、Flow 和 WorkManager 展开，而不是停留在裸线程层面。

### 7. 最小示例：新闻列表加载为什么不能直接在页面里做重工作

一个新闻列表页面如果在主线程直接做网络请求、JSON 解析和列表数据构建，界面很快就会卡顿。这个例子看似朴素，却说明了 Android 并发设计最重要的现实：页面层最该做的是组织状态，而不是承载耗时工作。

### 8. 最小实践任务

1. 列出你当前项目中所有明显耗时的操作。
2. 判断哪些绝不应运行在主线程。
3. 检查你的页面层是否仍在直接做数据库、文件或网络相关工作。
4. 记录一个你见过的卡顿场景，并分析它更可能是哪类工作放错了位置。

### 9. 常见误区

- 认为并发问题就是“开线程”。
- 只会把工作挪出主线程，但不管理任务生命周期。
- 把所有异步结果直接回调给页面，忽略状态组织。
- 以为卡顿只和动画或布局有关，不看主线程工作负载。

## 小结

线程基础章节的核心，不是让你掌握多少底层 API，而是先建立一个清晰底线：保护主线程、识别耗时工作、明确异步任务不能脱离生命周期和架构边界存在。后面的 Handler、协程和 WorkManager，都会建立在这层认知之上。
