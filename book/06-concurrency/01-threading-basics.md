# 线程基础

很多 Android 并发问题，第一次并不是以“线程模型”这种学术名词出现的，而是以很具体的故障出现: 点一下按钮页面卡住了，图片列表滚动不动了，数据库写入时界面掉帧了，两个异步结果互相覆盖了。读者往往会立刻去找“怎么开后台线程”，但真正的问题通常更早就已经出现了: 你还没有先把任务分清楚，哪些事情必须在主线程做，哪些事情不能在主线程做，哪些事情虽然放到了后台却仍然会把状态搞乱。

这一章不把线程当成单独的语法知识来讲，而是把它放回 Android 的核心约束里: 界面只能在主线程更新，耗时操作不能阻塞主线程，并发任务之间必须有清楚的归属和边界。只要这三件事理解透，后面的 `Handler`、协程、`Flow` 和 `WorkManager` 才不会只是工具堆砌。

## 学习目标

- 理解 Android 主线程为什么既重要又脆弱。
- 理解 CPU 密集型任务、I/O 密集型任务和 UI 任务的基本差异。
- 理解“放到后台”不等于“问题解决了”。
- 为协程、Flow 和后台任务章节建立并发直觉。

## 前置知识

- 已理解 Activity、Fragment 和页面状态的基本概念。
- 已接触网络请求、数据库访问或图片加载等耗时操作。

## 正文

### 1. 主线程为什么必须先被理解

Android 的主线程承担着几乎所有用户直接能感知到的工作:

- 分发输入事件。
- 执行界面更新。
- 驱动绘制和动画。
- 协调很多生命周期回调。

这意味着主线程既是界面的心脏，也是最容易出问题的地方。只要你把耗时工作放进去，用户感受到的就不是“代码执行中”，而是卡顿、掉帧、无响应甚至 ANR。

所以并发学习的第一步，不是学怎么“开线程”，而是先接受一个非常清晰的规则: 主线程要尽量只做与当前界面直接相关且必须立即完成的工作。

### 2. 为什么“不要阻塞主线程”不是一句口号

不少初学者听过“不要在主线程做耗时操作”，但并不真正理解它的后果。设想一下列表页加载:

- 如果网络请求在主线程执行，界面会直接卡住。
- 如果数据库大量写入在主线程执行，输入和滚动会明显迟滞。
- 如果图片解码在主线程完成，列表滑动会丢帧。

主线程的问题，不是“慢一点”而已，而是它一旦被耗尽，整个界面的感知质量都会下降。这也是为什么 Android 平台一直强烈区分“界面工作”和“耗时工作”。

### 3. 先把任务分成三类，比先学 API 更重要

并发设计最有用的起点，不是线程类名，而是任务分类。一个非常实用的分法是:

- UI 任务: 必须立即反馈到当前界面，例如点击反馈、显示 loading、切换控件状态。
- I/O 任务: 受网络、磁盘、数据库影响较大，常见于请求、读写和缓存。
- CPU 任务: 计算量大，例如大图处理、复杂排序、批量加解密、数据分析。

这三类任务的共性是，都可能涉及并发；差异在于，它们对线程、时机和取消的要求完全不同。只要分类先做对，后续工具选择会自然很多。

### 4. 放到后台并不等于万事大吉

很多项目表面上已经“把耗时操作移出主线程”，却仍然频繁出问题。原因通常是，开发者把“线程切换”误当成了“并发设计完成”。

举个例子:

- 用户快速连续搜索，旧请求晚于新请求返回，把新结果覆盖掉。
- 页面已经关闭，后台任务回来后仍尝试更新 UI。
- 多个写操作并发执行，导致状态顺序混乱。

这些问题都说明，真正困难的不只是把任务放到后台，而是决定任务归谁管、什么时候取消、谁有权更新当前状态。

### 5. Android 并发的难点，很多时候是生命周期

在桌面程序里，一个后台任务完成后更新界面，通常只是“回到 UI 线程”。但 Android 不一样，页面实例可能已经重建、销毁，用户也可能已经离开当前屏幕。

因此在 Android 中做并发，必须总是多问一句: 当任务完成时，原来的消费者还在吗？如果还不在，这个结果应该丢弃、缓存，还是转成别的状态？

这也是为什么现代 Android 后来逐渐把并发工具和生命周期、作用域、状态流绑定在一起看。单纯会开线程，并不足以解决页面一致性问题。

### 6. 为什么共享可变状态会让并发变得难

并发之所以容易出 bug，往往不是因为线程本身难，而是因为多个执行路径同时操作同一份可变状态。比如:

- 两个刷新请求同时改同一个列表。
- 一个任务刚把 loading 关掉，另一个任务又把页面切回错误态。
- 后台线程和主线程都在读写同一个集合。

只要共享可变状态没有被清楚约束，并发就会变成偶发、难复现、难调试的问题。现代 Android 越来越强调单向数据流和不可变状态对象，背后其实也有很强的并发原因。

### 7. 一个典型误区: 把所有耗时工作都当成同一种后台任务

例如:

- 页面打开时加载一页数据。
- 周期性同步远程配置。
- 长时间文件上传。

它们都“耗时”，但完全不是同一类问题。第一个更像页面内异步工作，第二个更像可延迟后台任务，第三个可能还涉及前台服务和用户可感知状态。线程基础这一章必须先把这件事讲清楚，否则后面工具永远会被误用。

### 8. 一个最小例子: 为什么简单线程切换还不够

假设你在搜索页里每次输入都发请求。如果只是简单把请求放到后台线程，你解决了“不卡 UI”，却没解决“旧结果覆盖新结果”。真正更完整的并发思维应该包括:

- 当前这次搜索是否还有效。
- 旧任务是否应被取消。
- 返回结果是否仍属于当前输入条件。

这正说明，并发问题真正复杂的部分是任务关系，而不只是线程位置。

### 9. 实践任务

起点条件:

- 已有至少一个涉及网络、数据库、图片或文件处理的页面。

步骤:

1. 列出页面里所有耗时操作。
2. 判断每个操作属于 UI、I/O 还是 CPU 任务。
3. 标出哪些任务页面关闭后应该停止，哪些任务可能仍有价值。
4. 检查是否存在多个任务同时读写同一份状态。
5. 为其中一个任务写下“它完成时谁还应该接收结果”。

预期结果:

- 你会把并发问题先看成任务分类和状态边界问题。
- 你会更清楚为什么仅仅“放到后台”并不够。
- 后面学习协程和后台调度时，会更容易理解它们在解决什么。

自检方式:

- 你能解释主线程为什么必须尽量轻。
- 你能判断某个任务是 I/O 还是 CPU 密集型。
- 你能说明“线程切换”为什么不等于“并发设计完成”。

调试提示:

- 页面不卡了但结果仍经常错乱，优先检查任务关系，而不是继续加线程。
- 多个任务同时更新一个状态时，优先先理清状态所有权。
- 如果你无法说出一个任务完成后结果应该交给谁，说明并发边界还没有设计好。

### 10. 常见误区

- 把并发学习简化成“怎么开后台线程”。
- 认为只要不阻塞主线程，任务设计就已经正确。
- 不区分页面内异步任务和真正的后台任务。
- 忽视共享可变状态带来的竞争问题。

## 小结

线程基础这一章真正要建立的，不是 API 记忆，而是 Android 并发直觉: 主线程必须轻，耗时任务必须分类，并发结果必须有明确归属。只要这三件事站稳，后面的 `Handler`、协程、`Flow` 和 `WorkManager` 才不会变成表面工具，而会成为有边界的解决方案。
