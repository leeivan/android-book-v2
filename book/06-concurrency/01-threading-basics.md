# 线程基础

并发问题是 Android 开发里最常见、也最容易被“会用 API”掩盖的问题之一。页面卡顿、主线程阻塞、重复请求、状态竞争、加载时机混乱，很多表面上看像 UI 或网络问题的故障，底层都和线程与执行模型有关。本章的目标，是先把最基础的一条线拉直：主线程为什么重要，什么工作不该阻塞它，以及为什么“会开线程”远远不等于“并发设计是对的”。

Android 官方在进程与线程文档中把这个原则说得非常清楚：默认情况下，同一应用的所有组件运行在同一个主线程里，而这个主线程同时负责分发 UI 事件和绘制界面。只要理解了这一点，你就能明白并发章节的第一原则始终不是“怎么开更多线程”，而是“怎么保护主线程”。

## 学习目标

- 理解主线程在 Android 中的特殊地位。
- 理解 CPU 密集和 I/O 密集任务的基本区别。
- 理解为什么阻塞主线程会直接破坏 UI 体验。
- 理解为什么并发设计必须和生命周期、状态管理一起考虑。

## 前置知识

- 已理解页面、生命周期和网络数据的基本特征。
- 已知道 Android 页面会持续响应用户输入和状态变化。

## 正文

### 1. Android 的主线程不是普通线程

Android 官方文档把主线程同时称为 UI thread。这个命名本身就在提醒我们，它不仅仅是一条默认执行线程，而是负责：

- 分发触摸和点击事件。
- 执行大多数 UI 回调。
- 驱动界面绘制与重绘。
- 维持页面整体可交互性。

这意味着主线程一旦被阻塞，用户最先感受到的不会是“程序内部有点慢”，而是点击失效、滚动卡顿、动画不流畅，甚至直接触发 ANR。对移动应用来说，这种影响是立刻可见的。

### 2. 什么工作不该放在主线程

一个很实用的判断标准是：凡是明显可能耗时、等待外部资源或需要大量计算的工作，都不应直接阻塞主线程。最典型的包括：

- 网络请求。
- 数据库读写。
- 大文件读写。
- 图片解码和压缩。
- 大规模 JSON 解析或复杂计算。

Android 官方甚至会在主线程网络访问时直接抛出 `NetworkOnMainThreadException`。这不是额外刁难，而是在强制你遵守最基本的 UI 响应性边界。

### 3. CPU 密集和 I/O 密集任务为什么值得区分

虽然入门阶段不需要做很细的调优，但理解这两类任务的差别非常重要。

CPU 密集型任务主要耗费计算资源，例如复杂图片处理、批量加密、较重的数学运算。它们的问题在于长时间占用处理器。

I/O 密集型任务则主要花时间等待外部资源，例如网络返回、磁盘读写、数据库查询。它们的问题在于等待时间长、不可控因素多。

为什么要分清？因为这会影响你如何看待调度、取消和线程切换。例如一个 I/O 任务的大部分时间都在等待，而不是在计算；这也是为什么现代 Android 更常用协程和调度器去承接这类工作，而不是简单“多开几个线程”。

### 4. “开个线程”为什么不是完整答案

早期教程经常把异步问题简化为“新开一个线程就好了”。这只解决了一个表层问题：任务暂时不再阻塞主线程。真正的复杂度很快会出现：

- 线程由谁管理。
- 页面关闭后任务是否应继续。
- 结果如何回到正确页面。
- 多个任务同时运行时如何取消旧任务。
- 失败和重试逻辑放在哪里。

换句话说，并发问题从来不只是“在哪条线程上执行”，还包括“谁拥有这项工作、谁消费结果、什么时候应该停止”。这就是为什么并发设计一定会和架构设计发生关系。

### 5. UI 卡顿很多时候不是因为线程少，而是因为工作放错地方

很多项目出现卡顿后，第一反应是“是不是要开更多线程”。但真实原因往往更简单，也更基础：

- 本应后台执行的工作跑到了主线程。
- 主线程上做了过多同步数据处理。
- 状态变化太频繁，引发反复重绘。
- 页面层承担了本该在 ViewModel 或数据层处理的耗时逻辑。

所以并发优化最先要做的，不是增加线程数量，而是重新审视工作边界。只要耗时任务仍在错误的层和错误的线程上，再多线程也只是让问题更难排查。

### 6. 并发设计从第一天起就和生命周期绑在一起

Android 项目中的线程问题很少是纯计算问题。你几乎总会同时面对这些现实因素：

- 页面会被销毁和重建。
- 用户可能重复点击触发相同请求。
- 旧任务可能在新状态之后才返回。
- 后台任务可能比当前页面活得更久。

因此，现代 Android 文档在推荐协程、Flow、WorkManager 时，强调的从来不只是性能，而是结构化取消、生命周期感知和系统调度。这说明“工作属于谁”本身就是并发设计的一部分。

### 7. 一个更接近真实项目的最小例子

以新闻列表页为例，如果你在页面初始化时直接在主线程做网络请求、JSON 解析、排序和列表构建，界面很快就会卡顿。就算你把其中一部分移到后台线程，如果页面销毁后任务还继续、旧请求结果还会覆盖新结果，问题仍然没有真正解决。

这个例子说明，并发设计至少要同时回答三件事：

- 这段工作该不该离开主线程。
- 这段工作应由谁发起和管理。
- 这段工作在页面状态变化后是否还应继续。

只有把这三件事一起回答，线程设计才算真正站住。

### 8. 实践任务

起点条件：

- 已有一个具备网络、数据库或文件操作的 Android 项目。

步骤：

1. 列出项目中所有明显耗时的操作。
2. 判断哪些绝不应运行在主线程。
3. 检查这些操作当前是由页面、ViewModel 还是数据层发起。
4. 记录一个你见过的卡顿场景，分析它更可能是 CPU 密集还是 I/O 密集工作放错了位置。
5. 思考页面关闭后，这些任务哪些应取消，哪些仍有继续价值。

预期结果：

- 你会更清楚地主线程应该只承担什么。
- 你会开始从“任务归属”和“生命周期”角度理解并发。
- 你能为后续 Handler、协程和 WorkManager 找到清晰背景。

自检方式：

- 你能解释：为什么 Android 主线程一旦阻塞，用户体验会立即受损。
- 你能区分：一个任务更像 CPU 密集还是 I/O 密集。
- 你能判断：某项异步工作到底应绑定页面、ViewModel 还是后台调度。

调试提示：

- 如果卡顿发生在点击后立即出现，优先检查是否有同步重工作跑在主线程。
- 如果旧请求经常覆盖新状态，优先检查任务取消和结果归属。
- 如果页面层直接处理大量数据库和网络逻辑，通常并发边界已经和架构边界一起失守。

### 9. 常见误区

- 认为并发问题就是“开线程”。
- 只会把工作挪出主线程，不管理任务生命周期。
- 把所有异步结果直接回调给页面，忽略状态组织。
- 以为卡顿只和动画或布局有关，不看主线程工作负载。

## 小结

线程基础章节真正要建立的，不是底层 API 列表，而是一条基本底线：保护主线程、识别耗时工作、明确任务归属，并承认并发设计从一开始就离不开生命周期和架构边界。后面的 Handler、协程和 WorkManager，都会建立在这层认知之上。
