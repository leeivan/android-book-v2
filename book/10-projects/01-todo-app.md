# Todo App

待办应用之所以适合作为 Android 综合案例，并不是因为它“简单”，而是因为它把大量真实能力放进了一个足够克制、足够能看清边界的产品里。你可以在它身上练习列表、表单、状态管理、本地存储、提醒、通知、后台调度，甚至再往上扩展同步、小组件和多设备支持。它不像聊天和复杂内容分发那样一上来就要求高并发和高实时性，却已经足够接近真实应用的工程形态。

这章的重点不是罗列功能点，而是演示一个教学案例该怎样被逐步做出来。你会看到为什么待办应用天然适合本地优先，为什么页面状态和实体模型必须分开，为什么提醒功能应该交给后台调度和通知协作，以及为什么一个看似简单的案例，恰恰最适合把前面章节的现代 Android 主线真正串起来。

## 学习目标

- 理解待办应用为什么是现代 Android 入门到综合实践之间最合适的案例之一。
- 理解如何从需求拆解出页面状态、数据模型和后台能力。
- 理解本地优先、提醒、通知和状态管理在同一应用里的协作方式。
- 学会把前面学过的章节知识组织成一个小而完整的项目。

## 前置知识

- 已理解 Room、ViewModel、StateFlow、Repository、WorkManager 和通知。
- 已能完成基本列表页和表单页。

## 正文

### 1. 为什么待办应用是“综合练习”的好起点

一个健康的教学案例，需要同时满足三个条件:

- 业务足够真实。
- 复杂度足够克制。
- 能自然带出核心工程能力。

待办应用刚好满足这三个条件。用户需求很容易理解: 新建任务、完成任务、查看待办、按时间提醒。与此同时，它又天然包含:

- 列表和详情 / 编辑页面。
- 本地存储和状态流。
- 提醒和通知。
- 背景调度。

这让它既适合做第一套完整项目，又不会因为业务噪音太大而掩盖教学重点。

### 2. 正确的起点不是先建数据库表，而是先定义用户动作

很多人一开始做案例，会先写 `TodoEntity`。这不是不对，但顺序更好的做法通常是:

1. 先问用户能做什么。
2. 再问页面要呈现哪些状态。
3. 最后才决定实体和数据库结构。

例如一个待办应用最核心的用户动作往往是:

- 新建任务。
- 修改任务。
- 标记完成。
- 查看今天或全部任务。
- 为某项任务设置提醒。

先从动作和状态出发，后面的实体模型才不会脱离产品语义。

### 3. 待办应用为什么天然适合本地优先

待办类应用最重要的体验之一，就是“打开就能立刻看到任务”。用户通常不希望:

- 打开列表先转圈等网络。
- 没网就完全不能记任务。
- 勾选完成后要等远程响应才更新界面。

这使它成为本地优先最自然的案例。Room 作为可信来源、Repository 作为数据入口、ViewModel 作为页面状态持有者，这条链路在待办应用里非常容易讲清楚，也非常容易让读者真正体会到它的价值。

### 4. 页面状态和任务实体为什么必须分开

待办应用看起来只有几个字段，但真正到 UI 层时，页面关心的并不只是实体本身。例如:

- 当前是不是首次加载。
- 列表为空是因为没有任务，还是因为筛选后为空。
- 编辑页是在新建还是修改。
- 某个提醒是否刚刚设置成功。

这些都不是 `TodoEntity` 应该直接承载的内容，而是页面状态。这个案例特别适合训练读者把“持久化数据”和“页面状态”分开，否则很快就会出现零散布尔值满天飞的问题。

### 5. 一个更健康的最小结构

一个适合教学的待办应用最小结构通常可以是:

- `TodoDao`: 本地读写任务。
- `TodoRepository`: 统一数据入口。
- `TodoListViewModel`: 负责列表页 `uiState`。
- `TodoEditViewModel`: 负责新建 / 编辑页状态。
- `WorkManager`: 负责延迟提醒调度。
- `Notification`: 负责用户提醒触达。

重点不在于类名，而在于边界:

- 数据从哪里来。
- 页面状态由谁组织。
- 提醒由谁调度。
- 用户被怎样通知。

### 6. 为什么提醒功能是这个案例最好的“进阶点”

第一版待办应用即使不做提醒，也已经能覆盖很多基础能力。但一旦加入提醒，这个案例就会自然把更多系统能力串起来:

- 用户在编辑页设置提醒时间。
- ViewModel 将其作为状态的一部分管理。
- Repository 保存提醒配置。
- WorkManager 负责在合适时间触发。
- Notification 负责把事件展示给用户。

这条链路非常适合教学，因为它把页面、数据、后台、系统通知放到了一次完整动作里。

### 7. 更适合教材的迭代顺序

比起一次性做“大而全待办应用”，更适合教学的迭代顺序通常是:

第一版:

- 任务列表。
- 新建 / 编辑。
- 标记完成和删除。

第二版:

- 搜索、筛选、排序。
- 空态和错误态优化。

第三版:

- 提醒和通知。
- 小组件或快捷方式。

这种顺序的好处是，每一版都能清楚验证上一阶段学到的架构和系统能力，而不会一开始就被需求压垮。

### 8. 一个最小案例该怎样验证自己是否做对

待办案例做完后，最值得检查的不是“功能齐不齐”，而是:

- 列表更新是否真的依赖单向数据流。
- 页面状态是否已经统一到 `uiState`。
- 本地数据是否足以支撑无网可用。
- 提醒是否真正通过后台机制而不是页面临时逻辑触发。

如果这些结构问题是对的，这个案例就已经达到了教学目标；反过来，即使功能再多，结构混乱也很难称为好案例。

### 9. 实践任务

起点条件:

- 已有一个空项目或基础应用骨架。

步骤:

1. 先写出待办应用的最小用户动作清单。
2. 画出列表页和编辑页各自的 `uiState`。
3. 用 Room 落地最小本地数据层。
4. 用 ViewModel + StateFlow 驱动页面状态。
5. 在第二阶段再把提醒能力接入 WorkManager 与通知。

预期结果:

- 你会真正用一套完整案例串起前面学过的核心能力。
- 你会清楚感受到本地优先和状态建模的价值。
- 你会更容易向更复杂项目过渡。

自检方式:

- 你能解释为什么待办应用适合本地优先。
- 你能判断哪些信息属于实体，哪些属于页面状态。
- 你能说明提醒功能为什么应交给后台调度而不是页面临时逻辑。

调试提示:

- 列表和编辑页共享太多字段，优先重新拆分页面状态。
- 勾选完成后页面不能立即反映，优先检查数据流是否单向。
- 提醒功能必须依赖页面打开才生效，说明后台边界还没立住。

### 10. 常见误区

- 一开始就把案例做成“大而全”的任务管理系统。
- 只关注数据库 CRUD，不认真设计页面状态。
- 把提醒逻辑直接写在页面层。
- 本地数据和 UI 状态混在一起。

## 小结

待办应用最适合作为综合案例，不是因为它简单，而是因为它足够小、足够真、足够能把现代 Android 主线串起来。只要你能在这个案例里真正理清本地数据、页面状态、后台调度和系统通知的边界，后面面对更复杂的应用场景时，就已经有了一套可迁移的思考方式。
