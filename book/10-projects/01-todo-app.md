# 待办应用案例

待办应用之所以适合作为 Android 综合案例，并不是因为它“简单”，而是因为它恰好能把大量真实能力放进一个足够克制的产品场景中：列表展示、增删改查、本地存储、状态管理、日期提醒、过滤与排序、持久化、通知，甚至还可以逐步引入同步和小组件。它不像聊天和复杂内容分发那样一开始就要求高强度网络与实时性，却又足够接近真实应用的工程形态。

因此，这一章的目标不是再做一个“功能清单式 Demo”，而是演示如何把前面学过的页面层、数据层、状态流、Room、WorkManager、通知和发布意识真正串起来。

## 学习目标

- 理解为什么待办应用是很好的 Android 综合练习。
- 理解如何从需求拆解到模块、状态和数据模型设计。
- 理解本地优先、页面状态和提醒能力如何组合。
- 学会用一个小而完整的项目串联前面章节。

## 前置知识

- 已理解 ViewModel、Room、StateFlow、Repository 和 WorkManager。
- 已能完成基本列表页、表单页和本地存储能力。

## 正文

### 1. 待办应用最适合练什么

一个成熟的待办应用至少可以覆盖这些核心主题：

- 列表与详情的基本 UI 组织。
- 本地数据建模与持久化。
- 表单输入和状态保存。
- 已完成 / 未完成、筛选和排序等状态转换。
- 到期提醒、通知和后台触发。

这说明它虽然业务简单，却足以覆盖大量 Android 核心能力，因此非常适合作为“从教材知识到小型产品”的第一站。

### 2. 先从需求语义出发，而不是先从数据库表开始

设计待办应用时，很多人会一上来就创建数据库实体，但更好的顺序通常是：

1. 明确用户能做什么。
2. 明确页面有哪些核心状态。
3. 再决定数据模型和持久化结构。

例如，一个任务至少会有标题、完成状态、可选截止时间、可选备注，以及某种排序或筛选需要的字段。只要你先从这些产品语义出发，后面的 Room 实体和页面状态才不会脱节。

### 3. 待办应用天然适合本地优先

待办类功能最符合“本地优先”思路，因为用户通常希望：

- 打开应用立即看到已有任务。
- 没有网络时仍然能添加和勾选任务。
- 数据操作能即时反映到列表。

这也是为什么待办案例特别适合用 Room、Repository 和 Flow 串起一条清晰的数据主线。即使后续再引入云同步，本地也仍然应保持强可用。

### 4. 页面状态比“几个布尔值”更重要

待办应用看似简单，但如果只用若干散落变量去管理页面，很快就会失控。例如：

- 列表为空和列表加载失败不是一回事。
- 新建任务和编辑任务虽然复用表单，但状态语义不同。
- 过滤条件和排序条件会改变列表展示，但不应直接污染实体层。

因此，这个案例特别适合练习统一的 `uiState` 思维，而不是零散变量拼页面。

### 5. 通知和后台提醒可以如何合理加入

如果待办应用支持截止时间提醒，那么它就是引入通知和 WorkManager 的理想时机。但关键不是“功能更多了”，而是要坚持边界：

- 用户应主动设置提醒。
- 通知应准确落到具体任务。
- 后台触发应交给合适的调度机制。

这让待办案例天然成为连接本地数据、系统组件和后台工作的好样本。

### 6. 一个合理的最小架构

对一个现代待办应用来说，更合理的最小结构通常是：

- `TodoDao` 负责本地读写。
- `TodoRepository` 负责统一数据入口。
- `TodoListViewModel` 负责列表状态。
- `TodoEditViewModel` 负责新建与编辑状态。
- `WorkManager` 负责延迟提醒。
- `Notification` 负责用户触达。

这个结构的重点不在“类名标准”，而在于边界清楚。只要边界清楚，案例就不容易退化成巨型 Activity。

### 7. 从第一版到第二版的合理演进

第一版待办应用完全可以只做：

- 本地任务列表。
- 新建、编辑、删除。
- 已完成切换。

第二版再逐步加入：

- 搜索与过滤。
- 到期提醒。
- 小组件。
- 多端同步。

这种演进方式比一开始全塞进去更合理，也更符合真实产品迭代路径。

### 8. 实践任务

起点条件：

- 已有最小 Android 应用骨架，或准备开始第一个综合项目。

步骤：

1. 先写需求而不是先写数据表，明确任务对象和主要页面。
2. 用 Room 建立最小本地数据层。
3. 用 ViewModel + StateFlow 建立列表页与编辑页状态。
4. 在列表页中实现筛选、完成状态切换和空态显示。
5. 为其中一个任务增加提醒能力，并通过通知落地。

预期结果：

- 你能把前面章节的能力组合成一个真实可用的小应用。
- 你会真正体验到本地优先、状态建模和后台提醒如何协作。
- 你会为后续更复杂案例打下实践基础。

自检方式：

- 你能解释：为什么待办应用更适合本地优先而不是先做重网络。
- 你能判断：哪些逻辑属于实体层，哪些属于页面状态层。
- 你能说出：为什么提醒功能应由调度和通知协同，而不是 UI 临时控制。

调试提示：

- 如果列表页和编辑页共用过多状态，优先重新区分它们的职责。
- 如果任务操作后列表不能及时更新，优先检查数据流是否仍然单向。
- 如果提醒功能必须依赖页面打开才有效，说明后台调度边界还没建立。

### 9. 常见误区

- 一开始就把待办应用做成“大而全”。
- 只关注数据库 CRUD，不认真建模页面状态。
- 把提醒逻辑写在页面层，而不是交给后台机制。
- 本地数据和 UI 状态边界混在一起。

## 小结

待办应用案例的价值，不在于功能数量，而在于它足够小、足够完整、足够适合练习现代 Android 主线。只要你能把本地数据、页面状态、后台提醒和系统组件真正串起来，这个案例就已经完成了它最重要的教学目标。
