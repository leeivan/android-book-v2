# Todo App

待办应用之所以适合作为 Android 综合案例，并不是因为它“简单”，而是因为它把大量真实能力放进了一个足够克制、足够能看清边界的产品里。你可以在它身上练习列表、表单、状态管理、本地存储、提醒、通知，甚至再往上扩展同步、小组件和多设备支持。它不像聊天和复杂内容分发那样一上来就要求高实时性，却已经足够接近真实应用的工程形态。

参考目录中的项目型书籍都很重视这一类“小而完整”的案例。它们的共同方法不是先堆功能，而是先把数据边界、页面状态和实现顺序安排清楚。本章就用这个思路来组织一个教材版 Todo App：不是追求一次做全，而是追求每一步都能验证前面学到的架构主线。

## 学习目标

- 理解待办应用为什么适合作为现代 Android 综合案例的第一站。
- 学会从用户动作拆出页面状态、数据模型和后台提醒能力。
- 学会用 Room、ViewModel、StateFlow、WorkManager 和通知构成一个完整小项目。
- 建立“先立结构，再扩需求”的项目推进顺序。

## 前置知识

- 已理解 Room、ViewModel、StateFlow、Repository、WorkManager 和通知。
- 已完成基本列表页、表单页或局部业务练习。

## 正文

### 1. 先定义用户动作，而不是先写数据库

很多人一做综合案例，第一步就是新建 `TodoEntity`。这不是错，但顺序更好的做法通常是先问：用户到底会做什么。对一个最小待办应用来说，最核心的动作通常是：

- 新建任务。
- 修改任务。
- 标记完成或取消完成。
- 删除任务。
- 为任务设置提醒时间。

只要动作先明确，后面的页面状态和数据结构就不会脱离产品语义。相反，如果一开始就围绕数据库字段设计，案例很容易退化成 CRUD 演示，而不是一个真正的应用。

### 2. 为什么 Todo App 天然适合本地优先

待办类产品最重要的体验之一，就是“打开就能立刻用”。用户不会接受：

- 打开列表先转圈等网络。
- 没网时完全不能记录任务。
- 勾选完成后要等服务器响应才能看到变化。

这让待办应用成为本地优先最自然的教材案例。Room 不是补充缓存，而是第一可信来源；Repository 不是顺手包一层，而是负责统一读写入口；ViewModel 不是多活一会儿的对象，而是把列表页和编辑页状态真正组织起来的地方。

### 3. 一个适合教材的最小结构

如果你想把这个案例做成一套可维护的小项目，最小结构可以非常朴素：

- `TodoEntity` / `TodoDao`：承载本地任务数据。
- `TodoRepository`：统一任务读写与查询。
- `TodoListViewModel`：负责列表页状态。
- `TodoEditViewModel`：负责新建与编辑页状态。
- `ReminderScheduler`：封装提醒调度。
- `WorkManager` + `Notification`：实现延迟提醒。

重点不在于类名，而在于边界是否清楚：页面不直接碰数据库，提醒不直接写在页面层，页面状态不直接等同于持久化实体。

### 4. 页面状态和实体模型为什么必须分开

这是 Todo App 最适合训练的一件事。任务实体只需要关心任务本身，例如标题、备注、完成状态、提醒时间；而页面状态还会关心：

- 当前是不是首次加载。
- 当前筛选条件是什么。
- 编辑页是在新建还是修改。
- 表单校验有没有通过。
- 提醒设置是否刚刚成功。

这些都不是实体层该承担的内容。只要一开始就把实体和页面状态混在一起，案例越往后做越容易变成一团互相覆盖的布尔值和临时变量。

### 5. 更适合教学的实现顺序

参考书里的项目案例普遍有一个优点：不会一开始就追求大而全。Todo App 更适合按下面的顺序推进。

第一阶段，先做最小核心：

- 任务列表页。
- 新建 / 编辑页。
- 本地持久化。

第二阶段，再加状态复杂度：

- 搜索、筛选和排序。
- 空态、错误态和加载态。
- 列表与编辑页之间的数据回流。

第三阶段，再加系统能力：

- 任务提醒。
- 通知跳转。
- 小组件或快捷方式。

这种顺序的好处是，每一步都能验证前一阶段的结构是否站得住，而不是在业务尚未稳定时就同时引入太多系统能力。

### 6. 为什么提醒功能是这个案例最好的进阶点

很多初学者会把提醒理解成“最后加一个通知”。其实它正好是这个案例最有教学价值的扩展点。因为一旦加提醒，你必须把整条链路连起来：

1. 用户在编辑页设置提醒时间。
2. ViewModel 把提醒作为页面状态的一部分处理。
3. Repository 保存提醒配置。
4. 调度层把配置转成后台任务。
5. 到点后由通知系统把任务带回用户面前。

这件事非常适合训练“页面、数据、后台、系统通知如何协作”。它不是额外点缀，而是把整本书主线串起来的一次完整练习。

### 7. 一个更像真实项目的目录建议

如果你不想案例最后变成“所有文件都塞在 app 里”，可以用一个轻量目录方案：

- `feature/todo-list`
- `feature/todo-edit`
- `core/database`
- `core/model`
- `core/notification`
- `core/work`

这还不必上升到复杂模块化，但已经足够让你习惯把“功能逻辑”和“基础设施能力”分开。等项目进一步增大，再考虑把它们提升为真正的 Gradle 模块，会自然得多。

### 8. 一个案例做完后，最值得检查的不是功能数，而是结构

Todo App 完成后，最有价值的复盘问题通常不是“还差不差日历视图、标签、云同步”，而是：

- 列表更新是不是依赖清晰的数据流。
- ViewModel 是否真正持有了页面状态。
- Room 是否承担了可信数据来源角色。
- 提醒是否由后台机制驱动，而不是页面临时逻辑。

如果这些问题回答得清楚，这个案例就已经达到非常高的教学价值。功能多寡反而是第二位的。

### 9. 实践任务

起点条件：

- 已有一个空项目或基础应用骨架。

步骤：

1. 用一句话写出 Todo App 的产品定位，只保留核心动作。
2. 为列表页和编辑页分别定义 `UiState`，不要一开始就只画数据库表。
3. 用 Room 实现最小任务读写链路，并用 StateFlow 驱动列表刷新。
4. 在不引入提醒功能之前，先确保列表、新建、编辑、完成状态都能闭环。
5. 第二阶段再把提醒接入 WorkManager 与通知系统。

预期结果：

- 你会真正完成一套“结构先行”的小项目。
- 你会体会到本地优先和页面状态建模在真实案例里的价值。
- 你会为更复杂项目建立可复用的项目组织方式。

自检方式：

- 你能解释为什么 Todo App 适合本地优先。
- 你能区分哪些信息属于实体，哪些属于页面状态。
- 你能说明为什么提醒功能更适合交给后台调度而不是页面逻辑。

调试提示：

- 列表和编辑页共享太多字段，优先重拆页面状态。
- 勾选完成后 UI 变化不稳定，优先检查可信数据来源是否唯一。
- 提醒只能在页面打开时生效，说明后台边界还没立住。

### 10. 常见误区

- 一开始就把案例做成大而全的任务管理系统。
- 只练数据库 CRUD，不认真设计页面状态。
- 把提醒逻辑直接写在页面层。
- 本地数据和 UI 状态混成一团。

## 小结

Todo App 最适合作为综合案例，不是因为它简单，而是因为它足够小、足够真、足够能把现代 Android 主线串起来。只要你能在这个案例里真正理清本地数据、页面状态、后台调度和系统通知的边界，后面面对更复杂的内容型应用和实时应用时，就已经具备了一套可靠的起步方法。
