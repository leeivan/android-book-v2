# Todo App

待办事项应用是 Android 学习里最经典的综合案例之一，不是因为它“简单”，而是因为它几乎可以用最小复杂度串起界面、状态、数据持久化、后台提醒和交互反馈等核心能力。它非常适合作为本书项目部分的第一个完整案例，因为它既足够小，便于读者真正做完，又足够真实，足以暴露结构设计、状态流转和数据组织中的关键问题。

## 学习目标

- 用一个最小项目串联列表、表单、数据存储和状态管理。
- 理解中小型应用从功能需求到工程分层的基本过程。
- 建立“先做稳定主链，再做功能扩展”的项目实现习惯。
- 为后续网络型和实时型项目建立可复用的工程骨架。

## 前置知识

- 已了解 UI、Room、ViewModel、Flow 和 WorkManager 等核心章节。
- 已知道一个完整应用需要同时考虑功能和结构。

## 正文

### 1. 为什么先做 Todo App

Todo 应用的优势在于需求边界清晰，核心功能天然适合现代 Android 技术栈：

- 列表展示适合 `RecyclerView` 或 Compose 列表。
- 数据持久化适合 Room。
- 页面状态管理适合 ViewModel。
- 提醒和延期处理适合 WorkManager 与通知。

这使它非常适合作为“第一套完整工程”。它不会一开始就把你拖进复杂网络协议和同步冲突，但又足够真实，能让你练到一个 Android 应用最关键的主链。

### 2. 这个案例真正训练的是什么

很多人以为 Todo 项目只是“做个列表”。实际上它训练的是一条完整工程链路：

- 用户如何新增和修改数据。
- 数据如何进入本地持久层。
- ViewModel 如何组织页面状态。
- 页面如何根据状态渲染列表、空状态和错误反馈。
- 后台提醒如何与数据状态保持一致。

也就是说，它练的不只是功能，而是“一个简单应用应如何被组织”。

### 3. 功能范围应如何控制

一个适合作为教学案例的 Todo App，不需要一次做成复杂产品。最小范围通常包括：

- 新增待办。
- 编辑和删除。
- 标记完成状态。
- 列表筛选或排序。
- 可选的提醒能力。

重点是把主链路打通，而不是不停堆功能。很多学习项目失败，不是因为不会写，而是因为功能边界从一开始就失控。

### 4. 推荐的结构设计

这个案例最适合用一条清晰的现代主线来组织：

- UI 层负责列表、表单和交互反馈。
- ViewModel 负责状态与事件编排。
- Repository 负责统一数据入口。
- Room 负责本地持久化。
- WorkManager 负责提醒或延迟处理。

这样设计的价值在于，读者能看到前面章节中的概念如何在真实项目里分工协作，而不是一堆知识点停留在孤立示例里。

### 5. 为什么状态流是这个案例的核心

Todo 应用虽然业务简单，但状态变化非常频繁：新增、完成、编辑、删除、筛选、搜索、提醒状态更新。正因为变化多，它特别适合训练“单向数据流”和“状态驱动界面”思维。

一旦状态设计混乱，这个应用会很快变成回调堆积场。也正因如此，Todo 项目很适合作为学习 ViewModel 与 StateFlow 的第一个综合练习。

### 6. 数据建模为什么比 UI 更早决定上限

很多人做这个项目时，最先投入大量时间的是界面样式；但从长期维护角度看，真正决定项目是否能扩展的，是数据建模是否合理。例如：

- 待办项是否需要分类。
- 提醒时间是否允许为空。
- 完成状态、排序和筛选如何表达。

这些问题一开始看似简单，实际会直接影响 Room 设计、状态流建模和后续功能扩展。

### 7. 实现顺序为什么建议分阶段

一个更稳妥的顺序通常是：

1. 先实现新增、列表展示和完成状态切换。
2. 再补本地持久化，让应用重启后状态仍在。
3. 最后再加入筛选、排序、提醒和扩展功能。

这个顺序的重要性在于：你会先得到一个可靠主链，而不是一开始就把自己埋进多个未完成分支里。

### 8. 如何把这个案例继续扩展

当最小版本完成后，你可以继续加入：

- 分类和标签。
- 截止时间与提醒。
- 多设备同步。
- 小组件或快捷方式。
- 深色主题与无障碍改进。

这些扩展会把它从“练习项目”推向“完整作品集项目”，也能帮助你把前面章节逐步串回真实工程。

### 9. 这个案例最容易暴露哪些结构问题

Todo 项目看起来小，但很容易暴露以下问题：

- 页面层过重。
- 状态和事件混在一起。
- 数据库直接被页面访问。
- 列表更新与数据身份设计不清。

这正是它适合作为全书第一个完整案例的原因。问题足够小，可控；但类型足够典型，具有代表性。

### 10. 最小实践任务

1. 先只实现新增、列表展示和完成状态切换。
2. 再补数据持久化，让应用重启后仍保留状态。
3. 最后再考虑提醒、筛选和排序，不要一开始同时铺开所有需求。
4. 为这个项目画出 UI、ViewModel、Repository 和 Room 的最小关系图。

### 11. 常见误区

- 一开始就做复杂功能矩阵，导致主链迟迟不稳定。
- UI、状态和数据访问写在同一层。
- 只追求“能运行”，不追求结构清晰。
- 没做提醒功能就先过度设计同步和多端。

## 小结

Todo App 是最适合训练完整工程思维的入门案例。它的真正价值不在于题材本身，而在于它能以最小复杂度把现代 Android 的主线能力连接起来，并让结构问题尽早显形。
