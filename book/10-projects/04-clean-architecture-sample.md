# Clean Architecture 示例

前面几个项目更关注具体业务场景，而本章的目标是把这些经验抽象成一套可复用的结构样板。所谓 Clean Architecture，在 Android 里不应被理解成教条式分层，而应被理解成一种帮助你控制依赖方向、隔离变化和提高可测试性的组织方法。本章的价值，不在于再画一张“经典三层图”，而在于把前面项目中已经出现的结构规律明确说出来。

## 学习目标

- 理解一个中型 Android 项目如何组织分层和依赖方向。
- 看清 UI、domain、data 和 infrastructure 在样板工程中的分工。
- 理解 Clean Architecture 的收益和使用边界。
- 学会从“依赖稳定性”而不是“层数多少”评价结构。

## 前置知识

- 已阅读架构、模块化和至少一个综合项目章节。
- 已知道项目复杂度增长后，仅靠页面层堆代码会迅速失控。

## 正文

### 1. 为什么需要一个结构样板

当项目还小时，很多问题可以靠经验和局部约定解决；但随着业务增长、模块增多、多人协作深入，缺少统一结构的代价会越来越大：

- 新功能不知道落哪。
- 边界越来越模糊。
- 重构影响面越来越难判断。
- 测试入口不清晰。

结构样板的意义，就是让新功能接入、旧功能演进和多人协作都拥有更稳定的轨道。

### 2. Clean Architecture 在 Android 中的实用理解

它的核心不在于层数多，而在于依赖方向稳定。常见理解可以简化为：

- UI 层负责展示和交互。
- Domain 层负责业务规则和用例编排。
- Data 层负责数据来源和实现细节。

只要依赖方向清楚，具体是不是严格三层、四层，并不是最关键的问题。

### 3. 这个案例最想回答什么

本章真正要回答的是：

- 一个页面事件如何进入 UseCase。
- UseCase 如何调用 Repository。
- Repository 如何协调本地与远程数据。
- 状态如何一路返回 UI。

只有这条链路真实存在，读者才能判断这套结构是否真的有价值，而不是只是目录看起来更整齐。

### 4. 为什么它特别适合作为“样板工程”而不是“第一项目”

Clean Architecture 的样板价值，在于帮助你总结规律；但如果把它作为第一项目模板，很多人会只看到大量目录、接口和层次，而看不见它们为什么存在。

因此更合理的学习顺序通常是：

1. 先通过 Todo、新闻、聊天等项目看到复杂度如何增长。
2. 再回到本章，把这些复杂度重新收束成更稳定的结构样板。

这样你看到的就不是“架构模板”，而是“复杂度治理工具”。

### 5. 什么时候不必过度引入

如果项目很小、需求变化快、团队人数有限，过于完整的 Clean Architecture 结构可能会让工程显得笨重。它的价值主要体现在：

- 中大型项目。
- 多人协作。
- 业务规则逐渐复杂。
- 测试和模块化要求提高。

因此这套结构更像“可升级的组织方式”，而不是所有项目一开始就必须照搬的模板。

### 6. 如何判断这套结构是否真的成功

一个好用的结构样板，至少应做到：

- 新增功能时落点明确。
- 业务逻辑不依赖具体 UI 实现。
- 数据来源可替换。
- 测试入口清晰。
- 依赖方向稳定，不反向污染。

如果只是目录更多、文件更多，但这些目标都没实现，那它就只是形式上的复杂化。

### 7. 样板工程应该保留哪些“教学可读性”

一个适合教学的 Clean Architecture 示例，不能只顾结构完整，还应注意：

- 类和层次不要过度冗长。
- 用一个清晰业务链路贯穿说明。
- 明确哪些层是“当前需要”，哪些层是“未来可扩展”。

否则读者很容易把它误以为“所有项目都必须先写出这么多骨架”。

### 8. 最小实践任务

1. 把 Todo 或新闻案例按 UI、domain、data 三层重新画一次依赖关系。
2. 选一个用户动作，追踪它从界面到数据层再返回界面的完整路径。
3. 判断这条路径中哪些依赖是稳定的，哪些依赖还可以继续收敛。
4. 检查你的项目里是否存在“名义上分层，实际上依赖到处穿透”的问题。

### 9. 常见误区

- 把 Clean Architecture 当成固定模板死记。
- 为小项目过早引入过重结构。
- 分了层却没有真正控制依赖方向。
- 用大量接口和空壳类代替真正的职责治理。

## 小结

Clean Architecture 示例的价值，是帮助你理解复杂 Android 工程应如何组织，而不是要求你机械复制某种目录形状。真正值得保留的，是清晰依赖方向、稳定边界和可演进的结构思维。
