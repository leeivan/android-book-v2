# Clean Architecture 示例

很多读者接触 `Clean Architecture` 时，会被一大堆同心圆图、层名和依赖倒置术语包围。结果最常见的两种反应是：要么觉得它高深抽象、离实际项目很远；要么把它机械翻译成“每个功能都必须有 entity / usecase / repository / datasource 四件套”。这两种理解都不够好。真正有价值的 Clean Architecture，不是类图模板，而是一种在复杂项目里持续保护边界的方式。

把它放在综合项目部分的最后一章，意义就在于帮助你回头看整本书的主线：页面状态、ViewModel、Repository、UseCase、依赖注入、模块化，这些东西为什么会被反复强调。参考目录中的几本项目书也都体现了同一个事实：真正成熟的项目，并不是“层越多越高级”，而是“边界越清楚，变化越可控”。

## 学习目标

- 理解 Clean Architecture 真正要保护的是依赖方向和业务边界。
- 理解它和 MVVM、Repository、UseCase、Hilt、模块化之间的关系。
- 学会判断哪些边界值得抽离，哪些地方不该过度模板化。
- 能把前面章节的知识组织成一套更稳定的项目结构。

## 前置知识

- 已理解 MVVM、Repository、UseCase、Hilt、模块化和综合案例主线。
- 已完成至少一个完整案例或主要章节学习。

## 正文

### 1. 为什么项目一复杂，就开始重新谈“边界”

项目小时，页面里多写一点逻辑、数据层里多做一点业务判断，问题通常还能忍。一旦系统复杂度上升，几个风险会迅速放大：

- 页面层和业务规则互相污染。
- 数据来源策略和页面状态逻辑混在一起。
- 依赖方向越来越乱。
- 功能越加越多，修改影响越来越难控制。

Clean Architecture 真正针对的，就是复杂度上升以后边界开始失控的时刻。

### 2. 它的核心不是“层数”，而是“依赖方向”

教材里最值得记住的，不是它有几层，而是一个更基础的判断：越靠近业务核心的部分，越不应该依赖越外层、越易变的实现细节。换句话说：

- UI 可以依赖业务动作。
- 业务动作可以依赖抽象的数据边界。
- 具体网络、数据库和框架细节不应反过来绑住业务核心。

这也是为什么 Clean Architecture 会自然和依赖注入、UseCase、接口抽象、模块化联系在一起。它们都在帮助依赖方向保持健康。

### 3. 它和本书前面内容并不是两套体系

如果你按本书顺序学下来，会发现很多内容其实都在为 Clean Architecture 打基础：

- `ViewModel` 让页面状态有清晰承载点。
- `Repository` 让数据来源和上层解耦。
- `UseCase` 让复杂业务动作从页面层抽离。
- `Hilt` 让依赖关系的装配变得可控。
- 模块化让边界从“概念”变成“构建与依赖上的现实存在”。

也就是说，Clean Architecture 不是忽然多出来的一套新规则，而是把前面这些结构进一步系统化。

### 4. 它最容易被误用成“模板工程”

最常见的误用，就是不管项目规模和问题类型，先把每个功能都拆成：

- entity
- repository
- datasource
- usecase
- mapper
- interface

然后层层转发。表面看很整齐，实际很多层没有承接真实复杂度，只是在制造跳转成本。教材里必须强调一点：Clean Architecture 不是要求你拥有更多类，而是要求你只在边界真的值得保护时，才引入额外抽象。

### 5. 一个更现实的判断：哪些边界真的值得“清理”

更值得认真保护的，通常是这些地方：

- 核心业务规则，不能被 UI 和框架细节绑死。
- 复杂业务动作，需要在多个页面或入口复用。
- 数据来源策略，不能直接泄漏到页面层。
- 模块间接口，需要长期稳定。

不太值得过度抽象的，通常是：

- 还在快速试错的小功能。
- 没有业务规则的简单透传。
- 只是为了“看起来分层”而存在的包装层。

这类判断能力，比会背“Clean Architecture 标准图”更重要。

### 6. 一个教学案例怎样更像“干净架构”

以新闻应用或待办应用为例，更接近 Clean Architecture 的结构通常表现为：

- 页面只消费 `uiState` 和事件。
- ViewModel 只组织页面动作与页面状态。
- UseCase 组织复杂业务流程。
- Repository 统一数据来源与同步策略。
- Retrofit、Room、通知调度等都留在更外层实现细节。

重点不是一定要出现每一层，而是依赖方向是否始终从外到内使用、从内到外不反依赖实现细节。

### 7. 为什么它和测试、模块化关系特别强

只要边界清楚、依赖方向健康、实现与抽象分开，测试就会容易很多，模块化也会更自然。反过来，一个项目如果一切都紧紧绑在页面和框架层上，不仅架构不“干净”，测试和拆模块也会一起变难。

这也是参考书项目案例最值得借鉴的一点：好的架构从来不只是类图好看，而是让替换、扩展、测试和团队协作变得更容易。

### 8. 一个更适合落地的实践顺序

如果你想在自己的项目里引入更“干净”的结构，不要一上来就全面重构。更成熟的顺序通常是：

1. 先找出当前最痛的一条功能流。
2. 画出它现在的依赖方向。
3. 找出页面层是否直接依赖了实现细节。
4. 只抽出真正值得复用或稳定的边界。
5. 删除那些只是为了形式存在、却没有实际收益的中间层。

这样做的好处是，你不是在“套模板”，而是在解决真实复杂度问题。

### 9. 实践任务

起点条件：

- 已有一个使用 ViewModel、Repository 和部分 UseCase 的项目或案例。

步骤：

1. 选一个复杂一点的功能流，例如搜索、同步、提醒或多页面共享数据。
2. 画出当前依赖方向图，标出页面层直接碰到的实现细节。
3. 判断哪些逻辑值得抽成更稳定的 UseCase 或抽象接口。
4. 删除一个只是为了“看起来有层次”却没有真实价值的中间层。
5. 再看修改后的结构是否让测试和替换变得更容易。

预期结果：

- 你会把 Clean Architecture 看成边界治理，而不是模板集合。
- 你能更清楚地区分“值得抽象”和“过度抽象”。
- 你能把本书前面章节的结构主线真正串起来。

自检方式：

- 你能解释为什么依赖方向比层数更关键。
- 你能判断某层抽象是否真的带来了边界收益。
- 你能说明它为什么与测试和模块化紧密相关。

调试提示：

- 类很多、层很多，但谁也说不清为什么存在，通常说明已经模板化了。
- 页面仍直接理解网络、数据库和系统细节，说明边界并没有真正清理出来。
- 一个小项目为了追求“纯净”加了太多层，往往比不加更难维护。

### 10. 常见误区

- 把 Clean Architecture 理解成固定分层模板。
- 不顾项目规模，强行让每个功能都套满所有层。
- 只谈层数，不谈依赖方向。
- 抽象很多，却没有真正减少耦合。

## 小结

Clean Architecture 真正要保护的，不是某张标准图，而是复杂项目里的依赖方向和业务边界。只要把它看成“在真正痛的地方清理结构”的方法，而不是“所有项目必须照抄的模板”，它就会和本书前面讲过的 ViewModel、Repository、UseCase、Hilt、模块化自然连成一套完整的工程思路。
