# Clean Architecture 示例

很多读者接触 `Clean Architecture` 时，会被一大堆同心圆图、层名和依赖倒置术语包围。结果最常见的两种反应是: 要么觉得它高深抽象、离实际项目很远；要么把它机械翻译成“每个功能都必须有 entity / usecase / repository / datasource 四件套”。这两种理解都不够好。真正有价值的 Clean Architecture，不是类图模板，而是一种在复杂项目里持续保护边界的方式。

把它放在综合案例最后一章，意义就在于帮助你回头看整本书的主线: 页面状态、ViewModel、Repository、UseCase、依赖注入、模块化，这些东西为什么会被反复强调。Clean Architecture 不是替代它们，而是把它们重新组织成一套更稳定的边界关系。

## 学习目标

- 理解 Clean Architecture 真正要保护的是依赖方向和业务边界。
- 理解它和本书前面讲过的 MVVM、Repository、UseCase、模块化之间的关系。
- 理解为什么 Clean Architecture 不该被机械模板化。
- 学会用一个综合案例判断哪些边界值得被“清理”，哪些不必过度抽象。

## 前置知识

- 已理解 MVVM、Repository、UseCase、Hilt、模块化和综合案例。
- 已完成至少一个完整案例或主要章节学习。

## 正文

### 1. 为什么很多项目一复杂，就开始重新谈“边界”

项目小时，页面里多写一些逻辑、数据层多做一点业务判断，问题通常还能忍。一旦系统复杂度上升，几个风险会迅速放大:

- 页面层和业务规则互相污染。
- 数据来源策略和页面状态逻辑混在一起。
- 依赖方向越来越乱。
- 功能越加越多，修改影响越来越难控制。

Clean Architecture 真正针对的，就是这种复杂度上升后的边界失控。

### 2. Clean Architecture 的核心，不是层数，而是依赖方向

它最值得被记住的，不是“有几层”，而是一个判断: 越靠近业务核心的部分，越不应该依赖越外层、越易变的实现细节。

换句话说:

- UI 可以依赖业务动作。
- 业务动作可以依赖抽象的数据边界。
- 具体网络、数据库和框架细节不应反过来绑住核心业务。

这也是为什么它会和依赖注入、接口抽象、UseCase、模块化自然联系在一起。它们都在帮助依赖方向保持健康。

### 3. 它和本书前面内容到底是什么关系

如果你按本书顺序学下来，会发现很多内容其实都在为 Clean Architecture 打基础:

- `ViewModel` 让页面状态有清晰承载点。
- `Repository` 让数据来源和上层解耦。
- `UseCase` 让复杂业务动作从页面层抽离。
- `Hilt` 让依赖关系以更可控的方式装配。
- 模块化让边界变成构建和依赖上的现实存在。

也就是说，Clean Architecture 不是突然多出来的一套新规则，而是把这些边界进一步系统化。

### 4. 为什么它最容易被误用成模板工程

最常见的误用，就是不管项目规模和问题类型，先把每个功能都拆成:

- entity
- repository
- datasource
- usecase
- mapper
- interface

然后层层转发。表面上很整齐，实际上很多层没有承接真实复杂度，只是在制造跳转成本。这就是“形式上很 clean，阅读和维护上却很脏”的典型情况。

所以教材里必须强调: Clean Architecture 不是要求你拥有更多类，而是要求你只有在边界真正值得保护时，才去引入额外抽象。

### 5. 一个更现实的判断: 哪些边界真的值得清理

更值得用 Clean Architecture 思路认真保护的，通常是这些地方:

- 核心业务规则，不能被 UI 和框架细节绑死。
- 复杂业务动作，需要在多个入口复用。
- 数据来源策略，不能直接泄漏到页面层。
- 模块间接口，需要长期稳定。

不太值得过度抽象的，则往往是:

- 一次性过渡实现。
- 没有复杂规则的简单透传。
- 还在快速变化、边界未成熟的小功能。

### 6. 一个综合案例怎样更像“干净架构”

以新闻或待办应用为例，更接近 Clean Architecture 的结构通常表现为:

- 页面只消费 `uiState` 和事件。
- ViewModel 只组织页面状态和页面动作。
- UseCase 组织复杂业务流程。
- Repository 组织数据来源和同步策略。
- 具体 Room、Retrofit、通知调度等都留在更外层实现。

这个结构的重点不在于一定要出现所有层，而在于依赖方向始终是从外到内使用、从内到外不反依赖实现细节。

### 7. Clean Architecture 为什么和测试也强相关

只要边界清楚、依赖方向健康、实现和抽象分开，测试就会容易很多。反过来，一个项目如果一切都紧紧绑在框架和页面层上，不仅架构不“干净”，测试也会变得异常困难。

这说明 Clean Architecture 的真正收益，不只是代码看起来更有层次，而是让变化、替换和测试都更可控。

### 8. 什么时候不要急着用它

如果项目还很小、业务边界还没稳定、团队也还在快速试错，那么过早追求完整 Clean Architecture 反而可能让开发效率下降。成熟的做法通常是:

- 先把真正痛的边界收出来。
- 再让结构随着复杂度自然增长。

而不是一开始就照着模板把所有层都摆满。

### 9. 实践任务

起点条件:

- 已有一个使用 ViewModel、Repository 和部分 UseCase 的项目或案例。

步骤:

1. 选一个复杂一点的功能流。
2. 画出当前它的依赖方向图。
3. 找出哪些地方是页面层在直接依赖实现细节。
4. 判断哪些边界值得真正抽离为更稳定的抽象。
5. 删除一个只是为了“看起来分层”却没有实际价值的中间层。

预期结果:

- 你会把 Clean Architecture 看成边界治理，而不是模板集合。
- 你能更清楚地区分“值得抽象”和“过度抽象”。
- 你能把本书前面章节的内容更系统地串起来。

自检方式:

- 你能解释 Clean Architecture 最核心的其实是依赖方向。
- 你能判断某层抽象是否真的带来边界收益。
- 你能说明它为什么与测试和模块化紧密相关。

调试提示:

- 类很多、层很多，但谁也说不清为什么存在，通常说明已经模板化了。
- 页面仍直接理解网络、数据库细节，说明边界并没有真正清理出来。
- 一个小项目为了追求“纯净”加了太多层，往往比不加更难维护。

### 10. 常见误区

- 把 Clean Architecture 理解成固定分层模板。
- 不顾项目规模，强行让每个功能都套满所有层。
- 只谈层数，不谈依赖方向。
- 抽象很多，却没有真正减少耦合。

## 小结

Clean Architecture 真正要保护的，不是某张标准图，而是复杂项目里的依赖方向和业务边界。只要把它看成“在真正痛的地方清理结构”的方法，而不是“所有项目必须照抄的模板”，它就会和本书前面讲过的 ViewModel、Repository、UseCase、模块化自然连成一套完整的工程思路。
