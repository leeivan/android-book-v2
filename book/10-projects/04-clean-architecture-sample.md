# Clean Architecture 综合示例

前面几个案例已经分别练习了本地优先、网络驱动和实时状态主线，而 Clean Architecture 综合示例的目标，是把这些能力再往前推一步：不仅功能能工作，还要让项目在规模增长后依然保持可理解、可测试、可演进。这个案例并不是要求你把所有项目都做成“层数越多越高级”，而是帮助你看清：在业务复杂度真正上来之后，分层和依赖方向为什么会开始变得重要。

一个好的综合示例，重点不是类图看起来多完整，而是是否能回答这些问题：页面状态由谁组织，业务动作由谁表达，数据从哪里来，依赖如何注入，测试如何落地，模块边界是否清楚。只要这些问题能说清，架构就不会变成模板崇拜。

## 学习目标

- 理解 Clean Architecture 综合示例真正想训练什么。
- 理解 UI、Domain、Data 三层如何在 Android 中落地。
- 理解 UseCase、Repository、DI 和测试在综合项目里的协作方式。
- 知道什么时候该引入更完整的层次，什么时候不应过度设计。

## 前置知识

- 已理解 MVVM、Repository、UseCase、Hilt、测试和模块化主线。
- 已完成前面的至少一个小型项目案例。

## 正文

### 1. 综合示例真正训练的是“边界能否长期成立”

在小项目里，即使分层不够清楚，很多问题也不会立刻爆炸。但一旦项目开始增长，你就会逐渐遇到：

- 一个业务动作需要协调多个数据来源。
- 多个页面复用同一条业务规则。
- 测试越来越难写。
- 页面和数据层之间的边界越来越模糊。

综合示例的意义，就是用一个相对完整但仍可控的项目，让这些问题在可学习范围内真实出现，然后被结构化解决。

### 2. UI 层、Domain 层、Data 层在这里分别承担什么

一个更实用的理解方式是：

- UI 层关心屏幕状态、用户事件和渲染。
- Domain 层关心业务动作、规则和跨数据源编排。
- Data 层关心数据来源、缓存、同步和对外数据接口。

Clean Architecture 在 Android 中真正有价值的地方，不是“层名好看”，而是依赖方向足够明确：上层依赖抽象，下层负责实现。

### 3. UseCase 在综合项目里为什么开始变得重要

前面我们已经强调过，UseCase 不是所有项目都必须有。但在综合示例中，如果一个业务动作确实要协调多个 Repository、包含明确规则或会被多个页面复用，那么 UseCase 的价值会明显体现出来。

这也是综合示例比前面案例更强调 Domain 层的原因：它不是为了多一层，而是为了让复杂业务动作拥有独立位置。

### 4. 依赖注入与可测试性在这里第一次真正绑定

在小案例里，你可能觉得 Hilt 和测试只是“工程化加分项”。但在综合示例里，一旦对象关系和边界复杂起来，依赖注入和测试就不再是附加能力，而是结构能否成立的证明：

- 依赖能否清楚替换。
- ViewModel 和 UseCase 是否可独立测试。
- 数据层实现能否被隔离。

如果这些都做不到，就说明架构只是层数增加了，而没有真正带来维护收益。

### 5. 模块边界会开始真正有意义

综合示例也是第一次适合认真引入模块化思维的地方，因为这时项目通常已经具备：

- 多个功能域。
- 多个共享基础设施能力。
- 多人协作或至少多层依赖。

feature、core、data 这类模块划分不再只是概念，而会直接影响构建速度、测试隔离和依赖可见性。

### 6. 一个合理的综合项目结构

一个更健康的综合示例通常会包含：

- feature 模块或屏幕级 UI 模块。
- domain 中的业务动作和模型抽象。
- data 中的 repository 实现、本地与远程数据源。
- core 中的网络、数据库、设计系统和公共能力。

真正重要的是依赖方向清晰，而不是目录数量多少。

### 7. 这个案例不该怎么做

最容易走偏的方式包括：

- 为了“显得专业”给每个函数都加 UseCase。
- 每一层都只是机械转发，没有新增语义。
- 模块很多，但依赖关系仍然混乱。
- UI 层明明还在直接依赖数据实现，却自称已经做了 Clean Architecture。

这些问题说明，综合示例更应该训练判断力，而不是类模板复制能力。

### 8. 实践任务

起点条件：

- 已完成至少一个中小型 Android 项目，准备把它整理成更可维护的结构。

步骤：

1. 选一个复杂业务动作，判断它是否需要 UseCase。
2. 画出当前项目的 UI、Domain、Data 依赖方向图。
3. 检查哪些层之间仍存在不该有的直接耦合。
4. 选择一个模块或一条业务链进行最小重构，而不是一次性全量重写。
5. 为重构后的关键用例补上一组能证明边界有效的测试。

预期结果：

- 你会更清楚架构层次真正带来的收益和代价。
- 你能判断哪些项目已经值得进入更完整的分层设计。
- 你能把 UseCase、Repository、DI、测试和模块化真正串成系统。

自检方式：

- 你能解释：为什么综合示例重点是依赖方向，而不是层名。
- 你能判断：当前项目哪些地方属于“增加层数但没有增加清晰度”。
- 你能说出：为什么测试是验证架构是否成立的重要手段。

调试提示：

- 如果每层都只是在转发，优先检查是否真的需要那一层。
- 如果 Domain 层已经开始知道 UI 细节，说明依赖方向已经反了。
- 如果重构后仍然很难测试，通常说明边界还没有真正清楚。

### 9. 常见误区

- 把 Clean Architecture 学成固定模板。
- 所有项目无差别引入全套层次。
- 只增加目录和类名，不真正整理依赖方向。
- 认为只要有 UseCase 和 Repository 就已经架构化。

## 小结

Clean Architecture 综合示例的真正价值，不是展示“一个标准答案长什么样”，而是帮助你建立对复杂度、边界和依赖方向的判断能力。只要这种判断能力建立起来，你就能在不同规模项目里做出更合理的取舍，而不是机械套模板。
