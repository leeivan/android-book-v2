# Clean Architecture 示例

前面几个项目更关注具体业务场景，而本章的目标是把这些经验抽象成一套可复用的结构样板。所谓 Clean Architecture，在 Android 里不应被理解成教条式分层，而应被理解成一种帮助你控制依赖方向、隔离变化和提高可测试性的组织方法。

## 学习目标

- 理解一个中型 Android 项目如何组织分层和依赖方向。
- 看清 UI、domain、data 和 infrastructure 在样板工程中的分工。
- 明白 Clean Architecture 的收益和使用边界。

## 前置知识

- 已阅读架构、模块化和至少一个综合项目章节。
- 已知道项目复杂度增长后，仅靠页面层堆代码会迅速失控。

## 正文

### 1. 为什么需要一个结构样板

当项目还小时，很多问题可以靠经验和局部约定解决；但随着业务增长、模块增多、多人协作深入，缺少统一结构的代价会越来越大。

结构样板的意义在于：

- 给新功能接入提供固定落点。
- 让依赖方向更清晰。
- 降低重构时的心智负担。

### 2. Clean Architecture 在 Android 中的实用理解

它的核心不在于层数多，而在于依赖方向稳定。常见理解可以简化为：

- UI 层负责展示和交互。
- Domain 层负责业务规则和用例编排。
- Data 层负责数据来源和实现细节。

只要方向清晰，具体是不是严格三层、四层，并不是最关键的问题。

### 3. 样板工程应展示什么

一个有价值的示例，不该只是画架构图，而应真正展示：

- 一个页面如何触发 UseCase。
- UseCase 如何调用 Repository。
- Repository 如何协调本地与远程数据。
- 状态如何一路返回 UI。

只有这条链路真实存在，读者才能判断这套结构是否适合自己的项目。

### 4. 什么时候不必过度引入

如果项目很小、需求变化快、团队人数有限，过于完整的 Clean Architecture 结构可能会让工程显得笨重。它的价值主要体现在中大型项目、多人协作和长期维护中。

因此这套样板更像“可升级的组织方式”，而不是所有项目一开始就必须照搬的模板。

### 5. 如何评价这套结构是否成功

一个好用的结构样板，至少应做到：

- 新增功能时落点明确。
- 业务逻辑不依赖具体 UI 实现。
- 数据来源可替换。
- 测试入口清晰。

如果只是目录更多、文件更多，但这些目标都没实现，那它就只是形式上的复杂化。

### 6. 最小实践任务

1. 把 Todo 或新闻案例按 UI、domain、data 三层重新画一次依赖关系。
2. 选一个用户动作，追踪它从界面到数据层再返回界面的完整路径。
3. 判断这条路径中哪些依赖是稳定的，哪些依赖还可以继续收敛。

### 7. 常见误区

- 把 Clean Architecture 当成固定模板死记。
- 为小项目过早引入过重结构。
- 分了层却没有真正控制依赖方向。

## 小结

Clean Architecture 示例的价值，是帮助你理解复杂 Android 工程应如何组织，而不是要求你机械复制某种目录形状。真正值得保留的，是清晰依赖方向和可演进的结构思维。
