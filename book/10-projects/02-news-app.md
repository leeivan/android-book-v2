# 新闻应用

如果说待办应用最适合练本地优先和基础状态管理，那么新闻应用则更适合把现代 Android 项目里的网络主线真正跑通。它会自然引出很多真实问题：列表和详情怎么分工，远程数据怎样拉取和缓存，搜索和分类怎样影响页面状态，断网时页面应该表现成什么样，刷新和离线缓存谁是可信来源。

参考目录里的多个案例书都把“内容型应用”作为训练网络与缓存的主战场，这很合理。因为新闻应用几乎天然覆盖了 `Retrofit`、Repository、Room、离线缓存、列表分页、详情页和搜索筛选这条主线。本章要做的，就是把这条链路整理成一套更适合教学复现的实现顺序。

## 学习目标

- 理解新闻应用为什么适合训练网络、缓存和页面状态主线。
- 学会把远程数据链路和本地可信来源一起设计。
- 学会把列表页、详情页、搜索和刷新组织成清晰的数据流。
- 能按阶段推进一个内容型 Android 项目，而不是一上来就堆功能。

## 前置知识

- 已理解 Retrofit、Repository、Room、StateFlow 和离线缓存。
- 已能完成基本列表页、详情页和状态管理练习。

## 正文

### 1. 先想清楚“用户看到什么”，再决定“接口怎么调”

做新闻应用时，最容易犯的错误就是先围着 API 文档打转，结果把项目做成“接口展示器”。更适合教材的起点应该是先问：

- 用户打开首页时想立刻看到什么。
- 断网时至少还能看到什么。
- 搜索和分类会怎样改变当前结果。
- 用户点击文章后，详情页需要独立承担哪些状态。

只要先从这些问题出发，你就不会把网络调用直接糊到页面层。

### 2. 新闻应用的核心，不是 UI，而是“远程 -> 本地 -> 页面”

一个健康的新闻项目主线通常是：

1. 远程接口返回文章列表或详情。
2. Repository 判断如何写入本地缓存。
3. Room 对外提供稳定的数据流。
4. ViewModel 把数据流转成页面 `uiState`。
5. 页面只消费状态，不直接理解 HTTP 结果。

这条链路之所以重要，是因为它一次解决了三个问题：离线可用、页面恢复和多页面共享可信数据来源。

### 3. 为什么列表页和详情页不能共享一份页面状态

新闻类应用里，列表页通常关心分类、搜索词、空态、错误态和文章摘要列表；详情页更关心某篇文章的完整内容、收藏状态和打开失败后的恢复。它们可以共享数据层，但不应共享页面状态。这个案例特别适合训练“共享的是数据，不是页面状态”这条边界。

如果这条边界没立住，最容易出现的现象就是：详情页刷新逻辑影响列表页、搜索词污染详情页、收藏状态到处同步不清楚。

### 4. 搜索、分类和刷新为什么最能暴露状态设计是否成熟

很多教学项目在“拉出一页列表”时看起来没问题，一旦加入搜索和分类，状态复杂度就迅速暴涨。你必须回答：

- 当前关键词是什么。
- 当前分类是什么。
- 旧请求是否应该被取消或覆盖。
- 搜索为空和搜索无结果是不是同一种状态。

这就是为什么新闻应用非常适合训练页面状态建模。因为它会强迫你从“字段拼装”升级到“真正建模页面状态”。

### 5. 一个更健康的最小结构

如果你希望这个案例可以继续扩展，一个很稳妥的最小结构通常是：

- `NewsApi`：远程接口定义。
- `ArticleDao`：本地文章缓存。
- `NewsRepository`：整合拉取、缓存、刷新和可信来源策略。
- `NewsListViewModel`：负责列表页状态。
- `ArticleDetailViewModel`：负责详情页状态。

这套结构的关键不在于分了几层，而在于页面不直接认识 Retrofit，数据层不直接碰界面，缓存不只是“顺手存一份”。

### 6. 更适合教材的迭代顺序

第一阶段先做最小链路：

- 文章列表。
- 文章详情。
- 基础远程请求。

第二阶段再加状态复杂度：

- 搜索。
- 分类筛选。
- 空态、错误态和刷新状态。

第三阶段再加数据策略：

- Room 缓存。
- 离线优先体验。
- 收藏或稍后阅读。

这种顺序能让你每次只引入一类复杂度，而不是从第一天开始就把网络、缓存、搜索、离线和收藏全塞进项目里。

### 7. 离线缓存的教学价值，远大于“看起来更高级”

新闻应用非常适合讲清楚：离线能力不是锦上添花，而是内容型项目结构成熟的标志。更合理的体验往往不是“有网看新闻，没网空白”，而是：

- 有缓存时先显示缓存。
- 后台尝试刷新。
- 刷新失败时保留已有内容并给出轻量提示。

这条体验路线能帮助读者真正理解 offline-first 和单一可信来源，而不是只会把缓存当作一个次要附加层。

### 8. 项目做完后最值得复盘的不是功能数，而是数据链

一个新闻案例完成后，比起“页面像不像资讯 App”，更值得检查的是：

- 页面是否仍直接依赖网络接口。
- 缓存是不是可信来源，还是只做了展示辅助。
- 搜索和分类是否进入统一状态模型。
- 离线和刷新体验是否经过设计，而不是发生错误时才临时补。

如果这条链路是清楚的，这个案例就已经非常有教学价值。

### 9. 实践任务

起点条件：

- 已具备列表、详情和基础网络请求能力。

步骤：

1. 先画出新闻应用的数据流，不急着写 UI。
2. 明确远程、本地和页面状态各自的职责。
3. 完成一个最小文章列表页和详情页。
4. 加入搜索或分类，观察页面状态复杂度如何上升。
5. 最后再接入 Room 缓存和刷新策略，验证离线体验。

预期结果：

- 你会真正理解内容型应用的核心在于数据链路，而不只是页面数量。
- 你会清楚 Repository 和本地可信来源在实际项目中的价值。
- 你能把网络篇、数据篇和架构篇组织成一套完整项目结构。

自检方式：

- 你能解释为什么新闻应用适合训练“远程 -> 本地 -> 页面”。
- 你能区分列表页状态和详情页状态。
- 你能说明缓存为什么不该只是“顺手存一下”。

调试提示：

- 页面直接持有 Retrofit 结果，优先先补数据层边界。
- 离线后一片空白，说明可信来源设计还没真正站住。
- 搜索和分类散落在页面层，说明状态模型还不够稳定。

### 10. 常见误区

- 直接把远程结果喂给页面，不建设本地可信来源。
- 列表页和详情页状态混在一起。
- 搜索和分类只靠零散字段硬撑。
- 缓存存在，但不承担真正的数据入口角色。

## 小结

新闻应用最大的教学价值，在于它迫使你认真处理网络、缓存、页面状态和离线体验之间的关系。只要“远程 -> 本地 -> 页面”这条主线真正建立起来，你面对绝大多数内容型 Android 项目时，就已经拥有了一套可复用的数据与状态组织方法。
