# 新闻应用

如果说待办应用最适合训练本地优先和基础状态管理，那么新闻应用更适合把现代 Android 项目里的网络主线真正跑通。它会自然引出一组比“把列表显示出来”更真实的问题：首页内容从哪里来，详情页要不要重复请求，搜索和分类怎么影响已有结果，断网时页面应该显示旧内容、空白还是错误，刷新时到底谁才是页面的可信来源。

它也正好处在 Todo App 和聊天应用之间。它不像待办应用那样天然以本地数据为中心，也不像聊天应用那样一开始就要处理实时通信和高频状态变化。新闻应用更像内容型产品的中间地带：远程内容是入口，本地缓存决定体验，页面状态决定读者是否能真正把“网络层、数据层、架构层”连成一条完整的工程主线。

参考目录中的项目书和架构书都在强调同一个事实：内容型应用最怕的不是接口多，而是数据链路不清楚。本章的目标，不是带你拼一个“看起来像资讯 App”的界面，而是把新闻应用整理成一套更适合教材推进的实现顺序，让你知道应该先搭什么、后加什么，以及每一步在验证哪条工程判断。

## 学习目标

- 理解新闻应用为什么适合训练网络、缓存和页面状态这条主线。
- 学会把远程请求、本地缓存和 UI 状态组织成一条可维护的数据链。
- 学会区分列表页、详情页、搜索与分页各自的职责边界。
- 能按阶段推进一个内容型 Android 项目，而不是一上来就堆满功能。

## 前置知识

- 已理解 Retrofit、Repository、Room、ViewModel、StateFlow 和离线缓存的基本角色。
- 已能完成基础列表页、详情页和简单状态管理练习。

## 正文

### 1. 新闻应用真正训练的，不是资讯界面，而是内容链路

很多人第一次做新闻应用，注意力会先落在卡片布局、Banner 图、分类标签和详情页排版上。这些当然重要，但它们不是本章最值得训练的部分。新闻应用真正有教学价值的地方，在于它迫使你正面回答“内容是怎么进入系统、怎样被缓存、怎样被页面消费、怎样在失败时恢复”这一整条链路。

这也是为什么同样是列表型项目，新闻应用比单纯的 RecyclerView 或 LazyColumn 示例更接近真实工程。读者看到的不是一个静态列表，而是一个持续变化的内容流。只要数据链路没有立住，首页刷新、搜索切换、详情打开、离线回退这些动作很快就会互相牵扯，项目也会迅速退化成“接口回调写到哪儿算哪儿”的拼装工程。

### 2. 先定义页面契约，再定义接口契约

更健康的起点不是先打开 API 文档，而是先定义页面到底要向用户承诺什么。首页打开时，用户希望立刻看到的是最近一次可用内容，还是强制等待远程返回；搜索切换时，是保留旧结果直到新结果回来，还是直接清空页面；详情页失败时，是回退到摘要信息，还是明确提示无法打开。这些问题如果不先回答，后面的数据层设计就会一直摇摆。

所以新闻应用的第一步，应该先把页面契约写清楚。列表页负责展示内容集合、筛选条件和加载反馈；详情页负责展示单篇内容、局部操作和打开失败后的恢复；搜索和分类负责改变当前查询条件，而不是直接承担数据存储职责。页面契约先稳定下来，接口只是实现这份契约的手段，而不是反过来决定页面长什么样。

### 3. 把“远程 -> 本地 -> 页面”立成主线

新闻应用最值得建立的认知，是页面不应该直接以远程响应作为长期数据来源。更稳妥的方式是：远程接口负责拿到最新内容，本地数据库负责保存和输出可消费的数据，ViewModel 再把这些数据整理成页面状态。也就是说，真正稳定的主线不是“接口 -> 页面”，而是“远程 -> 本地 -> 页面”。

这样做的好处有三层。第一，离线时页面依旧有内容来源，不会因为一次失败就整屏空白。第二，列表页和详情页可以共享同一份内容基础，不需要各自维护一套互不相认的数据。第三，系统回收页面、用户旋转设备或重新进入应用时，恢复的依据不再只是内存中的临时对象，而是一个更稳定的本地可信来源。

这里最关键的工程判断是：页面永远消费本地结果，远程请求只负责刷新本地内容。在线时，Repository 先从服务端拉取，再写入 Room，然后由 Room 的结果驱动 UI；离线时，则直接展示已经缓存的数据。如果本地为空且刷新失败，才进入“无数据可展示”的错误路径。只要这条主线立住，很多原本容易散掉的问题就会自然收束。

### 4. 列表页、详情页、搜索态，应该共享数据层，但不要共享页面状态

新闻应用很容易把“共享数据”误写成“共享一整个状态对象”。这通常会带来两个后果：列表页的刷新逻辑污染详情页，详情页的收藏或阅读状态又反过来牵扯列表页的显示更新。更成熟的拆法是共享数据层，不共享页面状态。

列表页关心的是查询条件、分页位置、当前结果集合、首次加载、下拉刷新、空结果和加载失败；详情页关心的是文章标识、完整内容、局部操作状态以及打开失败后的兜底信息。它们当然可以从同一个 Repository 和同一份本地数据出发，但 `UiState` 必须分开建模。否则搜索词、分类、分页尾部状态和详情页错误提示很快就会混成一团。

这也是新闻应用比普通“列表 + 详情”练习更适合训练状态建模的原因。它逼着你承认一件事：共享的是内容事实，不是页面上下文。只要这个边界没立住，项目后面再接分页、收藏、稍后阅读或阅读历史时，几乎一定会出现状态互相污染的问题。

### 5. 一个最小但健康的实现骨架

如果你希望这个案例既能跑通，又能继续扩展，一个很稳妥的最小骨架通常包含这几层：`NewsApi` 负责远程内容获取，`ArticleDao` 负责本地缓存与查询，`NewsRepository` 负责协调刷新与读取，`NewsListViewModel` 负责列表页状态，`ArticleDetailViewModel` 负责详情页状态。

下面这个最小示例，重点不在语法本身，而在于职责划分：

```kotlin
class NewsRepository(
    private val api: NewsApi,
    private val dao: ArticleDao,
) {
    fun observeArticles(
        query: String,
        category: String,
    ): Flow<List<Article>> =
        dao.observeArticles(query, category).map { items ->
            items.map { it.toDomain() }
        }

    suspend fun refresh(
        query: String,
        category: String,
    ) {
        val remote = api.getTopHeadlines(
            query = query,
            category = category,
        )
        dao.replaceArticles(
            query = query,
            category = category,
            articles = remote.articles.map { it.toEntity(query, category) },
        )
    }
}
```

这段代码适合放在数据层的 `repository` 与 `dao` 之间。它表达的不是“Room 和 Retrofit 怎么一起用”，而是“页面只观察本地内容，刷新动作只负责更新本地内容”。真正到了页面层，列表页只需要观察 `observeArticles()` 的结果并把它转成 `NewsListUiState`；详情页则根据文章 ID 单独观察或读取对应内容。

如果你的实体里还包含 `bookmarked`、`read`、`readLater` 这类本地字段，就更不能简单地把远程结果整表覆盖。远程数据只知道新闻内容，不知道用户在本地做过什么操作。刷新缓存时，必须明确保留本地字段，或者做局部更新；否则你每次联网刷新，都可能把用户刚刚做过的本地选择重新抹掉。

### 6. 搜索、分类、分页和刷新，不要同一天一起接进来

很多章节之所以显得“新闻应用很复杂”，并不是因为新闻应用本身超出能力，而是因为实现顺序过于激进。首页列表刚显示出来，就立刻加搜索、分类、分页、缓存、收藏、稍后阅读和下拉刷新，这会让读者在还没有立住主线时，先被一堆局部需求拖走。

更适合教材的顺序通常是三步。第一步先做最小内容链路：文章列表、文章详情、基础远程请求。第二步再引入查询复杂度：搜索词、分类筛选、空结果、错误态和刷新态。第三步再引入数据策略：Room 缓存、离线优先、收藏或稍后阅读、必要时再接分页。这样做的价值不只是“更容易写”，而是每一步都能清楚验证一个判断：你的边界到底立住没有。

分页尤其适合放在主线稳定之后再引入。因为分页带来的不是简单的“再多一页数据”，而是新的状态：是否正在加载下一页、是否已经到底、翻页失败怎么恢复、搜索条件切换时旧分页是否作废。如果前面的列表状态模型还没稳定，分页只会把问题放大。

### 7. 离线能力不是附加项，而是内容型项目的验收线

新闻应用最常见的误判，是把离线缓存看成“等项目差不多了再补一个高级功能”。实际上，对内容型应用来说，离线能力不是锦上添花，而是判断数据链路是否成熟的一条验收线。因为只要你认真做离线，就必须回答谁是可信来源、刷新失败时是否保留旧内容、没有本地内容时错误怎么表达、重新联网后如何更新而不破坏本地状态。

更合理的体验通常不是“有网显示内容，没网一片空白”，而是“有缓存先展示缓存，后台尝试刷新，刷新失败时保留旧内容并给出轻量提示”。这类体验之所以重要，不是因为它显得更专业，而是因为它迫使你把刷新和展示彻底解耦。只要展示链路还在直接依赖远程返回，离线体验就不可能稳定。

### 8. 一个新闻项目做完后，最该复盘的是数据链，而不是页面像不像产品

新闻案例完成后，最值得检查的不是首页是否足够像某个商业资讯 App，而是你是否真正建立了稳定的数据组织方式。页面是不是还直接理解 HTTP 结果，Repository 是否只是机械转发，Room 是不是只在离线失败时才被想起来，搜索和分类是不是仍然散落在几个零碎字段里，这些问题比界面装饰更能决定这个案例有没有教学价值。

如果你已经能清楚回答下面这些问题，这一章就算真正做成了：为什么列表页和详情页应该共享内容基础却分开 `UiState`；为什么刷新动作不应该直接把远程结果塞给页面；为什么本地字段需要在刷新时被保留；为什么分页属于“状态复杂度升级”而不是“多写一个 API 参数”。一旦这些判断稳住，后面的聊天应用才有一个足够可靠的起点。

### 9. 实践任务

起点条件：

- 已具备文章列表、详情页和基础网络请求能力。

步骤：

1. 先画出新闻应用的内容链路，明确远程、本地和页面分别承担什么职责。
2. 为列表页定义 `NewsListUiState`，把查询条件、首次加载、刷新态、空态和错误态写清楚。
3. 完成一个只包含列表页、详情页和基础远程请求的最小版本，不急着加搜索和分页。
4. 第二阶段加入搜索或分类筛选，观察哪些状态属于查询条件，哪些属于结果状态。
5. 最后接入 Room 缓存，并验证“离线仍可浏览已有内容、联网后可刷新”的体验是否成立。

预期结果：

- 你会真正理解内容型应用的主线在于数据链，而不只是页面数量。
- 你会清楚 Repository、本地缓存和页面状态在工程中的分工。
- 你能把网络篇、数据篇和架构篇的知识组织成一套完整项目结构。

自检方式：

- 你能解释为什么新闻应用适合训练“远程 -> 本地 -> 页面”。
- 你能区分列表页状态、详情页状态和查询条件状态。
- 你能说明为什么刷新失败时不应默认把页面清空。

调试提示：

- 页面直接持有 Retrofit 响应对象，优先先补数据层与 UI 层的边界。
- 每次联网刷新都会丢失收藏或阅读状态，优先检查本地字段是否被远程结果覆盖。
- 搜索、分类和分页都散落在页面回调里，说明状态模型还没有真正建立。

### 10. 常见误区

- 把新闻应用做成“接口展示器”，而不是内容型产品案例。
- 让页面长期直接依赖远程结果，而不是依赖本地可信来源。
- 把列表页和详情页写成一个共享的大状态对象。
- 在主线还没稳定时，就同时接入搜索、分页、缓存、收藏和刷新。
- 刷新缓存时整表覆盖，顺手把本地字段一起重置。

## 小结

新闻应用的教学价值，不在于它能做出一个资讯列表，而在于它迫使你把远程请求、本地缓存、查询条件、页面状态和失败恢复放进同一套工程判断里。只要“远程 -> 本地 -> 页面”这条主线真正建立起来，你面对大多数内容型 Android 项目时，就已经拥有了一套可复用的数据组织方法。

这也是它适合作为聊天应用之前那一站的原因。等你能稳定处理内容链路、离线体验和页面状态以后，再进入聊天场景里的实时连接、消息状态和通知协作，复杂度虽然会上升，但判断框架不会从零开始。
