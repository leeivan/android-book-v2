# 新闻应用案例

新闻应用是理解“网络驱动型 Android 应用”非常合适的综合案例。它比待办应用更强调远程数据、不稳定网络、缓存策略、内容列表、详情展示、分类切换和刷新体验，因此能把网络篇、数据篇和架构篇的大量知识点真正拉到一条完整主线上。它不是为了做一个资讯产品原型，而是为了练习现代 Android 项目在远程数据场景下的结构组织。

这类案例真正值得学的，不是把文章列表显示出来，而是如何处理：加载、空态、失败、本地缓存、详情页落地、分页与刷新策略，以及“旧内容还能不能先给用户看”。

## 学习目标

- 理解新闻应用为什么是网络主线的理想练习案例。
- 理解远程数据、本地缓存和页面状态如何协同。
- 理解列表页、详情页、分类页在数据流和导航上的组织方式。
- 学会把 Retrofit、Room、Repository、Flow 和页面状态组合成完整项目。

## 前置知识

- 已理解 Retrofit、API 结果包装、离线缓存和 Repository。
- 已能完成基本列表页、详情页和导航结构。

## 正文

### 1. 新闻应用最难的地方不是列表，而是“不确定性”

新闻应用天然面向不稳定网络环境，因此它的核心难点往往不在界面本身，而在于：

- 首次打开时有没有内容。
- 网络失败时是否保留旧内容。
- 分类切换如何避免频繁重拉。
- 详情页是否允许离线阅读缓存内容。

这也是为什么新闻案例特别适合练习网络篇的全部主线：请求、解析、缓存、状态包装和单一可信来源。

### 2. 列表页和详情页的数据策略不应完全相同

新闻列表页通常更适合：

- 优先展示本地缓存。
- 合适时机触发远程刷新。
- 允许短时间内容不是绝对最新。

而详情页则更强调：

- 点击时尽快能看到完整内容。
- 如果已缓存则允许离线阅读。
- 如果远程更新到了更完整内容，再回写本地。

这说明即使都属于同一个内容产品，不同页面的数据策略也不应简单复制。

### 3. 分类和搜索会改变整个数据流

只要加入分类切换、关键字搜索或分页，新闻应用的数据流就会立刻复杂很多。因为这时你需要明确：

- 当前页面正在看哪一类内容。
- 旧分类结果是否还能继续保留。
- 搜索是否属于临时会话状态。
- 列表分页是本地缓存还是远程实时拉取。

这类问题正好能帮助你把 Flow、Repository 和页面状态真正结合起来，而不只是停留在“能请求一个列表”。

### 4. 新闻应用非常适合练习离线缓存

新闻内容通常具备这些特点：

- 用户希望打开就看到东西。
- 旧内容多数情况下仍然有阅读价值。
- 网络失败时，显示上次内容比空白页更好。

因此它天然适合“本地作为可信来源、远程回填更新”的策略。这也是为什么新闻案例经常被用来演示 offline-first 或至少 local-first 的数据设计。

### 5. 页面状态设计比接口数量更重要

新闻应用里最容易走偏的地方，是把所有注意力都放在 API 数量和 DTO 上，而忽略页面状态本身。更重要的状态通常包括：

- 首次加载。
- 下拉刷新。
- 内容展示。
- 空分类或无结果。
- 请求失败但有旧内容。

如果这些状态不被认真建模，应用即使“请求成功”，用户体验也会很混乱。

### 6. 一个合理的最小结构

一个更健康的新闻应用案例结构通常是：

- `NewsApi` 负责远程内容获取。
- `ArticleDao` 负责本地缓存。
- `NewsRepository` 负责列表与详情的数据协调。
- `NewsListViewModel` 管理列表状态。
- `ArticleDetailViewModel` 管理详情状态。
- `Navigation` 负责从列表到详情和搜索页流转。

真正重要的是：页面不直接消费 Retrofit 结果，而是统一从状态层接收整理后的内容。

### 7. 从最小版到进阶版的演进

最小版新闻应用可以只做：

- 文章列表。
- 文章详情。
- 下拉刷新。
- 本地缓存。

进阶版再逐步加入：

- 分类切换。
- 搜索。
- 收藏。
- 小组件或通知推荐。

这种演进方式比一开始就追求多页面、多来源、多轨道同步更合理。

### 8. 实践任务

起点条件：

- 已有一个远程接口样例，或准备使用公开新闻 API 做练习。

步骤：

1. 先设计列表页、详情页和最小导航，不急着扩展功能。
2. 用 Retrofit + DTO 建立远程数据入口。
3. 用 Room 建立最小缓存层。
4. 通过 Repository 把远程结果统一写回本地，再由 UI 观察本地状态。
5. 明确“失败但有旧内容”与“失败且无内容”两类页面状态。

预期结果：

- 你能把网络、不确定性、缓存和页面状态真正放进一个项目里。
- 你会比做单页 Demo 更深刻理解离线缓存和结果包装的价值。
- 你能逐步演进出更像真实产品的内容应用结构。

自检方式：

- 你能解释：为什么新闻应用特别适合练离线缓存。
- 你能判断：列表页和详情页的数据策略为什么不应完全相同。
- 你能说出：为什么“有旧内容的失败状态”比普通错误页更重要。

调试提示：

- 如果每次进入列表都必须等远程返回，优先检查本地缓存是否真正成为主入口。
- 如果详情页和列表页各自维护一套互不相关的数据缓存，后续一致性会很难维护。
- 如果分类切换一多状态就混乱，优先检查是否缺少明确的当前查询条件模型。

### 9. 常见误区

- 只把新闻应用做成“请求列表然后展示”。
- 列表页和详情页都完全依赖远程实时返回。
- 不区分失败无内容和失败但有旧内容。
- 只关注 API 调通，不认真设计状态流和缓存策略。

## 小结

新闻应用案例最有价值的地方，在于它把“网络不确定性”和“内容可读性”同时带进了一个可控项目里。只要你能在这个案例中把远程、本地、状态和导航真正组织起来，网络主线就算真正学到位了。
