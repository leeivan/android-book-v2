# 新闻应用

如果说待办应用最适合练本地优先和基础状态管理，那么新闻应用则更适合把现代 Android 项目里的网络主线真正跑通。它会自然引出很多真实问题: 列表和详情怎么分工，远程数据怎样拉取和缓存，搜索和分类怎样影响页面状态，断网时页面应该表现成什么样，刷新和离线缓存谁是可信来源。也就是说，新闻应用之所以是经典案例，不是因为它“内容型”，而是因为它几乎天然覆盖了网络应用的核心架构问题。

这章的重点不是照着做一个资讯 Demo，而是让你理解一个教学案例该怎样把 `Retrofit`、`Repository`、Room、`StateFlow`、离线缓存和列表界面真正组织起来。新闻应用最能训练的，不是页面堆砌，而是“远程数据进入本地，再进入页面状态”这条链路。

## 学习目标

- 理解新闻应用为什么特别适合训练网络、缓存和页面状态主线。
- 理解列表、详情、搜索、刷新和离线缓存之间的关系。
- 理解新闻类项目为什么应该把远程数据链路和本地可信来源一起设计。
- 学会把网络篇、数据篇和架构篇的内容组织成一个完整案例。

## 前置知识

- 已理解 Retrofit、Repository、Room、StateFlow 和离线缓存。
- 已能完成基本列表页和详情页。

## 正文

### 1. 新闻应用最适合训练的，不是 UI，而是数据链路

新闻应用在产品上看似直观: 拉文章列表，点进去看详情，再支持搜索和收藏。但真正的教学价值在于它能很自然地逼你回答几个关键问题:

- 远程数据什么时候拉取。
- 页面看到的是直接网络结果，还是本地缓存结果。
- 搜索和分类怎样影响当前列表状态。
- 离线时用户还能看到什么。

这些问题一旦讲清楚，网络应用的核心结构也就基本立住了。

### 2. 列表页和详情页为什么不该共享同一份页面状态

新闻应用里，列表页通常关心:

- 当前分类或搜索词。
- 加载中、错误态、空态。
- 当前文章列表。

详情页则更关心:

- 某篇文章的完整内容。
- 收藏状态。
- 加载失败时的重试。

它们可以共享数据来源，但不应共享页面状态。这个案例很适合训练“共享的是数据层，不是页面层”的边界意识。

### 3. 为什么新闻应用特别适合练“远程 -> 本地 -> 页面”这条链

很多初学项目会直接把 Retrofit 结果喂给页面，这样很容易跑通，但很难走远。新闻应用更健康的主线通常是:

1. 远程接口拉取文章。
2. Repository 决定是否写回本地。
3. Room 作为可信来源对外提供流。
4. ViewModel 把数据转换为页面 `uiState`。
5. 页面只消费最终状态。

这条链路之所以重要，是因为它能同时解决离线、刷新、恢复和多页面共享数据来源的问题。

### 4. 搜索与分类是最好的状态建模训练场

新闻类应用很适合练搜索和分类，因为它们会立刻让页面状态变复杂。页面不只是“有没有列表”，还要考虑:

- 当前关键词是什么。
- 当前分类是什么。
- 切换分类时旧请求是否该失效。
- 搜索为空和搜索无结果是不是同一种状态。

这些问题逼着你把页面状态正式建模，而不是靠几个散落字段硬撑。

### 5. 离线缓存为什么在新闻应用里特别有教学意义

新闻应用并不像待办那样天然本地优先，但它很适合讲“离线可用性不是全有或全无”。很多时候更合理的体验是:

- 有缓存时先显示缓存。
- 同时尝试后台刷新。
- 失败时保留旧数据并给出轻量错误提示。

这条设计线很能帮助读者理解“网络应用也需要可信本地来源”，而不是非得在联网和空白之间二选一。

### 6. 一个更健康的新闻应用最小结构

一个适合教学的最小结构通常可以是:

- `NewsApi`: 负责远程请求。
- `ArticleDao`: 负责本地缓存。
- `NewsRepository`: 决定拉取、缓存和可信来源策略。
- `NewsListViewModel`: 负责列表页 `uiState`。
- `ArticleDetailViewModel`: 负责详情页状态。

这个结构的关键不是类名，而是“页面和网络逻辑被彻底解耦”。页面不应该直接理解 HTTP 细节，也不应该自己决定缓存策略。

### 7. 这个案例最适合怎样逐步扩展

第一阶段:

- 列表页。
- 详情页。
- 基础远程加载。

第二阶段:

- 搜索和分类。
- 收藏或稍后阅读。
- Room 缓存。

第三阶段:

- 离线优先体验。
- 多来源合并。
- 更完整的错误恢复和刷新策略。

这种渐进顺序能让网络主线逐步清楚，而不是一开始就被太多功能噪音淹没。

### 8. 新闻应用完成后最值得检查什么

比起“页面够不够像真实资讯 App”，更重要的是检查:

- 页面是否仍直接依赖网络接口。
- 缓存是不是可信来源，还是只是顺手存一下。
- 搜索和分类是否进入了统一状态模型。
- 离线和失败体验是否被认真设计。

只要这几个结构点做对，这个案例的教学价值就已经很高。

### 9. 实践任务

起点条件:

- 已具备基础列表、详情和网络请求能力。

步骤:

1. 先画出新闻应用的数据流，不直接写界面。
2. 明确远程、本地和页面状态各自的职责。
3. 做一个最小文章列表页，再接一个详情页。
4. 加入搜索或分类，观察页面状态复杂度如何变化。
5. 最后把缓存和刷新策略接入，验证离线体验。

预期结果:

- 你会真正理解网络应用的关键不只是“接口能调通”。
- 你会更清楚 Repository 和缓存策略在实际项目里的价值。
- 你会更容易过渡到真实内容型应用。

自检方式:

- 你能解释为什么新闻应用适合练“远程 -> 本地 -> 页面”的链路。
- 你能判断哪些状态属于列表页，哪些属于详情页。
- 你能说明缓存为什么不该只是“顺手存一下”。

调试提示:

- 页面直接持有 Retrofit 结果，优先先补数据层边界。
- 离线后一片空白，说明可信来源设计还没真正站住。
- 搜索和分类逻辑散落在页面层，说明状态建模还不够稳。

### 10. 常见误区

- 直接把远程结果喂给页面，不建设本地可信来源。
- 列表页和详情页状态混成一团。
- 搜索和分类只靠零散字段硬撑。
- 缓存存在，但不承担真正的数据入口角色。

## 小结

新闻应用最大的教学价值，在于它迫使你认真处理网络、缓存、页面状态和离线体验之间的关系。只要这条“远程 -> 本地 -> 页面”的主线真正建立起来，很多现代 Android 网络应用的核心结构也就建立起来了。
