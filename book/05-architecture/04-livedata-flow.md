# LiveData 与 Flow

在 Android 状态传播体系里，LiveData 和 Flow 都曾经或正在扮演重要角色。很多学习者第一次接触这一章时，会把它理解成“两个可观察对象 API 的比较”。这种理解太浅。真正重要的是：页面状态和异步数据流究竟应该如何被暴露、如何被收集，以及为什么现代 Android 更常把 StateFlow/Flow 作为主线，而 LiveData 更多出现在遗留项目或特定场景中。

## 学习目标

- 理解 LiveData 和 Flow 各自适合解决什么问题。
- 理解状态流、冷流和热流这些概念的基本意义。
- 理解为什么现代 Android 项目更常以 StateFlow/Flow 为主线。
- 为 ViewModel、Compose 和生命周期感知收集打基础。

## 前置知识

- 已理解 ViewModel 是页面状态容器。
- 已具备协程和异步结果的基本认知。

## 正文

### 1. 为什么页面状态需要“可观察”能力

页面状态不是一次性结果。网络返回、本地数据变化、用户输入、重试操作、筛选切换，都会让页面在运行过程中不断变化。

因此，页面层真正需要的是：

- 一个能持续接收新状态的机制。
- 一个能与生命周期协作的收集方式。
- 一个能与异步操作和数据层自然衔接的状态通道。

LiveData 和 Flow 都是在解决这个问题，只是侧重点不同。

### 2. LiveData 解决了什么问题

LiveData 的历史价值主要在于：它把生命周期感知的可观察数据引入了 Android 页面层，让页面不再需要自己管理大量订阅和解除订阅逻辑。

这在早期 Jetpack 体系中非常重要，因为它让 Activity 和 Fragment 能更自然地观察状态变化。

### 3. Flow 为什么更契合现代 Kotlin-first 项目

Flow 的价值不仅是“也能观察数据”，更重要的是：

- 它天然属于协程体系。
- 更擅长表达连续异步流。
- 更容易做转换、组合和错误处理。
- 与 Repository、Room 和网络层衔接更自然。

这也是为什么现代 Android 项目里，尤其是 Kotlin-first 项目中，更常把 Flow/StateFlow 作为主线。

### 4. LiveData 和 StateFlow/Flow 应如何理解差异

一个更实用的理解方式是：

- LiveData 更偏向页面层的生命周期感知状态观察。
- Flow 更偏向通用异步流表达。
- StateFlow 更适合承接“始终有当前值的页面状态”。

也就是说，它们不是简单的谁新谁旧关系，而是不同技术背景下的不同主线。但在今天的新项目里，StateFlow/Flow 通常更符合整体技术栈方向。

### 5. 页面状态和一次性事件为什么要区分

这是状态传播里非常容易混乱的点。页面状态通常表示“当前界面应该长什么样”，而一次性事件更像：

- 导航。
- 弹 Toast 或 Snackbar。
- 打开系统选择器。

如果这两类东西全都混在同一个状态流里，页面很快就会出现重复消费和语义混乱的问题。因此在设计状态传播时，必须先区分“稳定状态”和“一次性事件”。

### 6. 生命周期收集为什么仍然重要

即使你已经使用 Flow，也不意味着可以忽略生命周期。页面是否处于前台、视图是否还存在、是否应该继续收集状态，这些问题依然存在。

现代 Android 推荐的做法，是让状态流和生命周期感知收集配合，而不是只把“能发状态”理解成问题已经解决。

### 7. 最小示例：列表页状态为什么更适合 StateFlow

列表页通常始终存在一个“当前页面状态”，例如：

- 加载中。
- 有内容。
- 空状态。
- 错误。

这类状态具有“当前值明确”的特点，因此很适合用 StateFlow 来表达。它让界面始终能拿到当前页面应该显示什么，而不是只知道某个历史事件发生过。

### 8. 最小实践任务

1. 选一个页面，区分它有哪些是“稳定状态”，哪些是“一次性事件”。
2. 判断这个页面是否更适合用 StateFlow 来暴露状态。
3. 检查你当前项目里是否把页面事件和页面状态混在同一个流里。
4. 如果项目仍大量使用 LiveData，列出哪些地方是历史延续，哪些地方可以逐步迁移。

### 9. 常见误区

- 把 LiveData 和 Flow 理解成两个名字不同的同类 API。
- 不区分页面状态和一次性事件。
- 使用 Flow 后忽略生命周期收集边界。
- 只知道“收状态”，不知道状态本身是否建模清楚。

## 小结

LiveData 与 Flow 章节的重点，不是比较 API 写法，而是理解状态如何在现代 Android 项目中被稳定传播。对 Kotlin-first 的新项目来说，StateFlow/Flow 更常成为主线，但真正重要的是你是否把状态、事件和生命周期边界理清楚了。
