# ViewModel

ViewModel 是现代 Android 架构中最核心的组件之一。它不是“多加一层类”，也不是“写 Jetpack 就必须跟着做的样板”，而是专门用来承接界面相关状态的结构。本章把它单独拿出来讲，是因为它几乎会影响后续每一部分内容。

## 学习目标

- 理解 ViewModel 的职责边界。
- 理解为什么界面状态不应直接塞进 Activity 或 Fragment。
- 为 Flow、Repository 和综合项目章节提供中心支点。

## 前置知识

- 已理解 Activity 生命周期和 MVVM 的基本结构。
- 已理解页面状态会随着系统行为发生变化。

## 正文

### 1. ViewModel 解决的是状态托管问题

Android 界面开发中最容易出问题的一件事，就是把状态直接塞在页面实例里。这样一来，只要页面重建，很多状态就会丢失。ViewModel 的核心价值就在于：把界面相关但不该依附于页面实例的状态提取出来。

### 2. 什么应放进 ViewModel

通常更适合放进 ViewModel 的包括：

- 页面要展示的数据状态。
- 页面加载、成功、失败等状态表达。
- 用户操作触发后的状态更新逻辑。

它不应无边界地变成“什么都往里塞的万能类”，但它确实是界面状态的主要中心。

### 3. 什么不该放进 ViewModel

- 明显只属于某个 View 的极小局部展示细节。
- 与页面完全无关的全局平台能力调用细节。
- 本应属于 Repository 或数据源层的底层数据访问逻辑。

很多项目的问题不是没有用 ViewModel，而是 ViewModel 失去了边界。

### 4. ViewModel 和后续章节的关系

后面你会发现，很多主题都会自然回到 ViewModel：

- 列表状态怎么更新。
- 网络结果如何传到界面。
- 本地数据库变化如何反映到页面。
- 协程和 Flow 应该在哪一层被组织。

这也是为什么理解 ViewModel，是整本书后半部分能否顺畅的关键。

### 5. 最小实践任务

1. 找一个简单页面，把它要显示的状态列出来。
2. 判断哪些状态该放进 ViewModel，哪些只属于局部控件。
3. 尝试把一个界面层里的请求逻辑改成由 ViewModel 组织状态。

### 6. 常见误区

- 把 ViewModel 当成全局单例。
- 把一切逻辑都往 ViewModel 堆。
- 使用 ViewModel 但仍让界面直接操作数据源。

## 小结

ViewModel 的价值，在于让 Android 页面状态从“跟着页面实例一起漂移”变成“有明确承接层可管理”。它是现代 Android 架构里最重要的中轴之一。
