# ViewModel

ViewModel 是现代 Android 架构中最核心的状态承接点之一。它之所以重要，不是因为“官方推荐”，而是因为它恰好解决了一个非常现实的问题：页面实例会因为配置变化或生命周期切换不断变化，但页面的逻辑状态不应该每次都跟着重头再来。本章的重点，是把 ViewModel 从“一个带状态的类”提升到“页面状态容器”的正确理解上。

## 学习目标

- 理解 ViewModel 的核心职责和边界。
- 理解 ViewModel 与 Activity/Fragment/Compose 的关系。
- 理解为什么 ViewModel 适合承接页面状态而不是所有业务实现。
- 为 StateFlow、SavedStateHandle 和 Repository 协作打基础。

## 前置知识

- 已理解生命周期和 MVVM 基本协作关系。
- 已知道页面实例并不是长期可靠的状态容器。

## 正文

### 1. ViewModel 解决的到底是什么问题

ViewModel 主要解决的是：页面会变化，但页面逻辑状态需要更稳定地存在。

例如：

- 列表加载中、成功或失败的状态。
- 当前搜索关键字。
- 页面选中的筛选条件。
- 某次用户动作正在处理中。

这些状态如果全都塞在 Activity 或 Fragment 成员变量里，就会在配置变化和页面重建时非常脆弱。

### 2. ViewModel 最适合承接什么状态

从工程角度看，ViewModel 最适合承接的是页面级状态，也就是那些：

- 与某个页面或某个 UI 流程强相关。
- 不应随着页面重建立刻丢失。
- 不属于长期持久化业务数据本身。

它不是数据库，不是永久存档，也不是应用全局状态中心。它更像是“某个界面场景的状态组织者”。

### 3. ViewModel 与 Activity/Fragment/Compose 的关系

在现代 Android 中，ViewModel 与具体 View 技术并不强绑定。无论界面是：

- Activity。
- Fragment。
- Compose 页面。

ViewModel 承担的核心角色都类似：对上接收界面事件，对下调用数据层，对外暴露页面状态。

这也是为什么 ViewModel 是现代 Android 中非常稳定的架构支点。

### 4. ViewModel 为什么不是“新的万能大类”

很多项目在重构时，会把页面里的所有代码一股脑搬到 ViewModel，然后以为问题解决了。实际上，如果 ViewModel 里开始直接处理：

- 网络接口细节。
- 数据库细节。
- 模型转换细节。
- 一大堆互不相关的业务流程。

它就会迅速变成新的巨型类。ViewModel 的价值从来不是“替页面承受一切”，而是聚焦页面状态和页面事件组织。

### 5. 页面状态为什么应该被整体建模

一个成熟的 ViewModel 通常不会只暴露零散字段，而会逐步走向更稳定的状态建模。例如，一个列表页往往至少会有：

- 加载中。
- 有内容。
- 空状态。
- 错误状态。

如果这些状态只是散落在几个布尔值和几个变量里，页面逻辑很快就会混乱。ViewModel 的重要价值之一，就是帮助你把页面状态正式建模。

### 6. SavedStateHandle 为什么值得知道

页面状态并不都一样。有些状态只需要在普通重建下保留，有些状态需要在进程被系统回收后也能恢复关键输入。SavedStateHandle 的意义就在于，它为 ViewModel 提供了一种更合适的状态恢复协作方式。

入门阶段不必把它的所有细节吃透，但应先建立一个意识：页面状态也分层次，不是都靠同一种机制保存。

### 7. ViewModel 与协程、Flow 为什么天然适合组合

现代 Android 中，ViewModel 往往和协程、StateFlow/Flow 一起使用。原因很直接：

- 页面事件通常会触发异步操作。
- 页面状态需要以稳定、可观察的方式暴露。
- 生命周期变化要求状态层比页面实例更稳定。

这三点恰好让 ViewModel 成为协程和状态流的自然承载点。

### 8. 最小示例：列表页的 ViewModel 不该只是一堆变量

一个合理的列表页 ViewModel，重点不应是“保存列表数据”本身，而是组织下面这些状态和动作：

- 首次加载。
- 下拉刷新。
- 重试。
- 点击列表项。

它要回答的不是“页面上有哪些变量”，而是“这个页面在运行中可能进入哪些状态”。

### 9. 最小实践任务

1. 选一个页面，列出它当前所有页面级状态。
2. 判断哪些状态适合放在 ViewModel，哪些只属于瞬时 UI。
3. 找出 ViewModel 中一段明显属于数据层或业务层的逻辑，考虑是否应下沉。
4. 试着把一个零散的页面变量收束成更清晰的状态对象。

### 10. 常见误区

- 把 ViewModel 理解成“带状态的 Activity 替身”。
- 把所有逻辑都搬进 ViewModel。
- 只暴露零散字段，不做页面状态建模。
- 认为 ViewModel 可以替代本地存储或应用级状态管理。

## 小结

ViewModel 的真正价值，在于让页面状态获得一个比页面实例更稳定、更清晰的承载点。理解了这一点，再去看 StateFlow、SavedStateHandle、Repository 和 Compose 状态管理时，整个架构链路会自然很多。
