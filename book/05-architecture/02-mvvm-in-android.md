# Android 中的 MVVM

上一章解决了“为什么现代 Android 更常采用 MVVM”，这一章开始解决“它在 Android 中到底怎样落地”。关键不是把 View、ViewModel、Model 三个词背会，而是理解在 Android 的组件体系里，界面层、状态层和数据层究竟应如何协作，才能既符合生命周期现实，又不让页面代码继续膨胀。

## 学习目标

- 理解 MVVM 在 Android 中的具体职责分工。
- 理解 View、ViewModel 和数据层在现代项目中的协作方式。
- 理解单向数据流为什么常与 MVVM 搭配出现。
- 为 ViewModel、Flow、Repository 和状态建模打基础。

## 前置知识

- 已理解 Activity、Fragment、数据层和网络层的基本角色。
- 已理解 MVVM 相比 MVC、MVP 的整体优势。

## 正文

### 1. Android 中的 View 不只是 XML

在 MVVM 里，View 并不等于布局文件。更准确地说，View 指的是整个界面呈现层，包括：

- Activity。
- Fragment。
- Compose 界面。
- 与渲染和交互直接相关的轻量 UI 逻辑。

这意味着 View 的职责应尽量聚焦在：

- 展示状态。
- 接收用户输入。
- 触发页面事件。

而不是直接承担业务决策、数据协调和长期状态保存。

### 2. ViewModel 在 Android 中为什么如此关键

ViewModel 在 Android 中之所以重要，是因为它恰好填补了平台里最痛的一个空位：页面是短生命周期的，但页面状态不能总跟着实例一起消失。

ViewModel 的现实价值包括：

- 承接页面级状态。
- 避免把状态硬塞进 Activity 或 Fragment。
- 更自然地与协程、Flow、Room 和 Repository 协同。
- 在配置变化下提供更稳定的状态承载点。

因此，Android 里的 MVVM 很大程度上就是围绕 ViewModel 展开的。

### 3. Model 在 Android 中不只是“几个数据类”

很多入门者会把 Model 理解成一两个数据对象，这种理解太窄。在更完整的 Android 项目里，Model 往往包括：

- Repository。
- 网络数据源。
- 本地数据库。
- 模型转换逻辑。
- 某些业务规则或用例。

换句话说，MVVM 中的 Model 更接近“整个非界面数据与业务层”，而不只是一个被动数据类。

### 4. Android 中的 MVVM 最自然的链路是什么

一个更符合现代 Android 的 MVVM 链路通常是：

- View 捕获用户动作并展示状态。
- ViewModel 接收动作、组织页面状态。
- Repository 协调本地和远程数据来源。
- 数据层返回结果后，ViewModel 产出新的 UI 状态。
- View 依据状态重新渲染。

这条链路之所以重要，是因为它让“谁负责什么”变得比较稳定。

### 5. 单向数据流为什么几乎总会和 MVVM 一起出现

MVVM 在 Android 中落地时，一个非常高频的伴侣就是单向数据流。原因很简单：页面状态如果可以被多个地方任意修改，很快就会失控。

更稳定的方式通常是：

- 事件从 View 发给 ViewModel。
- 状态由 ViewModel 统一生成和暴露。
- View 只根据当前状态渲染。

这样，页面的变化来源更可追踪，也更符合 Android 页面不断重建和恢复的现实。

### 6. 界面层到底该“轻”到什么程度

“界面层要轻”是一句常被反复说的话，但如果不具体化，就会变成空话。更实际的理解是：界面层可以负责 UI 绑定和轻量交互，但不应负责：

- 多数据来源协调。
- 复杂业务分支。
- 长期状态持有。
- 网络和数据库直接调用。

只要这些问题还留在 Activity、Fragment 或 Composable 页面里，MVVM 通常就还没有真正落地。

### 7. ViewModel 也不是“业务垃圾桶”

这是另一个常见误区。很多项目把 Activity 里的逻辑全搬到 ViewModel，就以为已经完成架构升级。实际上，如果 ViewModel 里开始直接写大量数据库访问、接口拼装、模型转换和业务流程分支，它也会很快变成另一个巨型类。

这说明 MVVM 的重点不是“把逻辑搬家”，而是把逻辑放到真正适合的层。

### 8. 最小示例：列表页在 MVVM 中应如何流动

一个列表页的合理链路通常是：

- 页面发出“加载列表”事件。
- ViewModel 更新为加载中状态。
- Repository 读取本地或远程数据。
- ViewModel 把结果转换成内容状态、空状态或错误状态。
- 页面只根据这些状态显示界面。

这个例子看起来简单，但它已经把页面、状态和数据层边界清晰地分开了。

### 9. 最小实践任务

1. 以一个列表页为例，写出 View、ViewModel 和数据层各自的职责。
2. 检查你的页面中是否仍直接依赖 Repository、网络调用或数据库调用。
3. 找出一个你当前放在页面里的状态，判断它是否更适合由 ViewModel 承担。
4. 用自己的话描述“单向数据流”在这个页面里应如何体现。

### 10. 常见误区

- 把 MVVM 简化成“Activity + ViewModel”。
- 以为引入 ViewModel 就自动拥有良好架构。
- 界面层仍直接处理网络、数据库和复杂业务逻辑。
- 把所有复杂度从页面层机械搬到 ViewModel。

## 小结

Android 中的 MVVM 本质上是一种更适合平台现实问题的职责组织方式。它的关键不在于缩写，而在于是否真正让页面聚焦展示，让 ViewModel 聚焦状态，让数据层聚焦来源和规则。
