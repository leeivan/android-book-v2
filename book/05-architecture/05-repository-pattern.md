# Repository 模式

Repository 是现代 Android 项目里最常见、也最容易被机械套用的一层。很多人知道“应该有 Repository”，却说不清它为什么需要存在、它到底应该封装什么，以及哪些东西其实不该强行塞进 Repository。本章的目标，是把 Repository 从“为了分层而多出来的一层”还原为它真正的价值：统一数据入口、隐藏来源细节、稳定上层依赖。

## 学习目标

- 理解 Repository 为什么存在。
- 理解 Repository 与 ViewModel、数据源、本地数据库和网络层的边界。
- 理解什么场景值得引入 Repository，什么场景不需要过度包装。
- 为离线缓存、Result 包装和领域层设计打基础。

## 前置知识

- 已理解网络层和本地存储各自的角色。
- 已知道页面不应直接依赖多个底层数据来源。

## 正文

### 1. Repository 到底解决什么问题

Repository 最核心的价值是：把“数据从哪里来、怎样协调多个来源、结果怎样提供给上层”统一收束到一个更稳定的入口。

如果没有 Repository，常见情况会是：

- ViewModel 直接调用网络接口。
- 页面又直接读数据库。
- 某些配置状态在另一个地方单独读取。

很快，页面上层就会与多个来源直接耦合，结构会越来越难维护。

### 2. 为什么上层不应直接面对来源细节

对 ViewModel 和页面来说，它们真正关心的通常不是：

- 这是从网络来的还是数据库来的。
- 这次用了哪个 HTTP 客户端。
- 本地和远程谁先返回。

它们真正关心的是：我现在拿到了什么结果、当前页面应显示什么状态。Repository 的作用，就是帮上层屏蔽这些来源协调细节。

### 3. 一个好的 Repository 应该具备什么特征

一个更健康的 Repository 通常具有这些特征：

- 职责集中在某一类业务数据或能力上。
- 对上提供稳定接口。
- 对下可以协调本地、远程和缓存策略。
- 不把页面语义直接写死在内部。

换句话说，它应是“数据访问边界”，而不是“所有业务代码的收纳箱”。

### 4. Repository 和 DataSource 有什么区别

很多项目会同时出现 Repository 和 DataSource。一个更实用的区分方式是：

- DataSource 更贴近某个具体来源，例如网络、数据库、文件或配置。
- Repository 更贴近某类业务数据的统一入口。

Repository 可以组合多个 DataSource，但它不应退化成只是简单转发某个单一来源的壳层。

### 5. 什么时候不需要过度抽象 Repository

如果项目非常小、数据来源非常单一、页面数量也很少，那么过早引入过多 Repository 层可能反而增加复杂度。真正值得引入它的信号通常包括：

- 页面开始复用同一组数据。
- 本地与远程需要协调。
- 同一类数据存在多个读取入口。
- 上层越来越不应该知道来源细节。

架构层存在的意义是减少复杂度，而不是制造复杂度。

### 6. Repository 为什么会成为离线优先设计的关键

离线缓存、重试、刷新、本地优先显示、远程回填，这些都不是页面该做的事。它们非常适合由 Repository 统一承担，因为它最接近“协调来源”这个职责。

这也是为什么在很多现代 Android 项目里，Repository 是连接网络层和本地数据库的核心节点。

### 7. 最小示例：新闻列表的 Repository 为什么不是多余一层

新闻列表页面通常会面临：

- 首次打开先显示本地还是等待远程。
- 下拉刷新后如何回写本地。
- 请求失败时是否保留旧内容。

这些逻辑如果直接写在 ViewModel 中，很快就会膨胀；如果统一收敛到 ArticleRepository 中，页面和状态层都会更干净。

### 8. 最小实践任务

1. 找一个页面，列出它当前直接依赖了哪些数据来源。
2. 判断这些来源中哪些本应由 Repository 统一协调。
3. 检查你的某个 Repository 是否已经承载了明显不属于数据访问边界的页面逻辑。
4. 用一句话说明：这个 Repository 到底代表哪类业务数据。

### 9. 常见误区

- 觉得 Repository 只是“为了分层多加一层”。
- Repository 只做简单转发，没有真正统一来源。
- 所有业务代码都往 Repository 塞。
- 页面或 ViewModel 仍直接依赖多个来源，Repository 形同虚设。

## 小结

Repository 的价值，不在于目录里多一个名字，而在于它是否真的成为统一数据入口。只要这层站稳，上层就能更专注于状态与展示，而不是被各种来源细节拉扯。
