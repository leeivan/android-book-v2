# Repository 模式

Repository 是现代 Android 数据层里非常常见的一层。它既不是必须神化的架构名词，也不是纯粹的样板文件。它真正的意义，在于给界面层和 ViewModel 提供一个稳定的数据入口，把底层数据来源的复杂性藏起来。

## 学习目标

- 理解 Repository 在架构中的作用。
- 理解数据入口统一对维护性的价值。
- 为数据层设计和综合项目提供核心组织方式。

## 前置知识

- 已理解本地数据层、网络层和 ViewModel 的基本角色。
- 已具备“数据来自多个来源”的认知。

## 正文

### 1. 为什么数据入口需要统一

如果界面层直接分别访问：

- Retrofit 接口。
- Room 数据库。
- DataStore 配置。

那么只要页面稍微复杂，逻辑就会迅速分散。Repository 的核心价值在于把“数据怎么来”统一收口，让界面只关心“我要什么数据”。

### 2. Repository 最适合承接什么职责

- 协调本地和远程数据来源。
- 统一缓存策略。
- 提供稳定的读取和写入入口。
- 对外暴露更贴近业务语义的接口。

这类职责如果放在页面层，会让页面难以维护；如果放在更底层数据源里，又会让低层组件承担过多业务判断。

### 3. Repository 不该变成什么

- 万能工具箱。
- 所有业务逻辑的堆放处。
- 与界面层直接强耦合的“大总管”。

也就是说，Repository 虽然重要，但它同样需要边界。

### 4. 最小实践任务

1. 以“新闻列表”举例，写出一个 Repository 需要协调的两个数据来源。
2. 说明界面层如果直接访问这些来源，会出现哪些问题。
3. 用一句话描述 Repository 对外应暴露的接口应该更像“技术接口”还是“业务入口”。

### 5. 常见误区

- 为了“有架构”而机械增加 Repository，但不定义职责。
- 让 Repository 直接依赖界面层细节。
- 什么都往 Repository 塞，最后又变成大杂烩。

## 小结

Repository 的意义，在于给复杂数据流建立稳定入口。它是连接本地、远程和状态层的关键组织点，也是数据层从“可用”走向“可维护”的分水岭。
