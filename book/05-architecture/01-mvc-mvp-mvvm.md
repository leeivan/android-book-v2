# MVC / MVP / MVVM

架构模式章节最容易变成缩写对照表：MVC 是什么，MVP 是什么，MVVM 是什么，然后列几条优缺点，让读者机械比较。这样做有两个问题。第一，它很难解释为什么项目在真实复杂度下会失控；第二，它也很难帮你判断“这段代码到底该放哪一层”。真正重要的，不是背定义，而是理解这些模式为什么会出现，以及它们分别在解决什么痛点。

对 Android 来说，这个问题尤其现实。Activity、Fragment、Compose 页面、网络请求、本地数据库、状态恢复、配置变化和生命周期会同时作用在一段页面代码上。如果没有职责边界，页面类很快就会变成一切逻辑的容器。理解 MVC、MVP、MVVM 的意义，就在于看清复杂度被放到了哪里。

## 学习目标

- 理解架构模式存在的现实原因。
- 理解 MVC、MVP、MVVM 在职责划分上的关键差异。
- 理解为什么现代 Android 更常以 MVVM 为主线。
- 为后续 ViewModel、Repository、Flow 和 Hilt 建立背景。

## 前置知识

- 已理解页面层、数据层和网络层的基本角色。
- 已知道 Activity 和 Fragment 不应无限承担所有职责。

## 正文

### 1. 架构模式出现，是因为页面天然会膨胀

最小 Demo 往往看不出架构问题，因为一个按钮、一个文本、一次点击逻辑，用 Activity 全写也能跑通。但只要场景接近真实应用，复杂度就会迅速累积：

- 页面要加载远程数据。
- 页面要显示加载、成功、空数据和错误状态。
- 页面要响应旋转、返回栈、进程重建。
- 页面要处理搜索、筛选、刷新和局部更新。
- 页面还可能同时依赖本地数据库和远程接口。

当这些能力全塞进一个页面类，代码就会失控。架构模式本质上是在回答一个问题：面对不断增长的复杂度，职责应该如何拆分。

### 2. MVC 的价值，是让我们开始意识到分离职责

MVC 最重要的历史价值，并不是它今天仍然是 Android 的最佳选择，而是它帮助很多开发者第一次认识到：界面、数据和控制逻辑不应该全部混在一起。

在经典理解里：

- Model 负责数据与业务。
- View 负责展示。
- Controller 负责接收输入并协调两者。

这个思路本身没有问题。但落到 Android 时，很容易发生一件事：Activity 或 Fragment 既承担 View，又承担大量 Controller 职责，于是表面上有分层，实际上页面类仍然巨大。很多早期 Android 项目所谓的 MVC，最终都演变成了“巨型 Activity”。

### 3. MVP 是在修正“页面类过重”这个现实问题

MVP 曾长期流行于 Android，一个重要原因就是它试图把流程控制逻辑进一步从页面层拿出去。页面更像被动的 View，Presenter 负责协调数据、处理业务分支，再把结果推回界面。

它在特定历史阶段很有价值，因为那时生命周期感知组件、状态容器和 Jetpack 生态还不成熟，把逻辑抽到 Presenter 的确能明显减轻 Activity 负担。

但 MVP 也有现实代价：

- View 接口数量会快速膨胀。
- Presenter 与页面生命周期协作需要格外小心。
- 状态型页面往往会积累大量回调与条件分支。

它解决了一部分问题，同时也引入了新的维护成本。

### 4. MVVM 更适合现代 Android，不是因为它“更新”，而是更贴近平台现实

MVVM 在 Android 中成为主线，不是因为缩写更流行，而是因为它和 Jetpack 的核心能力天然契合。Android 官方架构建议强调：

- UI 应由数据驱动。
- 应坚持单一可信来源。
- 应采用单向数据流。
- 数据层应通过 Repository 暴露数据。

这些建议与 ViewModel、Flow、Room、Hilt、Navigation 等能力正好形成稳定主线，而这条主线最自然的落点就是 MVVM。

在 MVVM 中：

- View 负责展示和输入。
- ViewModel 负责页面级状态和 UI 层业务逻辑。
- Model 更接近数据层和领域逻辑，而不只是几个数据类。

这样的分工更适合现代 Android 的生命周期和状态驱动 UI。

### 5. 真正该比较的，是复杂度被放到了哪里

从工程角度看，MVC、MVP、MVVM 的核心差异，不在于缩写本身，而在于下面几个问题的答案：

- 页面承担多少逻辑。
- 长生命周期状态由谁维护。
- 异步结果如何回到 UI。
- 多数据来源由谁协调。
- 页面重建时状态如何恢复。

只要你开始用这些问题来比较模式，架构讨论就会从“名词比较题”变成真正有用的设计判断。

### 6. 为什么今天更适合把 MVVM 作为教学主线

这并不意味着 MVC、MVP 不值得了解。它们的历史价值和思想仍然重要，因为很多旧项目仍可见到它们的影子。但如果今天要为现代 Android 建立一条稳定学习路线，MVVM 更适合作为正文主线，原因很实际：

- ViewModel 已经成为官方推荐的屏幕级状态容器。
- StateFlow / Flow 更适合表达持续变化的 UI 状态。
- Repository 已成为数据层推荐入口。
- Hilt、Navigation、Compose 等现代工具都能自然接入这条链路。

所以本书后续章节会继续以 MVVM 为主线，同时保留对 MVC、MVP 的历史理解。

### 7. 模式不是宗教，边界才是重点

很多人以为目录里出现 `viewmodel/`, `repository/`, `usecase/` 就说明项目架构很好。这是最常见的误解之一。架构模式的价值不在于文件夹名称，而在于职责是否真的被放到了合适位置。

例如：

- 如果页面仍然直接发网络请求，那么叫不叫 MVVM 并不重要。
- 如果 ViewModel 变成新的巨型类，架构只是把复杂度搬家。
- 如果 Repository 只是机械转发，没有统一数据入口，也谈不上真正的数据层分工。

模式只是帮助你组织边界的工具，而不是自动生成好架构的咒语。

### 8. 一个列表页在三种模式下会怎样演化

以新闻列表页为例，最容易失控的写法是：Activity 直接发请求、解析结果、管理加载态、保存筛选条件、处理点击和缓存。这其实更像失控的 MVC。

如果改成 MVP，Presenter 会承担更多流程控制，页面轻一些，但回调与接口也会逐渐增多。

如果改成 MVVM，更自然的写法通常是：

- 页面只分发用户事件并渲染状态。
- ViewModel 暴露统一的 `uiState`。
- Repository 统一处理本地和远程来源。

这个例子说明，模式选择的重点不是“哪个名字更高级”，而是复杂度最终被放在了哪里。

### 9. 实践任务

起点条件：

- 已有一个 Activity、Fragment 或 Compose 页面稍微复杂的练习项目。

步骤：

1. 选择一个页面，列出它现在实际承担的所有职责。
2. 判断这些职责中哪些属于展示，哪些属于状态组织，哪些属于数据协调。
3. 用 MVC、MVP、MVVM 三种视角分别想一遍：这些职责会被放到哪里。
4. 找出当前页面中最不应该继续留在界面层的一段逻辑。
5. 用一句话说明：为什么这个页面更适合向 MVVM 收敛。

预期结果：

- 你会从“职责边界”而不是“缩写定义”来理解架构模式。
- 你能更容易识别页面类为什么会膨胀。
- 你能为后续章节中的 ViewModel 和 Repository 找到清晰位置。

自检方式：

- 你能解释：Android 中经典 MVC 为什么容易退化成巨型 Activity。
- 你能说出：MVP 当年解决了什么问题，又带来了什么新成本。
- 你能判断：MVVM 为什么更适合今天的 Jetpack 主线。

调试提示：

- 如果一个页面既调接口又写数据库还控制导航，说明职责边界基本已经失守。
- 如果你只能通过类名而不是职责来判断模式，说明理解还停留在表面。
- 如果所有复杂度都从页面搬到另一个类里，没有真正分层，架构并没有变好。

### 10. 常见误区

- 把架构模式学成定义和优缺点表格。
- 认为选了某个模式，代码自然就会整洁。
- 只看目录结构，不看真实职责分布。
- 把“逻辑搬家”误当成“架构优化”。

## 小结

MVC、MVP、MVVM 的意义，不在于比较谁更高级，而在于理解面对复杂 UI、生命周期和数据来源时，复杂度应该被放到哪里。对现代 Android 来说，MVVM 更容易与官方推荐的状态驱动和分层架构主线对齐，因此也更适合作为后续学习的基础。
