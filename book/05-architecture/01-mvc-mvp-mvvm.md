# MVC / MVP / MVVM

Android 学习走到这里，已经不能再只靠“页面 + 监听器 + 请求”把应用撑起来了。页面一旦增多、状态一旦复杂，就需要架构来帮助你组织职责。本章先从整体上比较几种常见模式，帮助读者理解为什么现代 Android 更常以 MVVM 为主线。

## 学习目标

- 理解 MVC、MVP、MVVM 的基本思想。
- 理解为何现代 Android 更常以 MVVM 为主线。
- 为 ViewModel、Repository 和状态流章节建立架构背景。

## 前置知识

- 已理解 UI、数据层和网络层的基础职责。
- 已感受到页面逻辑容易膨胀的问题。

## 正文

### 1. 架构模式真正解决什么问题

架构模式不是为了显得“专业”，而是为了回答下面这些现实问题：

- 哪层负责界面。
- 哪层负责状态。
- 哪层负责数据来源和业务逻辑。
- 当页面越来越多时，代码如何仍然可维护。

如果这些问题不提前组织，项目很容易变成“所有逻辑都堆在 Activity 或 Fragment 里”。

### 2. MVC 的直觉优势与现实限制

MVC 对初学者最友好，因为它看起来最接近直觉：界面、数据和控制逻辑分开。但在 Android 里，Activity/Fragment 天然承担了大量界面与控制职责，因此 MVC 很容易退化成“界面层过重”。

### 3. MVP 解决了什么

MVP 试图进一步把控制逻辑从界面层抽离出来，这在早期 Android 项目中非常常见。它解决了部分界面膨胀问题，但在现代 Jetpack 生态里，随着 ViewModel 和状态流工具成熟，很多原本由 Presenter 承担的问题有了更自然的表达方式。

### 4. MVVM 为什么更符合现代 Android

MVVM 更适合作为现代主线，主要因为它与 AndroidX/Jetpack 生态的耦合度更高：

- ViewModel 天然承担界面状态。
- Flow/LiveData 天然适合状态传播。
- Repository 天然适合承接数据层。

这使得 MVVM 在现代 Android 项目里不仅是理念，更有较成熟的配套工具链。

### 5. 最小实践任务

1. 选一个简单页面，标出其中的界面职责、状态职责和数据职责。
2. 思考：如果继续把它们都写在 Activity 里，会先在哪一步变得难维护。
3. 用自己的话比较 MVC、MVP、MVVM 最明显的区别。

### 6. 常见误区

- 把架构模式当作面试名词背诵。
- 只记定义，不看它们解决了什么现实问题。
- 认为只要用了某个模式名称，项目就自然整洁。

## 小结

本章的目标不是选出“唯一正确的模式”，而是帮助你理解：随着项目复杂度上升，职责拆分必须提前设计。后续章节会把 MVVM 作为主要落地方向展开。
