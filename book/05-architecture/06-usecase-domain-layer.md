# UseCase / Domain 层

当项目复杂到一定程度后，仅靠 ViewModel 和 Repository 之间的简单分工，可能已经不足以保持逻辑清晰。此时，很多团队会继续抽离 UseCase 或 Domain 层。本章讨论的不是“是不是所有项目都必须这样做”，而是“什么情况下，这种拆分开始变得值得”。

## 学习目标

- 理解 UseCase 和 Domain 层的适用场景。
- 理解何时需要进一步分离业务逻辑。
- 为 Clean Architecture 示例章节提供前置知识。

## 前置知识

- 已理解 MVVM、ViewModel 和 Repository 的基本分工。
- 已能感受到业务逻辑在页面层和数据层之间堆积的问题。

## 正文

### 1. 为什么会需要比 Repository 更高一层

当业务逻辑开始变复杂时，Repository 往往会遇到两个问题：

- 既要处理数据协调，又要处理复杂业务规则。
- 多个页面共享同一业务动作，但逻辑无处安放。

这时，UseCase 或 Domain 层就开始变得有价值，因为它能把“业务动作”从“数据来源”中进一步剥离出来。

### 2. UseCase 更像什么

UseCase 更像围绕一个明确业务目标组织的一段能力，例如：

- 获取今日待办。
- 提交用户登录。
- 同步离线收藏。

它不是简单工具函数，而是围绕用户意图或业务动作定义的调用单元。

### 3. 不是所有项目都必须引入 Domain 层

这点非常重要。小型项目或教学示例中，如果数据来源简单、逻辑清晰，过早引入过多抽象层只会增加理解负担。因此：

- 简单项目可以先不引入。
- 中大型项目、复杂业务规则项目更值得考虑。

### 4. 最小实践任务

1. 观察一个已有页面，判断它的业务逻辑是否已经开始挤压 ViewModel 或 Repository。
2. 试着把其中一个明确业务动作写成 UseCase 视角的描述。
3. 思考：这个项目目前真的需要 Domain 层，还是只是需要先把职责边界理清。

### 5. 常见误区

- 把 UseCase 当成“所有方法都包一层”。
- 在简单项目中过度设计层次。
- 只复制 Clean Architecture 结构，却不理解业务复杂度是否匹配。

## 小结

UseCase 和 Domain 层适合解决更高复杂度下的业务组织问题。它们不是入门必选项，但理解其价值，能帮助你在项目变大时更有方向地演进架构。
