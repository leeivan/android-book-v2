# UseCase 与 Domain 层

很多人接触 Clean Architecture 或领域层时，常常会陷入两个极端：要么觉得这是大型项目才需要的“高级架构”，完全不必了解；要么一开始就给所有小页面都强行加上 UseCase，结果工程立刻变重。更稳妥的做法，是先搞清楚 UseCase 和 Domain 层到底在解决什么问题，以及什么时候它们真的能带来收益。

## 学习目标

- 理解 UseCase 和 Domain 层的核心意图。
- 理解它们与 ViewModel、Repository 的边界关系。
- 理解 Domain 层在 Android 中是可选增强，而不是必须模板。
- 知道什么时候引入能降低复杂度，什么时候只会增加负担。

## 前置知识

- 已理解 MVVM、Repository 和数据层设计。
- 已知道页面和 ViewModel 不应无限堆叠复杂业务逻辑。

## 正文

### 1. 为什么会需要 Domain 层

随着项目增长，你可能逐渐发现一种情况：即使已经有了 Repository，某些业务逻辑仍然会变得很复杂。例如：

- 一个操作要串联多个 Repository。
- 一个业务规则会被多个页面复用。
- 某个操作包含较多决策分支和校验逻辑。

这时，如果继续把所有逻辑都堆在 ViewModel 里，或者让 Repository 同时承担“数据访问”和“业务编排”，结构很快会失衡。Domain 层出现的意义，就是为这些“更接近业务规则而非数据访问”的逻辑提供落点。

### 2. UseCase 解决的是什么问题

UseCase 可以理解成“一个明确业务动作的执行单元”。它更像是在表达：

- 用户到底要完成什么事情。
- 为完成这件事，需要协调哪些数据与规则。

它的重点不在于“多一个类”，而在于把复杂业务动作从页面状态组织中剥离出来。

### 3. Repository 和 UseCase 的边界是什么

一个更实用的理解方式是：

- Repository 关心数据从哪里来、如何写回去。
- UseCase 关心为了完成一个业务动作，应该怎样组合这些数据能力和规则。

这意味着，如果某个操作只是一次简单读取，直接调用 Repository 就够了；如果它涉及较多业务规则、多个来源协调或可复用业务流程，UseCase 就更有价值。

### 4. Domain 层为什么是“可选”，而不是“必须”

现代 Android 官方架构也明确表达过：Domain 层是可选的。它非常重要，因为它提醒我们：

- 不要为小项目过度分层。
- 不要为了追求“架构完整”而增加理解成本。
- 只有在复杂业务逻辑真正出现时，才值得引入额外层次。

换句话说，Domain 层是复杂度治理工具，不是模板崇拜对象。

### 5. 什么时候值得引入 UseCase

以下情况通常比较适合引入：

- 一项业务动作会被多个页面复用。
- 一次操作需要协调多个 Repository。
- 某些业务规则明显比单纯数据访问更复杂。
- 你希望这部分逻辑更容易独立测试。

如果还没有出现这些信号，强行把所有操作都包一层 UseCase，往往只会让代码更绕。

### 6. UseCase 也不该变成“伪装过的 Repository”

有些项目会写很多名义上的 UseCase，但内部只是简单转发 Repository 的一个方法。这样的 UseCase 价值很低，因为它没有新增任何业务语义，也没有降低复杂度。

UseCase 应该代表一个清晰的业务动作，而不是“每个函数再包一层”。

### 7. 最小示例：同步收藏列表为什么可能值得有 UseCase

假设“同步收藏列表”这个动作需要：

- 读取本地收藏。
- 拉取远程变化。
- 合并冲突。
- 回写本地。
- 返回同步结果。

这种情况下，用一个专门的 UseCase 来表达这个业务动作，通常会比让某个 ViewModel 或 Repository 独自承担更清晰。

### 8. 最小实践任务

1. 找出你当前项目中最复杂的一个用户动作。
2. 判断它是纯数据访问，还是明显包含业务编排。
3. 如果要引入 UseCase，写出它应代表的业务意图，而不是先写类名。
4. 检查你现有代码中是否存在“只是简单转发 Repository 的伪 UseCase”。

### 9. 常见误区

- 把 Domain 层当成所有项目都必须先搭好的模板。
- 所有小操作都强行包成 UseCase。
- UseCase 只是简单转发 Repository，没有新增业务语义。
- Repository 与 UseCase 的边界完全不清。

## 小结

UseCase 与 Domain 层的真正价值，在于为复杂业务规则提供更稳定的落点。它们不是越早越好、越多越好，而是当业务复杂度真正出现时，用来防止页面层和数据层被进一步拉扯的结构工具。
