# Intent

Intent 是 Android 组件通信模型里最基础的一层。很多初学者第一次接触它时，会把它理解成“页面跳转参数包”，这种理解不算错，但太窄。Intent 真正重要的地方在于：它把“我要做什么”和“由谁来做”解耦开了，因此既能用于应用内页面流转，也能用于系统能力调用和应用间协作。本章要建立的，不只是“会跳转”，而是对组件边界和动作语义的正确理解。

Android 官方文档把 Intent 描述为“对一个动作的抽象描述”。这句话非常关键，因为它提醒你：Intent 不是普通对象引用，也不是任意对象之间的调用桥梁，而是一种面向组件、面向动作的系统级请求描述。

## 学习目标

- 理解显式 Intent 和隐式 Intent 的区别。
- 理解 Intent 在组件通信、系统能力调用和结果回传中的位置。
- 理解 Intent Filter、Chooser 和最小必要参数的设计意义。
- 知道什么时候应该用 Intent，什么时候应直接使用本地对象调用。

## 前置知识

- 已了解 Activity 和系统组件的基本作用。
- 已知道 Android 组件不能像普通对象那样随意直接持有彼此。

## 正文

### 1. Intent 解决的到底是什么问题

Android 应用由多个组件组成，而这些组件经常处于不同生命周期和不同系统入口下。Intent 的作用，就是把一次“动作请求”结构化描述出来，再交给系统决定由谁处理。

如果把普通函数调用看成“我知道谁来做，现在就调用它”，那么 Intent 更像是“我知道要完成什么动作，但处理者由系统和匹配规则来决定”。这也正是它能同时服务两类场景的原因：

- 应用内部组件跳转。
- 系统或其他应用能力调用。

只要你抓住“动作请求”这个核心，Intent 就不会再只是“跳页面工具”。

### 2. 显式 Intent 和隐式 Intent 的边界

显式 Intent 会明确指出目标组件，例如从一个 Activity 跳到另一个 Activity。它最适合应用内部那些结构清晰、目标明确的调用，因为你知道究竟要把动作交给谁处理。

隐式 Intent 不直接指定组件，而是声明动作、数据类型和可选类别，由系统去匹配能处理这项请求的组件。它适合：

- 打开网页。
- 分享文本或图片。
- 选择照片或文件。
- 调起地图、电话、邮件等系统能力。

这条边界非常重要，因为它决定了你是在做“应用内明确导航”，还是在做“面向系统能力或外部应用的动作请求”。

### 3. Intent Filter 真正定义的是“我能处理什么动作”

Android 官方关于 intent filter 的文档强调，Intent Filter 负责声明一个组件愿意响应哪些 action、data、category 组合。对开发者来说，这意味着两件事：

- 你不应随意把组件暴露给过宽泛的外部匹配条件。
- 你必须非常清楚组件到底在对外承诺处理什么能力。

如果一个 Activity 的 filter 设计过宽，外部应用就可能以意料之外的方式触发它；如果设计过窄，又可能无法正确响应系统能力调用。Intent Filter 本质上是在定义组件公开边界。

### 4. Intent 不只是“跳转”，它也是系统能力调用入口

很多 Android 能力之所以优雅，就在于它们不要求你总是直接申请底层权限或自己实现一整套流程，而是允许通过系统入口完成。例如：

- 通过系统分享面板分享内容。
- 通过 Photo Picker 选择图片。
- 通过浏览器或地图应用完成外部跳转。

这种设计的价值非常大，因为它把复杂能力的实现和权限管理部分交给了更合适的系统组件。很多时候，优先使用系统入口比直接索取底层权限更符合现代 Android 的设计方向。

### 5. 参数设计应坚持“最小必要信息”

无论是显式还是隐式 Intent，一个非常值得养成的习惯是：只传递完成这次动作所必需的信息。例如：

- 详情页需要的是某个内容的 ID，而不是整大块对象。
- 分享动作需要的是文本或 URI，而不是整个复杂状态对象。
- 返回结果时只回传页面真正需要的关键字段。

这样做的好处很直接：

- 降低组件耦合。
- 减少大对象传递和序列化风险。
- 让组件职责更清晰。

一旦你开始在 Intent 里塞入越来越多与动作本身无关的状态，通常说明组件边界已经开始变得混乱。

### 6. Chooser 的价值，不只是“多一个弹窗”

当一个隐式 Intent 可能由多个应用处理时，Android 官方建议在合适场景下使用 chooser。它的意义不只是让用户选一个应用，而是：

- 避免默认应用直接拦截用户选择机会。
- 让用户清楚知道自己将把动作交给谁处理。
- 降低误跳转和错误应用接管的风险。

尤其在分享、打开文件或跨应用动作中，Chooser 往往是更尊重用户控制权的做法。

### 7. Intent 与普通本地对象调用的边界要分清

不是所有事情都适合用 Intent。如果逻辑完全发生在同一组件内部、同一层级、同一生命周期中，那么普通对象调用通常更直接，也更安全。

更适合使用 Intent 的场景通常具备这些特征：

- 涉及 Android 组件间边界。
- 涉及系统能力或外部应用。
- 涉及系统负责的动作分发与匹配。

如果你在应用内部普通对象之间也强行用 Intent 传来传去，往往说明层级和职责设计还不够清晰。

### 8. 一个更接近真实项目的最小例子

假设你有一个文章列表页，点击后跳转详情页。更合理的写法通常不是把整篇文章对象丢进 Intent，而是只传文章 ID：

```kotlin
val intent = Intent(this, ArticleDetailActivity::class.java).apply {
    putExtra(EXTRA_ARTICLE_ID, article.id)
}
startActivity(intent)
```

这个例子真正重要的，不是语法，而是边界：

- Intent 表达的是“打开详情”这个动作。
- 参数只包含完成动作所需的最小信息。
- 详情页再根据 ID 从自己的数据层读取完整内容。

这样做比“把整个页面状态都塞给目标页”更稳定，也更符合分层设计。

### 9. 实践任务

起点条件：

- 已有一个包含列表页、详情页或系统能力调用的 Android 项目。

步骤：

1. 找一个页面跳转场景，判断它更适合显式 Intent 还是隐式 Intent。
2. 检查当前传参是否超出了最小必要信息。
3. 选一个系统能力调用场景，判断是否应优先使用系统入口而不是自己实现。
4. 如果存在多个外部处理者，考虑是否应加入 chooser。
5. 检查项目里是否有用 Intent 代替本地对象调用的过度设计。

预期结果：

- 你会把 Intent 理解成“组件动作请求”，而不是简单跳转工具。
- 组件边界会更清晰，传参也会更克制。
- 你能更自然地区分应用内导航和系统能力调用。

自检方式：

- 你能解释：显式 Intent 和隐式 Intent 的根本区别。
- 你能判断：某个调用是否真的需要 Intent。
- 你能说出：为什么 Intent 参数应坚持最小必要原则。

调试提示：

- 如果某个组件需要依赖大量 Intent extra 才能工作，优先检查边界是否过重。
- 如果隐式 Intent 经常匹配不到组件，优先检查 action、data、category 是否一致。
- 如果跨应用动作没有用 chooser，用户体验可能比你想象的更不受控。

### 10. 常见误区

- 把 Intent 理解成“页面跳转参数包”。
- 在应用内部普通对象间也滥用 Intent。
- 向 Intent 里塞入过多与动作无关的状态。
- 不理解 Intent Filter，随意暴露组件能力。

## 小结

Intent 的核心价值，不是让页面能跳起来，而是为 Android 组件之间的动作请求提供统一语义。理解了显式与隐式 Intent、Filter、Chooser 和最小参数原则之后，你对系统组件边界的把握会明显更稳。
