# OkHttp

当我们说“Android 客户端发 HTTP 请求”时，真正承担底层通信工作的，往往不是 Retrofit 这种声明式接口工具，而是 HTTP 客户端本身。OkHttp 之所以长期成为 Android 生态里的主流 HTTP 客户端，不是因为它“流行”，而是因为它在连接复用、HTTP/2、缓存、拦截器和 TLS 支持等方面，提供了一套成熟、稳定、偏工程化的基础能力。

## 学习目标

- 理解 OkHttp 在 Android 网络栈中的位置。
- 理解 OkHttpClient、Request、Response 和 Interceptor 的基本职责。
- 理解为什么 HTTP 客户端应被集中配置和复用。
- 为 Retrofit、认证、日志和缓存设计打基础。

## 前置知识

- 已理解 HTTP 请求和响应的基本结构。
- 已知道网络层并不只是“拿到 JSON”。

## 正文

### 1. OkHttp 在网络栈中扮演什么角色

从职责上说，OkHttp 是 HTTP 客户端。它负责真正与服务器建立连接、发送请求、接收响应，并处理一系列与协议和连接相关的细节。

这意味着：

- 它关心“请求怎么发出去”。
- 它关心“连接如何复用和恢复”。
- 它关心“缓存、超时、TLS、拦截器”等底层通信问题。

而它并不天然关心：

- 返回体该映射成什么业务对象。
- 页面该显示怎样的错误提示。
- 本地与远程数据如何协调。

这正是它和 Retrofit、Repository 之间的职责边界。

### 2. 为什么 OkHttp 值得被看作工程基础设施

OkHttp 的价值并不只在“能发请求”，而在于它是一个比较完整的 HTTP 基础设施。它提供的能力包括：

- 连接池复用。
- HTTP/2 支持。
- 请求与响应缓存。
- 拦截器链。
- 较好的 TLS 和连接恢复能力。

这些能力很多时候都不会直接出现在页面代码里，但它们会显著影响整个应用的网络表现和稳定性。

### 3. 为什么 `OkHttpClient` 通常应该复用

这是 OkHttp 使用中最值得尽早建立的习惯之一。`OkHttpClient` 并不是“每次请求 new 一个就行”的临时对象，它通常应该作为可复用的客户端实例统一配置和共享。原因很简单：

- 连接池可以复用，减少重复建立连接的成本。
- 缓存、超时、拦截器等全局能力可以集中管理。
- 统一客户端更容易维护认证和日志策略。

如果每个请求都临时创建新客户端，很多底层能力会被无意义地重复初始化，结构也会迅速变乱。

### 4. 拦截器为什么是 OkHttp 的核心扩展点

拦截器的价值，在于它把横切需求放回了统一通道。很多你不想在每个接口声明里重复写的事情，都适合在这里处理，例如：

- 统一追加鉴权头。
- 打印日志。
- 统一处理某类响应头。
- 对请求做基础重试或标记。

这会让网络层的横切逻辑集中，而不是散落在各个接口或页面里。

### 5. 但拦截器不该承担什么

拦截器虽然强大，但也很容易被滥用。它更适合处理“所有请求或一类请求都共同需要的底层规则”，而不适合承载：

- 页面级业务判断。
- 某个单独接口的业务失败文案。
- 复杂状态流转逻辑。

一旦把太多业务含义塞进拦截器，网络层会开始污染业务层。

### 6. 最小示例：用 OkHttp 构造一次请求

下面这个例子只用来帮助你看清基本角色：

```kotlin
val client = OkHttpClient()

val request = Request.Builder()
    .url("https://example.com/articles")
    .get()
    .build()

client.newCall(request).execute().use { response ->
    if (response.isSuccessful) {
        println(response.body?.string())
    }
}
```

这个例子的重点不在于你以后要直接这样写所有请求，而是要看清：

- `OkHttpClient` 是通信基础设施。
- `Request` 描述了一次请求。
- `Response` 表示一次响应结果。

后面 Retrofit 其实也是建立在这套基础能力之上。

### 7. OkHttp 和 Retrofit 的关系

很多初学者会把 OkHttp 和 Retrofit 当成互斥选择，实际上它们更像不同层级的工具：

- OkHttp 负责 HTTP 客户端层。
- Retrofit 负责接口声明与对象映射层。

在很多项目里，Retrofit 底下用的就是 OkHttp。理解这层关系之后，你就不会再把它们看成“两个重复库”。

### 8. 最小实践任务

1. 画出你当前项目里 OkHttp、Retrofit 和 Repository 的关系。
2. 思考哪些横切能力适合由 OkHttp 统一处理，例如鉴权头或日志。
3. 检查你的项目里是否存在反复创建 HTTP 客户端实例的情况。
4. 判断你当前的拦截器是否已经承载了不属于底层通信的问题。

### 9. 常见误区

- 把 OkHttp 当成一次性请求工具，每次都新建客户端。
- 不理解它与 Retrofit 的层级关系。
- 把业务逻辑大量塞进拦截器。
- 只看“能请求成功”，不看连接、缓存和统一配置能力。

## 小结

OkHttp 是现代 Android 网络栈的重要基础设施。理解它的价值，不是为了在所有项目里手写底层请求，而是为了知道：底层 HTTP 客户端应该承担哪些职责，又不应该越界承担哪些业务含义。
