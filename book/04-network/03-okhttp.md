# OkHttp

OkHttp 是 Android 生态中非常常见的底层 HTTP 客户端。它并不负责替你定义业务语义，而是更专注于请求执行、连接管理和拦截处理。本章的目标是帮助你理解它在整个网络栈中的位置。

## 学习目标

- 理解 OkHttp 负责的底层能力。
- 理解拦截器、超时和连接管理等关键点。
- 为 Retrofit 章节提供实现基础。

## 前置知识

- 已理解 HTTP 与 REST 的基本概念。
- 已知道请求和响应的核心结构。

## 正文

### 1. OkHttp 解决的是哪一层问题

在网络栈里，OkHttp 更接近“执行请求”的一层。它的重点不是业务模型，而是：

- 建立和复用连接。
- 发送请求并接收响应。
- 处理超时、重定向、缓存和拦截器。

因此，学习 OkHttp 时不应把它和 Retrofit 混成一层。前者更底层，后者更偏接口组织和结果映射。

### 2. 为什么拦截器这么重要

拦截器之所以关键，是因为很多横切逻辑都适合放在这一层，例如：

- 日志输出。
- 统一请求头。
- 认证令牌附加。
- 错误记录。

如果这些逻辑直接散落在每个接口调用点，后续维护会非常困难。

### 3. 超时和重试为什么不是次要细节

网络不稳定是常态，不是例外。因此，合理配置超时和设计重试策略，本质上是网络体验设计的一部分。处理不好会出现两种极端：

- 过早失败，用户体验差。
- 过晚返回，页面长时间无响应。

### 4. OkHttp 和后续章节的关系

在现代 Android 项目里，OkHttp 常常不会单独大量暴露在业务层，而是作为 Retrofit 或更高层网络模块的底座。但理解它仍然有价值，因为很多日志、认证、缓存和问题排查都会回到这层。

### 5. 最小实践任务

1. 理解一次请求在 OkHttp 里最基本的组成。
2. 设计一个简单的日志拦截思路。
3. 思考哪些跨接口通用逻辑适合放进拦截器。

### 6. 常见误区

- 把 OkHttp 当成完整网络架构。
- 把所有业务判断塞进拦截器。
- 完全忽视超时和重试配置。

## 小结

OkHttp 是现代 Android 网络栈的重要底座。理解它的价值，在于知道“请求执行层”该承担什么、不该承担什么，这会让后续 Retrofit 和 Repository 的职责边界更清楚。
