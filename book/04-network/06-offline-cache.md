# 离线缓存

缓存经常被误解成“性能优化技巧”，但对移动应用来说，它往往首先是可用性设计。用户可能在网络不稳定、切后台再回来、重复进入页面甚至完全离线的情况下继续使用应用。本章讨论的，就是远程数据、本地缓存和用户体验之间的关系。

## 学习目标

- 理解缓存为何不仅是性能问题，也是可用性问题。
- 理解远程优先、本地优先和离线优先的区别。
- 为数据层和综合项目章节提供缓存思路。

## 前置知识

- 已理解 Room、网络层和结果封装的基础。
- 已知道页面状态并不只有“请求成功”一种情况。

## 正文

### 1. 为什么缓存是移动应用的基础能力

与桌面程序相比，移动应用更容易遇到：

- 网络波动。
- 后台切换。
- 短时断网。
- 重复打开同一页面。

如果完全依赖实时远程请求，应用会显得脆弱且缓慢。缓存的价值就在于：让用户体验不必每次都从零开始。

### 2. 三种常见思路

#### 2.1 远程优先

先请求远程，失败时再考虑本地。适合对实时性要求高的内容，但如果兜底设计不好，用户体验会比较脆弱。

#### 2.2 本地优先

先展示本地已有数据，再决定是否刷新远程。适合需要快速进入界面的列表型场景。

#### 2.3 离线优先

把本地数据层当作主视图来源，远程更多承担同步更新角色。适合需要高可用和较强离线能力的应用。

### 3. 缓存设计为什么不能只看“能不能存”

真正困难的问题是：

- 缓存多久算过期。
- 数据刷新时如何覆盖旧数据。
- 用户看到的是旧数据还是空白页。
- 刷新失败时是否继续展示旧结果。

这些都不是底层 API 能自动替你决定的，而是产品和工程共同要回答的问题。

### 4. 最小实践任务

1. 选择一个新闻列表或待办列表场景，判断更适合哪种缓存策略。
2. 写出“首次打开页面”和“第二次进入页面”的数据流差异。
3. 思考：远程刷新失败时，用户应看到旧数据、错误页还是两者结合。

### 5. 常见误区

- 认为缓存只是提速技巧。
- 只设计缓存写入，不设计失效和刷新。
- 远程失败时直接把本地可用数据也隐藏掉。

## 小结

离线缓存的核心不是省几次请求，而是让应用在真实网络环境下仍然可用。理解缓存的产品意义后，后续的数据层和综合项目设计会更接近真实应用。
