# 离线缓存

只要应用的数据依赖网络，离线缓存就不是可有可无的小优化，而是体验设计的一部分。用户不会因为你“理论上可以重新请求一次”就接受白屏、空页面和反复加载。对移动应用来说，弱网、无网、网络抖动、切后台和接口波动都是常态，缓存的价值正在于帮助应用在这些不稳定条件下仍保持可用。

很多人第一次做缓存时，思路是“请求成功后顺手存一份本地数据”。这当然比完全不存要好，但还远远不够。真正的难点不在“有没有存”，而在“页面最终信任谁”“远程结果何时写回本地”“什么时候算过期”“离线时该显示什么”。本章会把这些问题串起来，让缓存不再只是附属功能，而成为完整数据链路的一部分。

## 学习目标

- 理解离线缓存为什么直接影响应用可用性。
- 理解远程优先、本地优先和离线优先的区别。
- 理解单一可信来源在缓存设计中的意义。
- 知道离线缓存如何和 Repository、Room、Flow 形成稳定协作。

## 前置知识

- 已理解网络层、数据层和 Repository 的基本职责。
- 已了解 Room、Flow 和 API 结果包装的主线思路。

## 正文

### 1. 缓存真正解决的是“不确定网络下的可用性”

当网络良好时，很多缓存设计问题并不明显，因为用户几乎总能在短时间内拿到最新结果。可一旦进入真实环境，你就会立刻看到缓存的重要性：

- 首次进入页面时，用户希望尽快看到内容，而不是一直等远程返回。
- 网络失败时，旧内容往往仍比空白页更有价值。
- 频繁重复请求同一数据，会浪费流量、电量和等待时间。
- 用户切后台再回来时，通常不希望页面完全从零开始。

所以缓存并不是“以后再优化”的事情。对网络驱动型应用来说，它和加载态、错误态一样，属于基本交互设计。

### 2. 缓存与本地业务数据不是同一个概念

虽然两者都保存在本地，但它们面对的业务语义不同。缓存更强调“这份数据可以重新从远程拿回”，因此它通常允许替换、过期和清理；而某些本地数据即使最初来自远程，一旦进入用户心智，就不再只是缓存，例如收藏、草稿、离线下载和本地标记。

区分这两类数据非常重要，因为它会影响：

- 是否允许系统或应用主动清理。
- 页面是否必须在无网时仍能访问。
- 同步策略应该偏向“覆盖”还是“合并”。
- UI 对旧数据的容忍度到底有多高。

如果一开始不区分，后面很容易把用户高度依赖的数据当成临时缓存处理，结果就是清理策略和同步策略都出问题。

### 3. 远程优先、本地优先、离线优先分别意味着什么

这三个词经常被混用，但在实践中差异很大。

远程优先的核心是：页面尽量以服务端最新数据为准，本地更多只是短期加速或兜底。它适合对实时性要求很高的数据，例如某些价格、库存或强一致状态。

本地优先的核心是：页面先展示本地已有结果，再决定是否触发远程刷新。新闻列表、文章详情、普通资料页通常更适合这种策略，因为旧内容往往仍然有价值。

离线优先的核心是：即使没有网络，核心读取能力也应继续成立。Android 官方的 offline-first 指南明确指出，至少应保证读取在无网时可用，并建议网络型 Repository 始终拥有本地数据源，且本地数据源应成为上层读取的规范来源。

这三种策略没有绝对高低，关键在于业务语义和用户预期。

### 4. 单一可信来源为什么是缓存设计的关键

缓存设计最容易失控的情况，不是“没存数据”，而是“页面同时相信多个来源”。例如：

- 页面初始化先显示数据库结果。
- 远程请求回来后又直接把返回值渲染到 UI。
- 某些地方还会额外读内存缓存。

这样做的结果是，同一个页面会同时有多套状态来源在竞争。谁先返回谁显示，久而久之就会出现闪烁、不一致、滚动位置错乱和调试困难。

Android 官方的 offline-first 指导把本地数据源定义为 canonical source of truth，也就是单一可信来源。更稳妥的链路通常是：

1. 页面只观察本地数据源。
2. Repository 决定何时从远程拉取。
3. 远程结果先写回本地。
4. 页面通过本地变化收到更新。

只要这条链路稳定，页面就不会和远程结果直接耦合。

### 5. 缓存设计真正困难的是“何时刷新”

很多教程把缓存讲成“把数据写进数据库”，却很少谈刷新时机。但在真实项目里，刷新策略往往比存储动作更难，因为它必须同时回答这些问题：

- 首次进入页面时，是立刻请求，还是先读本地后判断是否刷新。
- 用户下拉刷新时，是强制跳过缓存，还是仍然允许用旧数据先占位。
- 页面重新进入时，是否总要自动刷新。
- 数据多久算过期。
- 后台同步和前台刷新发生冲突时，谁优先。

如果这些规则完全不清楚，即使数据库里有缓存，应用也可能表现得很不稳定。

### 6. 新闻列表为什么是学习缓存的理想案例

新闻列表非常适合练习“先本地后远程”的链路，因为它具备几个典型特征：

- 用户希望页面打开就看到内容。
- 旧内容在大多数情况下仍然有阅读价值。
- 网络失败时，显示上次内容通常优于空页面。
- 远程更新后，页面应自动反映新的本地数据。

因此，一个更合理的实现通常是：ViewModel 只观察本地 `Flow<List<Article>>`，Repository 在合适时机请求远程接口并把结果写回 Room。这样页面不需要知道远程是否刚刚成功，只需要始终根据本地可观察数据渲染。

### 7. 缓存不只是技术决策，也是一种产品决策

缓存多久过期，旧内容是否允许继续显示，失败时是否保留旧页面，并不只是工程选择，也是在表达产品态度。例如：

- 新闻内容缓存几个小时通常可以接受。
- 金融行情缓存几分钟可能都不可接受。
- 聊天消息如果长时间停留在旧状态，会严重误导用户。
- 草稿和收藏则往往应该长期保留，不应被简单当成临时缓存。

所以缓存策略必须和业务一起设计，而不是只从“减少请求次数”出发。

### 8. 一个更接近真实项目的最小链路

下面这个示例不是完整生产代码，但能清楚展示“本地作为唯一读取入口，远程只负责回填”的结构：

```kotlin
class ArticleRepository(
    private val articleDao: ArticleDao,
    private val articleApi: ArticleApi
) {
    fun observeArticles(): Flow<List<Article>> {
        return articleDao.observeAll()
    }

    suspend fun refreshArticles(): ApiResult<Unit> {
        return try {
            val remote = articleApi.getArticles()
            articleDao.replaceAll(remote.map { dto ->
                ArticleEntity(
                    id = dto.id,
                    title = dto.title,
                    summary = dto.summary.orEmpty()
                )
            })
            ApiResult.Success(Unit)
        } catch (e: Exception) {
            ApiResult.ParseError(e)
        }
    }
}
```

这个例子真正重要的不是类名，而是读取和写入的方向：

- 读统一从本地出来。
- 写可以来自远程刷新。
- 页面不会直接消费远程 DTO。

这就是缓存设计开始稳定下来的关键。

### 9. 实践任务

起点条件：

- 已有一个使用 Retrofit 和 Room 的练习项目，或者已有列表页示例工程。

步骤：

1. 选一个网络页面，判断它更适合远程优先、本地优先还是离线优先。
2. 明确页面最终读取的单一可信来源是什么。
3. 设计首次进入、下拉刷新、断网重进、后台恢复四种场景下的页面表现。
4. 把远程结果改成先写回本地，再由页面通过 Flow 感知变化。
5. 写下你的过期规则，明确什么情况下需要强制刷新。

预期结果：

- 你会开始从“数据链路”而不是“多存一份”来理解缓存。
- 页面不再同时直接依赖本地和远程结果。
- 你能更清楚地区分临时缓存和长期业务数据。

自检方式：

- 你能解释：为什么本地应该成为离线场景下的单一可信来源。
- 你能判断：某个页面到底适合哪种缓存策略。
- 你能说出：缓存的主要难点为什么是刷新和失效策略，而不是单纯写库。

调试提示：

- 如果页面同时直接订阅数据库和网络结果，后面通常会出现状态竞争。
- 如果你缓存了数据却始终不知道何时刷新，用户会看到过旧内容或频繁闪烁。
- 如果所有数据都被视为“临时缓存”，收藏、草稿等功能迟早会出问题。

### 10. 常见误区

- 认为缓存只是性能优化，不影响核心体验。
- 页面同时直接消费本地和远程结果。
- 不区分可丢失缓存和长期业务数据。
- 只考虑“怎么存”，不考虑“何时刷新”和“何时失效”。

## 小结

离线缓存的核心，不是把远程数据额外存一份，而是让网络不稳定时应用仍保持可读、可用、可恢复。只要单一可信来源、刷新策略和数据语义都明确下来，缓存就会从“附属功能”变成真正稳定的数据基础设施。
