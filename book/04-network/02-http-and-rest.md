# HTTP 与 REST

Android 开发者并不需要把自己训练成协议工程师，但如果不理解 HTTP 和 REST 的基本思想，后面的 OkHttp、Retrofit、错误包装和缓存设计都会显得像黑盒。本章的任务，是建立一套足够工程化的理解：请求为什么有方法和头，状态码为什么重要，REST 又到底在帮助我们组织什么。

## 学习目标

- 理解 HTTP 请求和响应的关键组成部分。
- 理解常见请求方法和状态码在工程中的意义。
- 理解 REST 是一种资源组织思路，而不是魔法规范。
- 为 Retrofit 接口声明和错误处理打基础。

## 前置知识

- 已理解网络层处理的是不确定性而不是单次成功调用。
- 已知道请求与响应是网络交互的基本结构。

## 正文

### 1. HTTP 为什么重要

在现代应用中，HTTP 是最常见的网络通信基础协议之一。对移动应用来说，理解 HTTP 的意义并不是为了手写底层报文，而是为了知道：

- 客户端究竟向服务端表达了什么。
- 服务端又如何明确回应。
- 为什么同样是“失败”，原因却可能完全不同。

这层理解会直接影响你后面如何设计接口声明、错误分类和缓存策略。

### 2. 一个 HTTP 请求通常包含什么

一个 HTTP 请求通常至少包含：

- 方法。
- URL。
- 请求头。
- 查询参数或请求体。

这些元素共同表达的是：我要对哪个资源做什么操作、带着什么上下文去做、同时还要传递哪些额外信息，例如鉴权、语言、内容类型等。

### 3. 常见请求方法为什么不是随便挑一个

入门时最常见的误解之一，是觉得 GET、POST、PUT、DELETE 只是不同名字的“发请求方式”。实际上它们表达的是不同意图：

- GET 更强调读取资源。
- POST 通常表示创建或提交动作。
- PUT/PATCH 更偏向更新。
- DELETE 更偏向删除。

理解方法语义的价值，在于它帮助客户端和服务端都保持更清晰的资源操作边界，而不是靠接口名字硬猜。

### 4. 状态码为什么值得认真对待

很多初学者会把状态码简单分成“200 成功，其他失败”，这在学习早期能用，但不够。状态码真正重要的地方在于：它帮助你区分失败性质。

例如，失败可能来自：

- 客户端请求格式问题。
- 鉴权失败。
- 资源不存在。
- 服务端处理异常。
- 临时性网络或网关问题。

只有把失败性质分清，后面你的错误提示、重试策略和监控信息才有可能合理。

### 5. 请求头和内容类型为什么不是边角知识

请求头经常被教程快速带过，但现实里它们非常重要。它们常常承载：

- 身份认证信息。
- 内容类型说明。
- 语言和本地化偏好。
- 缓存控制信息。

也就是说，请求头不是附加品，而是通信上下文的一部分。只看 URL 和参数，不看头信息，你对一次请求的理解往往是不完整的。

### 6. REST 是什么，不是什么

REST 常被讲得过于神秘。对 Android 客户端开发者来说，更实用的理解是：REST 是一种围绕资源组织接口的思路，它强调：

- 用 URL 表达资源。
- 用 HTTP 方法表达操作意图。
- 用统一约定让接口更可理解。

它不是一个只要贴上标签就天然“高级”的东西，也不保证每个自称 REST 的接口都设计得合理。学习 REST 的重点，是让你更容易读懂和设计资源式接口，而不是背概念。

### 7. “接口返回成功”不等于“业务一定成功”

这是 HTTP 学习中特别值得强调的一点。即使 HTTP 层面请求成功，也不代表业务一定满足预期。例如：

- HTTP 成功，但返回空列表。
- HTTP 成功，但业务字段提示权限不足。
- HTTP 成功，但返回的数据版本不符合当前客户端逻辑。

这说明应用需要区分：

- 协议层成功或失败。
- 业务层成功或失败。

后面的 Result 包装章节会专门处理这个问题。

### 8. 最小示例：文章列表接口为什么不是“只要 200 就结束”

一个文章列表请求即使返回 200，也仍然可能遇到：

- 数据为空。
- 某些字段缺失。
- 分页信息不完整。
- 当前用户无权查看部分内容。

这就是为什么学习 HTTP 时，不能只停留在“会看状态码”，而要进一步理解“协议结果如何进入业务结果”。

### 9. 最小实践任务

1. 找一个你常用的接口，写出它的请求方法、URL、主要请求头和关键响应字段。
2. 列出这个接口至少三种失败情况，并区分哪些属于协议层，哪些属于业务层。
3. 观察你当前代码里是否把所有非成功都简单当作同一种错误处理。

### 10. 常见误区

- 认为 HTTP 只是会发请求和拿响应。
- 把所有状态码都粗暴当成“成功”或“失败”两类。
- 不理解请求头和内容类型的角色。
- 把 REST 当成神秘名词，而不是资源组织思路。

## 小结

HTTP 和 REST 提供的是网络交互的基本语义框架。理解了请求方法、状态码、头信息和资源思维之后，后面的 OkHttp 和 Retrofit 才不会只剩下“照着模板写接口”。
