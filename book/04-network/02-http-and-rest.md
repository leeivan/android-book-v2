# HTTP 与 REST

上一章解释了“网络层是什么问题”，这一章继续回答“这些请求和响应通常以什么协议和风格组织”。在现代移动应用里，HTTP 和 REST 仍然是最常见的基础语义。

## 学习目标

- 理解请求方法、状态码和报文结构。
- 理解 REST 在移动应用接口设计中的常见模式。
- 为 OkHttp 和 Retrofit 提供接口语义基础。

## 前置知识

- 已理解网络请求与响应的基本概念。
- 已知道 JSON 常作为常见返回体格式。

## 正文

### 1. 为什么要先理解 HTTP 而不是直接学库

库可以换，协议语义通常不会轻易变。只有先理解 HTTP，你才能判断：

- 这个接口在做读取还是修改。
- 为什么会返回不同状态码。
- 某个请求是否应该携带参数或请求体。

如果跳过协议层，后续遇到任何网络问题都会变成机械排错。

### 2. HTTP 方法在移动端最常见的意义

- `GET`：读取资源。
- `POST`：创建或提交动作。
- `PUT/PATCH`：更新资源。
- `DELETE`：删除资源。

初学阶段不必追求把所有语义背得过细，但应知道：请求方法本身就在表达意图，不是随便选一个字符串。

### 3. 状态码为什么重要

状态码帮助你快速判断请求结果处于哪一类状态：

- 成功。
- 客户端问题。
- 服务端问题。

对移动端来说，状态码的价值在于：它决定你是该更新界面、提示用户、重试，还是记录错误。它不是日志里的装饰信息。

### 4. REST 的实用价值

REST 不是必须崇拜的教条，但它仍然是现代接口设计中非常常见的组织方式。它的价值在于：

- 资源边界清晰。
- 请求语义较稳定。
- 与客户端模型对应关系较容易理解。

对 Android 开发者而言，理解 REST 的重点不在于背诵定义，而在于看懂接口文档和调用语义。

### 5. 最小实践任务

1. 写出一个读取新闻列表的 `GET` 请求示意。
2. 写出一个新增待办事项的 `POST` 请求示意。
3. 为这两个请求分别思考成功和失败时最合理的界面反馈。

### 6. 常见误区

- 不理解请求方法，只会照抄接口示例。
- 不关心状态码，只要拿到返回体就继续处理。
- 把 REST 理解成必须完美遵守的形式，而不是常见资源设计方式。

## 小结

HTTP 和 REST 提供的是网络层的共同语言。理解这层语义之后，后续使用任何具体网络库都会更稳，也更容易定位请求设计问题。
