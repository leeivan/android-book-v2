# HTTP 与 REST

Android 开发者并不需要把自己训练成协议工程师，但如果不理解 HTTP 和 REST 的基本思想，后面的 OkHttp、Retrofit、错误包装和缓存设计都会显得像黑盒。本章的任务，是建立一套足够工程化的理解：请求为什么有方法和头，状态码为什么重要，REST 又到底在帮助我们组织什么。

这一章最容易学偏的地方，是把 HTTP 学成一张状态码表，把 REST 学成几个抽象口号。这样能在短期里背出一些名词，却很难帮助你真正看懂接口设计和错误语义。对 Android 客户端开发者来说，更实用的目标是：能读懂一次请求到底在表达什么，能区分协议层和业务层结果，能看出某个接口结构为什么会让客户端好用或难用。

## 学习目标

- 理解 HTTP 请求和响应的关键组成部分。
- 理解常见请求方法和状态码在工程中的意义。
- 理解 REST 是一种资源组织思路，而不是魔法规范。
- 为 Retrofit 接口声明和错误处理打基础。

## 前置知识

- 已理解网络层处理的是不确定性而不是单次成功调用。
- 已知道请求与响应是网络交互的基本结构。

## 正文

### 1. HTTP 为什么重要

在现代应用中，HTTP 是最常见的网络通信基础协议之一。对移动应用来说，理解 HTTP 的意义并不是为了手写底层报文，而是为了知道：客户端究竟向服务端表达了什么，服务端又如何明确回应，以及为什么同样是“失败”，原因却可能完全不同。RFC 9110 对 HTTP status code 和 method semantics 的定义非常明确，而这些定义会直接影响客户端如何设计请求、错误分类和缓存策略。

如果这一层不清楚，后面的 Retrofit 接口就会变成“照着后端文档把注解抄进去”，你知道怎么调用，却不知道为什么这样组织，更不知道当请求异常时该如何正确解释。

### 2. 一个 HTTP 请求通常包含什么

一个 HTTP 请求通常至少包含四类信息：方法、URL、请求头，以及查询参数或请求体。这些元素共同表达的是“我要对哪个资源做什么操作、带着什么上下文去做、还要传递哪些额外信息”。例如，URL 决定目标资源，请求方法决定意图，请求头承载认证、语言、内容类型等上下文，请求体或查询参数则承载本次交互所需的数据。

请求对应的响应也有类似结构：状态码、响应头和响应体。状态码告诉你这次协议交互处于什么结果类别，响应头提供更多协商与上下文信息，响应体则承载内容或错误详情。理解这套结构后，你再去看 OkHttp 或 Retrofit，就不会只看到一堆封装函数，而能看到它们背后真正处理的是哪一层网络语义。

### 3. 请求方法为什么不是随便挑一个

RFC 9110 对标准方法的语义定义得很清楚：GET 用于获取目标资源的当前表示，POST 请求目标资源按其特定语义处理请求内容，PUT 用于替换目标资源的当前表示，DELETE 则用于删除目标资源。对客户端来说，这些方法不是不同名字的“发请求方式”，而是在表达不同意图。

理解方法语义的价值，在于它帮助客户端和服务端都保持更清晰的资源操作边界。比如一个读取动作如果被设计成 POST，客户端就更难自然地理解它是否可缓存、是否只是查询；一个更新动作如果语义含混，后续重试和错误处理策略也会更难推导。对 Android 开发者来说，方法语义不是纯后端知识，它会直接影响接口可读性和错误判断。

### 4. 状态码为什么值得认真对待

很多初学者会把状态码简单分成“200 成功，其他失败”，这在学习早期勉强能用，但远远不够。RFC 9110 明确把状态码按首位数字分成五类：1xx 表示信息性响应，2xx 表示成功，3xx 表示重定向，4xx 表示客户端错误，5xx 表示服务端错误。这个分类非常重要，因为它让客户端首先能判断“失败的性质”。

只有把失败性质分清，后面的错误提示、重试策略和监控信息才有可能合理。比如 401/403 更接近鉴权或权限问题，404 更接近资源不存在，5xx 则意味着服务端出了问题。把所有非 200 结果都统一成“请求失败”，会让客户端丢掉大量本来可以利用的语义。

### 5. 请求头和内容类型，不是边角知识

请求头经常被教程快速带过，但现实里它们非常重要。它们常常承载身份认证信息、内容类型说明、语言与本地化偏好、缓存控制信息等上下文。也就是说，请求头不是附加品，而是通信上下文的一部分。只看 URL 和参数，不看头信息，你对一次请求的理解往往是不完整的。

Android 官方网络文档同样强调，网络安全和通信上下文应被认真对待。比如通过 HTTPS 传输、在正确的层处理认证、合理读取网络状态，这些都依赖你对请求和响应上下文有明确认识，而不是把请求简化成“一个 URL 加一段 JSON”。

### 6. REST 更像一种资源组织思路

REST 常被讲得过于神秘。对 Android 客户端开发者来说，更实用的理解是：REST 是一种围绕资源组织接口的思路，它强调用 URL 表达资源，用 HTTP 方法表达操作意图，用统一约定让接口更容易被理解和复用。它不是一个只要贴上标签就天然“高级”的东西，也不保证每个自称 REST 的接口都设计得合理。

学习 REST 的重点，是让你更容易读懂和设计资源式接口，而不是背概念。比如 `/articles/{id}` 明显在表达一个资源，`GET /articles` 和 `POST /articles` 也天然体现了读取集合与创建资源的不同语义。客户端一旦能够沿着“资源”和“操作意图”去理解接口，后面的调用和错误处理都会更顺。

### 7. 协议层成功，不等于业务一定成功

这是 HTTP 学习中特别值得强调的一点。即使 HTTP 层面请求成功，也不代表业务一定满足预期。例如 HTTP 返回 200，但响应体是空列表；HTTP 返回成功，但业务字段提示权限不足；HTTP 返回成功，但数据版本不符合当前客户端逻辑。这说明应用需要区分两层结果：协议层成功或失败，业务层成功或失败。

如果把这两层混在一起，客户端很快就会遇到混乱：某些请求明明协议没错，却要给用户提示失败；某些错误其实可恢复，却被当成完全不可用。这也是后面结果包装和错误分类章节要继续处理的问题。

### 8. 最小示例：文章列表接口为什么不是“只要 200 就结束”

一个文章列表请求即使返回 200，也仍然可能遇到数据为空、某些字段缺失、分页信息不完整，或者当前用户无权查看部分内容。这个例子说明，学习 HTTP 时不能只停留在“会看状态码”，而要进一步理解“协议结果如何进入业务结果”。200 只能说明本次 HTTP 交互成功完成，不能替你回答业务是否满足页面需求。

对客户端来说，这意味着你不能把“200”直接等同于“页面显示成功”。你仍然需要在更上层判断：空数据怎么展示，字段缺失如何兜底，业务错误码是否需要转换成 UI 可理解的状态。

### 9. 实践任务

起点条件：

- 已有一个你熟悉的接口样例，或者愿意用“文章列表/任务列表接口”作为练习对象。

步骤：

1. 写出这个接口的请求方法、URL、主要请求头和关键响应字段。
2. 判断它当前更像是在表达“资源读取”“创建动作”还是“更新动作”。
3. 列出这个接口至少三种失败情况，并区分哪些属于协议层，哪些属于业务层。
4. 观察你当前代码里是否把所有非成功都简单当作同一种错误处理。

预期结果：

- 你能从方法、状态码和头信息角度读懂一次请求。
- 你能区分协议语义和业务语义。
- 你会更容易看懂后面的 Retrofit 接口声明和错误包装设计。

自检方式：

- 你能解释：为什么请求方法不是随便挑一个。
- 你能说出：状态码分类为什么比“200/非200”更有工程价值。
- 你能判断：一个 HTTP 成功响应为什么仍可能导致页面进入空状态或业务失败状态。

调试提示：

- 如果你总把所有失败都提示成同一句文案，通常说明协议层和业务层结果没有拆开。
- 如果你无法解释某个接口为什么是 POST 而不是 GET，说明对方法语义的理解还不稳。
- 如果你从不看请求头，只盯 URL 和 body，常常会错过认证、语言和缓存协商问题。

### 10. 常见误区

- 认为 HTTP 只是会发请求和拿响应。
- 把所有状态码都粗暴当成“成功”或“失败”两类。
- 不理解请求头和内容类型的角色。
- 把 REST 当成神秘名词，而不是资源组织思路。

## 小结

HTTP 和 REST 提供的是网络交互的基本语义框架。理解了请求方法、状态码、头信息和资源思维之后，后面的 OkHttp 和 Retrofit 才不会只剩下“照着模板写接口”，而会真正成为你能推理、能设计、能调试的网络工具。
