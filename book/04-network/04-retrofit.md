# Retrofit

如果 OkHttp 解决的是“请求怎么执行”，那么 Retrofit 更偏向解决“接口怎么组织”。它把接口调用从零散的请求代码提升为更清晰的声明式结构，因此在 Android 项目中非常常见。

## 学习目标

- 理解 Retrofit 在网络层中的职责。
- 理解接口定义、序列化与调用之间的关系。
- 为 API 结果封装和离线缓存提供基础。

## 前置知识

- 已理解 HTTP、REST 和 OkHttp 的基本角色。
- 已理解 JSON 解析的基础意义。

## 正文

### 1. Retrofit 为什么常被作为主线入口

直接用底层客户端也能完成请求，但接口一多，问题就会出现：

- 每个请求代码结构都不同。
- 参数、路径和解析逻辑分散。
- 维护成本迅速上升。

Retrofit 的价值就在于把“接口是什么”表达得更稳定。这样一来，网络层的可读性和可维护性都会明显提升。

### 2. Retrofit 关注的不是底层连接，而是接口建模

学习 Retrofit 时，最重要的不是去记注解列表，而是理解它在做什么：

- 把接口语义写成清晰声明。
- 把返回数据映射到模型。
- 把调用方式变成更统一的结构。

这使得网络层更接近“定义能力”，而不是“每次手写请求过程”。

### 3. Retrofit 为什么适合和现代 Android 架构搭配

Retrofit 往往会和以下结构一起出现：

- OkHttp 作为执行底座。
- Repository 作为数据协调入口。
- ViewModel 作为状态承接层。

因此，学习 Retrofit 时要始终记住：它不是界面层工具，而是数据层与网络层之间的一部分。

### 4. 哪些问题不该让 Retrofit 独自承担

- 页面状态更新。
- 复杂缓存策略。
- 全部业务错误语义。
- 完整的领域模型转换。

如果把这些问题都压在 Retrofit 上，网络层会过度膨胀。更合理的做法是：Retrofit 负责接口定义和调用入口，结果组织和状态传播交给更高层。

### 5. 最小实践任务

1. 为一个新闻列表接口写出最小 Retrofit 接口声明思路。
2. 标出输入参数、返回模型和调用结果分别应落在哪一层。
3. 思考：如果接口失败，应该由 Retrofit、Repository 还是界面层负责主要反馈。

### 6. 常见误区

- 把 Retrofit 当成完整网络架构。
- 把页面状态处理直接放进接口层。
- 只会照抄接口注解，不理解接口建模。

## 小结

Retrofit 让网络调用从“零散请求代码”变成“可维护接口结构”。理解这一点后，后续 API 结果封装和离线缓存章节就会更容易组织起来。
