# Retrofit

如果说 OkHttp 解决的是“请求如何被可靠地发出去”，那么 Retrofit 解决的就是“如何把网络接口声明成更适合业务代码消费的形式”。它最大的价值，不在于帮你少写几行代码，而在于把 HTTP 接口从零散构造过程提升成了清晰、可复用、可阅读的声明式契约。本章会把 Retrofit 放回它正确的位置：接口声明层，而不是“万能网络框架”。

## 学习目标

- 理解 Retrofit 在 Android 网络栈中的职责。
- 理解接口声明、序列化转换和 `suspend` 支持的基本意义。
- 理解 Retrofit 与 OkHttp、DTO、Repository 的关系。
- 知道为什么 Retrofit 返回值不应直接一路传到页面层。

## 前置知识

- 已理解 HTTP、REST 和 OkHttp 的基本角色。
- 已知道页面层不应直接面对底层网络细节。

## 正文

### 1. Retrofit 到底解决了什么问题

如果直接使用底层 HTTP 客户端，你很快会发现自己在重复做这些事：

- 拼接 URL。
- 指定方法和参数。
- 加请求头。
- 拿到响应体再手工转对象。

Retrofit 的价值就在于把这套过程收束为接口声明。你用接口方法表达“我要调用哪个资源”，它再结合底层客户端和转换器完成具体工作。

这让网络接口从“过程代码”变成“契约描述”。

### 2. 声明式接口为什么特别适合工程协作

一个好的 Retrofit 接口，往往一眼就能看出：

- 请求方法是什么。
- 路径长什么样。
- 参数放在哪里。
- 返回结果大致是什么结构。

这种可读性并不是表面上的“优雅”，而是直接影响团队理解速度和后续维护成本。尤其当接口数量上升时，声明式契约比散乱的请求构造代码更容易管理。

### 3. Retrofit 与 OkHttp 的关系

Retrofit 并不取代 OkHttp。更准确地说：

- Retrofit 站在接口声明和对象转换这一层。
- OkHttp 站在实际 HTTP 通信这一层。

在实际项目里，Retrofit 往往复用你统一配置好的 `OkHttpClient`，因此日志、鉴权、缓存和超时等能力，通常仍由 OkHttp 这层统一提供。

### 4. `suspend` 接口为什么适合现代 Android

Retrofit 支持将接口方法直接声明为 `suspend` 函数，这使它能更自然地融入 Kotlin 协程体系。这样做的价值不是“语法更短”，而是：

- 网络请求可以直接进入协程链路。
- 页面和数据层更容易用结构化并发组织请求。
- 取消、异常和状态更新更容易与 ViewModel 协同。

也就是说，Retrofit 在 Kotlin 项目里的现代用法，天然和协程主线绑定。

### 5. 最小示例：接口声明的重点是什么

下面这个例子只用来观察“声明式接口”而不是完整网络架构：

```kotlin
interface ArticleApiService {
    @GET("articles")
    suspend fun getArticles(): List<ArticleDto>
}
```

这个例子的重点是：

- `@GET` 表达请求方法和路径。
- 方法签名表达这次调用的意图。
- 返回类型表达原始数据契约。

它看起来很简单，但已经把大量底层过程从业务代码里抽离出来了。

### 6. 为什么 Retrofit 返回值不该直接一路传到 UI

这是初学者非常容易走偏的一点。Retrofit 接口直接返回的对象，通常更接近接口契约，而不是页面最终需要的数据结构。它们可能：

- 字段命名更贴近后端。
- 包含页面根本不关心的字段。
- 在多个页面里的展示需求不同。

因此，更合理的链路通常是：

- Retrofit 返回 DTO。
- Repository 或数据层做转换。
- ViewModel 组织为 UI 状态。
- UI 只消费自己需要的结果。

### 7. 转换器和错误处理为什么不是小细节

Retrofit 往往会结合序列化转换器使用，把返回体转成对象。这一步虽然很方便，但也容易让人误以为“数据已经万事大吉”。实际上你仍然需要考虑：

- 返回字段可能缺失。
- 状态码成功不代表业务成功。
- 数据类型可能变化。

工具帮你减少了样板代码，但并没有替你解决边界和容错问题。

### 8. 最小实践任务

1. 选一个接口，把它改写成声明式 Retrofit 接口签名。
2. 明确这个接口返回的是 DTO、业务模型还是 UI 模型。
3. 检查你的项目里是否把 Retrofit 返回值直接交给页面层使用。
4. 思考哪些能力应放在 Retrofit 配置层，哪些应留在 Repository 层。

### 9. 常见误区

- 把 Retrofit 当成完整网络架构本身。
- 不理解它和 OkHttp 的层级关系。
- Retrofit 返回什么，页面就直接用什么。
- 只关注接口能不能调通，不关注模型转换和错误语义。

## 小结

Retrofit 的价值，是把网络接口从零散过程提升成清晰契约。真正成熟的使用方式，不是“所有网络问题都交给 Retrofit”，而是让它专注接口声明，再与 OkHttp、DTO 转换和 Repository 形成清晰分工。
