# API 结果封装

网络请求几乎不会只产生一种结果。除了成功之外，至少还会出现加载中、失败、空数据、部分可用等状态。如果这些结果没有被统一表达，界面层就会到处充满 `if/else` 和临时判断。本章讨论的就是如何把这些状态变成稳定的数据结构。

## 学习目标

- 理解结果封装为何能降低界面层复杂度。
- 理解错误处理、异常处理和空数据处理的基本边界。
- 为 ViewModel 与状态流设计提供支撑。

## 前置知识

- 已理解网络请求和 Repository 的基本角色。
- 已知道界面需要加载态、成功态和失败态。

## 正文

### 1. 为什么不能只返回“成功数据”

如果接口层只把成功数据往上传，界面层最终就不得不自己猜测：

- 当前是不是还在加载。
- 当前是失败还是空数据。
- 当前是否需要重试。

这种设计会让页面状态越来越分散。统一结果封装的意义就在于：把“请求发生了什么”变成一种稳定表达。

### 2. 结果封装真正解决的是状态问题

它不仅解决错误处理，更解决状态组织问题。一个良好的结果封装通常能帮助你表达：

- 正在加载。
- 加载成功。
- 加载失败。
- 可选地表达空数据或局部成功。

这样一来，界面层更容易根据一种稳定结构来渲染状态，而不是临时拼判断。

### 3. 异常和业务失败为什么要区分

很多开发者会把所有失败都塞进一个“错误”概念里，但实际常见的失败至少分两类：

- 技术性失败：如网络错误、超时、解析失败。
- 业务性失败：如接口明确返回失败语义或业务校验未通过。

只有区分这两类失败，界面和日志层才能做出更准确的反馈。

### 4. 结果封装为什么与 ViewModel 和状态流关系密切

在现代 Android 架构里，ViewModel 往往承担“向界面暴露状态”的职责。如果结果封装本身不稳定，ViewModel 就很难把网络状态转换成统一的页面状态。

因此，结果封装并不是网络层内部细节，而是连接网络层、数据层和界面层的重要桥梁。

### 5. 最小实践任务

1. 为一个列表请求设计三种结果：加载中、成功、失败。
2. 思考失败结果中至少应包含哪些信息，才能让界面和日志都可用。
3. 比较“直接返回数据”与“返回统一结果结构”的差异。

### 6. 常见误区

- 接口层只关心成功，失败一律抛给界面。
- 把所有失败都视为同一类错误。
- 结果结构过于复杂，反而增加使用负担。

## 小结

API 结果封装的目标是让不确定的网络结果变成可维护的状态表达。它不是为了增加抽象，而是为了让界面层和 ViewModel 更容易稳定地组织状态。
