# API 结果包装

网络请求真正麻烦的地方，从来不是“拿到一个对象”，而是“如何把请求结果以稳定语义传给上层”。如果页面直接接收底层响应、异常和空值，很快就会出现到处写 `try-catch`、到处判断 `null`、到处拼错误文案的局面。本章的重点，是建立结果包装意识：网络层返回的不是“原始响应本身”，而应是更适合上层消费的结果语义。

## 学习目标

- 理解为什么 API 返回结果需要做统一包装。
- 区分协议错误、业务错误、空数据和成功结果。
- 理解结果包装和 UI 状态不是一回事，但两者紧密相关。
- 为 Repository、ViewModel 和统一错误处理打基础。

## 前置知识

- 已理解 HTTP 状态码与业务成功不是同一层问题。
- 已理解 Retrofit 返回对象不应直接一路传到 UI。

## 正文

### 1. 为什么不能把原始响应直接丢给上层

如果网络层直接把原始响应对象、异常对象或裸数据一路往上抛，上层很快就会承担大量本不属于它的细节：

- 状态码怎么解释。
- 哪种错误该重试。
- 空列表算不算成功。
- 某个业务码该显示什么提示。

这会让页面层和 ViewModel 层充满底层通信细节，结构变得非常脆弱。

### 2. 结果包装真正解决的是什么问题

结果包装的本质，是把网络层复杂且零散的结果收束成更有限、更清晰的语义。例如：

- 成功并有数据。
- 成功但数据为空。
- 失败且可重试。
- 失败且需要登录。
- 失败且属于服务器异常。

它让上层面对的是“已经解释过一轮的结果”，而不是一次原始 HTTP 过程。

### 3. 协议错误、业务错误和解析错误为什么要分开

这三类错误经常被混成一种“请求失败”，但它们完全不是同一个层次。

- 协议错误：例如状态码非成功。
- 业务错误：HTTP 看起来成功，但业务字段表达失败。
- 解析错误：数据结构不符合预期或字段缺失。

只有先把错误层次分清，后面你才能决定该如何记录、重试、提示和恢复。

### 4. 结果包装和 UI 状态有什么关系

它们不是一回事。结果包装主要发生在数据层或网络边界，目的是让上层面对更清晰的语义；UI 状态则是页面最终要显示什么，例如：

- 正在加载。
- 显示内容。
- 显示空状态。
- 显示错误提示。

更自然的链路通常是：

- 网络层产生包装后的结果。
- ViewModel 把它转成 UI 状态。
- 页面只负责渲染 UI 状态。

### 5. 为什么统一结果模型能降低重复代码

在没有统一结果包装时，常见情况是：

- 每个 ViewModel 自己写一套错误判断。
- 每个页面自己拼一套加载和失败分支。
- 同一种错误在不同页面表现完全不同。

统一结果模型并不能让所有页面长得一样，但它能让同类错误至少从同一种语义出发，再根据页面上下文做不同展示。

### 6. 最小示例：新闻列表请求的结果不应只有“成功或失败”

一个新闻列表请求至少可能出现这些结果：

- 成功并返回文章。
- 成功但为空列表。
- 网络超时。
- 服务端异常。
- 鉴权过期。

如果你把它们都粗暴归成“请求失败”，页面就只能展示一种笼统错误；而如果你把它们收束成更清晰的结果模型，ViewModel 就能更准确地组织界面反馈。

### 7. 统一包装并不意味着丢失细节

结果包装常被误解为“把所有错误都包成一个大类”。真正好的包装不是抹平细节，而是在保留关键区分能力的前提下，把对上层无意义的噪声隔离掉。

换句话说，它做的是“语义提炼”，不是“信息消失”。

### 8. 最小实践任务

1. 列出你当前项目里一个接口可能出现的至少五种结果。
2. 把这些结果分成协议层、业务层和解析层三类。
3. 设计一个更统一的结果表达方式，让页面不再直接碰原始异常和响应。
4. 观察你的 ViewModel 是否仍承担了太多底层错误解释工作。

### 9. 常见误区

- 直接把原始响应或异常一路传到 UI。
- 把所有失败都当成一种错误。
- 结果包装和 UI 状态混为一谈。
- 统一包装时把所有关键细节也一起抹掉。

## 小结

API 结果包装的价值，是把复杂、零散的网络结果收束成更适合上层消费的语义。它不是额外负担，而是让页面和 ViewModel 从底层通信细节中解耦的关键一步。
