# 网络基础

一旦应用不再只是本地演示程序，网络几乎就不可避免。新闻内容、用户资料、登录态、商品列表、远程配置、同步任务，都依赖网络通信。本章的目标不是立刻上手某个库，而是先理解移动端网络开发最根本的问题：为什么网络层天然充满不确定性，为什么“请求成功”只是所有情况中的一种，以及为什么网络逻辑不能被当作页面里的一个普通函数调用来处理。

## 学习目标

- 理解移动端网络开发面临的核心不确定性。
- 理解请求、响应、延迟、超时、取消和错误处理的基本概念。
- 理解网络层为什么必须和页面状态、生命周期以及数据层设计一起考虑。
- 为 HTTP、REST、OkHttp、Retrofit 和离线缓存打基础。

## 前置知识

- 已理解数据层设计和 JSON 在远程数据中的常见角色。
- 已知道 ViewModel 和页面状态不应直接混入底层实现细节。

## 正文

### 1. 网络层为什么是独立问题

很多初学者第一次做网络请求，会把它理解成“从服务器拿一个 JSON 回来”。这当然是结果的一部分，但网络开发真正困难的地方在于：一旦请求离开设备，开发者就不再能完全控制执行过程。

网络天然伴随着这些事实：

- 请求可能延迟很久。
- 请求可能失败。
- 请求可能部分成功。
- 用户可能在等待过程中离开当前页面。
- 同一接口在不同网络环境下表现可能完全不同。

这意味着网络开发从第一天起就不能只围绕“成功路径”来设计。

### 2. 从最抽象角度看，请求和响应是什么

网络通信最基本的结构可以先简化成：

- 客户端发出请求。
- 服务端返回响应。

一个请求通常包含：

- 目标地址。
- 请求方法。
- 头信息。
- 查询参数或请求体。

一个响应通常包含：

- 状态码。
- 响应头。
- 响应体。

理解这层结构很重要，因为后面无论你使用 OkHttp、Retrofit 还是其他工具，最终都绕不开这些基本元素。

### 3. 移动端网络与本地函数调用最大的差异

本地代码往往可以假设：函数被调用之后，会很快得到返回结果。网络层不能这样想。只要请求走出设备，就必须默认考虑这些情形：

- 结果会晚到。
- 结果可能为空。
- 结果可能格式不符合预期。
- 请求可能因为超时或连接问题中断。
- 页面可能已经不再需要这次结果。

这也是为什么网络层几乎天然会与协程、状态流、取消机制、缓存和 UI 状态设计绑在一起。

### 4. 网络层真正处理的是不确定性

很多示例会把网络层写成“调用接口 -> 显示结果”。这在教材最小示例里足够直观，但从真实工程角度看，网络层真正要处理的是不确定性。你至少要为以下情况提前做准备：

- 首次加载中的界面该如何表达。
- 网络失败时，是否保留旧数据。
- 用户连续触发重复请求时，应如何去重或取消。
- 一次失败是立即提示，还是可以静默重试。

这说明网络层从来不只是“拿数据”，更是“组织不确定结果”。

### 5. 生命周期为什么会影响网络请求

页面是有生命周期的，而请求也是有生命周期的。用户可能在请求返回前切走页面、旋转屏幕、重新发起搜索或关闭应用。只要页面和请求的生命周期不同步，就会出现常见问题：

- 页面销毁后仍尝试更新 UI。
- 旧请求覆盖新请求结果。
- 重建页面后重复发起相同请求。

因此，现代 Android 网络设计通常不会让页面直接持有请求，而是把请求交给更稳定的状态层或数据层来管理。

### 6. 为什么“成功、失败、加载中”是网络章节必须谈的话题

网络结果不是只有“拿到了数据”和“没拿到数据”两种。对用户来说，一个完整的网络交互至少通常会经历下面这些状态：

- 正在加载。
- 成功拿到数据。
- 失败但可以重试。
- 失败但仍可显示旧内容。
- 数据为空但请求本身成功。

如果不把这些状态作为正式设计对象，页面往往只会在“成功时显示内容，失败时什么都没有”之间摇摆，体验会非常脆弱。

### 7. 网络层和数据层边界为什么要一起看

页面不应该既直接发网络请求，又自己决定缓存、重试、数据转换和错误文案。否则同一个接口一旦被多个页面使用，很快就会出现逻辑重复和行为不一致。

更合理的思路通常是：

- 网络层负责通信。
- 数据层负责协调来源和转换结果。
- ViewModel 负责把结果组织成 UI 可消费状态。
- 页面只负责显示状态和接收用户动作。

这样，网络请求才不会直接污染 UI 层。

### 8. 最小示例：新闻列表的一次请求到底包含哪些问题

一个新闻列表页面看似只是“点开就加载文章”，但它背后至少包含这些问题：

- 首次进入时是否显示骨架屏或进度指示。
- 请求超时后展示什么。
- 下拉刷新时是否保留旧列表。
- 用户退出页面后，请求是否仍应继续。

这个例子说明，网络章节的重点绝不只是“能不能把 JSON 打印出来”，而是你是否开始从结果不确定的角度设计页面和数据流。

### 9. 最小实践任务

1. 画出一条最简单的请求链路：UI -> ViewModel -> Repository -> 网络数据源。
2. 为一个列表请求设计成功、加载中、失败和空数据四种界面状态。
3. 判断用户离开页面后，这个请求是否仍有继续存在的价值。
4. 记录你当前项目里哪些网络逻辑还直接写在页面层。

### 10. 常见误区

- 认为网络层只是“会发请求”。
- 只处理成功路径，不处理失败和空数据。
- 把网络调用直接塞进点击事件或页面初始化代码里。
- 请求返回什么，就原样直接绑定到 UI，不做状态组织。

## 小结

网络开发的本质，是在不确定环境中稳定地组织请求、响应和状态变化。理解了这一点，再去学习 HTTP、REST、OkHttp 和 Retrofit，才不会只停留在“API 会不会写”的层面。
