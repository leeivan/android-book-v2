# 网络基础

一旦应用不再只是本地演示程序，网络几乎就不可避免。新闻内容、用户资料、登录态、商品列表、远程配置、同步任务，都依赖网络通信。本章的目标不是立刻上手某个库，而是先理解移动端网络开发最根本的问题：为什么网络层天然充满不确定性，为什么“请求成功”只是所有情况中的一种，以及为什么网络逻辑不能被当作页面里的一个普通函数调用来处理。

网络章节最容易学偏的地方，是一上来就背 Retrofit 接口和回调写法，而没有先建立“移动网络不是本地函数调用”的直觉。只要这层直觉没有立住，后面无论用 OkHttp 还是 Retrofit，都容易把请求写成只考虑成功路径的直线逻辑。本章先把这种直觉搭起来。

## 学习目标

- 理解移动端网络开发面临的核心不确定性。
- 理解请求、响应、延迟、超时、取消和错误处理的基本概念。
- 理解网络层为什么必须和页面状态、生命周期以及数据层设计一起考虑。
- 为 HTTP、REST、OkHttp、Retrofit 和离线缓存打基础。

## 前置知识

- 已理解数据层设计和 JSON 在远程数据中的常见角色。
- 已知道 ViewModel 和页面状态不应直接混入底层实现细节。

## 正文

### 1. 网络层为什么是一个独立问题

很多初学者第一次做网络请求，会把它理解成“从服务器拿一个 JSON 回来”。这当然是结果的一部分，但网络开发真正困难的地方在于：一旦请求离开设备，开发者就不再能完全控制执行过程。请求可能延迟很久，可能失败，可能部分成功，用户也可能在等待过程中离开当前页面，甚至同一接口在不同网络环境下表现都可能完全不同。

这意味着网络开发从第一天起就不能只围绕“成功路径”来设计。真正要组织的是“不确定结果如何进入页面和数据层”，而不是“如何把一次成功响应打印出来”。

### 2. 请求和响应，是网络通信最小但不够简单的结构

从最抽象角度看，网络通信可以先简化成：客户端发出请求，服务端返回响应。一个请求通常包含目标地址、请求方法、头信息，以及查询参数或请求体；一个响应通常包含状态码、响应头和响应体。理解这层结构很重要，因为后面无论你使用 OkHttp、Retrofit 还是其他工具，最终都绕不开这些基本元素。

但对移动端来说，知道“请求和响应长什么样”还不够。更关键的是，你要知道这两个对象的生命周期并不一定和页面一致。请求可能晚于页面需要才返回，也可能在页面离开后仍继续执行。这正是网络层必须被单独设计的原因。

### 3. 移动端网络与本地函数调用最大的差异

本地代码往往可以假设：函数被调用之后，会很快得到返回结果。网络层不能这样想。只要请求走出设备，就必须默认考虑这些情形：结果会晚到，结果可能为空，结果可能格式不符合预期，请求可能因为超时或连接问题中断，页面也可能已经不再需要这次结果。

Android 官方的“Connect to the network”文档对这点给出的提醒非常直接：不要在主线程上做网络操作，否则会抛出 `NetworkOnMainThreadException`。这不只是线程规则，而是在提醒你，网络天然是一个异步、不确定、可能阻塞的过程，不能被当作普通 UI 代码的一部分。

### 4. 网络层真正处理的是不确定性，而不是“拿到数据”

很多示例会把网络层写成“调用接口 -> 显示结果”。这在教材最小示例里足够直观，但从真实工程角度看，网络层真正要处理的是不确定性。你至少要为这些情况提前做准备：首次加载中的界面该如何表达，网络失败时是否保留旧数据，用户连续触发重复请求时应如何去重或取消，一次失败是立即提示还是可以静默重试。

只要这些问题没有被纳入正式设计，网络请求就会在页面中表现得非常脆弱。用户点了一下按钮，没有反馈；换个网络环境，界面就卡住；旋转屏幕后同一个请求又发了一遍；旧请求回来后把新数据覆盖掉。这些都不是“库不会用”，而是没有把网络当成不确定系统来组织。

### 5. 生命周期为什么会影响网络请求

页面是有生命周期的，而请求也是有生命周期的。用户可能在请求返回前切走页面、旋转屏幕、重新发起搜索或关闭应用。只要页面和请求的生命周期不同步，就会出现常见问题：页面销毁后仍尝试更新 UI，旧请求覆盖新请求结果，重建页面后重复发起相同请求。

因此，现代 Android 网络设计通常不会让页面直接持有请求，而是把请求交给更稳定的状态层或数据层来管理。页面只负责触发意图和显示状态，这也是为什么网络层总会和 ViewModel、Repository、取消机制、缓存和状态组织绑在一起。

### 6. 为什么“加载中、成功、失败、空数据”必须被当成正式状态

网络结果不是只有“拿到了数据”和“没拿到数据”两种。对用户来说，一个完整的网络交互至少通常会经历这些状态：正在加载、成功拿到数据、失败但可以重试、失败但仍可显示旧内容、数据为空但请求本身成功。如果不把这些状态作为正式设计对象，页面往往只会在“成功时显示内容，失败时什么都没有”之间摇摆，体验会非常脆弱。

后面你会看到，OkHttp 和 Retrofit 负责的是请求执行，而不是 UI 状态建模。真正把这些状态组织清楚的，仍然是数据层和 ViewModel。

### 7. 网络安全和权限，也是网络基础的一部分

Android 官方网络文档明确要求：要进行网络操作，应用清单中必须声明 `INTERNET` 权限；如果需要读取网络状态，还要声明 `ACCESS_NETWORK_STATE`。文档还强调，这两者都属于 normal permission，不需要运行时请求。另一方面，官方安全文档也明确建议应用通过加密通道传输数据，并指出从 Android 9 开始，常见 HTTP 客户端对明文流量默认是受限的。

这说明网络基础不仅是“怎么发请求”，还包括“请求是否安全”“应用是否具备正确权限”“是否遵守平台默认安全边界”。这些内容虽然不炫技，但决定了网络代码能不能进入真实项目。

### 8. 最小示例：一个新闻列表请求真正要考虑什么

一个新闻列表页面看似只是“点开就加载文章”，但它背后至少包含这些问题：首次进入时是否显示骨架屏或进度指示，请求超时后展示什么，下拉刷新时是否保留旧列表，用户退出页面后请求是否仍应继续。这个例子说明，网络章节的重点绝不只是“能不能把 JSON 打印出来”，而是你是否开始从结果不确定的角度设计页面和数据流。

### 9. 实践任务

起点条件：

- 已有一个需要远程数据的列表页面设想，例如新闻、待办同步或商品列表。

步骤：

1. 画出一条最简单的请求链路：UI -> ViewModel -> Repository -> 网络数据源。
2. 为一个列表请求设计加载中、成功、失败和空数据四种界面状态。
3. 判断用户离开页面后，这个请求是否仍有继续存在的价值。
4. 检查你当前项目或草稿里，是否还有网络逻辑直接写在页面层。

预期结果：

- 你能把网络理解成“状态和不确定性”问题，而不是单次函数调用。
- 你能说清网络请求为什么必须脱离主线程。
- 你开始把页面状态和网络状态分开组织。

自检方式：

- 你能解释：为什么网络层不能只设计成功路径。
- 你能判断：某个请求结果应进入 UI 状态、数据层，还是只是一次临时动作。
- 你能说出：为什么页面生命周期会直接影响网络请求组织方式。

调试提示：

- 如果页面销毁后还在更新 UI，先回头检查请求是否被页面直接持有。
- 如果旧请求覆盖新请求结果，说明请求和状态的协调还没做好。
- 如果你只在乎“接口通了没”，却没设计加载和失败状态，UI 往往会显得非常脆弱。

### 10. 常见误区

- 认为网络层只是“会发请求”。
- 只处理成功路径，不处理失败和空数据。
- 把网络调用直接塞进点击事件或页面初始化代码里。
- 请求返回什么，就原样直接绑定到 UI，不做状态组织。

## 小结

网络开发的本质，是在不确定环境中稳定地组织请求、响应和状态变化。理解了这一点，再去学习 HTTP、REST、OkHttp 和 Retrofit，才不会只停留在“API 会不会写”的层面，而会开始真正搭建可维护的网络数据流。

下一章我们会继续往下钻，进入 HTTP 与 REST，把请求方法、状态码和资源语义这些更底层的网络语言讲清楚。
