# 网络基础

一旦应用不再只是本地演示程序，网络几乎就不可避免。新闻内容、用户资料、登录态、商品列表、远程配置、同步任务，都依赖网络通信。本章的目标不是立刻上手某个库，而是先理解移动端网络开发的基本问题：请求是什么、响应是什么、为什么失败和延迟不能当作异常情况来忽略。

## 学习目标

- 理解移动端网络开发的核心挑战。
- 理解请求、响应、延迟和错误处理的基本概念。
- 为 HTTP、REST、OkHttp 和 Retrofit 提供背景。

## 前置知识

- 已理解数据层设计。
- 已知道 JSON 常作为远程数据交换格式。

## 正文

### 1. 网络层为什么是独立问题

很多初学者第一次做网络请求，会把它理解成“从服务器拿个 JSON 回来”。这当然是结果之一，但真正的网络开发远比这复杂，因为它天然伴随着：

- 延迟。
- 失败。
- 重试。
- 超时。
- 不稳定连接。

这意味着网络开发从一开始就不能只按“成功路径”来设计。

### 2. 请求与响应是什么

从最抽象的角度看，网络通信可以理解为：

- 客户端提出请求。
- 服务器返回响应。

请求里通常包含：

- 目标地址。
- 请求方法。
- 请求头。
- 参数或请求体。

响应里通常包含：

- 状态码。
- 返回头。
- 返回体。

理解这层结构很重要，因为后续无论你用 OkHttp、Retrofit 还是其他库，最终都绕不开这些基本元素。

### 3. 移动端网络与本地程序的最大差异

本地代码可以假设函数调用通常会立即返回结果；网络层则不能。只要请求走出设备，就必须默认考虑以下事实：

- 结果可能晚到。
- 结果可能失败。
- 结果可能为空。
- 用户可能在等待期间离开当前页面。

这也是为什么网络层几乎天然需要和协程、状态流、缓存和界面状态设计绑定在一起。

### 4. 网络层最容易被低估的问题

- 只处理成功，不处理失败。
- 请求直接写在页面点击事件里。
- 界面没有加载态和错误态。
- 不考虑重复请求和取消请求。

这些问题在最小 Demo 中不明显，但只要页面稍微复杂，就会迅速影响可用性。

### 5. 最小实践任务

1. 画出一条最简单的请求链路：界面 -> ViewModel -> Repository -> 网络层。
2. 写下一个请求成功、失败、超时三种情况下，界面分别该显示什么。
3. 思考：如果用户离开页面，请求是否还需要继续。

### 6. 常见误区

- 认为网络层只是“会发请求”。
- 只在控制台看到返回数据就算完成。
- 把所有网络逻辑直接塞进界面层。

## 小结

网络开发的本质是处理不确定性。理解这一点之后，再去学习 HTTP、REST、OkHttp 和 Retrofit，才不会只停留在“API 会不会写”的层面。
