# 符号说明

为了让正文结构更稳定、阅读节奏更清晰，本书统一使用下列标记方式。它们的目的不是增加形式感，而是帮助读者快速识别信息的层级和用途。

## 行文中的常见标记

- `代码`：表示命令、路径、类名、库名、关键 API 或配置项。
- 提示：表示容易忽略但实际非常常见的实现细节。
- 注意：表示兼容性、平台行为变更、安全或性能风险。
- 实践任务：表示建议读者亲手完成的最小练习。
- 常见误区：表示初学者或从旧项目迁移时最容易踩到的坑。

这些标记不是为了制造格式感，而是为了减少读者在长篇技术书中最常见的疲劳问题：分不清当前读到的是主结论、补充提醒、操作练习还是风险边界。

## 代码和概念的阅读方式

当正文中出现一段代码时，默认应把它当成“说明某个设计点的最小示例”，而不是一个可以直接覆盖所有场景的万能模板。也就是说，代码示例的重点通常在于：

- 展示一个核心 API 是如何参与整体流程的。
- 展示一个架构角色应该放在哪里。
- 展示某种写法为什么比旧方案更适合作为当前默认路径。

因此，阅读代码时不要急着把它当成“万能模板”复制到项目里，而应先问自己三件事：它解决了哪个问题，它放在这一层的理由是什么，它省略了哪些在真实项目里仍然需要继续补上的边界。

## 关于版本与平台说明

凡是涉及 Android 版本、平台行为变更、权限模型或 Google Play 发布规则的表述，都应优先理解成“平台边界说明”，而不是简单的背景知识。这类内容往往会直接影响代码设计，而不是只影响概念记忆。

如果一处内容同时出现“当前推荐做法”和“历史写法”，默认应把前者视为新项目主线，把后者视为理解旧项目和迁移路径所需的背景材料。

## 关于术语使用

本书正文会尽量统一采用以下写法：

- 用“页面状态”描述概念层，用 ``uiState`` 描述代码中的状态对象。
- 用“单一可信来源”描述数据主线原则。
- 用“数据层 / UI 层 / Domain 层”描述职责，而不是只列目录名。

阅读时如果遇到英文原词，多半是为了帮助你和官方文档对齐，而不是要求你背英文术语本身。

## 小结

读书时如果能先分清“这段是在讲概念、这段是在讲风险、这段是在讲练习”，会显著降低 Android 教程常见的阅读疲劳感。本书使用这些标记，就是为了让这种区分更稳定。
