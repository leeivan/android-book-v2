# 版本管理与维护

很多项目在 1.0 发布前，对“版本”的理解还停留在 `versionCode` 和 `versionName` 这两个字段上。真正进入持续发布以后，你才会发现版本管理远不只是“数字怎么涨”。它实际在回答一组长期问题：这次发布对应什么变化，用户如何平滑升级，哪些兼容性风险必须提前处理，什么时候应该停掉旧方案，平台和依赖升级如何融入发布节奏。

从工程角度看，版本是团队组织变化、控制风险和解释变化的一种语言。系统和商店需要它判断升级顺序，团队需要它回溯构建历史，测试需要它确认验证对象，用户也需要它理解这次更新大概意味着什么。只要版本策略仍然停留在“发版前把数字随手改一下”，后面的维护、排障和平台适配就很容易陷入混乱。

这一章会把版本管理放回“长期维护”语境里理解。重点不是发明一套花哨命名规则，而是回答几个更根本的问题：哪些字段服务平台，哪些字段服务人类沟通；为什么每次发布都要保证单调升级；为什么平台升级、依赖升级、崩溃回看和热修复边界，其实都属于版本管理的一部分。

## 学习目标

- 理解版本管理的核心是组织变化，而不只是填写版本号。
- 理解 `versionCode`、`versionName` 和升级链路之间的关系。
- 学会把平台升级、依赖升级和线上质量回看纳入维护节奏。
- 建立“发布之后才真正进入维护期”的意识。

## 前置知识

- 已理解应用签名、release 构建和 Google Play 上架流程。
- 已接触崩溃监控、日志回看和持续集成基础。

## 正文

### 1. `versionCode` 和 `versionName` 服务的是不同对象

Android 官方文档一直强调，这两个字段虽然同时出现在 `defaultConfig` 里，但职责并不相同。`versionCode` 主要服务系统与分发平台，用来判断升级顺序；它必须随着每次正式发布单调递增，而且当前官方文档仍然给出了上限 `2100000000`。`versionName` 则更偏向人类阅读，用来表达“这是什么版本”，例如 `1.4.0`、`2.0-beta1` 之类。

只要把这条边界分清楚，很多发布决策就会自然变稳：`versionCode` 首先要保证平台和升级关系不混乱，`versionName` 则承担对团队、测试和用户沟通变化的职责。它们经常一起改，但不应该被当成同一回事。

### 2. 版本号只是表面，真正重要的是“这次变化是什么”

一个成熟版本至少应该让人回答出三个问题：这次改动主要是什么，它属于新功能、修复还是兼容性维护，发布后最该观察的风险点是什么。如果版本号只是简单递增，平台当然能识别顺序，但团队和用户对变化的理解仍然可能混乱。

这也是为什么很多团队会在版本管理里同时维护发布说明、变更摘要和风险备注。你不一定非要强行套一整套复杂规范，但至少要让“这次为什么发布”和“这次不准备夹带什么”变得可回答。版本管理不仅在决定发什么，也在决定这一次不该顺手混进哪些额外变化。

### 3. 在 Gradle 里写对字段，只是第一步

从配置角度看，最小版本字段通常像这样：

```kotlin
android {
    defaultConfig {
        versionCode = 42
        versionName = "1.4.0"
    }
}
```

真正要注意的，不是这两行怎么写，而是它们在整个发布链路里怎么被维护。如果你的项目存在多构建变体、多渠道或不同测试轨道，那么所有会被提交到分发平台的正式产物都必须维持正确的升级顺序。也就是说，不能出现某个新构建的 `versionCode` 反而小于已经分发出去的旧构建，否则平台会直接拒绝升级链路。

因此，CI 自动递增也好，人工维护也好，核心标准都不是“看起来省事”，而是“不会破坏升级顺序，不会让不同产物之间的版本关系失控”。

### 4. 修复版必须学会克制，别把热修复做成功能拼车

真实团队里最常见的版本管理失控场景之一，就是修复版本顺手夹带新功能。表面上看这很高效，实际上它会迅速模糊发布边界：本来只是修一个崩溃，最后却把 UI 微调、文案优化、埋点修改和实验逻辑一起放进去。等线上出问题时，团队反而更难判断到底是哪一类变化引发了回归。

所以版本管理不只是编号管理，也是发布纪律管理。修复版更强调恢复稳定，功能版更强调可见变化，兼容性版更强调平台和依赖适配。边界越清楚，验证越精准，回滚和回溯也越容易。

### 5. 维护工作从第一个正式版本上线后才真正开始

很多团队会把 1.0 上线看成项目完成。现实通常恰好相反：1.0 只是维护的起点。之后你会持续面对崩溃和 ANR 回看、用户反馈、新机型兼容性、新系统行为变更、目标 API 提升要求、依赖库升级压力和旧代码清理窗口。

这说明版本管理不该只在“发版前一天”才出现。更成熟的节奏通常会把版本工作拉长到整个周期：需求阶段决定本次变化边界，开发阶段为发布说明和风险点做标记，发布阶段确定版本号和验证清单，上线后再回看质量指标与用户反馈。只要这条节奏存在，版本就会真正成为组织变化的工具。

### 6. 平台升级和依赖升级，也属于版本管理主线

Android 平台是持续变化的。`targetSdk`、权限模型、后台限制、通知规则、隐私要求和构建工具都会变化。如果团队只盯业务功能，不给平台升级和依赖升级留固定窗口，项目会越来越难维护。等到 Play 目标 API 截止日期逼近时，再集中补课，风险通常更大。

因此，更健康的维护方式是把这些工作前置进版本节奏：每个较大版本回看一次平台变化，每几个迭代整理一次关键依赖升级，每个正式发布之后检查一次是否有已经可以删除的兼容代码。它们也许不像新功能那样显眼，但直接决定项目一年后还能不能健康迭代。

### 7. 一个更稳妥的维护节奏长什么样

对大多数应用来说，不需要一开始就设计复杂的发布治理体系，但至少可以建立一套最小节奏。例如：

- 每次发布前确认版本边界、版本号和关键风险点。
- 每次发布后回看崩溃、ANR、评分和主要反馈。
- 每月或每个里程碑回看一次平台要求和关键依赖。
- 每个较大版本检查一次哪些旧兼容逻辑已经可以下线。

这种节奏的价值，在于把维护工作从“谁想起来谁处理”变成固定流程。版本管理一旦进入周期化节奏，很多技术债就不再只会在截止日期前集中爆炸。

### 8. 实践任务

起点条件：

- 已有至少一个对外发布版本，或正在建立持续发布节奏。

步骤：

1. 检查当前项目的 `versionCode` 和 `versionName` 规则，明确二者各自服务的平台或对象。
2. 回看最近两个版本，判断它们是否真的保持了清晰的发布边界。
3. 为下一次发布写一份最小变更说明，区分功能变化、修复变化和兼容性变化。
4. 建立一份维护回看清单，至少覆盖崩溃、ANR、用户反馈、依赖升级和平台要求。
5. 检查当前项目是否存在已经被拖延过久的 API 适配或依赖升级项，并把它们排进后续版本计划。

预期结果：

- 你会把版本管理理解成控制变化的策略，而不是改两个字段。
- 你能更清楚地区分功能版、修复版和维护版之间的边界。
- 项目会开始形成最小可持续的维护节奏。

自检方式：

- 你能解释 `versionCode` 和 `versionName` 分别服务谁。
- 你能判断某次发布为什么更适合作为修复版而不是功能版。
- 你能说明为什么目标 API 和依赖升级也属于版本管理的一部分。

调试提示：

- 修复版总是夹带额外变化，说明发布纪律不稳定。
- 版本号虽然递增，但没人说得清本次到底改了什么，说明版本管理还停留在表面。
- 平台和依赖升级总被无限期推迟，说明维护节奏实际上不存在。

### 9. 常见误区

- 把版本管理理解成两个字段怎么填写。
- 只追求发版频率，不追求发布边界和回看节奏。
- 把依赖升级和平台适配长期当成可以无限延期的技术债。
- 版本发布后完全不回看质量和用户反馈。

## 小结

版本管理与维护真正要解决的，不是编号本身，而是应用如何在真实用户环境中持续、稳定、可控地演进。只要把版本管理放回“组织变化、控制风险、安排维护”的语境里，你就会自然开始关心发布边界、回看节奏、平台升级和依赖治理，而不仅仅是下一次把数字改成多少。

到这里，发布篇的主线也就完整了：签名负责身份连续性，release 构建负责正式产物质量，Google Play 负责分发和放量，版本管理负责长期维护。后面的综合项目章节，正是把这整条链路真正落回到项目实践里。

## 练习题

1. 概念理解题：为什么 `versionCode` 和 `versionName` 虽然常常同时修改，却不应该被当成同一类字段？
2. 编码实现题：为当前项目设计一套可执行的版本递增规则，并验证它不会破坏现有升级顺序。
3. 拓展思考题：如果团队长期把平台升级和依赖升级推迟到最后一刻，最可能在什么节点集中付出代价？
