# 发布构建

发布构建并不等于“把 debug 改成 release 再打个包”。它是从开发态到交付态的一次正式切换，里面包含代码压缩、资源处理、签名、版本信息、日志策略、依赖检查和产物验证等一整套工程动作。真正成熟的团队不会把发布构建当成临时手工操作，而会把它视为一条可重复、可验证、可自动化的生产流程。

Android 官方的发布构建文档本质上就在做这件事：把“给用户的版本”与“给开发者自己用的版本”明确区分开。本章要建立的，就是这种“交付态构建”意识。

## 学习目标

- 理解发布构建和调试构建在目标上的差异。
- 理解压缩、混淆、资源优化和日志策略为什么会进入发布构建。
- 理解发布产物在生成前后分别需要关注什么。
- 为商店上传、灰度分发和版本维护打基础。

## 前置知识

- 已理解签名、Gradle 和构建变体。
- 已知道 debug 构建并不等于可交付版本。

## 正文

### 1. 发布构建的目标是“给用户稳定使用”，而不是“开发方便”

调试构建更关注的是：

- 可快速安装。
- 易于调试。
- 日志和调试开关更宽松。

发布构建则更关注：

- 稳定性。
- 安全性。
- 体积和性能。
- 可持续升级。

所以发布构建不是 debug 的“轻微修改版”，而是完全不同的交付目标。

### 2. 为什么发布构建会引入压缩与优化

代码压缩、资源优化和最小化处理之所以会进入发布流程，不是为了追求技巧，而是因为它们会直接影响：

- 包体积。
- 启动性能。
- 逆向可读性。
- 未使用代码与资源的冗余程度。

但这些优化也意味着发布版本的行为可能和 debug 有差异，所以更需要在发布前做稳定验证，而不能只在 debug 上跑通一次就放心。

### 3. 发布构建前最值得检查哪些内容

在真正生成产物前，最需要确认的通常包括：

- 版本号是否正确。
- 签名配置是否正确。
- 关键变体是否按预期构建。
- 日志和调试开关是否符合发布要求。
- 敏感配置是否没有误带入正式版本。

这些检查很多都适合自动化完成，但前提是你先知道自己到底要检查什么。

### 4. 发布产物生成后也需要验证

很多团队把“构建成功”误当成“发布就没问题”。实际上，产物生成后至少还应该验证：

- 包名与版本是否正确。
- 图标、应用名和资源是否对应正确渠道与环境。
- 是否能正常安装升级。
- 关键功能是否在 release 行为下仍正常。

因为 release 构建和 debug 构建往往存在实际差异，不做产物验证就很容易把问题带到用户手上。

### 5. 发布构建为什么必须和日志策略一起看

很多项目在 debug 阶段依赖宽松日志和调试开关，一旦切到 release，就会暴露出：

- 某些逻辑只在 debug 下成立。
- 日志依赖过重。
- 错误信息不再可见。

这说明发布构建不是单一的打包动作，它也在检验项目是否真的把调试逻辑和正式逻辑边界分清了。

### 6. 一个更接近真实项目的例子

假设一个应用在 debug 下运行稳定，但切到 release 后登录失败。根因可能不是“发布构建很神秘”，而是：

- 环境配置切换错误。
- 混淆规则遗漏。
- 调试开关遮蔽了真实行为。

这个例子说明，发布构建最大的价值之一，就是把开发态中被掩盖的问题提前暴露出来。

### 7. 实践任务

起点条件：

- 已具备可运行的 release 变体或准备首次生成正式产物。

步骤：

1. 列出当前项目中 debug 与 release 最关键的差异点。
2. 为 release 构建设计一份构建前检查清单。
3. 生成一次正式产物后，验证安装、版本号和关键功能。
4. 检查是否存在只在 debug 下才能工作的逻辑。
5. 思考哪些验证步骤应进入自动化流水线，而不是靠人工记忆。

预期结果：

- 你会把发布构建看成正式交付流程，而不是“最后打个包”。
- 你能更主动发现 debug 与 release 的真实差异。
- 你会为 Play 上传和灰度发布建立更稳的基础。

自检方式：

- 你能解释：release 构建和 debug 构建在目标上有什么根本差异。
- 你能判断：哪些检查必须在发布前完成。
- 你能说出：为什么产物生成后仍然需要验证。

调试提示：

- 如果 release 才暴露大量问题，优先检查 debug 依赖和环境切换边界。
- 如果每次发布都靠记忆完成检查，优先把检查清单和自动化补起来。
- 如果团队从没验证过正式产物安装和升级行为，发布风险仍然很高。

### 8. 常见误区

- 把发布构建理解成“debug 改个开关”。
- 构建成功就当作发布没问题。
- 不验证 release 下的真实行为。
- 把发布前检查全靠人工记忆。

## 小结

发布构建的重点，不是导出一个文件，而是把应用从开发态稳定切换到交付态。只要你能围绕这个目标组织检查、验证和自动化，后续商店发布和长期维护都会稳得多。

### 教材化延伸：为什么发布构建不能只学出包按钮

发布构建章节真正重要的，不是“如何生成一个 release APK/AAB”，而是理解调试构建和发布构建为什么必须有不同的约束、资源、日志和安全策略。教材化写法要让读者知道，release build 是工程边界的收束点，而不是 IDE 菜单里的最后一步。
