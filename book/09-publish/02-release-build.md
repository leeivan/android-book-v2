# Release 构建

很多项目在开发期运行得很好，真正切到 `release` 才开始暴露问题：日志没关、测试开关残留、资源收缩后崩溃、反射代码被混淆破坏、调试服务器地址混进正式版本。原因并不复杂，团队从一开始就没有把 release 当成“正式产物”，而只是把它当成 debug 的导出形态。

参考书在讲 release 时有一个很重要的共同点：它们不会把 release 当成打包选项，而会把它写成一套面向真实用户的质量门槛。也就是说，本章真正要建立的不是某个 Gradle 配置记忆，而是 release 心智：一个交给用户的版本，应该和开发包有不同的约束、验证方式和排错准备。

## 学习目标

- 理解 release 是独立构建目标，而不是 debug 的附属产物。
- 理解混淆、资源收缩、日志控制和调试能力在 release 中的意义。
- 学会为 release 设计最小验证清单，而不是依赖“debug 能跑”。
- 理解为什么 release 构建必须和签名、监控、发布流程一起看。

## 前置知识

- 已理解构建变体、应用签名和基本发布流程。
- 已接触日志、测试和错误追踪工具。

## 正文

### 1. 从用户角度重新理解 release

debug 面向的是开发者，release 面向的是用户。这个区别会直接改变你的判断标准。开发者更关心是否方便调试、日志是否够多、问题是否容易复现；用户更关心应用是否稳定、是否安全、是否足够轻、是否不会暴露奇怪的测试痕迹。

一旦把视角切到用户，你就会发现很多“开发期没关系”的东西，在 release 里都必须重新审视：调试面板、测试入口、详细日志、宽松网络配置、实验性依赖、未清理的假数据。

### 2. 为什么 release 独有问题特别难排查

release 最麻烦的地方不是它容易出问题，而是它出问题后不再像 debug 那样好观察。你可能拿不到完整日志，看不到未混淆前的堆栈，也不能直接依赖调试工具现场定位。因此 release 构建必须提前准备两类东西：

- 尽量减少 release 独有风险。
- 一旦出问题，保留足够的定位材料。

这也是为什么成熟团队总会同时关心混淆映射、崩溃上报、版本对应关系和候选发布验证。

### 3. 代码压缩和资源收缩不是单纯瘦身

初学者最容易把 `minifyEnabled` 和资源收缩理解成“让包更小”。它们当然有这个作用，但更重要的是，它们会改变运行时环境。被移除的类、被重命名的方法、被裁剪的资源，都可能影响反射、序列化、依赖注入、动态资源访问等逻辑。

因此教材里讲 release，不能只告诉你“打开开关就行”，而要强调：任何使用反射、代码生成、序列化和 WebView 桥接的地方，都应该在 release 条件下实际验证。

### 4. 一个更真实的候选发布流程

参考书里常见的做法不是“本地打个 release 包就上线”，而是先做候选版本。一个最小但实用的候选发布流程通常是：

1. 用正式签名或与正式发布一致的配置生成候选产物。
2. 在真机上验证登录、首页加载、搜索、通知、上传等关键路径。
3. 检查 release 日志和崩溃上报是否正常。
4. 保存本次构建对应的映射文件和版本记录。
5. 再进入测试轨道或灰度流程。

这样做的好处是，你在“真正交给用户之前”已经把 release 当成正式产品验证过一遍。

### 5. 为什么日志策略必须在 release 中重做

开发阶段多写日志通常是好事，因为它能帮你理解状态变化。但进入 release 后，日志要回答的就不再是“开发者想看什么”，而是“哪些观测对定位问题仍然必要，同时不会泄露用户数据或拖累性能”。

更成熟的 release 日志策略往往会做三件事：

- 删除临时调试日志。
- 保留关键故障定位信息，但避免敏感内容。
- 把真正面向线上定位的能力交给崩溃上报、性能指标和埋点系统。

### 6. release 验证为什么要围绕关键路径，而不是围绕页面数量

并不是每个页面都要做同样强度的 release 检查。更有效的做法是先找关键路径。对于待办应用，可能是创建、编辑、同步；对于新闻应用，可能是启动、列表加载、详情展示、离线缓存；对于聊天应用，可能是会话列表、发消息、通知跳转。

只要关键路径在 release 里稳定工作，发布风险就会明显下降。反过来，如果只做“点开每个页面看一眼”，很多真正致命的问题反而很容易漏掉。

### 7. 实践任务

起点条件：

- 已有能够稳定运行的 debug 构建，并准备做首次候选发布。

步骤：

1. 写出项目中 debug 与 release 的主要差异项，包括日志、网络、签名、压缩、调试开关和第三方工具。
2. 生成一份候选 release 产物，不要直接跳到正式发布。
3. 为候选版本列一份关键路径验证清单，并在真机上走通。
4. 保存本次构建的版本号、映射文件和构建时间，确保问题可追踪。
5. 记录 release 独有问题，反推哪些配置或代码边界需要调整。

预期结果：

- 你会把 release 当成独立的质量目标。
- 你会建立候选版本验证意识。
- 你能更早发现只在正式构建里暴露的问题。

自检方式：

- 你能解释为什么 debug 通过不等于 release 通过。
- 你能判断某个问题为什么只会出现在 release 下。
- 你能说明为什么映射文件和线上问题追踪要绑定版本。

调试提示：

- release 一出问题就只能重打包重试，说明追踪准备不足。
- 只有 debug 验证、没有候选版本，说明发布链路还太粗糙。
- 混淆和资源收缩一开就坏，优先检查反射、序列化和动态资源访问。

### 8. 常见误区

- 把 release 理解成 debug 的导出模式。
- 只盯签名，不盯构建差异和关键路径验证。
- 不保存映射文件和版本对应关系。
- 等正式上架后才第一次看到 release 真实行为。

## 小结

Release 构建真正代表的是“准备交给真实用户的正式产物”。只要把 release 当成独立的构建目标，你就会自然开始重视候选版本、关键路径验证、日志策略、映射文件和线上追踪。到这一步，发布才不再是碰运气，而是开始进入受控的工程流程。
