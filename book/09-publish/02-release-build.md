# Release 构建

能在本地跑起来，并不意味着已经准备好发布。Release 构建的任务，是把开发期工程转换成适合交付的正式产物。这个过程不仅涉及打包，还涉及资源收敛、日志控制、配置校验、性能检查和风险排查。

## 学习目标

- 理解 Release 构建和 Debug 构建的目标差异。
- 知道正式发布前应检查哪些关键项。
- 建立“发布产物是完整交付结果”的意识，而不只是一个可安装文件。

## 前置知识

- 已了解构建变体和签名。
- 已知道发布流程需要比开发流程更严格的控制。

## 正文

### 1. Release 构建为什么不能等同于 Debug 构建

Debug 构建强调开发效率，允许更多日志、调试开关和实验性配置。

Release 构建强调稳定性、体积、性能和安全性。它面对的是用户环境，而不是开发者的本地设备。因此任何只适合开发调试的东西，都应在正式构建中被明确处理。

### 2. 发布前应检查什么

一个合格的 Release 构建至少要确认以下几类问题：

- 构建是否稳定可复现。
- 环境配置是否指向正式服务。
- 调试入口和测试开关是否关闭。
- 日志和敏感信息是否被正确控制。
- 关键路径是否经过回归验证。

这也是为什么发布不该只由“点一下打包按钮”来定义。

### 3. Release 构建与质量门槛

发布构建本身其实就是一道质量门。很多线上问题并不是代码逻辑第一次出现，而是发布前缺少足够明确的检查清单。

如果团队没有固定的发布前验证机制，那么每一次发布都会更像一次碰运气。

### 4. 体积、性能和稳定性要一起看

正式构建不应只关注“能不能装”。你还应同步考虑：

- 包体大小是否合理。
- 启动和关键页面是否明显退化。
- 崩溃与异常上报是否正常。

因为发布版本一旦到达用户手中，修复成本就会显著高于开发阶段。

### 5. 发布说明也是交付的一部分

Release 不只是技术产物，也包括对内和对外的说明：

- 团队要知道这次改了什么。
- 测试要知道重点验证什么。
- 用户要知道更新带来了什么变化。

如果只交付一个包而没有清晰说明，协作成本会立刻上升。

### 6. 最小实践任务

1. 为你的项目写一份发布前检查清单。
2. 至少覆盖环境配置、日志开关、关键流程回归、崩溃上报和签名确认。
3. 把这份清单和构建流程绑定，而不是临时凭记忆执行。

### 7. 常见误区

- 认为 Release 只是关闭调试开关后的 Debug 包。
- 发布前没有固定检查清单。
- 只验证能安装，不验证关键业务链路。

## 小结

Release 构建是把开发产物转换成正式交付结果的过程。它要求的不只是构建成功，更是对环境、质量和风险的系统确认。
