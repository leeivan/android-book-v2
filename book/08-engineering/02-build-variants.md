# 构建变体

只要项目开始区分调试环境、测试环境、预发布环境、国内外渠道或免费版与完整版，构建变体就会从“可选高级配置”变成真正的工程主线。很多团队在项目早期忽略这件事，等到后面需要多环境切换时，才发现各种常量、域名、图标、签名和开关都混在一起，构建已经很难维护。

Android 官方关于 build variants 的文档给出的核心思想很明确：变体不是额外补丁，而是由 build types、product flavors 和 source sets 正式组合出来的不同应用版本。理解这点非常重要，因为它决定了你是在“临时改配置”，还是在“设计可维护的产品版本体系”。

## 学习目标

- 理解 build type、product flavor 和 build variant 的区别。
- 理解 source set 为什么是变体可维护性的关键。
- 理解多环境、多渠道和多版本差异应如何有结构地组织。
- 为签名、发布和 CI 中的多版本构建打基础。

## 前置知识

- 已理解 Gradle 和 Android 构建基础。
- 已接触过 debug 与 release 的基本差异。

## 正文

### 1. 构建变体解决的是“同一项目需要多个版本”的问题

Android 项目很少永远只有一个版本。现实需求通常包括：

- `debug` 和 `release`。
- `staging` 与生产环境。
- 免费版与付费版。
- 国内服务配置与海外服务配置。

如果这些差异只是靠注释代码、改常量或手工替换资源来实现，项目很快就会失控。构建变体的意义，就是把这种差异正式纳入构建系统，而不是依赖人工切换。

### 2. build type、flavor、variant 分别是什么

一个很稳定的理解方式是：

- build type 更偏向构建行为差异，例如是否可调试、是否压缩、是否启用某些日志。
- product flavor 更偏向产品形态差异，例如环境、渠道、功能集、品牌。
- build variant 是这两类差异最终组合后的具体产物。

例如 `stagingDebug`、`prodRelease` 都是具体变体，而不是单独配置项。只要这条关系清楚，后面你读多版本项目时就不会再觉得命名混乱。

### 3. source set 是变体可维护性的关键

Android 官方文档强调 source sets 是定制不同版本代码和资源的重要机制。它真正的价值不在于“目录又多了几个”，而在于：

- 差异代码有明确位置。
- 差异资源有明确优先级。
- 不同版本的边界不会长期污染主代码。

如果没有 source set 纪律，很多团队会把所有版本差异都写成 `if (isDebug)`、`if (isOverseas)`，最后主分支代码会越来越难读。

### 4. 变体设计首先是产品边界设计

很多项目把构建变体当成纯技术工具，结果 flavor 维度设计得非常混乱。更合理的顺序通常是：

1. 先明确哪些差异属于“构建行为”。
2. 再明确哪些差异属于“产品形态”。
3. 最后才落到 build type、flavor 和资源目录。

如果产品边界不清，Gradle 配置只会把混乱自动化，而不会帮你整理混乱。

### 5. 多环境配置最容易失控的地方

多环境通常是构建变体里最先出现、也最容易混乱的需求，例如：

- 开发环境域名。
- 测试环境域名。
- 预发布环境开关。
- 日志策略和埋点策略差异。

最糟糕的做法，是把这些差异散落在各种常量文件和手工改动里。更稳妥的做法，是让环境差异统一绑定在 flavor 或有清晰命名的 build type 上。

### 6. 变体不是越多越好

Gradle 会为 flavor 和 build type 的组合生成变体。组合一多，构建复杂度和维护成本也会迅速上升。因此，设计变体时必须克制：

- 只保留真正有工程价值的维度。
- 避免无意义组合爆炸。
- 对不需要的变体进行过滤。

Android 官方文档也专门提供了 variant filtering 机制，这本身就说明“变体过多”是一个非常现实的问题。

### 7. 一个更接近真实项目的例子

假设你的项目有：

- `debug` 与 `release` 两种构建行为。
- `staging` 与 `prod` 两种环境。

那么更合理的组织方式通常是：

- `debug/release` 负责可调试、压缩和签名差异。
- `staging/prod` 负责域名、应用名后缀或环境资源差异。

最终生成 `stagingDebug`、`stagingRelease`、`prodDebug`、`prodRelease`。这种结构远比“手工切换域名常量”更稳定。

### 8. 实践任务

起点条件：

- 已有一个存在多环境、多版本或多渠道需求的项目。

步骤：

1. 列出项目中所有“版本差异”。
2. 判断每个差异属于 build type 还是 product flavor。
3. 设计一个尽量少维度但足够表达需求的变体体系。
4. 把环境差异统一收回 source set 或构建配置，而不是散落在主代码。
5. 检查是否存在根本不需要生成的变体组合。

预期结果：

- 你能把“多版本需求”转成结构化构建配置。
- 主代码会比以前更少出现版本判断噪音。
- 你能更自然地把变体设计接到发布和 CI 流程上。

自检方式：

- 你能解释：build type、product flavor、build variant 分别是什么。
- 你能判断：某种差异应该落在哪一层。
- 你能说出：为什么 source set 比大量条件判断更可维护。

调试提示：

- 如果主代码里充满环境开关判断，优先考虑是否该收回变体。
- 如果变体数量已经让构建矩阵难以理解，优先检查 flavor 维度是否设计过多。
- 如果团队总靠手工改域名和图标打包，说明构建变体还没有真正落地。

### 9. 常见误区

- 把所有版本差异都堆到一个 flavor 或常量文件里。
- 不区分 build type 和 product flavor。
- 让主代码充满环境判断。
- 变体越配越多，完全不控制组合数量。

## 小结

构建变体真正解决的，是同一项目多版本共存时的结构问题。只要差异被清晰放到 build type、flavor 和 source set 中，后面的测试、发布和自动化流程都会更稳定。
