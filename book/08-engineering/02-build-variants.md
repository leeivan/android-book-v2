# 构建变体

同一个 Android 项目，很少只存在“唯一一种构建方式”。开发环境、测试环境、预发布环境、正式环境，常常需要不同的接口地址、包名后缀、图标、日志策略或开关配置。构建变体存在的意义，就是把这些差异系统化，而不是靠手工改代码硬切环境。

## 学习目标

- 理解构建变体为什么是工程必需品。
- 知道 build type 和 product flavor 的职责差异。
- 学会从“环境治理”角度看待变体设计。

## 前置知识

- 已了解 Gradle 的基本角色。
- 已知道一个项目常常需要多种构建产物。

## 正文

### 1. 为什么不能靠手工改配置切环境

手工改接口地址、手工注释日志、手工改应用名，这类做法在最小演示里看似方便，但在真实工程里非常危险。因为它不可追踪、不可复现，也很容易把测试配置带进正式包。

构建变体解决的正是这个问题：把环境差异写进工程规则，而不是留给人脑记忆。

### 2. build type 和 flavor 的区别

可以把 build type 理解成“同一产品在不同构建目的下的差异”，例如调试版和发布版。

可以把 flavor 理解成“同一代码基线面向不同业务维度的差异”，例如国内版、国际版、免费版、企业版。

最简单的判断方法是：

- 如果差异主要来自开发流程，先考虑 build type。
- 如果差异主要来自产品维度，先考虑 flavor。

### 3. 构建变体真正带来的价值

它的价值不只是“多打几个包”，更重要的是让以下问题都有工程化答案：

- 测试环境如何和正式环境隔离。
- 哪些能力只在调试时开启。
- 不同市场是否需要不同资源和配置。

一旦这些差异进入构建系统，你的团队协作、测试复现和自动化流程都会更稳定。

### 4. 变体设计不要过度复杂

虽然变体强大，但也不能无限叠加。维度一多，构建矩阵会迅速膨胀，测试成本和理解成本也会暴涨。

因此设计时要优先问两个问题：

- 这项差异是否足够稳定，值得进入构建系统。
- 它是长期产品差异，还是临时调试需求。

只有长期、明确、会反复使用的差异，才值得做成正式变体。

### 5. 资源与代码如何随变体变化

构建变体不仅能切配置，也能切资源、清单和部分代码来源。这意味着环境差异可以被限制在最合适的层级，而不是污染整个业务代码。

从可维护性角度看，这是它比运行时硬编码分支更有价值的地方。

### 6. 最小实践任务

1. 为你的项目定义一个开发环境和一个正式环境。
2. 写出这两个环境至少三项稳定差异，例如接口地址、应用名后缀、日志策略。
3. 判断这些差异应属于 build type 还是 product flavor。

### 7. 常见误区

- 手工改配置切环境。
- 用变体承载临时性、一次性的需求。
- 变体维度设计过多，导致构建和测试负担失控。

## 小结

构建变体的价值在于把环境差异从“人肉操作”变成“工程规则”。一旦这个基础打稳，自动化测试、持续集成和发布流程都会更可控。
