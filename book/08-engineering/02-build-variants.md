# 构建变体

同一个 Android 项目，很少只存在“唯一一种构建方式”。开发环境、测试环境、预发布环境、正式环境，往往需要不同的接口地址、应用名、图标、日志策略或功能开关。构建变体的意义，就是把这些差异系统化，而不是靠人肉改代码或临时注释来切环境。本章的重点，是建立“环境治理”的工程视角。

## 学习目标

- 理解构建变体为什么是工程必需品。
- 理解 build type 和 product flavor 的职责差异。
- 知道如何把环境差异纳入构建系统，而不是留给人工记忆。
- 为发布、测试和 CI 流程打基础。

## 前置知识

- 已理解 Gradle 的基本角色。
- 已知道一个项目常常需要多种构建产物。

## 正文

### 1. 为什么不能靠手工改配置切环境

手工改接口地址、手工注释日志、手工换图标、手工切包名，这类做法在最小演示里看似方便，在真实工程里非常危险。因为它不可追踪、不可复现，也很容易把测试配置带进正式包。

构建变体解决的正是这个问题：把环境差异写进工程规则，而不是留给人脑记忆。

### 2. build type 和 flavor 分别解决什么问题

可以把 build type 理解成“同一产品在不同构建目的下的差异”，例如调试版和发布版。

可以把 flavor 理解成“同一代码基线面向不同产品维度的差异”，例如国内版、国际版、免费版、企业版。

一个实用判断是：

- 如果差异主要来自开发和发布流程，先考虑 build type。
- 如果差异主要来自产品维度，先考虑 flavor。

### 3. 变体真正带来的价值是什么

它的价值不只是“多打几个包”，更重要的是让这些问题都有工程化答案：

- 测试环境如何和正式环境隔离。
- 哪些能力只在调试时开启。
- 不同市场或客户版本如何共享大部分代码。
- CI 如何稳定构建不同目标产物。

变体一旦进入构建系统，工程的可重复性会明显增强。

### 4. 变体设计为什么不能过度膨胀

虽然变体很强大，但也不能无限叠加。维度一多，构建矩阵会迅速膨胀，测试成本、理解成本和 CI 成本都会一起上涨。

因此设计变体时要优先问：

- 这项差异是否足够稳定，值得进入构建系统。
- 它是长期产品差异，还是临时调试需求。

只有长期、明确、会反复使用的差异，才值得做成正式变体。

### 5. 资源、清单和代码为什么都可能随变体变化

构建变体不仅能切接口地址，也能切：

- 应用名和图标。
- Manifest 条目。
- 资源文件。
- 某些编译时常量。

这说明变体的意义不只是“切一个环境变量”，而是让差异在最合适的层级被表达，而不是污染整个业务代码。

### 6. 变体和发布质量为什么紧密相关

很多线上事故并不是代码本身有问题，而是：

- 正式包打到了测试环境。
- 调试开关忘记关闭。
- 某个内部功能错误进入线上。

这些问题本质上都是构建变体治理不足。只要变体设计稳定，发布质量会显著提高。

### 7. 最小实践任务

1. 为你的项目定义一个开发环境和一个正式环境。
2. 写出这两个环境至少三项稳定差异，例如接口地址、应用名后缀、日志策略。
3. 判断这些差异应属于 build type 还是 flavor。
4. 检查你当前工程里是否仍存在人工切环境的操作。

### 8. 常见误区

- 手工改配置切环境。
- 用变体承载临时性、一次性的需求。
- 变体维度设计过多，导致构建和测试负担失控。
- 只切接口地址，不治理其他明显环境差异。

## 小结

构建变体的价值，在于把环境差异从“人肉操作”变成“工程规则”。只要这层治理站稳，测试、CI 和发布流程都会更可控。
