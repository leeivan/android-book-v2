# 构建变体

很多 Android 项目在最初只有一个版本时，看起来并不需要复杂的构建设计。开发环境和生产环境用同一个域名常量，调试和发布只靠手工改几个开关，图标差异靠临时替换资源，团队一开始往往觉得这样更快。真正的问题通常要等到项目进入多人协作、多环境联调、多渠道发布阶段才暴露出来。那时你会发现：同一个仓库里其实同时住着很多“不同版本的应用”，而这些差异如果没有正式进入构建系统，后期维护几乎注定混乱。

构建变体要解决的，正是“同一份代码库如何稳定地产生多个产品版本”这个问题。它不是高级配置，也不是发布前才需要碰的角落知识，而是工程结构的一部分。只要项目开始区分 `debug`、`release`、测试环境、预发布环境、不同品牌包或不同功能版本，构建变体就已经站到了主线上。

## 学习目标

- 理解 `build type`、`product flavor`、`build variant` 三者的职责。
- 理解 `source set` 为什么是多版本项目可维护性的关键。
- 掌握多环境、多渠道和多版本需求的基本组织方式。
- 理解为什么变体设计首先是产品边界设计，而不是 Gradle 语法题。

## 前置知识

- 已理解 Gradle、Android Gradle Plugin 和模块构建基础。
- 已知道 `debug` 与 `release` 的基本差异。
- 已接触过不同环境域名、包名后缀或资源替换这类实际需求。

## 正文

### 1. 为什么很多团队会在这里突然失控

项目初期常见的做法是：把开发环境域名写进某个常量文件，准备发版时手工改成生产域名；或者直接复制一份图标资源、改一下应用名，再打包出另一个版本。短期看这种方式“很快”，但它有两个致命问题。

第一，所有差异都依赖人工记忆。你必须记得改回哪些常量，替换哪些资源，关闭哪些日志，切换哪些签名。只要漏掉一步，发布产物就可能出错。

第二，差异没有结构。构建系统不知道哪些是环境差异，哪些是构建行为差异，哪些是品牌差异。于是主代码里开始出现越来越多的 `if (isDebug)`、`if (isDomestic)`、`if (isFreeVersion)`，代码很快就会被版本判断淹没。

构建变体的意义，不是“多一个配置文件”，而是让这些差异正式进入可维护结构。

### 2. 先把三层概念分开：build type、product flavor、build variant

教材里很多人记不住这些名词，不是因为它们太难，而是因为总是直接背定义，没有放回实际场景。

`build type` 更像“构建行为层”。它决定这个包是为了调试、测试还是正式发布而构建的。可调试、代码压缩、签名策略、日志开关，这些通常属于这一层。

`product flavor` 更像“产品形态层”。它描述的是产品本身有什么差异，例如面向测试环境还是生产环境，面向国内还是海外，面向免费用户还是付费用户。

`build variant` 则是前两者组合后的具体产物。`stagingDebug`、`prodRelease`、`freeDebug` 这些名字不是“额外起的别名”，而是构建系统真正能产出的版本。

只要把这三层区分清楚，后面很多配置都会变得自然：行为差异放 build type，产品差异放 flavor，最终产物由 variant 承接。

### 3. 一个真实需求：为什么环境差异不能再靠手工改常量

假设你的应用至少要支持下面这些版本：

- 开发联调用的 `staging`
- 线上真实环境的 `prod`
- 日常开发使用的 `debug`
- 提交市场用的 `release`

如果没有构建变体，你很快就会遇到一连串问题：开发时到底连哪个域名，调试包是否显示测试标记，发布包要不要保留日志，测试环境和线上环境的应用名是否应该区分，安装到同一台测试机时包名会不会冲突。

这些问题都不是“程序逻辑”，却又真真切切地决定了工程是否可维护。构建变体的价值，就是把这种差异从人工流程收回到构建系统里。

### 4. source set 为什么是这章里最重要的工程概念之一

一旦项目开始有多个版本，最危险的事情就是把差异代码混到主代码里。很多团队为了图省事，把环境判断和渠道判断直接散落到业务逻辑中。短期可以跑，长期一定变得难读、难测、难删。

`source set` 提供的核心能力，是让不同版本拥有明确的代码和资源落点。你可以把某些只属于 `staging` 的资源放进对应目录，把只属于 `debug` 的实现放到调试目录，把公共逻辑继续留在 `main`。这样主代码承担的是“共有部分”，而不是所有差异都往里塞。

要理解它的价值，可以把它想成“版本差异的边界线”。边界清楚，代码就能维持稳定；边界混乱，主分支就会变成一堆版本判断的集合。

### 5. 变体设计首先是产品设计，不是 Gradle 技巧

很多项目之所以把变体配乱，不是因为不会写 DSL，而是因为根本没想清楚“我们到底有哪些维度的差异”。开发团队常常把环境、渠道、品牌、功能开关、A/B 实验都堆进同一个 flavor，结果命名混乱、职责交叉、组合失控。

更稳妥的做法通常是先回答三个问题：

第一，哪些差异属于构建行为。例如是否可调试、是否压缩、是否使用正式签名。

第二，哪些差异属于产品形态。例如请求哪个环境、显示哪套品牌资源、提供哪组功能。

第三，这些差异是否真的需要通过构建时静态区分，还是可以放到运行时配置。

只有先把产品边界想清楚，再写 Gradle，构建变体才会真正服务工程，而不是把混乱自动化。

### 6. 一个最小但真实的配置例子

下面这个例子展示了一个常见的双环境配置：`staging` 与 `prod`，外加默认的 `debug` 与 `release`。

```kotlin
android {
    flavorDimensions += "environment"

    productFlavors {
        create("staging") {
            dimension = "environment"
            applicationIdSuffix = ".staging"
            versionNameSuffix = "-staging"
            buildConfigField("String", "BASE_URL", "\"https://staging-api.example.com/\"")
            resValue("string", "app_name", "News App Staging")
        }

        create("prod") {
            dimension = "environment"
            buildConfigField("String", "BASE_URL", "\"https://api.example.com/\"")
            resValue("string", "app_name", "News App")
        }
    }

    buildTypes {
        getByName("debug") {
            applicationIdSuffix = ".debug"
            isDebuggable = true
        }

        getByName("release") {
            isMinifyEnabled = true
            isShrinkResources = true
        }
    }
}
```

这段代码最值得学习的地方，不是每个 DSL 名词，而是它把不同类别的差异分开放了。

`staging` 和 `prod` 负责环境差异，回答“这个应用连哪个后端、显示什么应用名”。

`debug` 和 `release` 负责构建行为差异，回答“这个应用是否可调试、是否压缩、用什么发布策略”。

当这两个维度组合后，构建系统就能明确地产生 `stagingDebug`、`stagingRelease`、`prodDebug`、`prodRelease` 四个版本，而不是让团队靠手工切换常量去猜。

### 7. 为什么 applicationIdSuffix、resValue、buildConfigField 特别常见

这几个配置在多环境项目里几乎总会出现，因为它们分别对应三种很常见的差异。

`applicationIdSuffix` 解决的是安装共存问题。测试环境和正式环境如果包名完全相同，就很难在一台设备上并排安装。

`resValue` 适合一些轻量级资源差异，例如应用名后缀、简单文案、占位值。它的好处是差异跟着变体走，不需要手工改资源文件。

`buildConfigField` 常被用来注入基础地址、开关或构建时常量。不过教材里也必须提醒一句：别把一切配置都塞进 `BuildConfig`。凡是会涉及密钥安全、运行时远程控制或频繁变化的内容，都要重新评估边界。

### 8. 变体一多，复杂度会怎样失控

只要有两个 flavor 维度，每个维度再有两三个取值，变体数量就会很快膨胀。例如环境、品牌、付费等级三个维度一叠加，再乘以 `debug` 和 `release`，项目一下子就会出现十几个甚至几十个变体。不是所有组合都真的有业务意义，但构建系统如果不加控制，照样会为它们准备任务和产物。

这就是为什么成熟团队不会盲目追求“什么都放 flavor”。构建变体不是越全越好，而是越有边界越好。能合并的维度尽量合并，能运行时决定的差异不要强行静态化，不需要存在的组合就不要让它生成。

### 9. source set 在真实项目里怎样帮助你减噪

假设 `staging` 版本需要一个测试环境专用的 `network_security_config.xml`，而 `prod` 不需要；或者 `debug` 版本需要一个只用于内部调试的入口页面，而 `release` 不应该包含。最差的做法，是在主代码中通过一堆条件判断来分叉这些逻辑。

更稳妥的做法是把差异直接放进对应 `source set`。这样：

- `main` 保存所有版本共享的内容；
- `staging` 保存环境专用资源；
- `debug` 保存调试专用实现；
- 组合差异由构建系统自动处理。

这不是“目录技巧”，而是保持主代码干净的根本方式。教材必须把这一点讲透，否则读者只会记住 DSL，却不会真正用结构消除噪音。

### 10. 一个常见重构路径：把手工切换环境改成正式变体

很多团队不是从零设计变体，而是接手一个已经靠常量切环境的项目。这时不需要一口气大改，可以按下面的顺序重构。

先把环境相关常量收拢，确认哪些是纯环境差异，例如 `BASE_URL`、应用名后缀、测试图标。

然后建立一个最小 flavor 维度，例如 `staging` 与 `prod`，先只迁移最关键的几个差异。

接着检查主代码中所有 `if (isTestEnv)` 之类的判断，能收回构建层的尽量收回。

最后再决定是否真的需要更多维度。好的变体系统通常是从一个清晰的最小模型长出来的，而不是一开始就配置成巨大的矩阵。

### 11. 实践任务

起点条件：

- 已有一个存在多环境、多版本或多渠道需求的 Android 项目，或者至少有这样的需求草案。

步骤：

1. 列出项目当前所有版本差异，例如域名、图标、应用名、调试开关、签名策略。
2. 判断每个差异属于构建行为还是产品形态。
3. 设计一个最小可行的变体模型，例如只先引入 `staging` 与 `prod` 两个 flavor。
4. 把一个原本靠手工改常量切换的差异迁移到构建配置中。
5. 检查主代码里是否仍有多余的环境判断，并记录可以继续收回的部分。

预期结果：

- 你能把“版本差异”转成结构化构建配置。
- 环境切换不再依赖人工记忆。
- 主代码会比以前少很多和版本有关的噪音。

自检方式：

- 你能清楚解释 `build type`、`flavor`、`variant` 三者的区别。
- 你能说出某个差异为什么应该落在 flavor，而不是 build type。
- 你能指出项目中至少一处适合迁移到 `source set` 的差异实现。

调试提示：

- 如果每次打包前都要手工改域名、图标或应用名，说明构建变体还没有真正落地。
- 如果主代码里充满环境判断，优先考虑是不是该把差异收回 source set。
- 如果变体数量已经多到团队说不清每个包是干什么的，先减少维度，而不是继续加配置。

### 12. 常见误区

- 把所有差异都塞进一个 flavor，结果职责混杂。
- 把本该属于产品形态的差异写进 build type。
- 一看到运行时差异就想用 flavor 静态区分，导致矩阵膨胀。
- 只会写 DSL，不会先整理产品边界。

## 小结

构建变体真正解决的，是“同一份代码如何稳定地生出多个版本”的结构问题。`build type` 管行为，`product flavor` 管产品形态，`source set` 负责给差异一个清晰的落点。只要这套边界建立起来，多环境联调、渠道发布、自动化构建和后续维护都会顺得多。相反，如果差异始终靠人工切换和条件判断维持，项目越大，成本只会越高。
