# 测试

很多团队在项目初期不做测试，并不是因为不知道测试重要，而是因为“现在先把功能做出来更紧急”。这种选择在早期往往看起来是对的，直到项目开始进入另一个阶段：每次改动都害怕牵一发而动全身，重构几乎不敢做，线上回归越来越多，某些问题总靠手工点一遍才能确认。到了这个阶段，测试的价值才会一下子变得非常具体：它不是为了增加流程，而是为了让你敢于继续演进代码。

参考书在讲测试时，通常不会从工具列表开讲，而会先讨论“什么风险值得被自动化保护”。这个顺序很重要。因为测试真正要保护的不是覆盖率数字，而是变化边界。本章就沿着这个思路展开：先判断风险，再决定写哪层测试，再讨论怎样让代码更容易被测。

## 学习目标

- 理解测试真正解决的是变化风险，而不是流程完整性。
- 理解 Android 中不同层级测试分别适合保护什么。
- 理解为什么可测试性和架构边界高度相关。
- 学会从风险、回归成本和维护收益角度选择更值得写的测试。

## 前置知识

- 已理解 ViewModel、Repository、UseCase 和基本分层。
- 已接触过页面交互、数据层和系统组件。

## 正文

### 1. 测试不是为了证明“永远没 bug”

很多读者一听到测试，就会产生一种不现实的期待：只要测试够多，项目就应该没有 bug。这个期待本身就会把测试推向两个极端：要么因为做不到而干脆不做，要么为了追求“全都测到”写出大量价值不高的 case。

更稳妥的理解是：测试的目标不是证明代码绝对正确，而是在你改动之后，尽快告诉你关键行为有没有被破坏。只要抓住这点，测试就会从“形式任务”变成“变更保险”。

### 2. 为什么“手工点一遍”最终一定不够

手工测试当然有价值，尤其是探索性体验和视觉检查。但它有两个天然限制：

- 不能稳定重复每一条关键路径。
- 随着功能增多，回归成本会迅速上升。

一开始你可能只需要点几个页面；后面一旦涉及列表、登录、搜索、缓存、后台任务和通知，单纯靠手工回归就会越来越慢。项目看起来没有被测试拖慢，实际上已经被“不确定性”拖慢了。

### 3. 不同层测试的价值，不在工具，而在风险匹配

教材里最容易误导人的讲法，是直接按“单元测试、集成测试、UI 测试”列清单，然后让读者自己决定先写什么。更成熟的顺序应该是先问：

- 哪些逻辑最容易出错。
- 哪些错误一旦上线代价最高。
- 哪些代码变化频繁。

然后再决定这一层风险更适合用哪种测试去保护。例如状态转换逻辑通常更适合单元测试，而真正面向用户的关键操作路径更适合 UI 测试。

### 4. 单元测试最适合保护“结构清楚、变化频繁”的逻辑

在 Android 项目里，最值得优先写单元测试的往往不是每个工具函数，而是这些位置：

- ViewModel 的状态转换。
- UseCase 的业务规则。
- Repository 的数据整合与错误映射。
- 格式化、过滤和排序这类纯逻辑处理。

它们的共同特点是：不依赖整套运行环境，但一旦出错，页面表现会立刻异常。参考资料里很多示例都强调这一点：单元测试的性价比，来自它快、稳、贴近真实业务逻辑。

### 5. UI 测试要保护“用户最怕坏掉的路径”

UI 测试最常见的误区，是想把所有按钮都点一遍。这样写出来的脚本往往脆弱、昂贵，还不一定真的保护了关键价值。更成熟的选择通常是围绕几条核心路径：

- 登录是否能完成。
- 搜索和筛选是否还能走通。
- 表单保存是否成功。
- 待办创建与完成是否稳定。

也就是说，UI 测试最值钱的时候，不是覆盖所有界面，而是保护用户最在意、产品最不能出错的那几条主路径。

### 6. 可测试性首先取决于边界，而不是测试框架

如果页面直接 new 出 Repository，Repository 又直接拿着 Retrofit、Room、SharedPreferences 到处调，测试很快就会变得很难。不是因为你不会写测试，而是因为代码本身没有留出可替换、可隔离的边界。

这也是为什么良好架构和良好测试几乎总是一起出现。清晰的依赖注入、单向数据流、明确的 Repository 边界，本质上都在降低测试成本。换句话说，测试难写，很多时候不是测试工具问题，而是代码设计问题。

### 7. 假实现、测试替身和固定输入为什么重要

Android 项目大量依赖外部条件：网络、数据库、时间、系统服务、权限状态。如果测试每次都真的依赖这些对象，不仅慢，而且不稳定。因此教材里通常会引入这些概念：

- fake repository
- fake data source
- stub / mock
- 固定时间、固定输入数据

它们的目的不是“显得高级”，而是让测试只关注自己真正想验证的逻辑，而不是被外部环境干扰。

### 8. 一个更像真实项目的测试决策例子

假设你在做一个新闻列表页。现在最值得保护的，不一定是“按钮颜色对不对”，而可能是：

- 进入页面时是否先显示 loading。
- 拉取成功后是否进入 content 状态。
- 拉取失败后是否显示错误状态并允许重试。
- 切换关键词后旧请求结果是否不会覆盖新结果。

这组问题中，前 3 条非常适合 ViewModel 单元测试，最后一条可能需要状态流测试或更高层验证。这样选择测试，才是真正围绕风险，而不是围绕工具。

### 9. 一个更稳妥的测试推进顺序

如果你现在项目里几乎没有测试，最合理的推进方式通常是：

1. 先选一条最怕回归的业务逻辑。
2. 用单元测试先保护状态转换或规则判断。
3. 再补一条面向用户核心路径的 UI 测试。
4. 最后随着项目稳定度提高，再扩展到更多层次。

这样做的好处是，测试会很快产生实际收益，而不是先搭一大套体系却没人维护。

### 10. 实践任务

起点条件：

- 已有一个包含页面状态或业务规则的模块。

步骤：

1. 找出当前最怕回归的一段逻辑，而不是随便找一段最容易写测试的代码。
2. 判断它更适合单元测试还是 UI 测试，并写出理由。
3. 把外部依赖替换成假实现或可控输入。
4. 写出一条能覆盖关键状态变化的测试。
5. 修改这段逻辑后重新跑测试，验证它是否真的提供了信心。

预期结果：

- 你会从“风险和价值”而不是“工具清单”角度思考测试。
- 你能更清楚知道先测什么最值得。
- 你会更明显感受到架构边界和可测试性的关系。

自检方式：

- 你能解释测试为什么本质上是在保护变化。
- 你能判断某个逻辑更适合单元测试还是 UI 测试。
- 你能说明为什么没有清晰边界的代码很难测试。

调试提示：

- 一写测试就发现要启动一堆真实依赖，优先回查代码边界是否太耦合。
- 测了很多细枝末节却依然不敢改核心逻辑，说明测试重点选错了。
- 测试经常不稳定，优先检查是否依赖了真实时间、网络或环境。

### 11. 常见误区

- 把测试理解成上线前补流程。
- 只讨论覆盖率，不讨论风险和价值。
- 页面、数据源、系统依赖全都真跑，导致测试又慢又脆。
- 没有分层边界，却抱怨测试难写。

## 小结

测试的真正价值，不是让项目显得更规范，而是为变化建立安全边界。只要你围绕关键逻辑、关键路径和高回归风险去设计测试，再让架构边界支持替换和隔离，测试就会从额外负担变成项目持续演进时最稳的一道保险。
