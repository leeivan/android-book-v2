# 测试

很多 Android 项目在早期都容易把测试视为“时间够再补”的附加项，结果功能越做越多，回归成本越来越高，重构几乎不敢动，线上问题也越来越依赖人工排查。测试真正解决的，并不是“让代码看起来更规范”，而是给变化建立可验证的边界，让团队在持续修改中仍然敢于前进。

Android 官方对测试的组织方式非常清楚：单元测试、界面测试、集成测试各自解决不同层面的风险。本章的重点，不是穷举测试框架，而是帮助你建立“哪些东西应该被哪一层测试保护”的判断。

## 学习目标

- 理解为什么测试在 Android 工程中是变化保障，而不是仪式。
- 区分单元测试、集成测试和界面测试的目标。
- 理解测试与架构边界、依赖注入和可维护性的关系。
- 知道如何为页面状态、数据层和关键业务流程建立最小测试面。

## 前置知识

- 已理解 ViewModel、Repository 和分层架构。
- 已知道日志与调试不能替代系统性的回归保障。

## 正文

### 1. 测试真正保护的是“变化”

测试的价值不在于证明代码现在能跑，而在于你下一次修改它时还能知道有没有破坏原有行为。功能一多，最难的往往不是“写出来”，而是“确认没把别的地方弄坏”。测试就是为这种变化建立安全边界。

这也是为什么测试越晚补，成本越高。因为后期你要补的不是某个小函数，而是已经到处交织的行为。

### 2. 单元测试、集成测试、界面测试分别解决什么

一个很实用的理解方式是：

- 单元测试更适合保护纯逻辑、状态转换和边界条件。
- 集成测试更适合验证多个对象协作是否正确。
- 界面测试更适合验证用户关键流程和界面交互行为。

这三者不是谁能替代谁的关系。真正成熟的测试体系，通常是让不同层级各自覆盖最适合自己的风险，而不是把所有问题都推给最重的 UI 测试。

### 3. 架构越清晰，测试越容易落地

测试和架构的关系非常直接。一个页面如果直接依赖网络、数据库和系统服务，它就很难被隔离测试；而如果 ViewModel 只依赖明确接口，Repository 边界清晰，可替换性就会高很多。

这也是为什么依赖注入、Repository、UseCase 等结构不只是“为了好看”，它们也在决定项目到底是不是可测。

### 4. 什么最适合先写单元测试

在 Android 项目里，最值得优先被单元测试保护的通常包括：

- ViewModel 的状态转换逻辑。
- UseCase 的业务规则。
- 数据层的映射和结果包装逻辑。
- 纯函数和边界判断。

这些地方通常不依赖真实 UI，也更容易稳定复现，是建立测试收益的最佳起点。

### 5. UI 测试应该更关注关键流程，而不是覆盖一切点击

界面测试很有价值，但也通常更慢、更脆弱、更依赖环境。因此更合理的做法不是“每个按钮都点一遍”，而是优先覆盖最关键的用户流程，例如：

- 登录。
- 下单。
- 搜索并进入详情。
- 创建并提交关键内容。

如果把 UI 测试当成全面覆盖工具，维护成本往往会迅速上升。

### 6. 测试数据与依赖替换同样重要

很多团队写测试时真正卡住的，不是断言不会写，而是无法替换依赖、无法构造稳定输入、无法隔离网络和数据库。这说明测试真正依赖的是可控环境。

更成熟的做法通常会配合：

- 假实现或测试替身。
- 可控制的时间与输入。
- 独立测试数据库或内存实现。
- 清晰的依赖注入入口。

没有这些基础，测试往往只会写在最表层，难以深入关键逻辑。

### 7. 一个更接近真实项目的例子

假设文章列表页的 ViewModel 负责把 Repository 结果映射为 Loading、Content、Empty、Error 四种 UI 状态。那么最有价值的单元测试通常不是“页面有没有渲染按钮”，而是：

- 当仓库返回空列表时，UI 状态是否正确进入 Empty。
- 当仓库抛出网络错误时，是否进入 Error。
- 当刷新开始时，是否先进入 Loading。

这个例子说明，很多最值得保护的行为，其实正落在状态转换边界上。

### 8. 实践任务

起点条件：

- 已有一个 ViewModel、Repository 或关键页面流程。

步骤：

1. 选一个最容易因为改动而出问题的功能。
2. 列出它最关键的 3 到 5 个行为结果。
3. 判断这些行为更适合单元测试、集成测试还是 UI 测试。
4. 为其中一个 ViewModel 状态转换写出测试用例草案。
5. 检查当前项目中哪些地方因为依赖过重而很难测试。

预期结果：

- 你会从“变化风险”而不是“工具列表”来设计测试。
- 你能更清楚地区分不同层级测试的职责。
- 你会开始反过来用测试需求检验架构边界是否清晰。

自检方式：

- 你能解释：为什么测试真正保护的是未来修改。
- 你能判断：某个行为为什么更适合单元测试而不是 UI 测试。
- 你能说出：哪些架构问题会直接导致测试困难。

调试提示：

- 如果一个核心逻辑完全无法测试，通常说明依赖边界或职责边界出了问题。
- 如果 UI 测试数量很多但改一次就大面积失败，优先检查是否覆盖了太多细碎交互。
- 如果测试只能跑通 happy path，说明真正高风险的边界还没有覆盖到。

### 9. 常见误区

- 把测试看成“时间够再补”的工作。
- 想用一种测试类型覆盖所有问题。
- UI 测试写得过多，核心逻辑却没有单元测试保护。
- 测试写不动时，只怪框架，不反思架构边界。

## 小结

测试的真正价值，在于让项目在持续变化中仍保持可验证。只要你能把行为风险和测试层级对应起来，测试就不会再是额外负担，而会成为工程稳定性的基本组成部分。
