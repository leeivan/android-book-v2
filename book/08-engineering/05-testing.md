# 测试

很多团队在项目初期不做测试，并不是因为不知道测试重要，而是因为“现在先把功能做出来更紧急”。这种选择在早期往往看起来是对的，直到项目开始进入另一个阶段: 每次改动都害怕牵一发而动全身，重构几乎不敢做，线上回归越来越多，某些问题总靠手工点一遍才能确认。到了这个阶段，测试的价值才会一下子变得非常具体: 它不是为了增加流程，而是为了让你敢于继续演进代码。

这一章不把测试当成“补几条 case”的形式任务，而是把它放回 Android 工程里最现实的目标: 为变化提供安全边界。只要你从这个角度看，单元测试、UI 测试、假实现、测试作用域和用例选择都会更容易理解。

## 学习目标

- 理解测试真正解决的是变化风险，而不是流程完整性。
- 理解 Android 中不同层级测试分别适合验证什么。
- 理解为什么可测试性和架构边界高度相关。
- 学会从风险和回归成本角度选择更值得写的测试。

## 前置知识

- 已理解 ViewModel、Repository、UseCase 和基本分层。
- 已接触过页面交互、数据层和系统组件。

## 正文

### 1. 为什么“手工点一遍”最终一定不够

手工测试在早期当然有价值，尤其是探索性场景和交互体验检查。但它有两个天然限制:

- 不能稳定复现每一个回归风险。
- 成本会随着功能和分支快速上升。

当项目越来越依赖人工回归时，开发速度表面上没有被测试拖慢，实际上已经被不确定性拖慢了。你不敢改、不敢删、不敢重构，这就是缺少测试最真实的成本。

### 2. 测试真正提供的是“对改动的信心”

这句话特别重要。测试不是为了证明“代码绝对没问题”，而是为了在你改动之后，尽快知道:

- 关键逻辑是否仍然成立。
- 状态流转是否被破坏。
- 用户最核心的路径是否还能走通。

这也是为什么测试总是和重构、持续交付、发布质量绑在一起。没有测试，代码不是不能继续写，而是演进成本会越来越高。

### 3. Android 测试为什么不能只按工具分，而要按风险分

很多教程一讲测试，就按单元测试、集成测试、UI 测试去分。这个分类本身没错，但如果只停留在工具层，读者还是不知道“先测什么更值”。更成熟的顺序是先问:

- 哪些逻辑最容易出错。
- 哪些错误一旦上线代价最高。
- 哪些逻辑变化频繁。

然后再决定应该用哪一层测试覆盖它。

### 4. 单元测试最适合保护稳定业务逻辑

单元测试在 Android 工程里最有价值的地方，通常不在页面像不像真的跑起来，而在于它能以低成本、快速地保护:

- ViewModel 状态转换。
- UseCase 业务规则。
- Repository 数据处理逻辑。
- 工具层映射与格式化逻辑。

这些逻辑一旦被抽离得足够清楚，单元测试通常是性价比最高的一层。它快、反馈直接，也最适合成为“修改逻辑前后的第一道保险”。

### 5. UI 测试适合验证“用户真正关心的路径”

UI 测试不是为了把每个点击都机械重放一遍，而是为了保护那些真正关键的用户路径，例如:

- 登录能否完成。
- 搜索和筛选是否仍可用。
- 下单流程是否能走通。
- 待办创建和完成是否正常。

也就是说，UI 测试最值钱的时候，是它覆盖了用户最重要、最怕回归的场景，而不是页面上所有按钮都点一遍。

### 6. 可测试性往往先取决于架构，而不是测试框架

如果 ViewModel 里直接 new 依赖、页面直接碰数据库、状态变化到处可写，那么你会发现写测试非常痛苦。不是因为测试工具太难，而是因为代码本身没有给测试留边界。

这也是为什么良好架构和良好测试总是一起出现。清晰的依赖注入、单向数据流、Repository 边界、可替换的数据源，本质上都在让测试更容易落地。

### 7. 假实现和测试替身为什么重要

Android 项目很多逻辑都依赖网络、数据库、时间、系统组件。如果测试每次都真的去访问这些外部对象，不仅慢，而且不稳定。更成熟的测试方式通常会使用:

- fake repository
- fake data source
- stub / mock
- 固定时间或固定输入

它们的价值不在“技术上更高级”，而在于让测试只关注自己真正要验证的逻辑。

### 8. 一个现实例子: ViewModel 状态测试为什么很值

例如文章列表页刷新时，最值得保护的不是“按钮是否长得一样”，而是:

- 初始状态是否正确。
- 刷新开始后是否进入 loading。
- 成功后是否得到内容状态。
- 失败后是否得到错误状态。

这类逻辑如果有清晰的 `uiState`，单元测试往往很容易写，却能显著提高重构信心。这也说明测试收益和状态建模质量之间关系很强。

### 9. 什么时候不必追求“到处都有测试”

测试很重要，但并不意味着每一行代码都必须覆盖。以下情况更需要克制:

- 纯样板代码。
- 极不稳定、马上就会重写的过渡实现。
- 几乎没有业务价值的简单透传。

真正成熟的测试策略，不是追求到处铺满，而是优先保护关键逻辑和关键路径。

### 10. 实践任务

起点条件:

- 已有一个包含页面状态或业务规则的模块。

步骤:

1. 找出当前最怕回归的一段逻辑。
2. 判断它更适合单元测试还是 UI 测试。
3. 把外部依赖替换成假实现或可控输入。
4. 写出一条能覆盖关键状态变化的测试。
5. 在修改逻辑后，观察这条测试是否真的能给你信心。

预期结果:

- 你会从“风险和价值”而不是“工具清单”角度思考测试。
- 你能更清楚地知道先测什么最值得。
- 你会更明显地感受到架构边界和可测试性的关系。

自检方式:

- 你能解释测试为什么本质上是在保护变化。
- 你能判断某个逻辑更适合单元测试还是 UI 测试。
- 你能说明为什么没有清晰边界的代码很难测试。

调试提示:

- 一写测试就发现需要启动一大堆真实依赖，优先检查代码边界是否太耦合。
- 测了很多细枝末节却不敢改核心逻辑，说明测试重点错了。
- 测试经常不稳定，优先检查是否依赖了真实时间、网络或环境。

### 11. 常见误区

- 把测试理解成上线前补流程。
- 只讨论覆盖率，不讨论风险和价值。
- 页面、数据源、系统依赖全都真跑，导致测试又慢又脆。
- 没有分层边界，却抱怨测试难写。

## 小结

测试的真正价值，不是让项目显得更规范，而是为变化建立安全边界。只要你围绕关键逻辑、关键路径和高回归风险点去设计测试，再让架构边界支持替换和隔离，测试就会从额外负担变成项目持续演进时最稳的一道保险。
