# 模块化

很多团队第一次认真谈模块化，往往不是因为“想做更高级的工程架构”，而是因为单体工程已经开始持续制造成本: 改一点代码要等很久才能重新构建，一个模块的依赖变动会波及整个项目，多个开发者经常在同一个大模块里互相踩改动，公共能力越积越多却没有边界。到了这个阶段，问题早就不是“目录够不够整齐”，而是工程结构已经撑不住协作和演进了。

模块化真正要解决的，就是把职责边界、依赖边界和构建边界同时整理出来。它不是把代码机械切成更多 module，也不是项目一上来就必须做的仪式，而是一种在规模和复杂度增长时，用结构换可维护性的工程手段。

## 学习目标

- 理解模块化真正解决的是边界和协作问题，而不是目录美化问题。
- 理解 feature、data、core 等常见模块划分背后的逻辑。
- 理解模块化与构建速度、依赖治理、团队协作之间的关系。
- 学会判断什么时候值得开始模块化，什么时候不必过度拆分。

## 前置知识

- 已理解 Gradle、构建变体和基础分层。
- 已接触 ViewModel、Repository、UseCase 等职责边界。

## 正文

### 1. 先看单体工程为什么会越来越难维护

设想一个项目最初只有几个页面，所有代码都放在 `app` 模块里，问题不大。随着功能增加，新闻、搜索、个人中心、支付、设置、离线缓存、上传、分享都开始长出来，`app` 模块就会逐渐承担几乎一切职责:

- 功能代码都在这里。
- 公共 UI 和工具也在这里。
- 网络、数据库、埋点、调试工具也都在这里。

短期内这样当然能跑，但长期后果非常明显:

- 依赖会向整个项目扩散。
- 改动范围会越来越难控制。
- 构建速度会逐步恶化。
- 没有人能快速判断“这个改动会影响到哪里”。

模块化就是为了在这个趋势变得更糟之前，把结构重新切开。

### 2. 模块化首先解决的是边界，而不是规模

不少人会把模块化和“大项目”直接画等号。更准确的判断标准其实不是体量，而是边界是否已经明显混乱。只要你开始频繁遇到这些问题:

- 功能代码和共享代码混在一起。
- 一个小改动总会影响很多无关模块。
- 新人很难判断该把代码放在哪。
- 多个页面都在重复造轮子。

那么模块化就已经有价值。它的本质不是为了显示项目很大，而是为了让边界变清楚。

### 3. 模块化不是“切得越多越高级”

这也是最常见的误区之一。把项目拆成很多 module，并不会自动带来结构收益。真正决定质量的是:

- 每个模块职责是否稳定。
- 依赖方向是否清晰。
- 共享能力是否被合理抽离。

如果只是把原来混乱的代码原封不动分散到更多模块里，耦合并不会消失，只是变成“跨模块耦合”。这样一来，复杂度不但没降，反而还加了 Gradle 配置和协作成本。

### 4. 为什么 feature / data / core 这类划分会反复出现

现代 Android 项目常见的模块划分往往围绕三类角色:

- `feature`: 对应具体产品能力，例如首页、搜索、详情、设置。
- `data`: 对应数据入口和来源实现。
- `core`: 对应多个功能共享的稳定基础设施，例如网络、数据库、设计系统、通用 UI。

这种划分之所以常见，不是因为它是唯一标准答案，而是因为它同时对齐了三种边界:

- 产品边界
- 架构边界
- 构建边界

也就是说，它让“我们怎么理解这个应用”和“Gradle 怎么组织这个应用”尽量保持一致。

### 5. 模块化为什么会影响构建速度

模块化经常和构建速度一起被提起，但这件事必须讲清楚。模块化不是魔法，它不会因为你多建几个 module 就自动提速。真正能改善构建的，是这些条件成立以后:

- 模块边界稳定，修改一个模块不需要重编大量无关代码。
- 依赖方向清晰，避免大范围级联影响。
- 共享能力被收在明确位置，而不是散落在所有功能里。

如果边界划得不好，模块化反而可能让配置更重、依赖更复杂、构建更慢。所以构建收益是“边界设计正确”的结果，不是“模块数量增加”的结果。

### 6. 模块化同时也是团队协作设计

很多工程问题表面上是技术问题，实则已经进入协作问题。模块化最大的价值之一，就是让多人协作不再总围着同一个巨型模块打转。稳定模块边界意味着:

- 功能团队能围绕清晰接口协作。
- 共享能力有统一维护位置。
- 改动影响面更容易预估。

换句话说，模块化不仅在整理代码，也在整理团队工作方式。

### 7. 一个更贴近真实项目的拆分过程

假设你在做一个新闻应用，功能包括首页、搜索、收藏和设置。更健康的成长路径通常不是“一次拆成十几个模块”，而是:

1. 先识别哪些代码是真正共享的，例如网络层、数据库、通用组件。
2. 再识别哪些功能已经有清晰产品边界，例如 `feature:search`、`feature:bookmark`。
3. 先从最容易形成独立边界的一两个功能模块开始拆。
4. 观察依赖是否仍然清晰，再逐步推进。

这种渐进式模块化比一次性大手术更稳，也更容易得到真正收益。

### 8. `core` 模块为什么最容易变成新的大杂烩

很多项目一旦开始模块化，就会把“暂时不知道放哪的东西”都丢进 `core`。短期方便，长期很危险。因为当 `core` 什么都能放时，它就会重新长成新的巨型模块。

更稳妥的做法通常是:

- 只有明确会被多个模块稳定复用的东西才进入 core。
- `core` 内部也保持子边界，例如 `core:ui`、`core:network`、`core:database`。
- 无法证明共享价值的代码，先留在功能模块中。

克制地建设共享层，比急着“统一收纳”更重要。

### 9. 什么时候不必着急模块化

如果项目仍然很小，功能边界还在频繁变化，团队规模也不大，那么过早拆很多模块很可能只会增加维护成本。因为:

- 边界还没稳定，拆出来的模块也会反复推倒。
- 配置和依赖治理成本会提前出现。
- 开发者会把更多时间花在“代码该放哪”而不是“功能该怎么做好”上。

模块化最成熟的姿态，不是“越早越好”，而是“边界成熟到值得长期维护时再拆”。

### 10. 实践任务

起点条件:

- 已有一个功能数量较多、依赖明显增长的 Android 项目。

步骤:

1. 列出当前项目最明显的功能边界和共享能力。
2. 标出哪些代码只是“看起来通用”，哪些是真的被多个功能稳定复用。
3. 找出当前最重、最容易扩散依赖的模块。
4. 为它设计一个最小可行的拆分方案，而不是一次性全量重构。
5. 检查拆分后的依赖方向是否仍然清晰。

预期结果:

- 你会把模块化理解成边界治理，而不是目录整理。
- 你能更清楚地判断哪些模块划分是有价值的。
- 你会更克制地推进模块化，而不是追求模块数量。

自检方式:

- 你能解释模块化为什么首先解决的是边界问题。
- 你能判断一个模块拆分是否真的带来结构收益。
- 你能说明为什么 `core` 模块最容易失控。

调试提示:

- 模块拆得很多但依赖仍然互相缠绕，说明只是把混乱切开了，没有治理。
- 如果 `core` 成了所有“暂时没地方放的代码”的收容所，先暂停继续拆分。
- 如果一个项目还很小却已经有复杂模块矩阵，优先先回看收益是否成立。

### 11. 常见误区

- 把模块化理解成“把代码切成更多目录”。
- 认为模块越多越高级。
- 过早大规模拆分，边界还没稳定就先引入高配置成本。
- 让 `core` 模块无限膨胀。

## 小结

模块化真正要解决的，是工程规模增长后边界、依赖和协作一起失控的问题。它的收益来自清晰职责和健康依赖，而不是模块数量本身。只要从最小可行边界开始、保持依赖方向清楚、克制地建设共享层，模块化就会逐步变成真正有工程价值的结构，而不是形式上的拆分。
