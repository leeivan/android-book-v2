# 安全基础

安全问题在 Android 项目里最容易被推迟，因为它在功能开发早期往往不直接影响“能不能跑”。很多团队直到出现真实风险才开始认真补安全，例如组件被意外导出、调试日志泄露 token、WebView 加载了不可信内容、文件共享边界太宽、明文传输被抓到、密钥直接写在仓库里。到了那个阶段，安全就不再是“再优化一下”，而是必须回头补基础。

本地安全参考资料反复强调的一点，是 Android 安全模型首先不是一堆零散技巧，而是一套分层边界：应用默认运行在隔离进程和独立 UID 下，权限决定它能触达哪些系统能力，组件导出与外部输入决定攻击面有多宽，网络配置、WebView 和文件共享则决定这条边界在和外部世界交互时会不会被重新撕开。本章沿用这个视角，把安全理解成“持续缩小暴露面”，而不是只在最后时刻补几个加密选项。

## 学习目标

- 理解 Android 安全首先是边界治理问题，而不是单一加密技术问题。
- 理解组件导出、网络安全、密钥管理、文件共享和日志隐私这些基础风险面。
- 理解“最小权限、最小暴露、最小信任”的工程意义。
- 学会从日常开发流程中识别和减少安全风险。

## 前置知识

- 已理解系统组件、网络、权限、文件共享和发布流程。
- 已接触日志、调试和构建配置。

## 正文

### 1. 安全问题最常见的形态不是“被攻击”，而是边界放得太松

很多安全问题并不是因为项目没有上复杂加密，而是因为最基本的边界没收紧。Android 的隔离模型确实已经帮应用建立了第一道线，但这条线一旦因为 exported 组件、过宽权限、调试日志或不受控 WebView 被自己主动放松，风险就会重新落回应用代码层。例如：

- 不该对外的 Activity / Receiver 被导出了。
- 调试日志直接打印 token 或用户信息。
- 文件共享给了过大的访问范围。
- 明文流量仍被允许。

这些问题的共同点是，它们都来自“默认太宽、收得不够紧”。所以 Android 安全的第一原则，不是“加更多”，而是“先少暴露”。

### 2. 组件安全首先从 exported 和输入边界开始

任何跨组件边界暴露出去的东西，都要先问：

- 真的需要对外开放吗？
- 允许谁调用？
- 传进来的数据是否可信？

这对 Activity、Service、BroadcastReceiver、ContentProvider 都成立。很多组件安全问题，本质上不是组件本身危险，而是开发者默认所有外部输入都可信、所有入口都应该开放。真正成熟的默认值应该是“先把攻击面收窄，再为明确需要的调用关系开口子”。只要这个默认值改掉，很多基础风险会直接下降。

### 3. 网络安全不是“上线前记得开 HTTPS”

现代 Android 安全里，网络边界非常关键。真正需要长期建立的意识包括：

- 默认只信任加密传输。
- 不轻易允许明文流量。
- 服务端证书和网络配置本身就是安全边界的一部分。
- 敏感接口数据不应进入日志和调试输出。

换句话说，网络安全不是某个开关，而是“应用如何与外部世界建立可信连接”的持续策略。

### 4. 密钥和凭证为什么总是容易被放错位置

实际项目里最常见的错误包括：

- API key 直接写进仓库。
- 签名信息硬编码在脚本里。
- 本地配置文件里存放敏感内容却没有隔离。

这些做法之所以危险，不只是因为“看起来不专业”，而是因为一旦仓库、日志、构建产物泄露，敏感信息就可能一起外流。教材里必须反复强调一点：凭证不是普通常量，它们需要比业务代码更高等级的管理方式。

### 5. 日志、调试和分析工具也可能成为安全入口

为了排查问题，开发者很容易把：

- 用户手机号
- 精确定位
- access token
- 完整接口响应

直接打印出来。短期看排障更快，长期却会让调试工具本身变成泄露通道。尤其在发布版本中，任何调试输出都要重新审视是否越界。安全不是只防外部，也包括别让自己的开发习惯把边界撕开。

### 6. 文件共享和本地存储也有安全边界

只要应用涉及图片、文件、下载和导出，你就必须考虑：

- 这个文件该不该被别的应用看到。
- 如果要共享，是否通过受控 URI 而不是直接路径。
- 文件权限范围是否最小。

这也是为什么 `FileProvider` 和受控存储策略在今天如此重要。安全共享不是“不共享”，而是“只共享必须共享的那一点点”。

### 7. WebView、外部内容和输入校验为什么需要谨慎

很多看起来“只是展示内容”的地方，也可能变成安全边界。例如：

- WebView 打开不可信 URL。
- 外部传入的 Intent data 直接使用。
- 下载内容未经校验就展示或执行。

这些问题的共同点是，应用在无意识地扩大自己对外部输入的信任范围。尤其 WebView 这类既像展示容器又像浏览器入口的组件，一旦和不受控 URL、脚本能力、外部 Intent 数据直接拼接，往往会把原本局部的功能点变成整条安全链路上的薄弱环节。成熟的默认值应该是：外部输入默认不可信，需要被校验、约束和最小化使用。

### 8. 权限最小化本身就是安全设计

权限章节更偏向用户体验和平台边界，安全章节则要再往前一步：只要应用拿到的权限越多，潜在攻击面和误用面也越大。因此：

- 能不用的权限就别要。
- 能用系统选择器替代就别直接拿底层访问权。
- 能只暴露局部数据就别暴露整片数据。

最小权限不仅是体验问题，也是安全基线。

### 9. 一个更现实的安全工作方式

安全并不总是通过单独安全审计才开始。更成熟的日常工作方式通常是：

1. 每做一个新入口，就问它是否真的需要暴露。
2. 每加一项权限，就问能否用更小替代方案。
3. 每写一段日志，就问是否包含敏感信息。
4. 每接一份外部输入，就问它是否可信。
5. 每准备发布，就回看构建和签名配置是否泄露凭证。

只要这些问题进入日常开发，安全基线就会不断提高，而不是等出问题后再补。

### 10. 实践任务

起点条件：

- 已有一个具备网络、文件、系统组件或发布配置的 Android 项目。

步骤：

1. 列出项目当前所有对外组件入口。
2. 检查哪些入口实际上并不需要导出。
3. 检查日志里是否出现敏感信息。
4. 检查配置和构建文件里是否存有不该直接暴露的凭证。
5. 找出一项权限或共享行为，思考是否可以缩小边界。

预期结果：

- 你会把安全问题更多地看成边界和暴露面问题。
- 你能更早识别日常开发里的高风险习惯。
- 项目的基础安全基线会更容易被系统化提高。

自检方式：

- 你能解释为什么很多安全问题来自边界放得太松。
- 你能判断某个组件或配置是否暴露过度。
- 你能说明为什么日志和调试信息也会成为安全风险。

调试提示：

- 一切外部输入都默认可信，是非常危险的起点。
- 看到 token、手机号、定位出现在日志里时，不要等到发布前再想起来处理。
- 如果一个组件“可能以后会用到对外开放”，那在今天通常就不该先开放。

### 11. 常见误区

- 把安全理解成高深专项，与日常开发无关。
- 认为只要用了 HTTPS 就已经足够安全。
- 忽视组件导出、日志和配置文件里的基础风险。
- 过度信任外部输入。

## 小结

Android 安全基础真正要建立的，不是几个零散技巧，而是一套持续的边界意识：最小权限、最小暴露、最小信任。只要把组件、网络、日志、文件共享和凭证管理这些日常工程环节都纳入安全判断，项目的风险面就会显著缩小，而不是等到问题出现后再被动补救。
