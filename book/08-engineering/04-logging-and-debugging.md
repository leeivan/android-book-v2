# 日志与调试

很多初学者第一次调试问题，靠的是一种非常朴素的方法：哪里不对，就在那附近 `Log.d()` 几行，看看值是不是自己想的那样。这种方式在小问题上常常有效，但只要项目开始变复杂，问题就会变成另一种形态：某个状态只在部分机型复现，某个页面只在切后台再回来时出错，某个接口只有线上偶发超时，某个崩溃在你本地怎么都复现不了。到了这个阶段，日志和调试已经不是“插几行打印”那么简单，而是工程排障能力的一部分。

参考书在讲调试时有一个共同点：都不会只列工具名称，而是会先给出排查顺序。因为真正决定调试效率的，不是你会不会点 Logcat、Breakpoint 或 Profiler，而是你能不能把问题改写成一个待验证的假设。本章就按这个顺序来组织：先讲如何提出假设，再讲不同工具各自适合验证什么。

## 学习目标

- 理解日志和调试的核心是验证假设，而不是盲目输出。
- 理解日志、断点、Inspector、Profiler 和线上崩溃收集分别适合什么问题。
- 学会围绕“现象 -> 假设 -> 证据 -> 修复”建立更系统的调试顺序。
- 理解调试信息与隐私边界、发布构建和线上监控之间的关系。

## 前置知识

- 已接触页面状态、网络请求和本地存储。
- 已会使用 Android Studio 运行项目、查看 Logcat 和打断点。

## 正文

### 1. 调试的第一步不是加日志，而是先把问题说清楚

很多 bug 之所以会排查很久，并不是因为太复杂，而是因为问题一开始就没有被说清楚。比如“列表刷新不对”这句话几乎没什么价值。更有效的表述应该是：

- 刷新后有时显示旧结果。
- 只在快速切换关键词时出现。
- 页面不崩，但 UI 会回退到前一次搜索结果。

一旦问题被说成这种形式，你才能继续提出可验证的假设。例如：

- 网络返回顺序和页面当前关键词不一致。
- Repository 没有正确写回最新结果。
- ViewModel 仍然在收旧 Flow。

调试真正的起点，不是工具，而是问题被说清楚。

### 2. 日志的价值在于留下结构化证据

健康的日志不是“多打印一点”，而是留下能帮助你还原链路的证据。好的日志通常至少包含三类信息：

- 发生了什么。
- 它属于哪一条业务链或哪个页面。
- 与问题最相关的关键状态是什么。

例如下面两条日志，后者显然更有排障价值：

```kotlin
Log.d("News", "start refresh")

Log.d(
    "NewsList",
    "refresh start, requestId=$requestId, query=$query, currentPage=$page"
)
```

第一条只能证明“某处开始刷新了”，第二条则能帮助你判断是哪个请求、带着什么条件发起了刷新。参考资料里反复强调这一点：日志必须带上下文，否则它只是噪音。

### 3. 断点解决的是“这一刻到底发生了什么”

如果日志擅长看时间线，断点更擅长看单个瞬间。它特别适合回答这些问题：

- 这个对象现在到底是什么值。
- 这条分支为什么会走到这里。
- 异常抛出之前，调用链上有哪些关键参数。

例如一个待办编辑页保存后没有更新列表，你可以在 Repository 的写入函数和 ViewModel 的收集逻辑里打断点，看数据库有没有真的写进去、收集方有没有收到更新。这类问题如果只靠打印日志，也能查，但往往需要更久。

### 4. Layout Inspector、Database Inspector 和 Profiler 解决“代码看不见”的问题

不是所有 bug 都适合看源码和日志。有些问题本质上更适合“直接看运行时状态”：

- 页面明明只有几个控件，为什么布局层级这么深。
- RecyclerView 为什么掉帧。
- Room 里到底有没有写入数据。
- 某个后台任务为什么排队不执行。

这时候 Inspector 和 Profiler 的价值就比继续加日志更高。参考书里讲调试时通常会把这些工具当成“观察运行时事实”的手段，而不是高级附属功能。你越早习惯用它们，越不容易在脑子里硬猜运行时状态。

### 5. 线上问题和本地问题，应该用不同思路处理

本地问题通常有一个优势：你能主动复现。但真实项目里很多 bug 只会在线上出现，例如：

- 只在某几个 Android 版本上出现。
- 只在弱网下偶发。
- 只在某批真实用户数据下出现。

这类问题不能只靠本地 Logcat。你需要崩溃上报、错误聚合、版本关联和必要的埋点证据。换句话说，本地调试解决的是“我眼前能看到的问题”，线上监控解决的是“我暂时看不到，但用户已经遇到的问题”。

一旦问题跨越多层，比如“点击后发起请求，Repository 落库，页面最后却显示旧值”，日志就更需要有链路关联信息，而不是每层各写一条彼此独立的打印。请求标识、页面入口、关键业务 ID 或版本号这类最小关联字段，往往比再多十条零散日志更有价值。它们能帮助你把 UI、数据层和后台响应重新串回同一条时间线。

### 6. 调试必须围绕假设，而不是围绕工具

教材里最容易忽略的一点是：先选工具，再找问题，几乎总是低效的。更成熟的顺序应该是：

1. 明确现象。
2. 提出两个最可能的假设。
3. 为每个假设选最合适的观测手段。

例如“列表刷新后偶尔显示旧数据”，你可能提出两条假设：

- 假设 A：旧请求比新请求后返回，导致结果覆盖。
- 假设 B：数据库写回了旧结果。

验证 A 更适合看带 `requestId` 的日志；验证 B 更适合看 Database Inspector 或 Repository 断点。只要按这种方式组织调试，你就不会陷入“什么都试一下”的混乱状态。

### 7. 调试信息也有隐私和安全边界

为了图快，很多项目会在日志里直接打印：

- 用户手机号
- 精确位置
- Access Token
- 完整接口响应

短期看排查更方便，长期却会让日志本身变成泄露通道。真正成熟的日志策略通常会区分：

- 本地开发临时日志
- 长期保留的结构化日志
- 发布版本允许出现的最小观测信息

只要一条日志含有用户敏感数据，就应该先问自己：这条信息是否真的必须被记录，是否可以脱敏，是否只能留在本地调试环境。

### 8. 一个更稳妥的 Android 调试顺序

面对大多数问题，你都可以先按这套顺序走：

1. 用一句话描述现象。
2. 判断问题更像 UI、数据流、系统能力还是性能问题。
3. 提出两个最可能的假设。
4. 为每个假设选择一种最省力的验证方式。
5. 只补最少量的日志或断点，避免一次改太多。
6. 修完以后删掉临时调试代码，保留真正有长期价值的观察点。

这套顺序看起来不复杂，但它能让调试从“随机试错”变成“有证据的排查”。

### 9. 实践任务

起点条件：

- 已有一个能够稳定复现，或至少部分复现的问题。

步骤：

1. 用一句话准确写出问题现象，避免“页面有点不对”这种模糊表述。
2. 写下两个最可能的原因，并说明它们分别发生在哪一层。
3. 给每个原因选一种合适工具：日志、断点、Inspector、Profiler 或线上监控。
4. 只添加足以验证假设的最少量观察点。
5. 修复后回头删除临时调试代码，并总结这次问题以后该在哪个位置保留长期观测信息。

预期结果：

- 你会把调试看成验证假设的过程，而不是“多打一打日志”。
- 你能更清楚什么时候该看日志，什么时候该用可视化工具。
- 你的调试动作会更克制，也更容易复盘。

自检方式：

- 你能解释为什么日志不应该只是“打印更多字符串”。
- 你能判断某个问题更适合日志、断点还是 Profiler。
- 你能说明为什么调试也有隐私边界。

调试提示：

- 日志太多反而看不出重点时，先删到只剩围绕一个假设的关键点。
- 一直加日志却从不写出假设，说明排查方法已经偏了。
- 发布包里保留了大量调试输出，优先回查构建和日志策略。

### 10. 常见误区

- 把调试等同于“多打日志”。
- 不先提假设，就盲目四处观察。
- 只会看 Logcat，不会用运行时可视化工具。
- 为了排障泄露敏感信息。

## 小结

日志与调试的核心，不是工具越多越好，而是你能否围绕问题提出假设、选择合适观察点、快速收集证据并缩小范围。只要把调试看成一种验证流程，而不是随机打印和点击，排障效率和工程质量都会明显提升。
