# 日志与调试

很多开发者把调试理解成“出问题时打几个日志看看”，这种做法在小 Demo 阶段勉强够用，但一旦进入真实工程，日志与调试就会变成一套完整的质量保障能力。日志负责留下运行痕迹，调试负责重建问题上下文，而二者共同决定了你面对线上与线下问题时到底是“靠猜”，还是“能定位”。

Android Studio 和 Android 平台都提供了大量调试能力，但真正有价值的不是工具本身，而是你是否建立了稳定的排查习惯：日志是否有边界、调试信息是否足够、线上线下是否能追到同一个问题。只要这三点缺失，再多工具也只是摆设。

## 学习目标

- 理解日志在工程中的真正价值。
- 理解日志、断点、Profiler、App Quality Insights 分别解决什么问题。
- 理解为什么日志必须克制、可分类、可检索。
- 为测试、性能优化和发布后维护打基础。

## 前置知识

- 已理解基本构建流程和 Android Studio 开发环境。
- 已做过简单断点调试或查看 Logcat。

## 正文

### 1. 日志不是“临时打印”，而是运行证据

日志最核心的价值，是在问题发生后帮助你重建当时的运行过程。它回答的问题通常包括：

- 某段逻辑有没有执行到。
- 输入参数是什么。
- 任务在哪个阶段失败。
- 某个状态是如何演变的。

如果日志只是随手打印一串难以检索的字符串，那么它很快就会变成噪声，而不是证据。

### 2. 日志必须有边界和分类

更成熟的日志设计通常会区分不同层次，例如：

- 页面状态日志。
- 网络请求与响应日志。
- 数据层关键流程日志。
- 错误与异常日志。

这样做的意义，是让你在排查问题时能快速定位范围，而不是在一整屏无意义输出里盲找。日志的价值很大程度上取决于是否可检索、可理解、可归类。

### 3. 调试不是只有断点

很多人把调试等同于“下断点单步走”。断点当然重要，但并不是唯一手段。现代 Android 工程里，常见调试手段还包括：

- Logcat。
- Layout Inspector。
- Network Inspector。
- Database Inspector。
- Profiler。
- App Quality Insights。

更合理的做法是根据问题类型选择工具，而不是所有问题都先下断点。

### 4. 为什么日志必须克制

日志越多不一定越好。过量日志会带来几个问题：

- 真正重要的信息被噪声淹没。
- 调试时检索困难。
- 线上版本可能暴露隐私或实现细节。
- 网络与数据库日志可能泄露敏感数据。

因此，日志设计必须同时考虑可用性和安全性。工程里真正有价值的日志，往往是少而关键，而不是多而混乱。

### 5. 调试的重点是重建“上下文”

很多问题之所以难查，不是因为单个异常太复杂，而是因为你缺少上下文：用户做了什么、页面处于什么状态、这次请求属于哪条链路、错误发生前后系统环境如何变化。日志和调试工具真正想帮你做的，就是把这些上下文补齐。

也因此，一个成熟的排查流程通常不是只看一条报错，而是把：

- 日志。
- 栈信息。
- 状态变化。
- 设备环境。
- 关键输入。

一起放回同一条时间线上理解。

### 6. 线上问题和线下问题需要不同思路

线下调试更适合断点、Inspector 和本地复现。线上问题则更依赖崩溃报告、分组统计、质量面板和带上下文的日志。很多团队的调试体系不成熟，就在于只能处理“我本地能复现”的问题，而一旦进入线上环境，就只能依靠猜测。

这也是为什么日志与调试章节必须和测试、性能、发布后维护连起来看。

### 7. 一个更接近真实项目的例子

假设用户反馈“进入文章详情页偶发白屏”。更成熟的排查方式通常不是只打一个“进入详情页了”的日志，而是：

- 记录页面参数。
- 确认 ViewModel 状态流变化。
- 记录数据层请求与缓存命中情况。
- 检查是否存在解析失败或空数据兜底。

这个例子说明，日志的价值在于帮助你重建链路，而不是单点确认某一行代码有没有执行。

### 8. 实践任务

起点条件：

- 已有一个会发网络请求、读数据库或状态较复杂的页面。

步骤：

1. 列出这个页面最关键的 3 到 5 个状态节点。
2. 设计一组有分类、有上下文的日志，而不是随手打印。
3. 选一个问题类型，判断它更适合用 Logcat、断点、Profiler 还是 Inspector。
4. 检查当前日志中是否包含敏感数据或无意义噪声。
5. 模拟一次线上问题排查，试着仅靠日志与报告重建问题链路。

预期结果：

- 你会把日志设计成运行证据，而不是临时输出。
- 你能根据问题类型选择更合适的调试工具。
- 你会更清楚线上与线下排查的差异。

自检方式：

- 你能解释：为什么日志的核心价值是重建上下文。
- 你能判断：一个问题更适合断点还是更适合 Profiler/Inspector。
- 你能说出：为什么日志必须克制且避免敏感信息泄露。

调试提示：

- 如果一条问题链路没有统一日志上下文，后续排查会变得非常痛苦。
- 如果所有日志都打印在 UI 层，通常说明状态与数据边界还不够清楚。
- 如果线上问题只能靠本地猜测，说明调试体系没有覆盖发布后场景。

### 9. 常见误区

- 把日志当成临时打印，用完不整理。
- 日志越多越好，不做分类和筛选。
- 认为调试只有断点一种方式。
- 在线上版本输出过多敏感实现细节。

## 小结

日志与调试真正考验的，不是工具使用熟练度，而是你是否能持续留下可定位问题的证据，并在问题发生后重建完整上下文。只要这套能力建立起来，测试、性能分析和发布后维护都会轻松很多。
