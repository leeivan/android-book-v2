# 性能优化

性能问题很少会以“请做性能优化”这样正式的需求出现。它更常见的表达是：首页打开慢，长列表滑动掉帧，切页面时卡一下，图片一多就发热，老设备上内存占用越来越高，或者用户只会留下一句“这个功能用着不顺”。这也是性能章节最容易写偏的地方。如果只把它写成技巧清单，读者很快就会记住一些名词，却仍然不知道该先看哪里、先量什么、先改什么。

本地参考资料里关于性能有几个共识值得直接吸收进来。第一，性能不是单一指标，而是用户感知、资源使用和系统约束共同作用的结果。第二，性能工作必须先测量，再优化；模拟器能帮你调试，但在真正评估性能时，真机往往更有代表性。第三，很多表面上的“卡”，最终都不是一个局部 API 的问题，而是一整条链路没有分层清楚。本章就围绕这三点来建立性能优化的基本工作流。

## 学习目标

- 理解性能优化首先解决的是用户感知和资源效率问题。
- 理解为什么性能工作必须先测量，并知道真机与 Profiler 的基本价值。
- 理解启动、列表、位图、内存、网络和后台任务这些常见性能问题的判断入口。
- 学会把性能问题从局部症状追到链路和结构层面。

## 前置知识

- 已理解线程、协程、布局、数据层和调试工具。
- 已接触 Android Studio Profiler、RecyclerView 或 Compose 列表的基本用法。
- 最好已经了解图片加载、页面生命周期和后台任务的基础概念。

## 正文

### 1. 性能首先是用户体验问题，不是参数堆叠问题

技术视角里，性能常被拆成 CPU、内存、网络、帧率、I/O、耗电等很多维度；用户视角里，问题通常只有一个：这个应用为什么用起来不顺。只要这一点没有先说清，后面的所有工具和指标都容易变成“看起来很专业”的噪音。

因此，性能工作的第一步不是直接打开 Profiler，也不是先讨论某个框架，而是把现象翻译成用户语言。用户抱怨启动慢，通常对应首帧前工作过重；用户抱怨滑动卡，往往对应列表绑定、图片解码、布局层级或主线程数据处理；用户抱怨越用越热、越用越慢，往往已经牵涉内存、后台任务、重试策略和资源回收。只有先把现象说成一条具体的使用链路，后面的测量才有意义。

### 2. 先测量，再优化，而且尽量在真机上测量

本地参考书在讲调试与性能时都强调同一个顺序：先看事实，再动代码。没有测量就优化，最大的问题不是可能白做，而是你根本不知道自己修的是不是主因。很多“凭感觉优化”的结果是：代码改了不少，体验没有变化；或者某个局部确实更快了，但真正的瓶颈还留在原地。

这一步至少要建立两个习惯。第一个习惯是先做基线。你要知道当前版本的启动、滚动、内存或网络表现到底是什么样，否则后面的“优化成功”无法被证明。第二个习惯是尽量在真机上看性能。本地参考书明确提醒，模拟器适合功能验证，但在真正评估性能时，它并不能替代真实设备。真机上的 CPU、磁盘、热状态、系统调度和后台限制，才更接近用户会遇到的环境。

Android Studio Profiler 之所以重要，不在于它是一个“高级工具”，而在于它能把性能问题拆成可观察的资源时间线。CPU、Memory、Network、Energy 这几个视角，分别对应不同类型的瓶颈。你不一定每次都要做深入录制，但至少要习惯先看时间线、再看峰值、再把峰值对应回用户操作，而不是一开始就直接改实现。

### 3. 启动优化的核心，是把首帧前的工作分层

启动性能之所以总是最容易被用户感知，是因为它决定了第一印象。用户不会把问题描述成“冷启动阶段主线程负载过高”，他们只会觉得应用打开后白屏太久、第一屏迟迟不出现，或者一启动就明显顿一下。

真正值得建立的判断，不是“有没有做启动优化”，而是“哪些工作必须发生在首帧之前，哪些工作应该延后”。主题、首屏必要依赖、关键导航判断通常要尽早完成；而批量数据预热、非关键网络请求、重型日志初始化、分析上报等，则往往更适合在首帧后再逐步启动。很多启动慢的问题不是某一行代码太慢，而是没有把初始化按优先级拆开。

如果一个页面只有在把所有远端数据、图片和统计都准备好之后才肯展示第一屏，它看起来像是“启动慢”，本质上却是把不该阻塞首帧的工作全部提前了。启动优化最常见的收益，不是把所有东西都做快，而是让不必抢首帧的工作不要抢首帧。

### 4. 列表卡顿通常不是列表容器本身慢，而是绑定链路太重

很多开发者一看到掉帧，就先怀疑 RecyclerView 或 Compose 列表本身。但本地参考资料在讲 RecyclerView 时反复强调，列表真正的成本常常藏在条目绑定里，而不是藏在容器名字里。列表项布局过深、绑定阶段做了过多格式化、滚动时还在主线程做排序映射、图片解码尺寸失控、每次更新都整表刷新，这些都可能让用户最终只感受到一个结果：滑不动。

这也是为什么性能章节不能只停留在“UI 技巧”。你看到的现象可能是掉帧，真正的问题却分布在布局、数据处理、图片加载、网络节奏和更新策略上。RecyclerView 的价值之一是重用 ViewHolder，减少重复创建成本；但如果你的条目每次绑定都做大量额外工作，重用机制也救不了整体流畅度。Compose 虽然没有 RecyclerView 这一套命名，但同样受制于数据稳定性、重组成本和主线程负担。

实际排查时，一个很有效的顺序是：先确认是滚动本身卡，还是滚动过程中某个绑定动作卡；再判断问题更像布局、图片、数据加工还是整表更新策略；最后才进入具体实现。只要顺序反过来，你就很容易陷入“先改一点再看看”的盲修状态。

### 5. 位图和内存问题，往往在崩溃之前就已经伤害体验

很多团队直到 OOM 才把内存当成问题，但在真实应用里，内存过高通常会先以更隐蔽的方式伤害体验：页面切换变慢、后台保活变差、老设备上更容易抖动、长时间使用后越来越热。这正是为什么性能章节不能只把内存写成“避免崩溃”。

本地参考书在讲图片处理时给出了一个非常有代表性的判断：压缩文件很小，不等于加载进内存后也很小。位图对象保存的是像素数据，而不是压缩后的磁盘格式。一张拍照得到的大图，磁盘上也许只有几 MB，但如果按原尺寸直接解码进内存，体积会迅速放大。也就是说，图片性能问题很多时候不是“图片太多”，而是“你把不该用原尺寸加载的图片按原尺寸加载了”。

下面这个最小示例就体现了性能章节里非常实用的一类优化：先读取尺寸，再按目标显示区域缩放，而不是直接把原图完整解码进内存。

```kotlin
fun decodeScaledBitmap(path: String, destWidth: Int, destHeight: Int): Bitmap {
    val bounds = BitmapFactory.Options().apply {
        inJustDecodeBounds = true
    }
    BitmapFactory.decodeFile(path, bounds)

    val safeWidth = destWidth.coerceAtLeast(1)
    val safeHeight = destHeight.coerceAtLeast(1)
    val widthSample = (bounds.outWidth / safeWidth).coerceAtLeast(1)
    val heightSample = (bounds.outHeight / safeHeight).coerceAtLeast(1)
    val sampleSize = maxOf(widthSample, heightSample).coerceAtLeast(1)

    return BitmapFactory.decodeFile(path, BitmapFactory.Options().apply {
        inSampleSize = sampleSize
    })
}
```

这段代码的意义不在于记住 `BitmapFactory` 的参数，而在于建立一个性能判断：图片显示要按目标区域解码，而不是按原文件尺寸解码。只要这个判断建立起来，你在处理头像、列表缩略图、相册预览和拍照结果时，就会更容易规避内存尖峰。

内存问题的另一类常见来源是对象生命周期混乱。参考书在讲 ViewModel 时明确提醒，ViewModel 不应该持有 Activity 或 View 的引用，否则配置变化后很容易形成泄漏：旧页面本该被回收，却因为还被强引用着而留在内存里。很多“越用越慢”的问题，不是某个算法突然变差，而是旧对象一直没真正离场。

### 6. 网络、电量和后台任务也是性能的一部分

如果一个应用页面滚动看起来还算顺，但后台一直高频请求、无意义重试、反复下载可缓存内容，它在用户眼里依然可能是“性能差”的应用，因为它更耗电、更耗流量，也更容易在弱网环境下显得不稳定。性能不是只看帧率，它还包括资源是否被克制地使用。

这类问题最常见的本质，是做了太多不必要的事。网络请求是否真的需要在页面一打开就全部并发？缓存命中后是否还在重复请求？后台同步是否有明确约束，还是任何状态变化都试图立即触发？只要这些边界没有建立清楚，性能优化就很容易停留在“把已有工作做快一点”，却没能先回答“哪些工作其实根本不该做”。

因此，很多真实的性能收益并不来自更复杂的实现，而来自更清醒的克制：减少不必要的请求、减少不可见时段的工作、减少无效刷新、减少无意义的重试和轮询。性能工程的成熟度，往往体现为系统学会了少做错事。

### 7. 结构越清楚，性能问题越容易被定位

不少性能问题最终会追溯到结构问题，而不是局部代码语句问题。页面层承担了太多数据转换，列表绑定承担了本该在数据层完成的加工，后台任务没有统一调度边界，图片、网络和数据库各自为政，这些都会让性能排查越来越难。你今天修掉的是掉帧，明天又会以耗电、卡顿或内存增长的形式回来。

因此，性能优化不只是“把几个热点函数改快”。它还要求你不断回头检查：这段工作应该在哪一层完成？它应该被谁拥有生命周期？它是应该在首帧前执行，还是进入后台？它是页面状态，还是缓存策略，还是数据层职责？只要职责边界稳定，Profiler 看到的峰值就更容易对应回具体职责；否则性能问题永远像散落在系统里的雾。

### 8. 一个更现实的性能排查例子

假设首页是一个图片列表，用户的抱怨是“打开慢，滑动时还会卡”。比起立刻去“优化首页”，更成熟的做法是先拆链路。第一层，启动阶段做了什么：Application 是否初始化过重，首屏是不是被非关键工作阻塞。第二层，列表首屏渲染做了什么：是不是等待了太多网络结果，条目布局是不是过深。第三层，图片链路做了什么：是否按显示区域缩放解码，还是直接加载了大图。第四层，滚动时主线程还在做什么：是否一边滑动一边做映射、排序或格式化。

这样一来，问题就不再是一个抽象的“慢”，而是一条可以逐段观察的用户链路。你可以先在真机上复现，再用 Profiler 看 CPU、Memory、Network 的变化，然后判断究竟是启动分层出了问题、条目绑定过重、位图解码过大，还是后台请求节奏不合理。性能优化真正怕的，不是问题复杂，而是问题一直停留在含糊的描述里。

### 9. 实践任务

起点条件：

- 已有一个可复现的性能问题，例如慢启动、列表掉帧、图片导致内存升高、发热或异常耗电。

步骤：

1. 先用一句用户语言写出问题现象，而不是笼统地写“性能不好”。
2. 判断它更接近启动、滚动、位图内存、网络还是后台任务问题。
3. 尽量在真机上复现，并用 Profiler 建立一次基线观察。
4. 只选择一条最可能的关键链路，不同时改很多地方。
5. 做一处针对性调整后重新测量，确认体验和指标是否真的改善。

预期结果：

- 你会把性能优化理解成观察、定位、验证的闭环，而不是经验清单。
- 你会更清楚性能问题为什么常常跨越 UI、数据和资源边界。
- 你能用更具体的语言描述性能问题，而不是停留在“感觉有点卡”。

自检方式：

- 你能解释为什么模拟器不能完全替代真机做性能判断。
- 你能说出 Profiler 里 CPU、Memory、Network、Energy 分别在帮助你看什么。
- 你能判断一个图片问题为什么可能最后变成内存问题。
- 你能说明为什么性能问题往往和结构边界有关。

调试提示：

- 没有基线就开始优化，最后通常很难证明收益。
- 一次改太多地方，回测时就很难知道真正起作用的是哪一步。
- 只盯着某个局部实现，不回到整条用户链路，很容易错过主因。

### 10. 常见误区

- 凭感觉优化，不先测量。
- 在模拟器里看起来“还行”，就当真实设备也没问题。
- 只把性能理解成帧率，不看内存、网络和电量。
- 把图片文件大小误当成内存占用大小。
- 让 ViewModel、页面或后台任务持有不该持有的引用与职责。
- 优化后不回测，导致结果无法证明。

## 小结

性能优化真正关心的，是用户是否感到应用顺畅、稳定、克制并且可持续使用。启动、滚动、位图、内存、网络和电量只是不同观察面，它们最后都会落回同一个工程问题：你有没有先把事实看清，再按链路和结构做针对性处理。只要坚持“先测量、后定位、再优化、最后回测”的顺序，并优先用本地参考资料里反复强调的真机测量、Profiler 观察、位图缩放和生命周期边界这些基本判断，性能工作就会从零散技巧变成稳定能力。
