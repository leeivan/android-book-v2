# 性能优化

性能问题很少会以“请做性能优化”这样正式的需求出现。它更常见的形式是: 首页打开有点慢，长列表滑动掉帧，切页面时明显卡一下，电量掉得快，老设备上内存占用偏高，某个功能用户一直抱怨“用着发热”。性能之所以难，不只是因为指标多，而是因为它最终表现为用户感知，而用户不会关心你是 CPU、I/O、布局层级还是对象分配出了问题。

这一章的目标，不是列出一串优化技巧，而是建立一种性能工作方式: 先明确现象，再测量，再定位瓶颈，再做有针对性的优化。只要跳过测量、直接凭经验乱改，性能优化很容易变成“做了很多，却说不出到底有没有变好”。

## 学习目标

- 理解性能优化真正关注的是用户感知和资源效率。
- 理解启动、滚动、内存、网络、电量等常见性能维度。
- 理解“先测量后优化”为什么是工程原则。
- 学会从 Android 运行时工具和架构边界角度定位性能问题。

## 前置知识

- 已理解线程、协程、布局、数据层和调试工具。
- 已接触 Android Studio Profiler 或基本性能指标。

## 正文

### 1. 性能问题首先是用户问题，不是技术问题

技术视角里，性能常被拆成很多指标；用户视角里，只有一个问题: “这个应用用起来顺不顺”。因此性能优化的第一步不是讨论框架或技巧，而是先把现象翻译成用户语言:

- 打开页面要等多久。
- 滑动是否顺畅。
- 切换操作有没有明显顿挫。
- 长时间使用后会不会发热或掉电。

只有先从用户感知出发，后面的指标才有意义。

### 2. 为什么“先测量后优化”是铁律

没有测量就做优化，最大的问题不是可能白做，而是你根本不知道自己在解决什么。很多“凭感觉优化”的后果是:

- 改了一堆代码，体验没变化。
- 修了一个并不是真瓶颈的地方。
- 引入了新复杂度，却没有拿到可见收益。

因此成熟的性能工作流通常是:

1. 先确认现象。
2. 用工具定位瓶颈。
3. 再做针对性优化。
4. 回测确认收益。

### 3. 启动性能在很多应用里最容易先被感知

用户对启动的耐心非常有限。如果应用打开后长时间白屏、卡在首帧前、或者主线程初始化太重，用户的第一印象就会被直接拉低。

启动性能问题常见来源包括:

- Application 初始化过多。
- 首屏页面做了太多同步工作。
- 首次绘制前就加载大量非必要资源。

这也是为什么工程上越来越强调区分“首帧必须完成的工作”和“可以稍后再做的工作”。

### 4. 滚动性能往往暴露 UI 和数据链路的共同问题

列表掉帧不一定只是 UI 问题，它可能同时暴露:

- 布局层级过深。
- 列表项绑定过重。
- 图片解码和加载策略不当。
- 滚动时仍在主线程做过多数据处理。

所以性能章节必须强调: 你看到的是滚动卡顿，但真正的问题可能出在布局、图片、线程、数据转换甚至日志输出上。

### 5. 内存问题不是只有 OOM 才算问题

很多团队直到应用崩溃才开始关心内存，其实过高内存占用早就会影响:

- 页面切换流畅度。
- 后台保活能力。
- 老设备上的稳定性。
- 长时间使用后的发热和性能退化。

因此内存优化不应只理解成“避免崩溃”，而是应用持续稳定运行的一部分。

### 6. 网络和电量也属于性能

性能不只是帧率和内存。一个总在后台高频请求、频繁重试、下载大量无必要内容的应用，即使表面滚动流畅，也可能在用户眼里很“差”。因为它会:

- 更耗流量。
- 更耗电。
- 更容易在弱网下表现不稳定。

这说明性能优化和数据策略、后台策略、缓存策略其实深度相关。

### 7. 结构清晰往往比局部技巧更能长期改善性能

不少性能问题最终会追溯到结构问题，例如:

- 页面层承担了太多数据处理。
- 后台任务分类混乱。
- 图片、网络、数据库各做各的，不共享策略。

这也是为什么性能优化不能只靠几个微技巧。好的架构边界、合理的缓存、正确的线程归属、克制的初始化，本身就是长期性能优化。

### 8. 一个现实例子: 首页卡顿为什么常常不是单点问题

首页打开慢，表面上像“启动性能”。真正排查后，往往会发现:

- Application 做了过多初始化。
- 首屏页面启动就并发请求多个接口。
- RecyclerView 首屏还要做复杂图片加载和映射。
- 主线程上存在不必要的数据整理。

这说明性能优化常常不是修一个函数，而是把一整条关键链路减重。

### 9. 实践任务

起点条件:

- 已有一个用户可感知的性能问题，例如慢启动、掉帧、发热或内存偏高。

步骤:

1. 用一句用户语言描述问题现象。
2. 决定它更可能属于启动、滚动、内存、网络还是电量维度。
3. 选择合适工具先测量，而不是先改代码。
4. 找出一条最可能的关键链路瓶颈。
5. 做一处针对性优化后重新测量，确认是否真的改善。

预期结果:

- 你会把性能优化看成定位与验证过程，而不是经验清单。
- 你会更清楚指标和用户感知之间的关系。
- 你能把局部症状追到更底层的结构问题上。

自检方式:

- 你能解释为什么性能优化必须先测量。
- 你能判断某个问题更像启动、滚动还是资源消耗问题。
- 你能说明为什么架构边界也会影响性能。

调试提示:

- 没有基线就开始优化，结果通常无法证明有效。
- 只盯着一个函数不看整条链路，很容易错过真正瓶颈。
- 优化后没有回测，说明你还没有完成性能工作闭环。

### 10. 常见误区

- 凭感觉优化，不先测量。
- 只把性能理解成帧率。
- 把所有性能问题都归结为某个局部实现。
- 优化后不验证效果。

## 小结

性能优化真正关注的，是用户是否感到应用顺畅、稳定、节能和可持续使用。启动、滚动、内存、网络和电量只是不同观察面。只要坚持“先测量、后定位、再优化、最后回测”的顺序，再把问题放回结构和链路里看，性能工作就会从零散技巧变成稳定工程能力。
