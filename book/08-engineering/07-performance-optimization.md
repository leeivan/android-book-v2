# 性能优化

性能问题很少会以“请做性能优化”这样正式的需求出现。它更常见的形式是：首页打开有点慢，长列表滑动掉帧，切页面时明显卡一下，电量掉得快，老设备上内存占用偏高，某个功能用户一直抱怨“用着发热”。性能之所以难，不只是因为指标多，而是因为它最终表现为用户感知，而用户不会关心你是 CPU、I/O、布局层级还是对象分配出了问题。

参考资料里讲性能时有一个特别重要的共同原则：先测量，再优化。这个顺序看起来朴素，却能直接拉开成熟做法和经验主义做法的差距。本章不会把性能写成技巧清单，而是把它写成一条完整工作流：先明确现象，再建立指标，再定位瓶颈，最后做针对性优化并回测。

## 学习目标

- 理解性能优化真正关注的是用户感知和资源效率。
- 理解启动、滚动、内存、网络和电量这些常见性能维度。
- 理解“先测量后优化”为什么是工程原则。
- 学会从 Android 运行时工具和架构边界角度定位性能问题。

## 前置知识

- 已理解线程、协程、布局、数据层和调试工具。
- 已接触 Android Studio Profiler 或基本性能指标。

## 正文

### 1. 性能首先是用户问题，而不是技术指标问题

技术视角里，性能常被拆成很多指标；用户视角里，只有一个问题：这个应用用起来顺不顺。因此性能工作的第一步，不是讨论框架、缓存或优化技巧，而是先把现象翻译成用户语言：

- 启动要等多久。
- 滑动是否顺畅。
- 页面切换是否顿挫。
- 长时间使用后会不会发热或掉电。

只有先从用户感知出发，后面的指标才有意义。

### 2. 为什么“先测量后优化”是铁律

没有测量就做优化，最大的问题不是可能白做，而是你根本不知道自己在解决什么。很多“凭感觉优化”的后果是：

- 改了一堆代码，体验没变化。
- 修了一个并不是真正瓶颈的地方。
- 引入了新复杂度，却没有拿到可见收益。

成熟的性能工作流通常是：

1. 确认现象。
2. 建立基线。
3. 用工具定位瓶颈。
4. 再做针对性优化。
5. 回测确认收益。

这个流程听起来慢，实际上比“先试试看”更节省时间。

### 3. 启动性能为什么总是最容易被用户先感知

用户对启动的耐心非常有限。如果应用打开后长时间白屏、首帧前卡住、或者 Application 初始化过重，用户的第一印象就会直接被拉低。教材里讲启动性能时，最值得建立的判断是：哪些工作必须在首帧前完成，哪些工作可以延后。

例如：

- 必须尽早完成的：主题、首屏必要依赖、关键导航判断。
- 更适合延后的：大批数据预热、非关键网络请求、重型分析初始化。

很多启动问题并不是“某一行代码太慢”，而是没有把初始化按优先级分层。

### 4. 列表掉帧通常暴露的是整条链路问题

长列表不流畅时，开发者最容易先盯着 RecyclerView 或 Compose 列表本身。但真实情况往往更复杂。掉帧可能来自：

- 布局层级过深。
- 列表项绑定逻辑过重。
- 图片解码和加载策略不合理。
- 滚动时仍在主线程做数据映射或排序。

这就是为什么性能章节不能只讲 UI 技巧。你看到的可能是“滑动卡”，真正的问题却可能在网络、图片、线程或数据处理。

### 5. 内存问题不是只有 OOM 才算问题

很多团队直到应用崩溃才开始关心内存，其实过高内存占用早就会影响：

- 页面切换流畅度。
- 后台保活能力。
- 老设备上的稳定性。
- 长时间使用后的发热和性能退化。

换句话说，内存优化不只是“避免崩溃”，而是让应用在长时间运行中保持稳定。参考资料里讲调试和性能时经常把内存与泄漏、对象生命周期、图片加载一起讨论，原因就在这里。

### 6. 网络和电量也属于性能问题

性能不只是帧率和内存。一个总在后台高频请求、频繁重试、下载大量无必要内容的应用，即使页面滚动看起来流畅，也可能在用户眼里很差。因为它会更耗流量、更耗电，也更容易在弱网环境下表现不稳定。

这说明性能优化和数据策略、缓存策略、后台任务策略其实深度相关。很多时候真正的性能收益，来自“少做不必要的事”，而不是“把必须做的事做得更快”。

### 7. 结构清晰往往比局部微调更有长期收益

不少性能问题最终会追溯到结构问题，例如：

- 页面层承担了太多数据处理。
- 后台任务分类混乱。
- 图片、网络、数据库各做各的，缺少统一策略。

这也是为什么教材里要不断强调架构边界。好的边界不只让代码更清晰，也让性能问题更容易被定位和修复。否则你每次都只能做局部补丁，问题很快还会回来。

### 8. 一个更现实的性能排查例子

假设首页打开很慢。更成熟的排查方式不是立刻去“优化首页”，而是先拆出这条链路：

- Application 启动做了什么。
- 首屏页面在首帧前做了什么。
- 哪些网络请求是同步阻塞的。
- 列表首屏渲染是否依赖重型图片或数据转换。

这样你就不再是在修一个抽象的“慢”，而是在拆一条具体的用户链路。性能优化最怕的问题，就是现象抽象、定位模糊。

### 9. 实践任务

起点条件：

- 已有一个用户可感知的性能问题，例如慢启动、掉帧、发热或内存偏高。

步骤：

1. 用一句用户语言描述问题现象，而不是直接说“性能不好”。
2. 判断它更像启动、滚动、内存、网络还是电量维度。
3. 选择合适工具建立基线，不先改代码。
4. 找出一条最可能的关键链路瓶颈。
5. 只做一处针对性优化，然后重新测量，确认是否真的改善。

预期结果：

- 你会把性能优化看成定位与验证流程，而不是经验清单。
- 你会更清楚指标和用户感知之间的关系。
- 你能把局部症状追到更底层的结构问题上。

自检方式：

- 你能解释为什么性能优化必须先测量。
- 你能判断某个问题更像启动、滚动还是资源消耗问题。
- 你能说明为什么架构边界也会影响性能。

调试提示：

- 没有基线就开始优化，最后通常无法证明有效。
- 只盯着一个函数不看整条链路，很容易错过真正瓶颈。
- 优化后没有回测，说明你还没有完成性能工作闭环。

### 10. 常见误区

- 凭感觉优化，不先测量。
- 只把性能理解成帧率。
- 把所有性能问题都归结为某个局部实现。
- 优化后不验证效果。

## 小结

性能优化真正关注的，是用户是否感到应用顺畅、稳定、节能和可持续使用。启动、滚动、内存、网络和电量只是不同观察面。只要坚持“先测量、后定位、再优化、最后回测”的顺序，再把问题放回结构和链路里看，性能工作就会从零散技巧变成稳定工程能力。
