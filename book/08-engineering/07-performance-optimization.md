# 性能优化

性能优化最容易走偏的地方，是把它理解成“若干零散技巧合集”。实际上，性能问题背后往往对应更深层的系统设计问题，例如不必要的工作、错误的线程分配、失控的内存占用或不合理的数据流。真正有效的优化，不是盲目套技巧，而是先建立度量和定位。本章的重点，就是把性能从“玄学”还原成可验证的工程活动。

## 学习目标

- 理解 Android 性能问题的主要类型。
- 建立“先测量、再定位、后优化”的工作方式。
- 理解启动、渲染、内存、电量和网络都是性能的一部分。
- 为调试、Profiler 和发布质量治理打基础。

## 前置知识

- 已理解 UI、并发、网络和数据层的基本原理。
- 已接触日志与调试工具。

## 正文

### 1. 为什么性能优化不能靠感觉

用户感受到卡顿、耗电或发热时，根因可能完全不同。只凭肉眼和猜测，很容易把时间花在次要问题上。

因此性能优化的第一原则是：先测量，再讨论。没有数据支撑的“优化”，很多时候只是代码层面的自我安慰。

### 2. 启动性能关注什么

启动阶段最重要的是减少不必要工作，尤其是那些用户在首屏出现前根本感知不到价值的初始化。例如：

- 不必要的同步初始化。
- 过早加载大资源。
- 把所有 SDK 和配置一次性堆到启动点。

启动性能问题很多时候不是“手机慢”，而是工作发生在了错误时机。

### 3. 渲染性能的核心是什么

界面性能的核心是让主线程保持轻量，避免在绘制和交互路径上堆积重任务。常见风险包括：

- 列表项过重。
- 布局层级过深。
- 主线程做数据库或网络工作。
- 状态变化过于频繁，导致无意义重绘。

很多 UI 卡顿并不是某个控件本身慢，而是系统被迫做了太多不该在当前时机做的工作。

### 4. 内存、电量和网络为什么也属于性能问题

性能不是只有“帧率”。如果应用频繁申请大对象、持有不必要引用、持续后台轮询、重复下载同样数据，最终都会反映为用户层面的卡顿、耗电和不稳定。

因此看性能时必须把资源成本一起纳入，而不是只盯着界面有没有掉帧。

### 5. 优化优先级应如何判断

一个更实用的顺序通常是：

1. 先解决用户明显能感知的问题。
2. 再解决高频路径上的系统性浪费。
3. 最后才处理边角场景和微优化。

这能避免团队在价值很低的优化项上投入过多时间。

### 6. 为什么性能优化往往会反推架构问题

一旦你真正开始定位性能问题，就会发现很多根因其实在架构层：

- 页面状态变化过于混乱。
- 数据层反复重复加载。
- 后台任务和页面任务界限不清。
- 列表更新策略粗暴。

这说明性能优化不是独立章节，而是对整个工程设计的一次反向体检。

### 7. 最小实践任务

1. 选一个最常用页面，记录它从打开到可交互的关键步骤。
2. 标出其中哪些工作是首屏必须完成的，哪些可以延后。
3. 检查这个页面是否存在主线程做重工作的情况。
4. 再判断一个你熟悉的“卡”问题究竟更像启动、渲染还是数据层问题。

### 8. 常见误区

- 没有测量数据就开始优化。
- 一味追求底层技巧，忽略业务流程中的大块浪费。
- 把性能问题只理解成“界面卡不卡”。
- 把所有优化都集中到发布前临时处理。

## 小结

性能优化是一种基于事实的工程活动。只要你坚持从测量出发、围绕高频路径下手，很多性能问题都会从“玄学”重新变成可以拆解和验证的工程问题。
