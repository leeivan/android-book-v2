# CI/CD

当项目从个人练习走向团队协作时，“在我电脑上能跑”已经远远不够。持续集成和持续交付的意义，就是把构建、检查、测试和发布流程标准化，让每次改动都在相同规则下被验证，而不是靠人工记忆维持质量。本章的重点，是让你把 CI/CD 看成工程稳定性的自然延伸，而不是只有大团队才需要的“高级配置”。

## 学习目标

- 理解 CI 和 CD 分别解决什么问题。
- 知道 Android 项目里一条基础流水线应该包含哪些检查。
- 理解自动化不是炫技，而是减少重复性人为错误。
- 为测试、发布和版本维护打基础。

## 前置知识

- 已理解构建系统、测试和发布的基本流程。
- 已知道多人协作时手工流程非常容易失控。

## 正文

### 1. CI 和 CD 的本质是什么

CI 关注的是：每次改动都能被自动验证。例如：

- 构建是否通过。
- 单元测试是否通过。
- 静态检查是否出现回归。

CD 关注的是：通过验证的产物如何更稳定地交付，例如：

- 生成候选构建。
- 分发给测试或灰度用户。
- 接入发布流程。

两者的共同目标都不是更复杂，而是更可重复。

### 2. Android 流水线最小应该包含什么

对大多数项目来说，一条有价值的基础流水线至少应覆盖：

- 依赖准备和构建。
- 单元测试。
- 静态检查。
- 关键产物归档。

如果连这些都还依赖人工本地执行，那么团队质量会高度受制于个人习惯。

### 3. 自动化最大的收益是什么

很多人以为自动化的收益只是省时间，实际上更关键的是降低不确定性。只要每次提交都经过相同检查，你就能更早发现：

- 某个依赖变更破坏了构建。
- 某个分支逻辑破坏了测试。
- 某个模块在特定变体下无法通过。

这会显著减少“临近发布才集中爆雷”的概率。

### 4. Android 场景下的 CI/CD 有哪些特殊点

Android 的流水线并不只是跑单元测试，它还常常涉及：

- 多变体构建。
- 签名材料管理。
- 产物命名和归档。
- 发布轨道或测试分发。

因此流水线必须和变体、签名、发布策略一起考虑，而不是孤立存在。

### 5. 自动化边界在哪里

自动化很重要，但不代表所有决策都应自动完成。比如：

- 是否发布这个版本。
- 发布节奏和灰度范围。
- 线上回滚判断。
- 文案和商店资料准备。

这些仍然需要明确的人为工程判断。自动化优先替代的是重复、机械、易错环节，而不是替代决策本身。

### 6. 为什么小团队也应尽早建立最小流水线

很多人觉得 CI/CD 只有大团队才需要。实际上，小团队更容易受到单点依赖和人工遗漏影响，因此更适合尽早建立最小流水线。哪怕只是：

- 每次提交自动构建。
- 自动跑核心单元测试。
- 自动产出候选包。

也已经能显著降低工程波动。

### 7. 最小实践任务

1. 写出你的项目当前一次发布前需要人工执行的步骤。
2. 找出其中最重复、最机械、最容易漏掉的三个环节。
3. 设计一条最小流水线，把构建、测试和产物归档先自动化起来。
4. 检查当前流程中是否有“只有某个人电脑能完成”的关键步骤。

### 8. 常见误区

- 把 CI/CD 当成大厂专属能力。
- 只自动化打包，不自动化检查和测试。
- 流水线过度复杂，团队没人能维护。
- 以为用了 CI/CD 就不再需要发布判断和质量门槛。

## 小结

CI/CD 的价值，在于把质量验证和交付流程从“依赖个人经验”变成“依赖明确规则”。对 Android 工程而言，它是把项目从能开发推进到能稳定交付的关键一步。
