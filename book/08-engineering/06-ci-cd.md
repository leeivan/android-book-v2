# CI/CD

很多团队第一次认真引入 `CI/CD`，并不是因为想追求“更现代”的标签，而是因为工程已经开始被另一类问题拖住：某个开发者本地可以构建，别人却拉不起来；单元测试有人跑、有人没跑；发版依赖某个人在 Android Studio 里手工点一遍；问题进入主分支以后才被发现，回滚和定位都很狼狈。到了这个阶段，持续集成和持续交付就不再是锦上添花，而是把“团队已经认同的工程规则”交给机器稳定执行。

参考资料对 CI/CD 的一个处理很值得保留：它们不会把流水线单纯写成某个平台教程，而会把它理解成“工程纪律的自动化”。这个视角比背配置字段更重要。CI 要回答的是“这次改动是否达到可合并的最低质量门槛”，CD 要回答的是“达到门槛之后，构建产物怎样稳定、安全、可追踪地流向测试和发布环境”。平台可以替换，规则思路不该变。

## 学习目标

- 理解 CI/CD 真正解决的是工程规则执行和交付稳定性问题。
- 理解持续集成与持续交付在目标上的分工差异。
- 理解 Android 项目里哪些检查适合优先进入流水线。
- 学会从速度、可信度和可追踪性角度设计更健康的流水线。

## 前置知识

- 已理解 Gradle、构建任务、测试和发布基础。
- 已接触过 Git、分支协作和基本版本管理流程。
- 最好已经了解签名、版本号和 Android 构建产物的基本概念。

## 正文

### 1. “本地没问题”从来不是足够条件

开发者本地构建成功，只能证明“在这台机器、这个时间点、这个环境里，大概率能跑”。它不能证明依赖是否完整、构建是否可重复、别人拉下代码是否也能通过，更不能证明团队规定的最低质量门槛真的被执行了。项目一旦进入多人协作，“本地能跑”就只能算起点，不能再算标准答案。

CI 的价值就在这里。它把“构建必须通过”“关键测试必须通过”“关键检查必须通过”这些规则，从个人习惯提升成仓库级自动化门槛。只要每次 push、每次 pull request 都在同一套环境里执行同样的检查，主分支的健康状态才开始真正可依赖。

### 2. 持续集成真正保护的是“主干可用”

很多人把 CI 理解成“服务器上自动跑几条命令”。这个理解太轻了。持续集成真正保护的是主分支不要被一连串看似小、实则不稳定的改动慢慢侵蚀。为此，一条最小但健康的 Android CI 通常会做几件事：

- 拉取仓库代码。
- 设置正确的 JDK 版本。
- 复用 Gradle 缓存，保证速度可接受。
- 跑一次稳定构建。
- 跑关键静态检查和关键单元测试。

这些步骤的重点不在于“看起来齐全”，而在于它们是否真的对应团队最怕漏掉的风险。构建不过就不该合并，关键测试不过就不该进主干，这才是 CI 的真正职责。

### 3. 从最小流水线开始，比一上来堆满步骤更健康

参考资料里列出的 GitHub Actions 示例非常适合拿来理解“最小可用流水线”的概念。Android 项目的起点通常不需要特别复杂，先把最关键的事情稳定自动化就够了：`actions/checkout` 拉代码，`actions/setup-java` 配 JDK 17，开启 Gradle 缓存，给 `gradlew` 加执行权限，然后执行 `assembleDebug`、`testDebugUnitTest` 或少量静态检查。

```yaml
name: Android CI

on:
  push:
    branches: ["main"]
  pull_request:
    branches: ["main"]
  workflow_dispatch:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: zulu
          java-version: "17"
          cache: gradle
      - name: Grant execute permission
        run: chmod +x gradlew
      - name: Assemble debug
        run: ./gradlew assembleDebug
      - name: Run unit tests
        run: ./gradlew testDebugUnitTest
```

这段配置本身并不复杂，但它背后的工程意义很清楚：任何人提交到主分支或发起合并请求，都必须通过同一套最小门槛。相比“什么都还没稳定，就先把所有任务都塞进流水线”，这种起点更容易被团队真正长期依赖。

### 4. 触发时机决定了流水线是在补救，还是在预防

如果流水线只在代码已经进入 `main` 之后才运行，那么它更像事后补救；如果它能在 pull request 阶段就执行关键检查，那么很多问题会在真正进入主干之前被挡住。参考资料也强调了这一点：把 `pull_request` 事件加入触发条件，是让流水线从“合并后报警”变成“合并前把关”的关键一步。

这背后其实是一个工程态度问题。CI 不该只是“告诉你坏了”，更应该尽量在最早、最便宜的阶段把问题拦下来。对团队来说，这意味着更少的回滚、更少的主干污染，也意味着每个人都能更放心地基于主分支继续工作。

### 5. Android 项目里最适合优先自动化的检查

并不是所有检查都该第一天就自动化。更稳妥的策略，是优先把以下几类高频、刚性、可重复的规则放进流水线：

- 基础构建是否成功。
- 关键单元测试是否通过。
- 代码风格和静态检查是否满足团队底线。
- 候选产物是否能被稳定生成。

参考资料在这方面给出了常见的 Android 任务：`ktlintCheck`、`detekt`、`testDebugUnitTest`，再往上才是基于模拟器的 `connectedCheck`。这很符合真实项目规律。连接模拟器的 UI 或仪表测试价值当然高，但它们成本更高、耗时更长，更适合作为关键链路检查、夜间任务或特定 PR 的门槛，而不是在项目初期一股脑全部压上。

### 6. 流水线速度和可信度同样重要

一条太慢的流水线会逼人减少提交、拖延合并，甚至寻找绕过办法；一条经常误报的流水线则会让团队逐渐把红灯当成噪音。真正成熟的 CI/CD，不是“步骤很多”，而是满足三个条件：

- 足够快，能融入日常开发节奏。
- 足够稳，失败结果值得信任。
- 足够清楚，失败后能迅速定位原因。

这也是为什么 Gradle 缓存、拆分任务、区分快速检查与重型检查都很重要。流水线本身也是工程产品，不是写好 YAML 就能永远不管。你需要持续观察哪些步骤最慢、哪些失败最常见、哪些任务已经失去价值，再调整它们在流程中的位置。

### 7. 持续交付保护的是“产物如何稳定往下游流动”

当代码已经达到可合并标准，接下来需要被自动化保护的就不只是代码质量，还有产物管理。谁来生成 release 包，版本号如何递增，映射文件如何保存，签名在哪个阶段完成，构建结果如何进入内部测试、封闭测试或正式发布，这些问题都属于 CD 的范围。

参考资料里的 GitHub Actions 方案给出了一个很典型的 Android 发布链路：

- 在 workflow 中自动 bump `versionCode` 和 `versionName`。
- 生成 release AAB。
- 通过仓库密钥注入签名和服务账号配置。
- 把产物推送到 Google Play Internal Testing 轨道。

这套链路最关键的收益并不是“可以一键发版”，而是构建产物终于脱离了个人电脑和手工记忆，变成一条可追踪、可回放、可审计的发布流程。

### 8. 密钥、签名和发布权限，是 CD 能否落地的真实门槛

很多团队在讲 CD 时只谈 workflow 文件，却忽略了真正敏感的部分其实是发布凭证。Android 发布要想真正自动化，至少需要理清几类信息：

- keystore 内容和对应密码。
- key alias 及其密码。
- Google Play 服务账号 JSON。
- 不同轨道的发布权限和审批策略。

这些内容不应该硬编码在仓库里，而应该通过仓库 secret 或专门的机密管理方案注入。只有当密钥管理、签名策略和权限边界都稳定下来，CD 才能真正落地。否则所谓“自动化发布”只会停留在能跑 demo 的阶段。

### 9. 一个更现实的 Android 发布链路，通常是分层的

对大多数团队来说，最健康的发布链路不会是“每次合并都直接上正式版本”，而通常会分层：

- 普通提交：验证构建、快速检查和关键单元测试。
- 候选版本：生成 release 产物并关联版本号、mapping 文件和 commit。
- 内部测试：自动推送到 Internal Testing 或企业测试渠道。
- 正式发布：在验证通过后再进入灰度或生产轨道。

这比“本地打个正式包上传一下”更接近真实团队，也更便于追溯问题来源。你可以清楚回答：哪个提交产生了哪个 AAB，哪个版本进入了哪个测试轨道，哪个发布步骤失败了，失败时该找哪一层修复。

从工程治理角度看，流水线最好还能把“提交、产物、版本号、mapping 文件和发布记录”串成一条可追踪链。这样当线上崩溃、回滚或热修复需求出现时，团队不会只拿到一个模糊的“最近那个包”，而是能明确定位到具体 commit、具体构建配置和对应的调试符号。CI/CD 的成熟度，往往就体现在这种追踪能力上，而不是单纯有没有自动上传功能。

### 10. 什么时候不该急着把东西塞进流水线

并不是所有任务都应该第一时间自动化。以下情况更适合先手工稳定，再考虑接入：

- 还在频繁变化、尚未收敛的实验性流程。
- 运行成本很高，但短期收益不清晰的检查。
- 经常误报、尚未被团队信任的任务。

CI/CD 成熟的标志，不是自动化项目越多越好，而是自动化的每一步都真正在替团队减少风险。凡是还没有稳定规则支撑的动作，过早塞进流水线通常只会把噪音自动化。

## 实践任务

起点条件：

- 已有基本的 Android 构建脚本和至少一部分测试任务。

步骤：

1. 列出团队每次提交和发版前理论上必须做的检查。
2. 从中挑出已经足够稳定、最怕被遗漏的几项，作为最小 CI 门槛。
3. 为仓库补一条在 `push` 和 `pull_request` 上都能运行的 CI workflow。
4. 再设计一条最小 CD 流程，明确 release 产物、版本号、签名和内部测试轨道。
5. 观察流水线是否够快、够稳、够清楚，必要时拆分重型任务和快速任务。

预期结果：

- 你会把 CI/CD 理解成工程纪律自动化，而不是平台配置题。
- 团队的最低质量规则会更稳定地被执行。
- 发布流程会更可追踪，也更少依赖个人手工操作。

自检方式：

- 你能解释为什么 CI 更关注主干可用，而 CD 更关注产物流转。
- 你能判断哪些任务应该优先进入最小流水线。
- 你能说明为什么流水线速度和可信度同样重要。

调试提示：

- 流水线总是太慢，先缩小到最关键的检查，再考虑分层执行。
- 红灯经常是假警报，优先修复任务稳定性，而不是让团队习惯忽略失败。
- 发版仍依赖某个人本地操作，说明 CD 还没有真正承接发布风险。

### 常见误区

- 把 CI/CD 理解成某个平台的 YAML 教程。
- 一开始就堆满所有任务，导致流水线又慢又脆。
- 只把它当发布工具，不把它当质量门槛。
- 把 workflow 写好就不维护，结果逐渐失去可信度。

## 小结

CI/CD 真正要解决的，不是“有没有一条看起来现代的流水线”，而是工程规则能否被稳定执行、构建产物能否被可重复地交付下游。持续集成保护主干可用，持续交付保护产物流转可追踪。只要先围绕团队最怕漏掉的规则建立最小可依赖流水线，再逐步扩展，CI/CD 就会从额外流程变成整个 Android 项目最可靠的一套工程支架。
