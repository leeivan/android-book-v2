# CI / CD

当项目还只有一个人开发、改动也很少时，手工运行测试、手工打包、手工上传看起来都还能接受。但只要项目开始进入多人协作、发布频率提高、版本分支增多，CI/CD 就不再是“锦上添花”的自动化，而是质量和节奏能否稳定的关键基础设施。

CI/CD 的价值，不是“显得工程化”，而是把重复、易错且必须一致执行的动作交给机器完成，让团队把注意力放在决策而不是重复劳动上。本章会重点讲清楚：持续集成到底在保护什么，持续交付又为什么不能等同于“自动上线”。

## 学习目标

- 理解 CI 和 CD 分别解决什么问题。
- 理解自动化构建、测试、静态检查和产物管理为什么重要。
- 理解发布流水线与构建变体、签名、版本策略的关系。
- 知道什么样的项目已经值得引入 CI/CD。

## 前置知识

- 已理解 Gradle、构建变体、测试和发布基础。
- 已接触过多人协作或至少有明确的版本发布需求。

## 正文

### 1. CI 解决的是“每次改动都能被一致验证”

持续集成的核心不是“有个远程脚本在跑”，而是：

- 每次代码改动都能被标准化构建。
- 关键测试和检查都能自动执行。
- 问题能尽早在合并前暴露，而不是堆到发布日。

只要没有 CI，团队就很容易依赖“我本地跑过了”“应该没问题”这种不稳定判断。

### 2. CD 解决的是“产物从通过验证到可发布的路径稳定可重复”

持续交付和持续部署经常被混用。更实用的理解方式是：

- 持续交付强调：代码通过验证后，随时都能以稳定方式生成可发布产物。
- 持续部署更进一步，强调：通过验证后自动进入最终上线环节。

对大多数 Android 团队来说，持续交付往往比“全自动上线”更现实也更稳，因为应用发布还涉及商店审核、灰度策略、业务窗口和政策检查。

### 3. 自动化最值得先覆盖哪些事情

如果资源有限，CI/CD 最先应该自动化的通常是这些：

- 构建是否成功。
- 单元测试是否通过。
- 基本静态检查是否通过。
- 是否生成了正确产物。
- 关键变体是否都能打包。

这些动作重复、容易遗漏、且每次都必须一致，非常适合交给流水线。

### 4. CI/CD 和构建变体、签名、版本号天然相连

流水线从来不是独立存在的。它一定会和这些工程规则绑在一起：

- 哪些变体需要构建。
- 哪些变体需要上传。
- 签名材料如何安全管理。
- 版本号如何自动生成或校验。

如果这些规则本身还很混乱，流水线只会把混乱放大。因此很多团队不是“不会配 CI”，而是构建与发布规则先天不清。

### 5. 安全管理是 CI/CD 里最容易被忽略的部分

一旦进入自动化发布，你就不得不面对敏感信息问题，例如：

- 签名密钥。
- 发布凭证。
- API token。
- 上传渠道认证信息。

这意味着 CI/CD 不只是自动化脚本，还包含密钥管理、权限隔离和环境变量治理。越到发布阶段，这部分越不能靠“先写死，后面再说”。

### 6. 流水线真正要追求的是可重复和可观测

成熟的流水线不只是“能跑通一次”，而是应该具备：

- 相同输入得到相同结果。
- 出错时能快速看到哪一步失败。
- 产物和日志都能回溯。
- 不依赖某个开发者本地机器的特殊环境。

这说明 CI/CD 的本质，其实是在把工程行为从“个人机器经验”转成“团队可复现流程”。

### 7. 一个更接近真实项目的例子

一个更健康的 Android CI 流程通常至少会包含：

- 拉取代码。
- 执行关键变体构建。
- 执行测试与静态检查。
- 生成产物。
- 把产物和报告保存下来。

如果还进入 CD 阶段，则会进一步连接签名、版本信息和商店分发流程。这个例子说明，CI/CD 不是一条神奇命令，而是一整条工程验证链路。

### 8. 实践任务

起点条件：

- 已有一个具备最小测试和可打包能力的 Android 项目。

步骤：

1. 列出当前每次发布前手工重复做的动作。
2. 选出其中最机械、最容易漏掉的 3 项，作为第一批自动化对象。
3. 设计一条最小 CI 流程：构建、测试、检查、产物。
4. 明确哪些密钥和凭证绝不能直接写进仓库。
5. 思考当前项目距离“随时可出一个可验证产物”还缺什么。

预期结果：

- 你会从“降低人工不确定性”角度理解 CI/CD。
- 你能更清楚地区分持续集成和持续交付。
- 你能为后续发布章节搭起自动化主线。

自检方式：

- 你能解释：CI 和 CD 分别在保护什么。
- 你能判断：当前项目最应该先自动化哪些步骤。
- 你能说出：为什么流水线必须同时考虑密钥和产物管理。

调试提示：

- 如果流水线只能在某个人机器上复现，说明自动化价值还没有真正建立。
- 如果 CI 只会打包不会跑测试，问题仍会大量延后暴露。
- 如果发布密钥被临时散落在本地目录里，优先补安全管理而不是继续扩大流水线。

### 9. 常见误区

- 把 CI/CD 当成“项目做大以后再说”的事情。
- 认为只要自动打包就算有 CI。
- 持续交付被误解成必须全自动上线。
- 忽略签名、密钥和环境变量的安全管理。

## 小结

CI/CD 的真正价值，在于让构建、验证和交付过程变得稳定、可重复、可回溯。只要这条链路建立起来，团队发布节奏和工程质量都会明显提高。
