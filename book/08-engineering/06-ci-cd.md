# CI/CD

很多团队第一次认真引入 `CI/CD`，并不是因为想追求“更现代的工程流程”，而是因为项目已经被另一种问题拖住了：本地能跑，换台机器就不行；有的人提交流水线全绿，有的人却漏了测试直接把问题带进主分支；发布依赖某个开发者手工点一遍 Android Studio，步骤多、出错也难追。到了这个阶段，持续集成和持续交付就不再是锦上添花，而是把“工程规则能否被稳定执行”这件事正式系统化。

参考书在讲 CI/CD 时有一个很值得保留的视角：它们不会把流水线写成某个平台教程，而会把它当成“工程纪律的自动化”。本章也沿用这个视角。CI 负责回答“这份改动有没有达到可合并的最低质量标准”，CD 负责回答“达到标准后，产物怎样稳定、安全地往下游走”。平台可以换，规则思路不变。

## 学习目标

- 理解 CI/CD 真正解决的是工程规则执行和交付稳定性问题。
- 理解持续集成与持续交付分别关注什么。
- 理解流水线为什么和测试、签名、发布策略、团队协作绑在一起。
- 学会判断什么应该进入流水线，什么不应过早复杂化。

## 前置知识

- 已理解 Gradle、构建变体、测试和发布基础。
- 已接触多人协作和版本管理。

## 正文

### 1. “本地没问题”为什么从来不是足够条件

单个开发者本地跑通功能，只能证明“在这个环境下，这次大概能跑”。它无法证明：

- 依赖是否完整。
- 构建是否可重复。
- 别人的机器是否也能通过。
- 最低质量门槛是否真的被执行了。

一旦项目进入多人协作，本地成功就不再是最终标准。CI 的存在，就是把“最低可合并标准”从个人习惯变成自动化规则。

### 2. 持续集成真正保护的是“主干可用”

很多读者会把 CI 理解成“跑几条命令”。更准确的说法是：CI 保护的是主分支不要被不稳定改动慢慢侵蚀。为此，一条最基础的 CI 流水线通常会在每次提交或合并请求中执行：

- 依赖解析与构建。
- Lint 或关键静态检查。
- 单元测试。
- 必要时的关键 UI 或仪表测试。

这些动作的意义，不在于形式完整，而在于防止问题悄悄进入主干，让整个团队一起承担后果。

### 3. 持续交付关注的是“产物怎样稳定往下游流动”

当代码已经达到可合并标准，接下来要解决的是产物管理问题：

- 谁负责打包。
- 哪个提交生成了哪个候选版本。
- 签名在什么阶段完成。
- 测试版、预发布版、正式版如何流转。

这就是 CD 的核心：让构建产物的流转不再依赖个人电脑、个人记忆和手工步骤，而是进入可追踪、可回放、可审计的流程。

### 4. 流水线真正自动化的，应该是“团队已经认同的规则”

有些团队一上来就把流水线做得很复杂，却没先想清楚哪些规则真的值得自动化。一个更稳妥的原则是：只有那些团队已经认同、而且一旦漏掉会造成明显损失的规则，才值得优先放进流水线。例如：

- 提交前必须能构建通过。
- 关键单元测试必须通过。
- release 包必须使用正确签名策略。
- 发布产物必须能追溯到具体版本和提交。

流水线不是为了展示工具能力，而是为了让“本来就该做的事”稳定执行。

### 5. 更适合 Android 项目的最小 CI 流程

对于大多数 Android 项目，一个健康的起点通常不是“把所有测试都塞进去”，而是这样几步：

1. 检查项目能否稳定构建。
2. 跑最快、最关键的单元测试。
3. 跑 Lint 或最基本的静态检查。
4. 输出调试包或候选发布包。

这个起点的关键在于可依赖。只要流水线足够快、足够稳，团队就更愿意真正依赖它。反过来，如果一开始就加进过多重任务，流水线会很快变得又慢又脆，大家只会想着绕过它。

### 6. 流水线速度和可信度同样重要

一条非常慢的流水线，会逼开发者减少提交频率、拖延合并，甚至寻找跳过办法；一条经常误报的流水线，则会让团队逐渐把“红灯”当成噪音。因此成熟的 CI/CD 不只是“步骤很多”，而是满足三个条件：

- 足够快，能融入日常开发节奏。
- 足够稳，结果值得信任。
- 足够清楚，失败后能迅速定位。

这说明流水线本身也是一项工程产品，需要被持续维护，而不是“一次搭好就永远不用管”。

### 7. 一个更现实的 Android 发布链路

对 Android 项目来说，CD 最有价值的地方通常不是“直接一键上架”，而是把发布链路分层。例如：

- 普通提交：只验证构建与基础测试。
- 候选版本：生成 release 构建，关联版本号与映射文件。
- 测试轨道：推送到内部测试或封闭测试环境。
- 正式版本：在满足验证条件后再进入灰度或正式发布。

这种分层方式比“本地打一个正式包直接上传”更接近真实团队，也更能把风险控制在合适阶段。

### 8. 什么时候不该急着把东西塞进流水线

并不是所有检查都应该第一天就自动化。以下情况通常更适合先手工稳定，再考虑接入：

- 还在频繁变化、尚未稳定的实验性流程。
- 运行时间极长、且价值暂时不清晰的检查。
- 经常误报、但团队还没搞清楚原因的任务。

CI/CD 成熟的标志，不是“自动化越多越好”，而是“自动化的每一步都真正在替团队减少风险”。

### 9. 实践任务

起点条件：

- 已有基本的 Android 构建脚本和至少一部分测试。

步骤：

1. 列出团队每次提交和发版前理论上必须做的检查。
2. 标出其中哪些已经稳定到值得交给流水线自动执行。
3. 设计一条最小 CI 流程，至少覆盖构建和关键测试。
4. 设计一条最小 CD 流程，明确候选版本、产物命名和版本关联方式。
5. 回看流水线是否快到足以被团队日常依赖，是否清楚到失败时能迅速定位。

预期结果：

- 你会把 CI/CD 看成工程纪律自动化，而不是平台配置任务。
- 团队关键规则会更容易被稳定执行。
- 发布流程会更可重复、可追踪。

自检方式：

- 你能解释 CI 和 CD 在目标上的区别。
- 你能判断什么步骤该优先放进流水线。
- 你能说明为什么流水线的速度和可信度同样重要。

调试提示：

- 流水线总是很慢，优先先缩小到最关键的检查。
- 规则全靠文档提醒，说明自动化还没承接真正风险点。
- 发布仍依赖单人本地操作，说明 CD 还没有真正落地。

### 10. 常见误区

- 把 CI/CD 理解成某个平台的使用教程。
- 一开始就塞进过多步骤，导致流水线又慢又脆。
- 只把它当发布工具，不把它当质量门槛。
- 流水线搭完就不维护，结果逐渐失去可信度。

## 小结

CI/CD 真正要解决的，是工程规则是否能被稳定执行，以及构建产物能否被可重复地交付下游。持续集成保护主干可用，持续交付保护产物流转稳定。只要先围绕团队最怕漏掉的规则建立最小流水线，再逐步扩展，它就会从“额外流程”变成整个项目最可靠的一套工程支架。
