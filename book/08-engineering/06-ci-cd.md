# CI/CD

很多团队第一次认真引入 `CI/CD`，并不是因为想追求“更现代的工程流程”，而是因为项目已经被另一种问题拖住了: 本地能跑，换台机器就不行；有的人提交流水线全绿，有的人却漏了测试直接把问题带进主分支；发布依赖某个开发者手工点一遍 Studio，步骤多、出错也难追。到了这个阶段，持续集成和持续交付就不再是锦上添花，而是把“工程规则能否被稳定执行”这件事正式系统化。

这一章不把 CI/CD 当成某个平台教程来讲，而是把它看成工程纪律的自动化。CI 负责回答“这份改动是否达到可以合并的最低质量标准”，CD 负责回答“达到标准后，怎样把产物稳定、安全地往下游传递”。只要从这个角度理解，平台和工具会变化，但核心思路不会变。

## 学习目标

- 理解 CI/CD 真正解决的是工程规则执行和交付稳定性问题。
- 理解持续集成与持续交付各自关注什么。
- 理解流水线为什么和测试、签名、发布策略、团队协作绑在一起。
- 学会判断什么应该进入流水线，什么不应过早复杂化。

## 前置知识

- 已理解 Gradle、构建变体、测试和发布基础。
- 已接触多人协作和版本管理。

## 正文

### 1. “本地没问题”为什么从来不是足够条件

单个开发者本地跑通功能，只能证明“在这个环境下，这次大概能跑”。它无法证明:

- 依赖是否完整。
- 构建是否可重复。
- 别人的机器是否也能通过。
- 测试和质量门槛是否都被执行了。

只要项目进入多人协作，本地成功就不再是最终标准。CI 的存在，就是把“最低可合并标准”从个人习惯变成自动化规则。

### 2. 持续集成真正要做的是“让主干保持可用”

CI 的核心目标不是“跑一堆任务”，而是让主分支尽量保持在可构建、可测试、可继续开发的状态。为此，流水线常常会在每次提交或合并请求中执行:

- 依赖解析和构建。
- 代码格式与静态检查。
- 单元测试。
- 关键 UI 或仪表测试。

这不是为了增加门槛，而是为了防止问题被悄悄带入主干，让后续所有人一起承担成本。

### 3. 持续交付关注的是“产物怎么稳定往下游走”

当代码已经通过基本质量门槛，接下来要解决的是产物管理问题。例如:

- 谁来打包发布版。
- 签名在什么环节完成。
- 测试版、预发布版、正式版怎么流转。
- 哪个产物对应哪次提交。

这就是 CD 的核心: 让构建产物的流转不再依赖人工记忆和个人电脑习惯，而是进入可追踪、可回放、可审计的过程。

### 4. 流水线不是“多做几步”，而是把口头规则写成系统规则

很多团队平时会说:

- 提交前要先跑测试。
- 发版前要先做 lint。
- 发布包一定要用 release 签名。

这些话如果只是写在文档里，最终很容易变成“有时候做，有时候忘”。流水线的价值，就是把这些规则从口头约定变成默认执行。只要规则已经被认同，就值得考虑让机器替人记住它。

### 5. 什么应该最先进入 CI

不是所有事情都要一开始就塞进流水线。更成熟的顺序通常是:

1. 先保证项目能稳定构建。
2. 再把最关键、最快速的测试纳入。
3. 再加入静态检查和格式校验。
4. 最后才考虑更重的 UI 测试和多环境发布流程。

这样做的好处是，你不会一上来就把流水线做得又慢又重，导致团队所有人都想绕过它。

### 6. 流水线速度和可信度同样重要

如果一条流水线运行非常慢，开发者就会本能地降低信任和依赖，甚至想办法跳过它。反过来，如果它虽然快，但经常误报或随机失败，团队也会把它当成噪音。

所以健康的 CI/CD 不只是“功能多”，而是:

- 足够快，能融入日常节奏。
- 足够稳，结果值得信任。
- 足够清晰，失败时知道该修哪里。

这也说明流水线本身也是一项工程产品，需要持续维护，而不是“一次搭好就不管”。

### 7. 发布不是流水线末尾“顺手打个包”

一旦进入 CD，签名、变体、渠道、版本号、灰度发布、回滚策略就会进入同一条链路。此时发布不再是某个人本地点击菜单导出 APK，而是一次正式交付动作。

这意味着流水线不仅要生成产物，还要帮助你回答:

- 这是哪个分支、哪个提交构建出来的。
- 它面向哪个环境和人群。
- 是否已经经过必要测试。
- 出问题时能否快速定位和回滚。

### 8. 一个健康的 Android 流水线最小模型

对很多 Android 项目来说，一个最小但有价值的流水线通常包括:

- 拉取代码和依赖。
- 执行 lint 和关键静态检查。
- 跑单元测试。
- 生成调试包或候选发布包。
- 在需要时生成带签名的发布产物并推到测试轨道。

关键不在于步骤是否豪华，而在于它是否真正承接了团队当前最怕漏掉的规则。

### 9. 实践任务

起点条件:

- 已有基本的 Android 构建和测试脚本。

步骤:

1. 列出团队现在每次提交和发版前“理论上应该做”的检查。
2. 标出其中哪些已经稳定到值得交给流水线自动执行。
3. 设计一条最小 CI 流程，至少覆盖构建和关键测试。
4. 再设计一条最小 CD 流程，明确产物输出和版本关联方式。
5. 检查流水线是否快到足以被团队日常依赖。

预期结果:

- 你会把 CI/CD 看成工程纪律自动化，而不是平台配置任务。
- 团队关键规则会更容易被稳定执行。
- 发布流程会更可重复、可追踪。

自检方式:

- 你能解释 CI 和 CD 在目标上的区别。
- 你能判断什么步骤该优先放进流水线。
- 你能说明为什么流水线的速度和可信度同样重要。

调试提示:

- 流水线总是很慢，优先先缩小到最关键的检查。
- 规则全靠文档提醒，说明自动化还没承接真正风险点。
- 发布仍依赖单人本地操作，说明 CD 还没有真正落地。

### 10. 常见误区

- 把 CI/CD 理解成某个平台的使用教程。
- 一开始就塞进过多步骤，导致流水线又慢又脆。
- 只把它当发布工具，不把它当质量门槛。
- 流水线搭完就不维护，结果逐渐失去可信度。

## 小结

CI/CD 真正要解决的，是工程规则是否能被稳定执行，以及构建产物能否被可重复地交付下游。持续集成保护主干可用，持续交付保护产物流转稳定。只要先围绕团队最怕漏掉的规则建立最小流水线，再逐步扩展，它就会从“额外流程”变成整个项目最可靠的一套工程支架。
