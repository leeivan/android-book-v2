# Android 概述

本章是全书真正的出发点。很多人第一次接触 Android 时，会把它想象成“一个可以做手机界面的开发框架”。这种理解不能说错，但远远不够。Android 更准确的身份，是一个由操作系统、应用模型、硬件差异、权限安全、工具链和分发机制共同构成的平台生态。只有先把这张全景图建立起来，后面的界面、数据、网络和架构章节才不会变成零散的 API 记忆。

从学习顺序上说，本章不追求让你马上写出复杂应用，而是先回答四个更根本的问题：Android 到底是什么；为什么开发者更应该关注 `API Level` 而不是版本宣传名；一个 Android 应用为什么不是“一个入口函数加若干页面”；以及在今天继续学习 Android 时，为什么应该直接把 Kotlin、AndroidX 和 Jetpack 当作默认主线。

## 学习目标

- 建立对 Android 平台、设备生态和应用模型的整体认识。
- 理解版本名、`API Level`、`minSdk`、`targetSdk`、`compileSdk` 之间的区别。
- 理解 Android 四大组件为什么是系统进入应用的正式入口。
- 明确本书采用的现代 Android 学习基线。

## 前置知识

- 具备任意一门编程语言的基础语法能力。
- 知道类、对象、函数、集合等基础概念。
- 不要求事先掌握 Kotlin 或 Android 开发经验。

## 正文

### 1. Android 首先是平台，不只是界面框架

从开发者视角看，Android 至少同时具有三层身份。第一，它是一个面向多种设备形态的操作系统平台。第二，它是一套应用运行、安装、更新和分发的完整生态。第三，它是一条包含 SDK、构建工具、调试工具和设计规范的开发工具链。只要这三层里有一层被忽略，你对 Android 的理解就会变得片面。

如果只从用户角度看，Android 最直观的载体当然是手机和平板。但从平台视角看，Android 早已覆盖折叠屏、平板、电视、车载、穿戴和其他专用设备。设备形态一旦变多，开发就不再只是“把页面做出来”这么简单，而要同时面对尺寸变化、输入方式差异、资源限制和系统约束。你后面会反复遇到的生命周期、权限、后台执行、通知、存储和发布问题，本质上都来自这些平台边界。

这也是 Android 和很多通用前端框架最不一样的地方。Android 当然也有 UI，但它的难点从来不只在 UI。它要求你在系统规则之内组织应用，而不是把应用当成一个由自己完全控制的前台程序。

### 2. 为什么 Android 学习不能只盯着版本名

很多入门者喜欢问“现在最新的 Android 是哪个版本”。这个问题并非没有意义，但对于开发者来说，真正决定工程行为的通常不是宣传名，而是 `API Level` 和平台行为变更。官方 SDK、构建工具和文档也都是围绕 `API Level` 来组织的。

更稳定的开发坐标，永远是“哪个 API Level 能用什么能力、受什么限制”，而不是“某个版本名字听起来像什么”。因为真正影响你写代码的，是系统接口是否存在、某个行为变更是否生效、某类权限是否收紧，以及应用声明自己适配到哪一代平台规则。

### 3. `minSdk`、`targetSdk` 和 `compileSdk` 在回答不同问题

这三个值经常同时出现在工程配置里，所以初学者很容易把它们理解成“同一件事的三个写法”。实际上，它们分别回答完全不同的问题。

`minSdk` 回答的是：应用最低能安装到哪些系统版本上。它决定兼容下界，也决定你在实现功能时要不要考虑更老设备的替代方案。`compileSdk` 回答的是：当前工程编译时可以看到哪些平台 API。它影响的是“编译器认不认识某个接口”，而不是“用户设备一定已经支持这个行为”。`targetSdk` 回答的则是：应用声明自己已经适配到哪一代平台行为要求。系统会根据这个值决定是否对应用启用某些新的行为约束和兼容策略。

只要把这三者混为一谈，后面很多平台问题就会越学越乱。比如，能编译通过不代表老设备一定能跑；能装到设备上不代表系统会按旧规则放过你；把三个值都设成同一个数字，也不代表你的兼容策略就合理。真实项目里，它们通常需要结合产品范围、平台要求和升级节奏分别决定。

### 4. Android 的变化不只是“新增了几个 API”

很多人理解 Android 更新时，第一反应是“又多了一些新方法”。但近年的平台变化，往往更重要的并不是新增多少类，而是系统在权限、后台执行、通知、存储和应用可见性上不断加强边界。也就是说，Android 的变化越来越多地体现在“你不能再默认那样做了”，而不只是“你现在多了一个可选功能”。

这对学习路线有直接影响。如果一本教程只讲“怎么把代码写通”，却不主动区分历史写法和当前主线，读者就很容易学到一些还能跑、但已经不该当默认方案的做法。现代 Android 学习必须带着时间维度，知道什么是平台主线，什么是历史机制，什么只适合读旧项目时识别而不适合新项目直接采用。

### 5. Android 应用为什么不是“一个 main 函数加若干页面”

很多人初学 Android 时，会带着桌面程序或命令行程序的直觉进入：程序从一个入口开始，由开发者持续掌控执行流程。Android 不是这样。在 Android 中，系统才是真正的总调度者。系统决定何时创建 Activity、何时回收进程、何时因配置变化重建页面、何时允许后台执行某类任务。开发者写的代码，更像是在系统提供的组件入口和回调中完成工作，而不是从头到尾独占流程控制权。

这也是为什么 Android 开发里，生命周期和组件模型不是“额外知识”，而是主干知识。没有这层认识，后面看到的页面状态、权限申请、后台工作、通知、导航和存储边界都会显得像零散规则。理解系统和应用之间的这种主从关系，是后续所有章节的基础。

### 6. Android 四大组件，是系统进入应用的正式入口

根据 Android 官方应用基础文档，Android 应用有四类核心组件：Activity、Service、BroadcastReceiver 和 ContentProvider。它们不是为了记忆而存在的名词表，而是系统按需激活应用能力的正式入口。

Activity 通常承载一个对用户可见的交互场景。很多入门者会把它简单理解成“一个页面”，这种理解足够入门，但还不够完整。更准确地说，Activity 是系统与界面场景之间的桥梁，它既承接 UI，也服从系统生命周期管理。

Service 用来承载那些不直接依赖界面显示、但仍需要在应用上下文中执行的工作。它不是线程，也不是“后台永驻”的代名词。现代 Android 对后台执行有严格约束，因此 Service 的正确理解必须建立在平台规则上，而不是停留在旧教程里“开个 Service 就能一直跑”的想象上。

BroadcastReceiver 负责接收系统或应用发出的广播事件。它是一个轻量入口，重点在“接到事件并快速响应或转发”，而不是直接在里面执行冗长任务。ContentProvider 则负责以统一、受控的方式暴露数据访问能力。它的意义不是“帮你存数据库”，而是定义数据如何被系统或其他应用安全访问。现实项目里，读者更常见的情况是使用系统已有的 Provider，而不是一开始就自己写 Provider。

这四类组件共同说明了一件事：Android 应用不是一个始终在内存里自我运行的独立程序，而是一组由系统按需激活、按规则协作的能力入口。

### 7. 为什么 Kotlin、AndroidX 和 Jetpack 已经成为默认学习路线

如果你今天第一次学习 Android，最不划算的做法之一，就是先完整学习一整套基于早期 Java 和旧 API 的教程，再在心里补一句“现代项目不这么写”。这样会让你同时维护两套互相冲突的认知，学习成本非常高。更合理的做法，是直接从当前主线出发。

Android 官方已经把 Android 开发定位为 Kotlin-first。它的含义不是“Java 不能用了”，而是新文档、新示例和新训练材料会优先围绕 Kotlin 展开。Kotlin 的空安全、数据类、更少样板代码，以及与协程和现代状态建模的自然结合，都非常适合移动端这种 UI 与状态密集场景。因此，本书正文以 Kotlin 为主线，Java 只在附录和迁移视角中出现。

AndroidX 则是现代 Android 库体系的实际坐标系。Fragment、RecyclerView、Lifecycle、Navigation、Room、WorkManager 等主线能力，都应放在 AndroidX 语境中理解。继续把旧支持库或早年教程的包名当主线，只会让学习内容和真实项目长期脱节。

Jetpack 进一步解决的是一整组现代工程问题：生命周期与状态如何组织、页面导航如何统一、本地数据如何持久化、后台任务如何合规执行、界面如何在配置变化中保持稳定。学习 Jetpack 的重点，不是记住多少库名，而是理解这些库为什么会出现在现代 Android 工程中。

### 8. 本书采用的技术基线

为了避免章节之间来回切换路线，本书统一采用以下基线：语言主线是 Kotlin；组件体系以 AndroidX 为基础；页面基础是 Activity 与 Fragment；状态管理以 ViewModel 为主线；列表主线采用 RecyclerView；本地结构化存储采用 Room；异步与状态流使用 Coroutines 和 Flow；后台任务主线使用 WorkManager；结果回调采用 Activity Result API。

这里需要提前说明的是：为什么本书同时保留 View/XML 和 Compose。原因很现实。大量现有项目仍然以 View/XML 为主，大量新项目已经采用 Compose，而真实工作中工程师经常要同时阅读这两类代码。因此，本书前半段先用 View/XML 建立平台和传统界面结构认知，再单独引入 Compose，帮助读者获得现代项目阅读能力和迁移能力。

### 9. 一个极简场景，也会暴露 Android 的平台复杂度

假设你要做一个最简单的“今日待办”应用。表面上看，它似乎只需要一个列表界面。但只要稍微接近真实场景，平台问题就会立刻出现：入口界面需要 Activity；列表和详情可能要通过 Fragment 或导航组件组织；本地保存需要 Room 或 DataStore；到时提醒涉及通知；延迟刷新可能涉及 WorkManager；用户分享待办内容时会碰到 Intent。

这个例子说明，Android 学习从第一天起就不是单一页面问题，而是“一个应用如何在平台能力之间协作”的问题。后续各章，其实就是在逐步拆解这个简单示例背后的系统性复杂度。

### 10. 学习 Android 时最容易走偏的三条路线

第一条偏路，是只记 API，不理解平台边界。这样学出来的人往往“会写很多例子”，但一到权限、生命周期、后台执行、通知或兼容性问题时就立刻失去判断力。第二条偏路，是先系统学习一整套旧写法，再试图补现代迁移。这样当然不是完全不能走，但成本很高，读者会把大量精力花在历史包袱上，而不是建立当前主线。第三条偏路，是只会搭界面，不会组织应用。Android 开发的真正难点，从来不只是把控件摆出来，而是能否在平台规则下把页面、状态、数据、权限和后台行为组织成稳定工程。

起点条件：

- 已打开本书目录。

步骤：

1. 通读 `SUMMARY.md`，把章节粗分成“平台基础”“界面与交互”“数据与网络”“系统组件与工程实践”四类。
2. 在纸上写出你当前理解的四大组件，并尝试为每一个组件各举一个现实应用场景。

预期结果：

- 你能说出 Android 不只是 UI 框架，而是平台。

自检方式：

### 12. 常见误区

- 认为 Android 开发就是“会写 Activity 就够了”。
- 先完整学一套旧 Java 教程，再考虑现代化。
- 把版本宣传名看得比 `API Level` 更重要。
- 把 Android 学习理解成“会摆控件就会做应用”。

## 小结

本章真正完成的，不是知识点罗列，而是视角校准。你应该开始把 Android 看成一个平台：系统拥有调度权，应用通过正式组件入口参与协作，版本坐标要靠 `API Level` 和平台行为来理解，而现代学习主线应直接建立在 Kotlin、AndroidX 和 Jetpack 之上。

从下一章开始，我们会把视角从“平台全景”收窄到“如何搭一套真正可用的开发环境”。只有平台认知先站稳，后面的工具、项目、页面和数据章节才不会变成互不相干的技巧清单。
