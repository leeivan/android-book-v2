# Android 概述

本章是全书真正的出发点。很多人第一次接触 Android 时，会把它想象成“一个可以做手机界面的开发框架”。这种理解不能说错，但远远不够。Android 更准确的身份，是一个由操作系统、应用模型、硬件差异、权限安全、工具链和分发机制共同构成的平台生态。只有先把这张全景图建立起来，后面的界面、数据、网络和架构章节才不会变成零散的 API 记忆。

从学习顺序上说，本章不追求让你马上写出复杂应用，而是先回答四个更根本的问题：Android 到底是什么；为什么开发者更应该关注 `API Level` 而不是版本宣传名；一个 Android 应用为什么不是“一个入口函数加若干页面”；以及在 2026 年继续学习 Android 时，为什么必须直接把 Kotlin、AndroidX 和 Jetpack 当作默认主线。

## 学习目标

- 建立对 Android 平台、设备生态和应用模型的整体认识。
- 理解版本名、`API Level`、`minSdk`、`targetSdk`、`compileSdk` 之间的区别。
- 理解 Android 四大组件为什么是系统进入应用的正式入口。
- 明确本书采用的现代 Android 学习基线。

## 前置知识

- 具备任意一门编程语言的基础语法能力。
- 知道类、对象、函数、集合等基础概念。
- 不要求事先掌握 Kotlin 或 Android 开发经验。

## 正文

### 1. Android 到底是什么

从开发者视角看，Android 至少同时具有三层身份：

1. 它是一个面向多种终端形态的操作系统平台。
2. 它是一套应用运行、安装、更新和分发的完整生态。
3. 它是一条包含 SDK、构建工具、调试工具和设计规范的开发工具链。

如果只从用户角度看，Android 最直观的载体当然是手机和平板。但从平台视角看，Android 早已覆盖折叠屏、电视、车载、穿戴和更多专用设备。设备形态一旦变多，开发就不再只是“把页面做出来”这么简单，而要同时面对尺寸变化、输入方式差异、资源限制和系统约束。

这就是 Android 的第一个特点：它不是一个纯界面开发平台，而是一个“平台边界很多”的应用平台。你后面会反复遇到的生命周期、权限、后台执行、通知、存储和发布问题，本质上都来自这些平台边界。

### 2. 为什么 Android 学习不能只盯着版本名

很多入门者喜欢问“现在最新的 Android 是哪个版本”。这个问题不是没有意义，但对于开发者来说，真正决定工程行为的通常不是宣传名，而是 `API Level` 和平台行为变更。

#### 2.1 版本名主要用于交流，`API Level` 才是开发坐标

Android 的版本宣传名、市场名称和 SDK 版本并不是一回事。官方 SDK 平台、文档和构建工具里，开发者真正面对的是 `API Level`。它直接决定：

- 某个 API 能否被调用。
- 某个系统行为变更是否生效。
- 某个兼容分支是否必须处理。

因此，工程上更稳定的表达方式永远是 “API 多少”，而不是“哪个版本代号”。

#### 2.2 `minSdk`、`targetSdk` 和 `compileSdk` 解决的是不同问题

这三个值经常同时出现在项目配置里，但它们绝不是一个意思：

- `minSdk`：应用允许安装的最低系统版本。
- `targetSdk`：应用声明自己已经适配到的目标平台行为级别。
- `compileSdk`：编译时使用的 SDK 版本，决定你能否访问某些新 API。

它们分别回答的问题是：

- 最低要兼容到哪里。
- 系统应该按哪一代规则要求你。
- 代码编译时能看到哪些平台接口。

很多初学者会误以为三个值“尽量保持一样”最安全，实际上这是一种偷懒的理解。真实项目里，它们往往需要结合平台要求、兼容策略和产品目标分别决定。

#### 2.3 Android 的变化不只是“新增 API”

Android 每年最重要的变化，往往不是多几个类、多几个方法，而是平台在下面这些方面不断加强约束：

- 隐私和权限边界。
- 后台任务与前台服务限制。
- 通知行为与用户控制。
- 存储访问模型。
- 组件导出和应用可见性规则。

这意味着 Android 学习天然带有“时间维度”。历史上可行的做法，不一定适合作为今天的默认做法。一本现代 Android 教程如果不主动区分“旧机制”和“当前主线”，读者很容易学到能运行但不该优先采用的实现。

### 3. Android 应用为什么不是普通桌面程序的翻版

很多人初学 Android 时，会带着桌面程序或命令行程序的直觉进入：程序从一个入口开始，由开发者持续掌控执行流程。Android 不是这样。

在 Android 中，系统才是真正的总调度者。系统决定何时创建 Activity、何时回收进程、何时因配置变化重建页面、何时允许后台执行某类任务。开发者写的代码，更像是在系统提供的组件入口和回调中完成工作，而不是从头到尾独占流程控制权。

这也是为什么 Android 开发里，生命周期和组件模型不是“额外知识”，而是主干知识。没有这层认识，你后面看到的页面状态、权限申请、后台工作、通知、导航和存储边界都会显得像零散规则。

### 4. Android 四大组件：系统进入应用的四个正式入口

根据 Android 官方应用基础文档，Android 应用有四类核心组件。它们不是“为了考试而背的名词”，而是系统与应用协作的正式入口。

#### 4.1 Activity：界面交互入口

Activity 通常承载一个对用户可见的交互场景。很多入门者会把它简单理解成“一个页面”，这种理解足够入门，但还不够完整。更准确地说，Activity 是系统与一个界面场景之间的桥梁。它既要承接 UI，也要服从系统生命周期管理。

#### 4.2 Service：持续任务入口

Service 用来承载那些不直接依赖界面显示、但仍需要在应用上下文中执行的工作。它不是线程，也不是“后台永驻”的代名词。现代 Android 对后台执行有严格约束，因此 Service 的正确理解必须建立在平台规则上，而不是旧教程里那种“开个 Service 就能一直跑”的想象上。

#### 4.3 BroadcastReceiver：事件接收入口

BroadcastReceiver 用来接收系统或应用发出的广播事件，例如状态变化、计划触发或某个系统行为完成。它是一个轻量入口，重点在“接到事件并快速分发”，而不是在里面直接做复杂长流程。

#### 4.4 ContentProvider：数据共享入口

ContentProvider 负责以统一、受控的方式暴露数据访问能力。它存在的意义不是“帮你存数据库”，而是“定义别人如何安全地访问你的数据”。绝大多数业务应用更常见的场景，是使用系统已有的 Provider，而不是自己实现一个新的 Provider。

这四类组件共同说明了一件事：Android 应用不是一个始终在内存里自我运行的独立程序，而是一组由系统按需激活、按规则协作的能力入口。

### 5. 为什么 Kotlin、AndroidX 和 Jetpack 已经成为默认学习路线

如果你今天第一次学习 Android，最不划算的做法之一，就是先系统学习一整套基于早期 Java 和旧 API 的教程，再在心里补一句“现代项目不这么写”。这种路线会让你同时维护两套互相冲突的认知，学习成本非常高。

更合理的做法，是直接从当前主线出发。

#### 5.1 Kotlin-first 不等于 Java 无效，但意味着学习主线已经变化

Android 官方已经明确把 Android 开发定位为 Kotlin-first。它的含义不是“Java 不能用了”，而是：

- 新文档、新示例和新训练材料会优先围绕 Kotlin 展开。
- 很多现代能力与 Kotlin 的结合更自然，例如协程、扩展函数和更简洁的状态建模。
- Kotlin 的空安全、数据类和更少样板代码，对移动端这种 UI 和状态密集场景非常有帮助。

因此，本书正文以 Kotlin 为主线。Java 会在附录和迁移视角中出现，但不再作为新项目的默认入口。

#### 5.2 AndroidX 是现代 Android 的坐标系统

Fragment、RecyclerView、Lifecycle、Navigation、Room、WorkManager、Activity Result API 等主线能力，今天都应放在 AndroidX 体系里理解。继续把旧支持库或早年包名当主线，只会让真实项目和学习内容长期脱节。

#### 5.3 Jetpack 解决的是一整组现代工程问题

Jetpack 不是“若干方便库”的集合，它解决的是一整组现代 Android 工程问题：

- 生命周期与状态如何组织。
- 页面导航如何统一。
- 本地数据如何持久化。
- 后台任务如何合规执行。
- 界面如何在配置变化和状态切换中保持稳定。

因此，学习 Jetpack 的重点从来不是记住多少库名，而是理解这些库为什么会出现在现代 Android 架构中。

### 6. 本书采用的技术基线

为了避免章节之间来回切换路线，本书统一采用以下基线：

- 语言主线：Kotlin。
- 组件体系：AndroidX。
- 页面基础：Activity + Fragment。
- 状态管理：ViewModel。
- 列表主线：RecyclerView。
- 本地结构化存储：Room。
- 异步与状态流：Coroutines、Flow。
- 后台任务：WorkManager。
- 结果回调：Activity Result API。

这里还有一个需要提前说明的问题：为什么本书同时保留 View/XML 和 Compose？

原因很现实：

- 大量现有项目仍然以 View/XML 为主。
- 许多新项目已经采用 Compose。
- Android 工程师在真实工作中经常要同时阅读这两类代码。

因此，本书前半段先用 View/XML 建立平台和传统界面结构认知，再单独引入 Compose，帮助读者获得现代项目阅读能力和迁移能力。

### 7. 学习 Android 时最容易走偏的三条路线

#### 7.1 只记 API，不理解平台边界

这样学出来的人往往“会写很多例子”，但一到权限、生命周期、后台执行、通知或兼容性问题时就会立刻失去判断力。

#### 7.2 先学整套旧写法，再试图补现代迁移

这条路当然不是完全不能走，但成本很高。你会把大量精力花在历史包袱上，而不是建立当前推荐做法的主线。

#### 7.3 只会搭界面，不会组织应用

Android 开发的真正难点，从来不只是把控件摆出来，而是能否在平台规则下，把页面、状态、数据、权限和后台行为组织成稳定工程。

### 8. 一个极简例子：为什么“待办列表”也不是简单页面题

假设你要做一个最简单的“今日待办”应用。表面上看，它似乎只需要一个列表界面。但只要稍微接近真实场景，你立刻就会碰到平台问题：

- 入口界面需要 Activity。
- 列表和详情可能要通过 Fragment 或导航组件组织。
- 本地保存需要 Room 或 DataStore。
- 到时提醒涉及通知。
- 延迟同步或到期刷新可能涉及 WorkManager。
- 用户分享待办内容时会碰到 Intent。

这个例子说明，Android 学习从第一天起就不是单一页面问题，而是“一个应用如何在平台能力之间协作”的问题。后续各章，其实就是在逐步拆解这个简单示例背后的系统性复杂度。

### 9. 学习这本书时应保持怎样的视角

从这一章开始，你最好逐步养成一种稳定的学习视角：

- 先问“这个能力解决什么平台问题”。
- 再问“当前官方更推荐怎样做”。
- 最后才问“具体 API 怎么写”。

只要顺序反过来，后面就很容易变成“把大量技巧塞进脑子”，而不是建立完整开发判断力。

1. 阅读本书目录，标出哪些章节处理界面问题，哪些处理数据问题，哪些处理系统和工程问题。
2. 在纸上写出你理解的四大组件，并尝试为每一个组件各举一个现实应用场景。
3. 记录你目前还不理解的关键词，例如 `API Level`、`targetSdk`、`ViewModel`、`Room`、`WorkManager`。后续学到对应章节时，再回来补上自己的理解。

### 11. 常见误区

#### 11.1 “Android 开发就是会写 Activity”

不对。Activity 只是系统进入应用的一类入口，真正的 Android 应用远不止 Activity。

#### 11.2 “先学 Java 版旧教程，再说现代化”

不建议。更高效的路径，是直接以现代做法为主线，再补历史机制和迁移知识。

#### 11.3 “版本名比 `API Level` 更重要”

对开发者来说通常相反。版本名更便于交流，`API Level` 更便于工程决策。

#### 11.4 “只要会用控件就会做 Android 应用”

不对。真正困难的部分通常在状态、生命周期、权限、后台执行和数据组织上。

## 小结

本章完成了三件事：第一，建立了 Android 作为平台而非单纯界面框架的整体认识；第二，解释了为什么 `API Level`、组件模型和平台行为变更比版本宣传名更值得开发者关注；第三，明确了本书的技术基线是 Kotlin、AndroidX 和现代 Jetpack 实践。

从下一章开始，我们会把视角从“平台全景”收窄到“怎样搭一套真正可用的开发环境”。当平台认知先站稳以后，后面的工具、项目、页面和数据章节都会更容易落地。
