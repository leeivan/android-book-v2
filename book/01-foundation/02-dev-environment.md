# 开发环境搭建

Android 学习中，环境问题往往不是“入门小事”，而是决定后续是否能稳定推进的前提。如果环境不可靠，你会把大量精力浪费在 IDE 起不来、Gradle 同步失败、模拟器启动异常、真机连不上、日志看不懂这些本不该成为学习主问题的地方。本章的目标，就是先把开发链路搭稳，让后续每一章的实践都有一个可靠起点。

从出版写作的角度看，环境章节最容易被写成一串会过时的截图说明。本章会尽量避免这种短寿命写法，而是围绕“工具各自负责什么、如何判断环境是否真的可用、常见问题该从哪一层开始排查”来组织内容。这样即使 Android Studio 版本继续更新，你也能保留稳定的理解框架。

## 学习目标

- 理解 Android Studio、SDK、ADB、模拟器和构建工具各自的角色。
- 完成一套可工作的 Android 开发环境。
- 建立判断环境问题所属层级的基本方法。
- 知道最常见的环境故障通常出在哪里。

## 前置知识

- 已阅读“Android 概述”。
- 知道 IDE、编译器和 SDK 是什么。

## 正文

### 1. 一个可用的 Android 开发环境包含哪些部分

很多初学者会把“安装 Android Studio”理解成环境搭建完成，但这只是第一步。一个真正可用的 Android 开发环境，通常至少由下面五层组成：

- Android Studio：日常开发、调试、布局预览、Logcat、Profiler 和设备管理的主要工作台。
- Android SDK：平台 API、构建工具、平台工具和系统镜像的总集合。
- Android Emulator：本地虚拟 Android 设备运行环境。
- ADB：连接 IDE、真机和模拟器的桥梁。
- Gradle 与 Android Gradle Plugin：负责依赖管理、构建任务和打包流程。

当你后面遇到问题时，先判断故障发生在哪一层，通常比直接重装 IDE 更有效。

### 2. 为什么 Android Studio 仍然是最合适的默认入口

理论上，Android 开发工具链并不完全依赖 Android Studio。你也可以用命令行工具、其他编辑器和自定义脚本组合开发流程。但在教学和大多数实际项目中，Android Studio 仍然是最合适的默认入口，原因主要有三点：

- 它把代码编辑、SDK 管理、Logcat、设备管理、Profiler、Layout Inspector 和构建流程整合在一起。
- 它与官方文档、示例项目和训练材料的工作流最一致。
- 它能显著降低初学者同时理解多套工具的认知负担。

这也是为什么本书后续所有实践都默认以 Android Studio 为主，而不是把每个步骤拆成离散命令行操作。

### 3. JDK、SDK、Platform Tools 和 Build Tools 分别负责什么

这些名字经常同时出现，但职责并不相同。

#### 3.1 JDK

Android 项目虽然主要写 Kotlin，但底层构建工具链仍然依赖 Java 运行环境。JDK 负责提供编译和运行构建工具所需的基础环境。

#### 3.2 SDK Platform

SDK Platform 对应特定 `API Level` 的平台接口和相关支持资源。它决定你在编译时能访问哪些系统 API。

#### 3.3 Platform Tools

Platform Tools 里最常见的成员是 `adb`。当你连接真机、列出设备、安装应用、查看进程或转发端口时，本质上都在和这一层打交道。

#### 3.4 Build Tools

Build Tools 参与资源处理、字节码转换、打包和生成最终产物。它更贴近“构建过程”，而不是“设备交互过程”。

把这些层次分清楚之后，很多环境错误就不会都被模糊地归为“Android Studio 报错”。

### 4. 一套稳定环境的最低验证标准

环境是否可用，不应靠“看起来已经装好了”来判断，而应靠一条完整链路是否跑通。至少要验证下面五件事：

1. 能创建并打开 Android 项目。
2. 能同步依赖并完成一次构建。
3. 能启动一个模拟器。
4. 能把应用安装到模拟器或真机。
5. 能看到运行日志并定位到自己的应用进程。

只要这五步中任何一步不稳定，你的环境就还不能算真正可用。

### 5. 模拟器和真机各自有什么价值

初学者常问“到底该用模拟器还是真机”。正确答案通常不是二选一，而是两者各有职责。

#### 5.1 模拟器的优势

- 可控：可以快速切换屏幕尺寸、API 级别、网络状态和地理位置。
- 易复现：方便复现某个固定系统版本下的问题。
- 成本低：不需要准备大量实体设备。

#### 5.2 真机的优势

- 更接近真实性能、输入体验和厂商差异。
- 后台限制、通知行为和权限交互更符合真实用户环境。
- 能更早暴露连接、安装和系统权限相关问题。

更合理的学习顺序通常是：

1. 先用模拟器跑通最小项目。
2. 再用一台稳定真机确认真实运行链路。
3. 后续按章节主题决定优先在哪类设备上验证。

### 6. ADB 为什么值得入门时就认识

很多人把 ADB 视为“高级工具”，其实它是 Android 开发中最值得尽早认识的基础能力之一。原因很简单：无论你用的是 Android Studio 图形界面，还是终端命令，本质上很多设备交互最终都会落到 ADB 这一层。

一个最常用的命令是：

```bash
adb devices
```

它可以帮你确认当前能被识别的设备列表。如果这里就已经异常，后面“为什么运行按钮点了没反应”通常也就不难理解了。

### 7. 环境搭建时最常见的问题

#### 7.1 Gradle 同步失败

常见原因包括：

- 网络或代理配置异常。
- JDK、Gradle 和 AGP 版本不匹配。
- 某些 SDK 组件缺失。

遇到这类问题时，先判断它属于“依赖下载失败”“构建脚本错误”还是“本地组件缺失”，通常比反复点击同步更有用。

#### 7.2 模拟器无法启动或启动过慢

常见原因包括：

- 机器未开启硬件虚拟化。
- 分配给模拟器的资源过高或过低。
- 磁盘空间、内存或图形加速能力不足。

#### 7.3 真机无法识别

常见原因包括：

- 未开启开发者选项和 USB 调试。
- 数据线只支持充电不支持数据传输。
- Windows 驱动或系统授权未处理。
- ADB 授权弹窗未确认。

### 8. 环境稳定后应形成哪些习惯

好的开发环境不只是“装好了”，还包括好的使用习惯：

- 先看错误属于同步、构建、安装还是运行阶段。
- 尽量避免在环境刚稳定时一次性升级过多工具。
- 保留至少一台稳定真机或一个长期可用模拟器。
- 学会看 Logcat，而不是只盯弹窗提示。

这类习惯会显著降低你后面每一章的排障成本。

### 9. 最小示例：用命令确认设备和连接链路

即使你主要通过 Android Studio 开发，也可以用最小命令确认底层链路：

```bash
adb devices
```

如果输出里能看到设备序列号和状态，说明 ADB 到设备这一层已经通了。这个例子之所以值得在环境章节里出现，是因为它提醒你：Android Studio 的“运行”按钮背后，其实是更底层的设备与工具链在协同工作。

### 10. 最小实践任务

1. 打开 Android Studio，并确认 SDK Manager 可以正常工作。
2. 创建一个模拟器并启动到桌面。
3. 使用 Android Studio 设备管理器或 `adb devices` 确认至少存在一个可用设备。
4. 准备一台真机，完成开发者选项和 USB 调试配置。
5. 记录你的环境问题清单，区分哪些属于 IDE、SDK、设备或构建层。

### 11. 常见误区

- 只安装 IDE，不检查 SDK 组件是否完整。
- 只依赖模拟器，不做任何真机验证。
- 一出错就重装 IDE，而不是先定位问题发生在哪一层。
- 看到 Gradle 报错就认为“网络不好”，不去具体判断失败阶段。

## 小结

本章解决的是“工具链是否真正可用”这个最现实的问题。后续所有实践都依赖这套环境，因此现在多花一点时间把创建项目、同步依赖、启动设备、运行应用和查看日志这条链路跑稳，能为后面节省大量无效排障时间。
