# 开发环境搭建

在 Android 学习中，环境问题往往不是“入门小事”，而是后续一切实践的前提。如果开发环境不稳定，你会把大量时间浪费在模拟器起不来、Gradle 同步失败、设备连不上、日志看不懂这些本不该成为学习主问题的地方。本章的目的，就是尽快建立一套可靠、可复用的开发环境。

## 学习目标

- 理解 Android Studio、SDK、ADB、模拟器和真机调试的基本角色。
- 完成一套可工作的 Android 开发环境。
- 知道最常见的环境问题出在哪里，以及如何初步排查。

## 前置知识

- 已阅读“Android 概述”。
- 知道 IDE、编译器和 SDK 是什么。

## 正文

### 1. Android 开发环境由哪些部分组成

一个可用的 Android 开发环境通常至少包含下面几层：

- Android Studio：日常开发和调试的主要工作台。
- Android SDK：提供平台 API、构建工具和设备工具。
- Emulator：在本地模拟 Android 设备运行环境。
- ADB：负责连接真机、模拟器和开发工具的桥梁。
- Gradle 与 AGP：负责构建、依赖和打包。

很多初学者会把“装好 Android Studio”理解成环境搭建完成，但实际上它只解决了入口问题。真正的环境是否可用，要看你能否完成下面这条完整链路：创建工程、同步依赖、启动模拟器、运行应用、查看日志、连接真机。

### 2. 为什么 Android Studio 是当前默认入口

理论上 Android 开发工具链并不完全依赖 Android Studio，但在教学和实际开发中，它仍然是最合适的默认入口，原因主要有三点：

- 它把代码编辑、SDK 管理、设备管理、Logcat、Profiler 和布局工具整合在一起。
- 它与官方文档和示例的工作流最一致。
- 它降低了初学者同时理解多套工具的负担。

这也是为什么本书后续所有工程演示都默认以 Android Studio 为主，而不是拆成若干零散命令行工具分别讲。

### 3. SDK、Platform Tools 和 Build Tools 分别做什么

这几个名字很像，但职责不同：

- SDK Platform：提供特定 Android API Level 的平台接口和系统镜像支持。
- Platform Tools：主要包含 `adb` 等与设备交互相关的工具。
- Build Tools：参与资源处理、打包和构建产物生成。

理解这层区别很重要。很多环境错误表面上看是“Android Studio 报错”，实际上是某个 SDK 组件缺失或版本不匹配。

### 4. 模拟器和真机各自有什么价值

模拟器的优势在于可控、方便、易复现。你可以快速切换屏幕尺寸、系统版本、网络状态和虚拟位置，也可以在没有大量实体设备的情况下做初步验证。

真机的优势在于更接近真实使用场景。特别是在性能、输入体验、厂商差异、通知行为和后台限制方面，真机经常会暴露模拟器不明显的问题。

因此，正确的学习顺序通常不是“模拟器和真机二选一”，而是：

1. 先用模拟器跑通最小项目。
2. 再用真机确认真实运行链路。
3. 在后续章节里根据主题选择更适合的验证方式。

### 5. 最常见的环境问题

#### 5.1 模拟器启动慢或无法启动

常见原因包括：

- 机器未开启硬件虚拟化。
- 模拟器镜像过重。
- 内存或磁盘资源不足。

#### 5.2 Gradle 同步失败

常见原因包括：

- 网络或代理配置问题。
- JDK、Gradle、AGP 版本不匹配。
- SDK 组件缺失。

#### 5.3 真机无法识别

常见原因包括：

- USB 调试未开启。
- 驱动问题。
- ADB 授权未确认。
- 数据线只支持充电不支持数据传输。

### 6. 环境搭建时应形成什么习惯

比“装好工具”更重要的是形成稳定习惯：

- 遇到错误先看报错主体是同步、构建、安装还是运行时。
- 先判断问题属于 IDE、SDK、设备连接还是工程配置。
- 用 Logcat 和设备管理器，而不是只盯着弹窗。
- 环境一旦稳定，就尽量避免无意义地同时升级多项工具。

### 7. 最小实践任务

1. 打开 Android Studio 并确认 SDK Manager 可正常工作。
2. 创建一个模拟器，并成功启动到系统桌面。
3. 使用 `adb devices` 或 Android Studio 设备管理器确认至少一个可用设备。
4. 为后续章节准备一台真机或一个稳定模拟器作为默认测试设备。

### 8. 常见误区

- 只装 IDE，不检查 SDK 组件是否齐全。
- 只依赖模拟器，不做真机验证。
- 一出错就重装 IDE，而不是先定位问题层次。

## 小结

本章解决的是“工具链是否可用”这个最实际的问题。后续章节会不断依赖这套环境，因此现在多花一点时间把基础链路跑稳，能省下后面大量无效排错成本。下一章开始，我们会在这套环境上真正创建并运行第一个 Android 应用。
