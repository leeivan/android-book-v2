# 第一个 Android 应用

环境搭好之后，最重要的不是立刻做复杂功能，而是用一个足够小、足够完整的应用把整个开发闭环走一遍。所谓“第一个 Android 应用”，真正要完成的不是界面本身，而是这条闭环：创建工程、理解最基础的项目骨架、运行到设备、修改界面、再次安装并观察变化。只要这条链路真正跑通，后面再学生命周期、布局和数据，都会更有抓手。

## 学习目标

- 创建并运行一个最小 Android 应用。
- 理解最小应用由哪些核心文件构成。
- 完成一次从修改代码到在设备上看到效果的完整迭代。
- 建立“应用不是孤立代码，而是资源、清单和构建共同作用”的第一层认知。

## 前置知识

- 已完成开发环境搭建。
- 已了解 Activity 作为界面入口的基本概念。

## 正文

### 1. 为什么第一应用一定要足够小

入门时最常见的错误之一，就是刚创建工程就急着塞进复杂需求。结果是：一旦运行失败，你很难判断问题来自环境、项目配置、页面代码还是业务逻辑。

因此，第一个应用的目标应该非常克制：

- 只保留一个最小界面入口。
- 先不引入复杂导航和网络。
- 把重点放在“工程如何运行”而不是“功能多不多”。

这样做的意义不是偷懒，而是给后续学习建立一个稳定的参考基线。

### 2. 创建工程时你到底做了什么

当你通过 Android Studio 创建一个新项目时，IDE 实际上帮你完成了几件事：

- 创建了应用模块。
- 生成了最小 Activity 和布局文件。
- 生成了应用清单。
- 准备了资源目录和构建脚本。

也就是说，创建工程不是“生成一堆神秘文件”，而是搭起了一个 Android 应用的最小骨架。你现在不必立刻理解每个文件的所有细节，但应知道：一个 Android 应用不是只靠 `main` 函数运行起来的，而是由多个入口文件和资源共同组装。

### 3. 一个最小 Android 应用至少包含什么

一个最小应用通常至少会包含下面四类核心元素：

- 入口 Activity：负责承接第一个界面场景。
- 布局资源：定义最小页面结构。
- 应用清单：声明应用与组件信息。
- 构建配置：告诉工具链如何编译和打包这个应用。

你可以把它理解成：

- Activity 负责“做什么界面入口”。
- XML 布局负责“界面长什么样”。
- Manifest 负责“系统如何认识这个应用”。
- Gradle 负责“工具如何把它组装成可安装产物”。

### 4. 从模板工程到真正运行

第一次点击运行按钮时，Android Studio 背后会完成一整条链路：

1. 检查项目配置和依赖。
2. 执行构建任务。
3. 生成 APK 或 App Bundle 的本地可安装产物。
4. 连接目标设备。
5. 安装并启动应用入口。

这条链路看似“只点了一个按钮”，但它正是 Android 开发工作流的缩影。以后无论项目变得多复杂，这条主链路本质上都不会消失。

### 5. 第一次修改，应该改什么最有价值

第一次修改应用时，最有价值的通常不是引入新页面，而是做三件最小但关键的变更：

- 改一个文本内容。
- 改一个控件颜色或间距。
- 在代码里加一段简单交互或日志。

这样做能帮助你快速理解三类改动的来源不同：

- 文本可能来自资源。
- 样式和间距可能来自布局或主题。
- 行为变化来自 Kotlin 代码。

这比一开始就堆更多控件更能帮助你认识工程结构。

### 6. 为什么“能看到变化”是重要里程碑

对初学者来说，第一次在设备上看到自己改动生效，是非常关键的里程碑。因为它说明：

- 工程是可构建的。
- 设备连接是正常的。
- 代码和资源修改能进入最终产物。
- 你已经拥有后续所有实践的基本操作能力。

之后每一章学到的新能力，都会在这个闭环上叠加，而不是重新开始。

### 7. 最小示例：修改欢迎文本

一个最典型的入门改动，是把模板生成的欢迎文本改成你自己的内容。例如把初始文字替换为“你好，Android”。

这个改动看起来微不足道，但它实际上验证了三件事：

- 你找到了界面文本的来源。
- 你完成了一次修改、构建和重装。
- 你开始理解“界面并不是全靠代码硬写出来”的 Android 工作方式。

### 8. 为什么要在这一章就认识 Logcat

很多教程会把日志放到更后面才讲，但在第一个应用阶段就认识 Logcat 其实很重要。原因是：只要你的应用出现异常、崩溃或运行结果与预期不符，日志就是最直接的反馈通道。

即使你现在还不会做复杂调试，也应该先形成习惯：运行应用时，知道去哪里看运行反馈，而不是只看屏幕有没有变化。

### 9. 第一个应用之后最该建立的能力

学完这一章后，你不需要马上会写复杂业务，但至少要具备以下能力：

- 知道如何新建、运行和修改一个应用。
- 知道 Activity、布局、Manifest 和 Gradle 在最小工程里的基本分工。
- 知道运行失败时应先看构建信息还是运行日志。

这三点会成为你进入下一章项目结构分析的基础。

### 10. 最小实践任务

1. 创建一个最小 Android 工程并成功运行到设备。
2. 修改首页显示文本，并重新安装确认变化生效。
3. 在入口 Activity 中添加一条简单日志，运行后在 Logcat 中找到它。
4. 用自己的话解释 Activity、布局文件、Manifest 和构建脚本各自负责什么。

### 11. 常见误区

- 一开始就在模板工程里堆复杂功能，导致问题来源无法判断。
- 只知道点运行按钮，不知道背后经历了构建、安装和启动过程。
- 改了界面却不知道文本、样式和行为分别来自哪里。
- 应用运行失败时不看日志，只反复重新运行。

## 小结

第一个 Android 应用的真正价值，不是做出了一个简单页面，而是让你完成了一次完整开发闭环：创建工程、理解最小结构、运行到设备、做出修改并看到结果。接下来我们会顺着这条闭环，进一步拆开项目结构，理解这些文件为什么会这样组织。
