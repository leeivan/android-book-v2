# 第一个 Android App

如果前两章解决的是“这是什么平台”和“工具是否就绪”，那么这一章解决的就是“第一次真正把应用跑起来”。这一步看似简单，实际上非常关键，因为它会把很多抽象概念串到一起：工程创建、布局文件、资源、Activity、构建、安装和运行。

## 学习目标

- 创建并运行一个最小 Android 工程。
- 理解一个空白工程里最重要的几个文件各自负责什么。
- 完成一次界面修改、重新构建和重新运行的闭环。

## 前置知识

- 已完成开发环境搭建。
- 知道 Activity 和布局的基本概念，但还不需要深入理解。

## 正文

### 1. 为什么第一个应用很重要

第一个应用的意义，不在于它能做多少事，而在于它能让你第一次看到 Android 工程的完整工作流。很多后续章节讲的内容，其实都藏在这个最小项目里：

- 入口 Activity。
- 布局文件。
- 资源引用。
- 构建与安装。
- 运行日志。

如果这一章只被当成“跟着点几下向导”，那么后面很多内容会继续显得零散。正确的做法是借这个最小工程，把最关键的概念先和真实文件对应起来。

### 2. 创建工程时应注意什么

创建项目时，最重要的不是模板多漂亮，而是参数是否符合你后续的学习主线。建议至少注意以下几点：

- 使用 Kotlin 作为默认语言。
- 选择适合当前教学路径的基础模板。
- 记录 `minSdk` 的取值，不要把它当成随手一填的数字。

对于本书的后续学习，最合适的起点通常是一个足够简单、但仍保留 View/XML 工程结构的模板。这样既便于理解 Activity 与布局文件的关系，也为后续 Jetpack 和 Compose 章节保留迁移空间。

### 3. 一个最小工程里最重要的文件

在第一个应用里，读者应至少能认出以下几类文件：

- Activity 源文件：负责页面入口和基础逻辑。
- 布局文件：负责页面结构。
- `AndroidManifest.xml`：负责组件声明和应用元信息。
- `strings.xml`：负责文本资源。
- Gradle 脚本：负责依赖与构建配置。

这时不要求把所有目录都搞懂，但至少要形成一个稳定印象：Android 工程不是“代码文件 + 若干杂项”，而是代码、资源、清单和构建脚本协同组成的整体。

### 4. 第一次运行时到底发生了什么

当你点击运行按钮时，IDE 表面上只是“帮你启动应用”，但底层其实发生了完整链路：

1. 构建系统编译源码并处理资源。
2. 生成安装产物。
3. 把产物安装到模拟器或真机。
4. 系统根据清单找到入口 Activity。
5. Activity 被创建并加载布局。

理解这条链路很重要，因为以后遇到问题时，你才能知道错误发生在“构建之前”“安装阶段”“启动阶段”还是“运行时逻辑”。

### 5. 第一次修改要看什么

第一次修改项目时，建议只做两类最小改动：

- 改一个界面文案。
- 改一个按钮或布局属性。

这样做的目的，是让你第一次看到资源修改、布局更新和应用重运行之间的关系。不要急着一开始就加复杂逻辑，因为此时你最需要建立的是“文件改动会如何反映到应用”这条认知链路。

### 6. 这个最小项目已经暴露了哪些后续知识点

虽然这个应用功能几乎为空，但它已经埋下了后续所有重要主题的入口：

- Activity 生命周期会在基础篇继续展开。
- 布局和控件会在 UI 开发部分继续展开。
- 项目结构和构建脚本会在后续章节继续拆开。
- 日志与调试会在工程实践部分继续展开。

也就是说，第一个应用并不是一个孤立教学任务，而是整本书的缩略图。

### 7. 最小实践任务

1. 创建一个空白 Android 工程并成功运行。
2. 修改标题文本并重新运行，确认改动生效。
3. 在 Activity 中增加一条日志输出，并在 Logcat 中找到它。
4. 记录你在运行过程中看到的主要文件，尝试说出它们的职责。

### 8. 常见误区

- 只跟着向导点完，不看任何生成文件。
- 第一次就试图做复杂业务逻辑，导致基础链路没看清。
- 运行失败时只看弹窗，不去区分构建、安装和运行时错误。

## 小结

这一章的目标不是做出一个有价值的产品，而是建立第一次完整的 Android 工程体验。下一章我们会正式拆开这个工程，看看一个 Android 项目为什么会是现在这样的目录结构。
