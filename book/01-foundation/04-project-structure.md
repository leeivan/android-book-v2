# 项目结构

跑通第一个应用之后，很多人会立刻遇到第二个问题：工程里为什么会有这么多目录和文件，它们到底各自做什么。如果这一层没有尽早理顺，后面学习布局、导航、数据和构建时就会长期处于“知道要改文件，但不知道为什么在这里改”的状态。本章的任务，就是把最小 Android 项目的结构拆清楚，让你对工程组织形成稳定认知。

## 学习目标

- 理解 Android 项目中最常见目录和文件的职责。
- 知道代码、资源、清单和构建脚本为什么要分开管理。
- 建立从“看目录”到“理解工程边界”的基本能力。
- 为后续布局、资源、导航和构建章节打基础。

## 前置知识

- 已完成第一个 Android 应用。
- 已知道工程至少包含 Activity、布局文件、Manifest 和 Gradle 配置。

## 正文

### 1. 为什么 Android 项目天然比普通脚本目录复杂

一个普通命令行程序，可能只需要几个源文件就能表达全部逻辑。但 Android 应用不是这样，因为它需要同时管理多种类型的内容：

- Kotlin 源码。
- XML 资源。
- 图片和图标资源。
- AndroidManifest 清单声明。
- 构建和依赖配置。
- 可能还包括测试代码、生成代码和不同环境的资源覆盖。

换句话说，Android 项目目录复杂，不是因为框架“爱堆文件”，而是因为移动应用本身就是多类型资源共同参与的系统工程。

### 2. 先看最常见的四类目录

在一个最小 Android 应用里，最值得优先认识的是下面四类位置：

#### 2.1 `src/main/java` 或 `src/main/kotlin`

这里放的是主要业务代码和组件代码，例如 Activity、Fragment、ViewModel、数据层类等。它解决的是“逻辑如何表达”。

#### 2.2 `src/main/res`

这里放的是资源文件，例如布局、字符串、颜色、样式、图片和图标。它解决的是“界面和资源如何管理”。

#### 2.3 `src/main/AndroidManifest.xml`

这是应用清单文件，负责声明应用与系统协作所需的重要信息，例如组件入口、权限、主题、应用名称等。它解决的是“系统如何认识这个应用”。

#### 2.4 Gradle 脚本

Gradle 脚本负责依赖、构建参数、插件和构建行为配置。它解决的是“工具如何把这个应用构建出来”。

只要把这四类边界分清，后面大多数“这个改动该改哪里”的问题就已经能回答一半。

### 3. 资源目录为什么要单独存在

初学者经常会好奇：为什么文本、颜色、尺寸、布局不直接写在 Kotlin 代码里？

原因很简单，资源单独管理能解决很多工程问题：

- 便于国际化和多语言。
- 便于主题切换和样式复用。
- 便于多屏幕尺寸适配。
- 便于设计资源和业务逻辑分离。

也就是说，资源目录不仅是为了“好看”，更是为了把界面表达从逻辑层拆出来，让工程更易维护。

### 4. 为什么 Manifest 不能被忽略

很多初学者在前几周只把 Manifest 当成“启动页写在里面的文件”，但它的重要性远高于此。Manifest 是系统层面对应用的总说明书。很多系统相关能力都要在这里体现边界：

- 哪些组件是应用入口。
- 哪些能力需要权限。
- 应用使用什么主题。
- 某些组件是否对外暴露。

因此，Manifest 不是“配置角落”，而是 Android 平台边界的核心文件之一。

### 5. 代码目录如何随着项目增长演进

最小项目里，你可能只看到一个 Activity 文件，这会让人误以为 Android 项目代码本来就很扁平。实际上，随着项目增长，你通常会逐步引入更清晰的分层和分包，例如：

- `ui`
- `data`
- `domain`
- `feature`
- `common`

这些结构后面会在架构章节里展开。但从现在开始，你就应该知道：项目结构不只是文件放哪儿的问题，它会直接影响依赖方向和可维护性。

### 6. Android Studio 里的视图和磁盘上的真实目录不完全一样

IDE 往往会用更适合人阅读的方式展示工程，例如把某些目录分组合并、隐藏生成内容或按逻辑层次展开。但磁盘上的真实目录结构和 IDE 展示结构并不总是一一对应。

这一点很重要，因为后面你在查找文件、看构建输出或处理资源时，必须逐步学会从“IDE 表象”过渡到“真实目录认知”。

### 7. 一个健康项目结构应具备什么特征

一个好的项目结构通常具备这些特征：

- 职责清晰：代码、资源、配置和测试边界明确。
- 命名稳定：目录和包名能表达实际职责。
- 易于扩展：新增页面或功能时，能快速找到合适落点。
- 易于协作：其他开发者能较快理解目录意图。

反过来，如果所有代码都堆在一个包里，所有资源都随意命名，项目很快就会变得难以维护。

### 8. 最小示例：顺着一个首页文本改动看工程结构

假设你想修改首页显示文本。这个看似简单的改动，通常至少会涉及两个位置：

- 文本资源所在的 `res/values`。
- 引用这个文本的布局文件或代码文件。

如果你进一步给页面加点击逻辑，还会进入 Activity 或 Fragment 代码。这说明 Android 项目中的一次功能变更，经常天然跨越代码和资源两个层面，而不是只改一个文件。

### 9. 最小实践任务

1. 打开你的项目，分别定位 Kotlin 源码、布局资源、字符串资源、Manifest 和 Gradle 脚本。
2. 为每一类文件写一句话，说明它负责什么。
3. 做一个小改动，例如修改标题文本和页面背景色，观察分别改动了哪些文件。
4. 画出你当前项目的最小目录图，帮助自己建立结构认知。

### 10. 常见误区

- 把所有内容都当成“代码问题”，忽视资源和 Manifest 的边界。
- 只会在 IDE 里点文件，不知道磁盘上的真实目录结构。
- 项目一大就把所有类堆进同一个包。
- 把构建脚本当成“不要碰的黑盒”。

## 小结

本章解决的是“Android 项目为什么要这样组织”的问题。只要代码、资源、Manifest 和构建脚本这四类边界先理清，后面学习布局、导航、数据和构建系统时，你就不会一直处于“知道现象、不懂结构”的状态。
