# Activity 生命周期

如果说前几章主要解决“平台是什么、环境如何搭、项目怎么跑起来”，那么这一章开始，Android 的核心难点才真正出现：界面不是一直存在的，系统会在不同时间点创建、暂停、停止、销毁并重建你的页面。理解 Activity 生命周期，是理解 Android 为什么不能像普通脚本那样线性编程的第一道门槛。

很多人第一次学生命周期时，会把它学成一张回调顺序表。这样当然能帮助你在早期认出几个方法名，但远远不够。真正重要的不是背出 `onCreate()`、`onStart()`、`onResume()` 的先后，而是理解系统为什么要在这些时机发出回调、每个阶段适合放什么逻辑，以及页面状态到底应该放在哪里，才不会在配置变化和进程回收后出问题。本章会把生命周期从“回调清单”还原为“状态管理问题”。

## 学习目标

- 理解 Activity 生命周期的主要阶段和回调时机。
- 理解生命周期为何直接影响界面状态、资源和逻辑组织。
- 理解配置变化、进程回收和页面重建为什么是 Android 开发的常态。
- 为 Fragment、导航、ViewModel 和状态管理章节提供前置知识。

## 前置知识

- 已创建并运行第一个 Android 应用。
- 已理解项目结构和 Activity 作为页面入口的基本角色。

## 正文

### 1. 为什么生命周期是 Android 的主问题之一

在很多平台上，页面逻辑更接近“打开就执行，关闭就结束”。Android 不是这样。系统可能因为用户切到后台、来电话、分屏、多任务切换、屏幕旋转或资源紧张，在你没有主动要求的情况下改变 Activity 的状态。也就是说，页面并不是你独占控制的对象，而是系统随时可能重新安排其存在方式的组件。

这会迫使开发者持续回答几个关键问题：什么时候初始化页面依赖，什么时候开始监听，什么时候暂停动画、相机、定位或播放器，什么时候保存用户当前输入，什么时候恢复那些不该丢失的状态。如果这些问题没有提前设计清楚，最常见的后果就是状态丢失、重复请求、界面错乱、内存泄漏和运行时崩溃。生命周期之所以难，不在于回调数量，而在于它把“页面代码怎么写”变成了“页面状态怎么托管”的问题。

### 2. 生命周期不是回调背诵题，而是状态变化地图

学习生命周期时，更推荐把它理解成一张状态变化地图，而不是一串方法名。系统通过这些回调想表达的，其实是 Activity 与用户可见性、交互性和资源占用之间的关系正在变化。你真正需要理解的是：此刻页面是否已经建立、是否已经可见、是否正在与用户交互、是否即将进入后台、是否可能被系统销毁。

一旦你从“状态变化”而不是“方法顺序”去理解生命周期，很多 API 设计就会自然得多。`onCreate()` 更像页面首次建立时的一次性初始化入口；`onStart()` 和 `onResume()` 更像页面重新进入前台时的准备与激活；`onPause()` 和 `onStop()` 则是在提醒你别再把这个页面当成当前活跃界面；而 `onDestroy()` 并不总等于“用户主动退出”，它可能只是一次普通的重建过程。

### 3. 页面第一次建立时，`onCreate()` 应该放什么

`onCreate()` 最适合承担“页面建立所必需的一次性初始化”。典型内容包括绑定布局、准备页面级依赖、建立与 ViewModel 的连接，以及恢复最基础的轻量 UI 状态。它适合做“页面从无到有”必须完成的事，但不适合把所有和页面有关的逻辑无差别塞进去。

初学者最常见的问题，就是把 `onCreate()` 当成一个万能入口：请求数据、注册监听、启动动画、创建长生命周期对象、订阅所有事件、甚至开启后台任务都往里堆。短期看这样写很顺手，长期看却会迅速失控。因为页面每次重建时，这些逻辑都会被重新执行，而你自己很快就分不清哪些行为是“页面第一次建立必须做”，哪些行为其实只应该在用户真正回到前台后才启动。

### 4. 为什么要区分“已创建”“可见”“可交互”

生命周期学习中最容易被忽略的一点，是“页面已创建”“页面可见”“页面可交互”不是一回事。页面完成 `onCreate()`，并不代表它已经处于用户正在实际感知和操作的状态。很多资源和行为只有在页面真正进入前台时才应该启用，例如开始某些动画、恢复前台可见监听、刷新依赖实时交互的 UI。

这也是为什么仅靠“我已经在 `onCreate()` 里做了初始化”往往不够。你的代码不仅要知道页面有没有建立起来，还要知道页面是不是正在被用户看到、是不是此刻适合占用昂贵资源、是不是已经失去焦点却尚未销毁。只要这个区分不清晰，很多“偶现问题”在真机和复杂场景下都会反复出现。

一个更容易落地的判断方法是，把工作和生命周期阶段配对来看。页面实例级的一次性初始化，通常放在 `onCreate()`；和“页面对用户可见”强绑定的工作，更适合跟随 `onStart()` / `onStop()`；而那些要求用户正处于前台交互状态的资源，例如相机预览、独占输入焦点或某些高频动画，更适合跟随 `onResume()` / `onPause()`。这样思考的价值，在于你不再是“看到回调就随手写代码”，而是在按资源和交互的边界安排职责。

### 5. 进入后台时，代码应该先想什么

当页面进入 `onPause()` 或 `onStop()` 阶段时，系统是在提醒你：用户已经不再把这个 Activity 当成当前操作界面。这里最重要的，不是急着猜“页面会不会立刻销毁”，而是先处理那些与当前前台状态强绑定的内容。比如暂停动画、停止不必要的持续监听、释放短时独占资源、尽快保存用户尚未提交但又不想丢失的轻量状态。

这个阶段最容易出现两个极端。第一个极端是什么都不做，导致页面进后台后仍然继续占用资源，甚至在用户看不到的时候还继续执行不必要工作。第二个极端则是过度清理，把本应保留到下一次返回的状态全部扔掉，导致用户只是短暂切出应用，再回来时页面却像冷启动一样重新开始。真正合理的策略，不是“一律全停”或“一律不管”，而是根据资源成本和状态重要性做分层处理。

### 6. 配置变化和进程回收，为什么必须提前考虑

很多初学者第一次真正意识到生命周期的重要性，往往是在屏幕旋转之后。因为旋转会触发配置变化，系统默认会销毁并重建 Activity。只要你把关键状态直接塞在 Activity 成员变量里，它就会在这类场景下暴露问题。这也是为什么官方文档会反复强调：不要把 Activity 实例当成长期可靠的状态容器。

进程回收则更进一步。用户切到后台后，系统在资源紧张时可能会回收应用进程。等用户重新回到应用时，你看到的往往不是“原来的页面继续存在”，而是系统根据已有信息重新创建了页面。对于读者来说，这意味着你必须从一开始就接受一个事实：页面实例并不稳定，真正稳定的应该是状态托管策略。

### 7. 生命周期必须和状态托管一起理解

生命周期真正难的地方，不在于方法名，而在于状态该放在哪里。工程上可以把页面状态大致分成三类。第一类是很轻、很短暂的 UI 状态，例如输入框当前内容、滚动位置或临时筛选项。第二类是与页面逻辑紧密相关、但不应随着 Activity 重建而丢失的屏幕状态。第三类则是需要长期保存的业务数据，例如本地记录、账户信息、离线内容或数据库实体。

这三类状态的托管方式不应完全相同。轻量 UI 状态可以借助 `onSaveInstanceState()` 等机制做最小保存；页面级逻辑状态通常更适合交给 ViewModel；需要长期保存的数据则应该进入本地存储或数据层。Android 官方文档当前也明确建议，在系统可能销毁 Activity 的场景下，结合 `ViewModel`、`onSaveInstanceState()` 和本地存储来保存和恢复界面状态。你后面学到 ViewModel、Room 和数据流时，会发现它们都和这一章的问题紧密相关。

这里还有一个很重要的边界：`Bundle` 适合放轻量、可序列化的界面状态，不适合承担大型对象、网络结果缓存或复杂业务模型的长期托管。很多生命周期代码之所以变得脆弱，就是因为开发者试图把所有东西都塞进 `savedInstanceState`。更稳妥的做法是把 `Bundle` 当成“帮助系统在页面重建后快速恢复界面表象”的兜底机制，而不是把它当成完整业务状态仓库。

### 8. 最典型的错误：把所有逻辑都堆进 `onCreate()`

这是生命周期相关代码最常见的问题之一。原因很简单：`onCreate()` 是最早能下手的地方，所以很多人自然会把几乎所有逻辑都塞进去，包括界面初始化、数据请求、订阅事件、开启动画和后台任务。刚写时似乎很方便，因为所有代码都集中在一个入口里；但稍微复杂一点之后，问题就会同时出现：职责混乱、重建重复执行、排查困难、逻辑边界模糊。

更稳妥的做法，是把“页面建立”“页面可见”“页面交互中”“页面退到后台”这些状态分开思考。只有这样，你才能把逻辑放进真正合适的位置，而不是因为 `onCreate()` 先出现，就把一切都塞进去。

### 9. 最小示例：把生命周期变成可观察的日志

学习生命周期最有效的第一步，不是背流程图，而是把系统行为打印出来，让它变得可观察。下面这个示例会在关键回调中记录日志，并演示如何保存一个轻量状态：

```kotlin
class MainActivity : AppCompatActivity() {

    private val tag = "LifecycleDemo"
    private var currentDraft: String = ""

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        currentDraft = savedInstanceState?.getString("draft").orEmpty()
        Log.d(tag, "onCreate, restoredDraft=$currentDraft")
    }

    override fun onStart() {
        super.onStart()
        Log.d(tag, "onStart")
    }

    override fun onResume() {
        super.onResume()
        Log.d(tag, "onResume")
    }

    override fun onPause() {
        Log.d(tag, "onPause")
        super.onPause()
    }

    override fun onStop() {
        Log.d(tag, "onStop")
        super.onStop()
    }

    override fun onSaveInstanceState(outState: Bundle) {
        outState.putString("draft", currentDraft)
        super.onSaveInstanceState(outState)
        Log.d(tag, "onSaveInstanceState")
    }

    override fun onDestroy() {
        Log.d(tag, "onDestroy")
        super.onDestroy()
    }
}
```

这个例子有两个教学点。第一，它让你在切前后台、旋转屏幕和重新进入页面时，直观看到回调顺序。第二，它展示了 `onSaveInstanceState()` 更适合保存轻量 UI 状态，而不是承担整个页面逻辑的长期托管。示例中的 `currentDraft` 只是最小化演示；真正复杂的屏幕状态，后面应交给 ViewModel 等更合适的结构。

### 10. 最小实践任务

起点条件：

- 已有一个最小 Activity 工程。
- 已知道如何在 Logcat 中过滤当前应用进程。

步骤：

1. 在入口 Activity 中加入关键生命周期日志，至少覆盖 `onCreate()`、`onStart()`、`onResume()`、`onPause()`、`onStop()` 和 `onDestroy()`。
2. 运行应用，记录首次进入页面时的日志顺序。
3. 按 Home 键切到后台，再回到应用，记录这一次的日志顺序。
4. 旋转屏幕或触发其他配置变化，观察 Activity 是否被销毁再重建。
5. 额外保存一个轻量 UI 状态，例如输入框文本或当前筛选关键字，比较保存前后的行为差异。

预期结果：

- 你能清楚看到不同场景下生命周期回调并不完全相同。
- 你会发现配置变化可能导致 Activity 重建。
- 你能观察到轻量状态在保存与不保存时的差异。

自检方式：

- 你能用自己的话解释：页面切后台不一定等于页面销毁。
- 你能指出：为什么 Activity 成员变量不适合承担长期状态保存。
- 你能说出：哪些内容更适合 `onSaveInstanceState()`，哪些内容更适合 ViewModel 或数据层。

调试提示：

- 如果旋转屏幕后没有看到预期日志，先确认设备的自动旋转和配置变化是否真的发生。
- 如果日志很多，看不清顺序，先只过滤当前应用进程和 `LifecycleDemo` 这个 tag。
- 如果你在 `onCreate()` 里放了过多逻辑，日志会帮助你发现它们在重建时被重复执行。

### 11. 常见误区

- 认为页面销毁只发生在用户主动退出时。
- 把生命周期理解成“回调顺序背诵题”，而不是“状态管理题”。
- 把所有初始化都集中在 `onCreate()`，不区分职责。
- 假设 Activity 实例天然可靠，可以长期保存关键状态。

## 小结

Activity 生命周期是 Android 开发的第一道真正门槛，因为它迫使你接受一个事实：页面不会永远按你预期的线性流程存在，系统会根据前后台切换、配置变化和资源压力不断改变它的状态。真正需要建立的，不是对几个回调名字的机械记忆，而是对“页面状态应该放在哪里、何时启动、何时暂停、如何恢复”的稳定判断。

下一章及后续相关章节里，我们会继续把这个判断扩展到 Fragment、ViewModel、导航和数据层。只有在生命周期这一层打稳基础，后面的现代 Android 结构才不会变成一堆互不相干的 API。
