# Activity 生命周期

如果说前几章主要解决“平台是什么、工程如何组织、应用怎样跑起来”，那么这一章开始，Android 的核心难点就真正出现了：界面不是一直存在的，系统会在不同时间点创建、暂停、停止、销毁并重建你的页面。理解 Activity 生命周期，是理解 Android 应用为何不能像普通脚本那样线性编程的关键一步。

## 学习目标

- 理解 Activity 生命周期的主要阶段和回调时机。
- 理解生命周期为何直接影响界面状态、资源和逻辑组织。
- 为 Fragment、导航、ViewModel 和后台任务章节提供前置知识。

## 前置知识

- 已创建并运行第一个 Android 应用。
- 已理解项目结构和 Activity 作为页面入口的基本角色。

## 正文

### 1. 为什么生命周期是 Android 的核心问题

在很多平台上，页面逻辑更接近“打开就执行，关闭就结束”。Android 不是这样。系统可能因为旋转屏幕、资源紧张、用户切后台或多任务切换，在你没有主动要求的情况下改变 Activity 的状态。

这意味着，开发者必须回答几个问题：

- 什么时候初始化界面和数据。
- 什么时候暂停动画、监听器或任务。
- 什么时候释放资源。
- 页面被系统重建后，哪些状态需要恢复。

如果这些问题没有被清晰处理，应用就会出现最常见的几类问题：状态丢失、重复请求、内存泄漏、界面错乱和崩溃。

### 2. 生命周期的主要阶段

虽然生命周期回调很多，但初学阶段最重要的是理解状态变化的意图，而不是死记方法名。

#### 2.1 创建阶段

在 `onCreate()` 中，通常会完成最基础的页面初始化，例如：

- 绑定布局。
- 初始化页面级对象。
- 建立与 ViewModel 或其他状态源的连接。

这里最适合放“一次性初始化”的逻辑，但不适合把所有事情无差别塞进去。

#### 2.2 可见与可交互阶段

当 Activity 进入前台并可交互时，用户会真实感知它的存在。这个阶段应重点考虑：

- 界面是否准备好显示。
- 输入和监听是否应该开始工作。
- 需要与用户同步的轻量状态是否已经就绪。

#### 2.3 暂停与停止阶段

当页面失去焦点、被其他页面遮挡或切到后台时，应用并不一定立刻销毁，但已经不能再假设“用户正在看着它”。这个阶段通常要思考：

- 哪些工作应暂停。
- 哪些资源应继续保留。
- 哪些内容应持久化或保存。

#### 2.4 销毁与重建

Activity 被销毁不一定是因为用户退出，也可能是因为配置变更或系统资源回收。这里最重要的认知是：Activity 实例本身不是长期可靠的状态容器。

### 3. 生命周期和状态管理为什么绑在一起

生命周期真正难的地方，不在于回调顺序，而在于状态该放哪里。

很多初学者会把页面状态直接塞进 Activity 成员变量里。短期看起来最方便，但一旦页面重建，这种状态就可能消失。更合理的做法通常是：

- 瞬时、轻量、界面级状态：结合保存机制处理。
- 与页面逻辑强相关但不应随重建丢失的状态：交给 ViewModel。
- 需要长期保存的数据：交给本地存储或远程数据层。

也正因为如此，生命周期章节和后续 ViewModel、导航、数据层章节其实是连在一起的。

### 4. 生命周期中最容易出现的问题

#### 4.1 在错误时机做重操作

例如，每次页面重建都重复发起昂贵请求，却没有判断是否真的需要重新加载。

#### 4.2 把页面实例当成永久存在

一旦系统重建页面，这种假设就会失效。

#### 4.3 只会在 `onCreate()` 里堆逻辑

这会让代码变成一个巨大入口，后续很难维护，也不利于把逻辑分层。

### 5. 应该带着什么视角进入后续章节

学完生命周期后，你需要逐步形成一种新的编程直觉：Android 页面开发首先不是“怎么把按钮点通”，而是“如何在系统不断改变页面状态的前提下，让界面和数据仍然稳定协作”。

后续几章会继续把这个问题拆开：

- Fragment 会让页面结构更灵活，但也带来自己的生命周期问题。
- Navigation 会让页面流转更清晰，但也依赖正确状态管理。
- ViewModel 会成为承接状态的关键角色。

### 6. 最小实践任务

1. 在一个简单 Activity 中打印关键生命周期日志。
2. 运行应用后切换前后台，观察回调顺序。
3. 尝试旋转屏幕，记录页面是否重建、哪些状态丢失。

### 7. 常见误区

- 认为页面销毁只发生在用户主动退出时。
- 把生命周期理解成“背顺序题”，而不是“状态管理题”。
- 把所有初始化都集中在 `onCreate()`，不区分职责。

## 小结

Activity 生命周期是 Android 开发的第一道真正门槛。它让你意识到：系统并不会永远按你预想的线性流程运行页面，状态必须被主动设计和托管。理解了这一点，后续学习 Fragment、Navigation、ViewModel 和数据流时，才不会只停留在 API 表面。
