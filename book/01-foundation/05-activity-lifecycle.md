# Activity 生命周期

如果说前几章主要解决“平台是什么、环境如何搭、项目怎么跑起来”，那么这一章开始，Android 的核心难点才真正出现：界面不是一直存在的，系统会在不同时间点创建、暂停、停止、销毁并重建你的页面。理解 Activity 生命周期，是理解 Android 为什么不能像普通脚本那样线性编程的第一道门槛。

很多人第一次学生命周期时，会把它学成一张回调顺序表。这样当然能应付最初的理解，但远远不够。真正重要的不是背出 `onCreate()`、`onStart()`、`onResume()` 的顺序，而是理解：系统为什么会在这些时机触发回调；每个阶段适合放什么逻辑；以及页面状态到底应该放在哪里，才不会在重建和切后台后出问题。

## 学习目标

- 理解 Activity 生命周期的主要阶段和回调时机。
- 理解生命周期为何直接影响界面状态、资源和逻辑组织。
- 理解配置变化、进程回收和页面重建为什么是 Android 开发的常态。
- 为 Fragment、导航、ViewModel 和状态管理章节提供前置知识。

## 前置知识

- 已创建并运行第一个 Android 应用。
- 已理解项目结构和 Activity 作为页面入口的基本角色。

## 正文

### 1. 为什么生命周期是 Android 的主问题之一

在很多平台上，页面逻辑更接近“打开就执行，关闭就结束”。Android 不是这样。系统可能因为用户切到后台、来电话、分屏、多任务切换、屏幕旋转或资源紧张，在你没有主动要求的情况下改变 Activity 的状态。

这意味着，开发者必须持续回答这些问题：

- 什么时候初始化界面和依赖。
- 什么时候开始或停止监听。
- 什么时候暂停动画、摄像头、定位或其他敏感资源。
- 页面被重建后，哪些状态应该恢复。

如果这些问题没有被清晰设计，最常见的后果就是状态丢失、重复请求、界面错乱、内存泄漏和崩溃。

### 2. 生命周期不是“回调列表”，而是“状态变化地图”

学习生命周期时，更推荐把它理解成一张状态变化地图，而不是一串方法名。系统通过这些回调向你表达的，其实是页面与用户可见性、交互性、资源占用之间的关系正在发生变化。

也就是说，生命周期真正要传达的不是“你应该背什么顺序”，而是“你应该在什么可见性和交互状态下做什么事”。

### 3. 创建阶段：页面第一次建立时该做什么

在 `onCreate()` 中，通常会完成最基础的一次性初始化，例如：

- 绑定布局。
- 准备页面级依赖。
- 建立与 ViewModel 的连接。
- 恢复最基础的界面状态。

这里最适合放“页面建立所必需的初始化”，但不适合把所有事情无差别塞进去。尤其是昂贵操作、重复可触发逻辑和与可见性强绑定的监听，不应只因为“顺手”就全部放在 `onCreate()`。

### 4. 前台可见阶段：页面真正与用户交互时

当 Activity 进入前台并开始与用户交互时，系统表达的是：用户正在真实感知这个页面。这个阶段要重点考虑：

- 哪些界面状态应立即刷新。
- 哪些轻量监听或动画应开始工作。
- 哪些与前台交互强绑定的资源需要启用。

很多“看起来没问题”的代码，之所以在真实设备上出问题，就是因为没有区分“页面已创建”和“页面已真正进入前台”这两个状态。

### 5. 暂停和停止阶段：不是销毁，但已经不能再假设用户正在看它

当页面失去焦点或进入后台时，Activity 可能并不会立刻销毁，但它已经不再适合作为“当前活跃界面”来思考了。这个阶段应重点判断：

- 哪些交互应暂停。
- 哪些资源应暂时释放。
- 哪些数据应尽快保存。

这里最容易犯的错误，是要么过度释放，导致返回页面时重新初始化成本过高；要么完全不释放，导致后台仍持续占用资源。

### 6. 销毁与重建：为什么 Activity 不适合保存长期状态

Activity 被销毁并不一定意味着用户“退出应用”。它可能只是因为：

- 配置变化，例如旋转屏幕。
- 系统回收资源后重新创建页面。
- 页面栈中的实例被替换或清理。

这也是一个关键结论：Activity 实例本身不是长期可靠的状态容器。把重要状态直接塞进 Activity 成员变量里，迟早会在重建场景下暴露问题。

### 7. 生命周期和状态管理为什么必须一起理解

生命周期真正难的地方，不在于方法名，而在于状态该放在哪里。

从工程角度看，页面中的状态大致可分成三类：

- 很轻、很短暂的界面状态。
- 与页面逻辑紧密相关，但不应随重建丢失的状态。
- 需要长期保存的数据。

更合理的策略通常是：

- 短暂界面状态，使用合适的保存与恢复机制处理。
- 页面级逻辑状态，交给 ViewModel。
- 长期数据，交给本地存储或数据层。

这也是为什么生命周期章节会和 ViewModel、Fragment、Navigation、Room 等后续章节天然联动。

### 8. 配置变化为什么值得单独强调

很多初学者第一次真正意识到生命周期的重要性，往往是在屏幕旋转之后。旋转屏幕会触发配置变化，这种变化常常导致 Activity 被销毁再重建。

它之所以是很好的学习样本，是因为它把“页面实例不可靠、状态需要托管”这个问题暴露得非常直接。你在旋转时丢失的内容，往往就是未来在更复杂场景里也会丢失的内容。

### 9. 典型错误：把所有逻辑都堆进 `onCreate()`

这是生命周期相关代码最常见的问题之一。原因很简单：`onCreate()` 是最早能下手的地方，所以很多人会把几乎所有逻辑都塞进去，包括：

- 初始化界面。
- 请求数据。
- 注册监听。
- 开启动画。
- 启动后台任务。

短期内似乎很方便，长期看却会带来严重问题：职责混乱、重建重复执行、代码难以维护，也让你很难分辨哪些逻辑与页面创建有关，哪些逻辑其实和前台可见性或用户动作有关。

### 10. 最小示例：观察一个简单页面的生命周期

最值得做的第一个实验，是在一个简单 Activity 里打印关键生命周期日志，然后做三件事：

1. 打开页面。
2. 切到后台再返回。
3. 旋转屏幕。

你会直观看到：

- 页面何时重新创建。
- 哪些回调会在前后台切换时触发。
- 为什么“Activity 还在不在”不是一个靠主观猜测就能回答的问题。

这个实验的价值不在于背顺序，而在于让系统行为第一次变得可观察。

### 11. 学完本章后应建立什么新直觉

学完生命周期后，你需要逐步形成一种新的编程直觉：Android 页面开发首先不是“怎么把按钮点通”，而是“如何在系统不断改变页面状态的前提下，让界面、状态和数据仍然稳定协作”。

有了这层直觉，后续 Fragment、Navigation、ViewModel、Flow 和数据层这些概念才不会变成互不相干的技术点。

### 12. 最小实践任务

1. 在一个简单 Activity 中打印关键生命周期日志。
2. 运行应用并切换前后台，记录回调顺序。
3. 旋转屏幕，观察页面是否重建，哪些状态丢失。
4. 把一个重要页面状态从 Activity 成员变量转移到更合适的托管位置，并比较效果。

### 13. 常见误区

- 认为页面销毁只发生在用户主动退出时。
- 把生命周期理解成“回调顺序背诵题”，而不是“状态管理题”。
- 把所有初始化都集中在 `onCreate()`，不区分职责。
- 假设 Activity 实例天然可靠，可以长期保存关键状态。

## 小结

Activity 生命周期是 Android 开发的第一道真正门槛。它让你意识到：系统不会永远按你预期的线性流程运行页面，状态必须被主动设计、托管和恢复。理解了这一点，后续学习 Fragment、Navigation、ViewModel 和数据流时，才不会只停留在 API 表面。
