# Full-Prose Standard

Use this reference when a chapter is structurally correct but still reads like notes, prompts, or a review outline.

## Symptoms of an outline-heavy chapter

- Many headings, but each heading has only one short paragraph or a few bullets
- The chapter states conclusions without explaining why
- Practice tasks say "try this" without a starting point or expected outcome
- Code appears without placement, context, or explanation
- The recap repeats headings instead of synthesizing the lesson

## Target chapter anatomy

1. Opening problem framing
- Explain what real problem the reader is about to solve
- Explain why the topic matters now in the learning path

2. Concept and decision model
- Explain the core concept in plain language
- Explain tradeoffs, scope, and common confusion

3. Recommended path
- Show the modern default approach
- Mention legacy paths only when needed for migration or historical context

4. Minimal implementation
- Provide the smallest example that demonstrates the idea
- Explain file placement, important lines, and what success looks like

5. Practice loop
- Give steps
- Give expected results
- Give one or two debugging hints

6. Recap and bridge
- Summarize what the reader now knows
- Point to the next chapter naturally

## Rewrite moves that usually help

- Merge shallow bullets into a paragraph with cause-and-effect logic
- Replace "there are three points" wording with direct explanation
- Add one concrete scenario before abstract terminology
- Turn "best practice" claims into scoped statements with reasons
- Add a short "what to observe" note after runnable steps

## Done checklist

- The chapter can be read without needing hidden author notes
- A learner can try at least one concrete action after reading
- The prose explains both the "why" and the "how"
- The chapter sounds like book prose, not meeting notes
