# 完整正文整理标准

本文档用于把“结构正确但内容偏薄”的章节，提升为能支撑学习和实践的正式书稿。

## 目标

一章合格的正文，不只是告诉读者“有哪些点”，而是要让读者理解：

- 这章在解决什么问题。
- 为什么今天推荐这样做。
- 最小实现路径是什么。
- 实践时应该观察到什么结果。
- 常见错误会出现在哪里。

如果一章只能提供标题、条目和结论，它仍然更像提纲，而不是可阅读、可实践的正文。

## 典型薄稿特征

- 标题很多，但每个标题下只有 1 到 2 句过渡话。
- 用 bullet list 代替背景解释、概念展开和判断过程。
- 给出“实践任务”，但没有写清起点、步骤和预期结果。
- 给出代码片段，却没有说明代码放在哪里、为什么这样写、运行后应看到什么。
- 只有结论，没有边界，没有错误案例，也没有与前后章节的衔接。

## 完整正文的最低要求

### 1. 开篇要有问题意识

章节开头至少要交代三件事：

- 本章为什么存在。
- 读者在真实开发中会遇到什么问题。
- 本章和前后章节是什么关系。

### 2. 核心概念必须展开

不要只给定义。至少要解释：

- 概念本身是什么。
- 它解决什么问题。
- 适用边界在哪里。
- 为什么容易被误解。

并且要尽量避免把概念解释写成“文档口吻”的说明书。更好的做法是补入：

- 真实问题情境。
- 常见误解来源。
- 为什么官方推荐会演进到今天的方案。
- 一个能帮助初学者建立直觉的类比或反例。

这些内容如果有必要补充，应该直接融进正文已有小节，而不是在章末另起“教材化延伸”“资料路线”“为什么只看官方文档不够”之类的元说明区。

### 3. 推荐做法要能落地

如果本章适合代码示例，示例至少应交代：

- 代码属于哪个文件或层次。
- 关键语句的作用。
- 运行后应该看到什么行为。

如果本章不适合代码，则要给出可观察的操作流程、结构图或验证任务。

如果某个主题已有高质量官方 codelab、示例项目或经典最小案例，应吸收其教学节奏，而不只是照着 API 文档写“步骤说明”。

### 4. 实践任务必须可执行

实践任务至少包含：

- 起点条件。
- 明确步骤。
- 预期结果。
- 最小自检方式。
- 必要时的调试提示。

### 5. 小结必须完成收束

小结不应只是把标题再列一遍。它应回答：

- 本章真正建立了什么认知。
- 这些知识为什么重要。
- 下一章会基于它继续解决什么问题。

## 建议的改写顺序

1. 先补开篇，明确本章任务。
2. 再把“只列结论”的小节改成自然段解释。
3. 然后补最小示例和实践任务。
4. 最后统一术语、语气、前后衔接和现代实践口径。

## 禁止的伪教材化做法

- 在章节主体写完后，另起“教材化延伸”“资料路线”“为什么只看文档不够”等元说明小节。
- 用“推荐去看哪些资料”代替对当前概念的正文解释。
- 把教材化理解放进章末补充，而不是前移到开篇、概念展开、示例解释和实践任务中。
- 让正文主体仍然像说明书，只在结尾补一段作者点评来弥补。

## 审稿检查清单

- 是否存在连续多个小节主要由 bullet list 组成。
- 是否解释了推荐做法背后的原因，而不是只说“建议这样写”。
- 是否给出最小实践路径，而不是只给抽象原则。
- 是否明确区分当前推荐方案与历史方案。
- 是否让读者知道学完后下一步该做什么。
- 是否主要像“说明书”而不是“教材”。
- 是否至少吸收了规则类资料之外的示例、课程或项目经验。
