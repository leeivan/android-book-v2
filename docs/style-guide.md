# 写作规范

## 语言目标

- 面向中文读者，表达清晰、直接、专业。
- 先解释问题和动机，再解释概念、机制和用法。
- 正文以自然段为主，不用条目代替解释。
- 术语首次出现可中英对照，后续保持统一。

## 正文章节的深度要求

- 章节开头用 2 到 4 段说明本章解决什么问题、为什么重要，以及与前后章节的关系。
- 核心小节应给出背景、判断框架和适用边界，而不是只列结论或 API 名称。
- 除学习目标、前置知识、常见误区、练习题等天然适合列表的部分外，正文不应长期由 bullet list 组成。
- 读者在读完整章后，应能回答：为什么要这样做、何时这样做、怎么开始做、怎么验证是否做对。

## 技术基线

- 正文优先采用 Kotlin。
- 优先采用 AndroidX、Jetpack 和当前推荐实践。
- 旧 API 需要明确标注使用边界、历史背景和迁移方向。
- 涉及平台行为变更、权限模型和发布规则时，以官方资料为准。

## 章节组织

- 每章开头说明本章定位和学习目标。
- 正文至少包含：问题背景、核心概念、推荐做法、最小示例、实践任务、常见误区、小结。
- 小结不能只是重复标题列表，要总结本章收获，并说明下一章如何承接。
- 练习题尽量区分概念理解、编码实现和扩展思考。

## 示例与实践

- 示例代码要短，但必须支撑正文论点。
- 代码前交代场景，代码后解释关键设计点。
- 实践任务必须写清起点、步骤、预期结果和最小自检方式。
- 如果主题不适合代码示例，可以改用操作步骤、结构图或观察任务，但仍要让读者知道如何验证。

## 风格约束

- 不写营销式表达，不故作夸张。
- 不把社区经验包装成官方规范。
- 不把 API 清单当成正文主体。
- 同类章节保持同一叙事重心、术语写法和代码风格。
