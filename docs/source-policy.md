# 来源策略

## 基本原则

- 优先使用官方文档、官方示例和一手资料。
- 涉及平台行为变更时，必须以当前官方说明为准。
- 引用社区资料时，只能作为补充解释，不能替代主依据。
- 正文不应只是把官方文档转述成中文说明书，而应把多类资料综合成更适合教学的解释。

## 推荐来源类型

### A 类：平台与规则主依据

- Android Developers 官方文档。
- Kotlin 官方文档。
- Google Play / Play Console 官方帮助文档。
- Material Design 官方文档。
- AndroidX / Jetpack 官方 API 与库文档。

这类资料用于确认：

- 平台行为。
- 权限与发布规则。
- 当前推荐做法。
- API 语义和限制。

### B 类：教学与实践主依据

- Android 官方 Codelabs。
- Android 官方课程与路径化教程。
- Android 官方示例项目，例如 `android/architecture-samples`、`android/nowinandroid`。
- Kotlin 官方教程与 JetBrains 官方实践文档。
- 相关库的官方文档，例如 OkHttp、Retrofit、Coroutines、Serialization。

这类资料用于补强：

- 教学路径。
- 最小实现步骤。
- 代码组织方式。
- 从概念到实践的落地桥梁。

### C 类：补充解释来源

- 高质量社区技术文章。
- 官方团队成员的技术分享。
- 经典书籍、课程和公开演讲。
- 规范、RFC、AOSP 源码或设计说明。

这类资料可以用于：

- 历史背景解释。
- 概念类比。
- 不同方案的取舍说明。
- 补足官方文档中偏说明书化、偏 API 化的部分。

但它们不能替代 A 类资料来定义平台事实。

## 教材化使用方式

- 一个章节如果只依赖 API 文档，通常会更像说明书，不像教材。
- 涉及实践操作的章节，尽量同时参考“规则类资料”和“教学/示例类资料”。
- 涉及架构、工程化和设计判断的章节，尽量补入案例、反例、迁移背景或真实项目语境。
- 如果一个主题有高质量官方示例项目或 codelab，应优先吸收其教学结构，而不是只摘 API 定义。

## 不推荐的写法

- 只把官网概念段落换成中文表达。
- 只列 API、属性和步骤，不解释问题背景和取舍原因。
- 把历史文章当成当前推荐方案。
- 只引用二手博客，而不回到一手来源确认边界。

## 使用方式

- 不直接大段复制来源内容。
- 必要时以自己的话重述，并保留出处信息。
- 对已经过时的资料，需要明确指出其历史背景与适用边界。
- 如果使用多类来源，应在正文中完成综合解释，而不是把出处堆成资料清单。
